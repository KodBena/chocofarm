## Lens 1 — contract-fidelity

Lens: EXECUTOR CONTRACT FIDELITY. Does CppActorExecutor match EXACTLY what exit_loop.run requires of an
executor, as ParallelExecutor defines it? Check method-by-method: (1) generate(net, version, worlds, lam,
explore_plies, lam_blend, n_step, hot_search=, max_steps=) — same positional/kw shape as exit_loop's call
site? Does it return a flat list of (feat, pi, mask, g) records the loop's `for feat, pi, mask, g in
recs_all` unpacking consumes (right element types: feat/pi/mask numpy rows, g a python float)? (2)
evaluate(net, version, worlds, lam, hot_search=, max_steps=) -> (totR, totT, list_of_T) — same shape the
loop unpacks as `totR, totT, ets`? (3) close(), .run, .cores all present and used safely by the
construction banner + the finally? (4) Are the (X,PI,M,Y) blocks read with the right dims (in_dim/n_slots)
+ dtype (RESULT_DTYPE), matching what the C++ runner's write_results emits and what the Python net trains
on? (5) The runner CLI it builds — are ALL the flags real (grep main.cpp): --run/--phase/--version/
--res-token/--episodes/--lam/--max-steps/--seed/--policy/--gumbel-m/--gumbel-n-sims and the threaded
--gumbel-<hot knob>? A wrong/ignored flag is a silent mis-config. Report any divergence.

## Lens 2 — correctness-failure-honesty

Lens: CORRECTNESS + FAILURE MODES + HONEST SEMANTICS. (1) The Part-B guard
`if (n_step is not None) or (lam_blend is not None and lam_blend < 1.0)` — does it fire on EVERY blend
request and NOT over-fire on the pure-MC default (lam_blend=1.0, n_step=None)? Any edge (lam_blend=None)?
(2) _read_records: empty-episode handling (the runner skips empty episodes — are gaps in idx handled?),
n derived from len(Y)//itemsize, the reshape dims, key deletion — any off-by-one / mis-shape / leak? What
if a block is present but another is None (partial write)? (3) Failure modes (ADR-0002 fail-loud): a dead
redis, a runner non-zero exit, a runner timeout, a missing result key — are they all LOUD (raise), never a
silent empty buffer that trains on nothing? (4) evaluate() correctness: it runs a Python GumbelPolicy on
the net — is this the SAME quantity exit_loop's serial eval computes (lines ~459-466)? Is the
fixed-across-iterations eval seed defensible, and does ignoring the eval_seed exit_loop drew the
eval_worlds from cause any mismatch (note: worlds ARE passed in; only the search RNG is local)? (5) The
'worlds is used only for COUNT; the runner draws its own worlds from base_seed+version' semantics — is that
an honest, documented divergence from ParallelExecutor (which uses the parent's exact worlds), or a silent
reproducibility trap? (6) Resource lifecycle: is the redis connection always closed; is the import-cycle
(exit_loop <-> cpp_executor) actually broken by the lazy imports? Report concretely.
