# Firewall — residual block "worse loss with more capacity"

Independent analytical firewall. Question: a residual block before the output heads (+131k
params, ~2x the net) did NOT improve the ExIt-loop policy CE and apparently *worsened* it
(residual it21 CE 0.756, the worst in the comparison table). A strictly more expressive net
should reach <= the baseline loss on comparable data, so worse loss points to an optimization
flaw. Is the flaw real, and if so what is it?

**Verdict: no genuine flaw. The "worse loss" is a confound** — convergence-stage plus
target-distribution. On a fixed dataset (the decisive controlled test), residual-ON reaches
*lower* CE than baseline at every learning rate and on two independent datasets, exactly as
added capacity should. The shipped block is sound; the ExIt-loop numbers were never an
apples-to-apples comparison.

All experiments: float64 training (the codebase trains in f64 and only infers in f32, so f32 is
not implicated), cores 0-3 under `timeout`, venv `/home/bork/w/vdc/venvs/generic`. Probes under
`probes/residual_firewall/`. Frozen datasets + result JSON under `/tmp/fw_residual/`.

---

## 1. Is the phenomenon real after controlling for the confounds?

**No.** The decisive test is a fixed-dataset supervised A/B: freeze ONE dataset of
`(features, π′, legal-mask, value-target)`, then train residual-OFF vs residual-ON on that SAME
data with identical init-seed / LR / batch order / steps. On fixed data, a strictly more
expressive net MUST reach <= the baseline loss; if it does, there is no optimization flaw.

The trunk and both heads are **bit-identical** between OFF and ON (the block params are drawn
from the constructor rng *after* the heads, exactly so the ablation is clean — verified
`np.array_equal` on W1/b1/W2/b2/Wv/bv/Wp/bp). Only the extra block differs.

Frozen dataset 1 (1495 decisions, generated by the converged `sims256` net), final CE after 60
epochs:

| LR     | OFF (baseline) CE | ON (residual) CE | Δ CE (ON − OFF) |
|--------|-------------------|------------------|-----------------|
| 3e-4   | 0.7112            | 0.6292           | **−0.082**      |
| 1e-3   | 0.5332            | 0.4672           | **−0.066**      |
| 3e-3   | 0.4247            | 0.3888           | **−0.036**      |

Frozen dataset 2 (1474 decisions, generated by the `longtail` net), lr=1e-3: OFF 0.5977 →
ON **0.4956** (Δ −0.102). On combined CE+MSE the residual arm wins in all four experiments.

The advantage is present from epoch 0 onward — there is no early-epoch penalty from the block's
init. Residual-ON is at worst tied with baseline at every epoch and generally ahead:

```
epoch |   off    on
    0 | 2.7534 2.2657
    2 | 1.6083 1.3062
    5 | 1.1798 1.1501
   19 | 0.7529 0.6951
   59 | 0.5332 0.4672
```

**On fixed data, more capacity reaches lower loss, exactly as theory requires.** The reported
"worse loss" does not reproduce once the confounds are removed.

## 2. What the confound actually was (with evidence)

Two confounds, both real and both quantified from the run histories.

**(a) Convergence stage.** The residual run was stopped at **it21**; the peers it was compared
against ran to it34-69. The raw table compared a less-converged run against more-converged ones.
At MATCHED iteration 21, residual sits in the same pack as its same-`n_sims` peers:

| iter | residual | combo | m24 | sims256 | longtail |
|------|----------|-------|-----|---------|----------|
| 15   | 0.806    | 0.702 | 0.748 | 0.645 | 0.740 |
| 21   | **0.755** | 0.703 | 0.720 | **0.613** | 0.725 |
| 34   | (n/a)    | 0.691 | 0.691 | 0.618 | 0.689 |

The residual CE trajectory (1.52 → 0.98 → 0.81 → 0.76) has the same healthy decreasing shape as
every other run. Nothing pathological; it simply had not reached it34 yet.

**(b) Target distribution.** Each ExIt run generates its OWN π′ targets, and `n_sims` differs:
`sims256` searches with n_sims=256 vs everyone else's 128, producing **sharper, lower-entropy π′
targets that are intrinsically easier to fit to low CE** regardless of architecture. That is why
sims256 (0.613 at it21) leads the table — it is the expert quality, not the apprentice capacity.
Comparing the residual run's CE against sims256's is comparing two different loss functions.
The fixed-dataset A/B removes exactly this by giving every arm the *same* targets.

So the table that motivated the investigation crossed both axes at once: a less-converged net
(it21 vs it34-69) fitting a harder target distribution (n_sims=128) than the run it looked worst
against (n_sims=256). Neither axis has anything to do with the residual block.

## 3. Point-gradient and init diagnostics (the named candidates, ruled out)

- **Gradient correctness.** Independent finite-difference check confirms the block participates
  in the loss with live, nonzero grads on all four block params (Wr1/Wr2/br1/br2); the shipped
  commit's check (max rel err 3.67e-6) stands. All 22 `tests/test_az_loop.py` pass on this
  branch. Not a gradient bug.

- **Init-scale hypothesis — investigated and CONTRADICTED.** The block is He-initialized, so at
  init it is *far* from identity: on the frozen data `||head_in − a2|| / ||a2|| = 1.23` (the
  block output zr2 has larger std than the trunk signal a2 it should pass through; the outer
  ReLU clips ~32% of the sum to zero). This *looks* like the classic "residual block should init
  near zero" pathology. But the fixed-data A/B falsifies it as a *training* problem: the He-init
  block is the **best CE arm** (besides removing the outer ReLU), and identity-style inits do not
  help CE:

  | variant (ds1, lr1e-3) | CE | MSE |
  |-----------------------|------|------|
  | off (baseline)        | 0.533 | 0.043 |
  | on (He-init, shipped) | **0.467** | 0.044 |
  | on_zeroinit (Wr2,br2=0 → exact identity start) | 0.532 | 0.039 |
  | on_smallinit (Wr1,Wr2 ×0.1) | 0.566 | 0.044 |
  | on_noouterrelu (head_in = a2+zr2, no outer ReLU) | **0.458** | 0.044 |

  Identity-init (`zeroinit`) lands at baseline CE — it starts equal to baseline and, on this
  small dataset and budget, the block does not depart far enough from identity to add much CE
  value (it does modestly help MSE). `smallinit` is slightly *worse* on CE. The large init
  deviation is real but benign: the optimizer fits from it without trouble. So the init scale is
  not a flaw to fix.

- **Outer ReLU.** `out = ReLU(h + block(h))` does gate the skip's negative half, so it is not a
  pure pre-activation residual. Removing it (`on_noouterrelu`) is consistently the best arm
  (CE 0.458 vs 0.467, and best combined loss on both datasets). This is a small, optional
  refinement, not a defect — the shipped form already beats baseline.

## 4. The fix

There is nothing to fix in the residual block — it works. The real defect is **methodological**:
the comparison that produced the table was not controlled. The corrective is to compare at
matched conditions:

1. **Run the residual config to the same iteration count** as its peers (>= it34) before
   reading its CE. At it21 it is mid-descent.
2. **Only compare CE across runs that share `n_sims`** (same expert/target sharpness), or use the
   fixed-dataset A/B for any architecture comparison. CE across different `n_sims` is not a
   like-for-like number.
3. If a CE *improvement* from extra capacity is the goal, the fixed-data A/B already demonstrates
   it (−0.07 to −0.10 CE). To squeeze a little more, **drop the outer ReLU**
   (`head_in = a2 + zr2`), which was the best arm in every experiment. This is an optional
   one-line change, not a correctness fix.

**Demonstrated:** the fixed-data A/B (§1) already shows residual-ON reaching <= baseline CE under
the requested fix-frame (matched data/LR/init/steps), at three LRs and two datasets.

## 5. Honest caveats

- **The fixed-data budget is heavier than the loop's.** 60 epochs on ~1500 rows is far more
  gradient/data than the ExIt loop's 2 epochs/iter on freshly-generated data. The A/B answers the
  representability/optimization question definitively (can more capacity reach <= loss on the SAME
  data? yes), but it does not by itself prove the block improves the *online* loop, where data is
  non-stationary and each net shapes its own future targets. The right way to settle the online
  question is to run residual-ON and a baseline to the SAME iteration count with the SAME `n_sims`
  and compare eval rate, not cross-config CE.
- **MSE is mixed.** Residual-ON sometimes shows slightly higher value MSE than baseline (ds2:
  0.069 vs 0.053) while winning CE; the optimizer trades the two heads off. Combined CE+MSE still
  favors ON everywhere. If value accuracy regresses online, consider a higher `beta` or a separate
  value-side budget — but this is tuning, not a flaw.
- **Datasets were generated by already-converged baseline nets**, so the targets are reasonably
  sharp. Very early in training (high-entropy targets) the capacity advantage may be smaller; the
  it0-it5 loop CE is dominated by target entropy, not architecture.
- **I did not re-run the online ExIt loop to it34 with the residual block** (each such run is
  ~hours; out of the bounded firewall budget). The matched-iteration table in §2 is read from the
  existing histories, which is sufficient to show the it21-vs-it34+ confound, but a clean online
  A/B to equal iterations would be the confirming follow-up if the maintainer wants the online
  number.
