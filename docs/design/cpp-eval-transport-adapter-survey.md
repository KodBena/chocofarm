# Prior-art survey — C++->JAX eval transport adapter

The six domain surveys and the completeness critic that the design note synthesizes. Rendered from the
workflow journal (wf_66209f19-914); the synthesis itself is `cpp-eval-transport-adapter.md`.

## production-batching-servers

**Systems surveyed:** KataGo NNEvaluator (cpp/neuralnet/nneval.cpp) — the closest analog: multi-thread MCTS feeding a single-GPU batched NN, NVIDIA Triton dynamic batcher (preferred_batch_size + max_queue_delay + instance_group), Triton ragged batching (variable-size batches without padding to max), Orca (OSDI'22) iteration-level / continuous batching, vLLM continuous batching (PagedAttention scheduler), Clipper (NSDI'17) adaptive AIMD batching to a latency target, Clockwork (OSDI'20) predictable batch scheduling from deterministic DNN timing, SEED RL (centralized batched inference for distributed RL actors)

### Queue-drain server (waitPopUpToN): decouple S, D, and E by draining whatever-is-available up to a cap

- **Mechanism:** KataGo's nneval.cpp is the canonical solution to OUR EXACT problem (N search threads, 1 single-device batched NN). Each search thread calls evaluate(), pushes an NNResultBuf onto a shared queryQueue (forcePush), increments numOngoingEvals, then blocks on a per-buffer condvar (while(!buf.hasResult) buf.clientWaitingForResult.wait(lock)). A SEPARATE server thread loops: desiredBatchSize = min(maxBatchSize, currentBatchSize); gotAnything = queryQueue.waitPopUpToN(resultBufs, desiredBatchSize). waitPopUpToN grabs WHATEVER is currently queued up to the cap — it does NOT wait for a full batch. It then runs ONE forward at numRows = resultBufs.size() (the ACTUAL count, not padded) via NeuralNet::getOutput(..., numRows, ...), then notify_all on each buffer. This is precisely the missing sweet spot: send-batch (S) is implicit — many producers enqueue single leaves, the server COALESCES them into one big eval-batch (E) opportunistically; in-flight depth (D) = numOngoingEvals is naturally large because every blocked thread is one outstanding request. S, D, E are three independent quantities, not one knob. Single-device handling: numNNServerThreadsPerModel lets you run >1 server thread over the same device handle to overlap copy/launch, but the forward itself stays serialized per device.
- **Applies to us:** This is the design point to adopt. Our two adapters are the two degenerate corners of this scheme: strict-barrier = S=K,D=1 (the server-drain coalescing is thrown away because the driver blocks after each send), greedy-async = S=1,D=K but with PER-MESSAGE framing so coalescing never happens server-side. KataGo's answer is: producers enqueue cheaply (S=1 logically) BUT a server-side drain coalesces into E=whatever-is-queued, so D is large (every parked leaf is in flight) WITHOUT per-leaf wire frames driving the batch. For us this means: stop making the C++ driver decide the eval batch. Have C++ stream leaves with high in-flight depth (D>1, cheaply-framed, ideally many leaves per ZMQ frame so S>1 amortizes framing), and let the PYTHON server drain its recv queue with a 'pop up to E_max' loop, running forward at the ACTUAL drained count. Our GEN ceiling is 76k leaves/s/core and SERVE is 190k-264k/s, so a single server that coalesces a few cores' worth of leaves per forward sits far below the serve wall — the win is removing the serialization barrier and the per-leaf framing simultaneously, which is exactly what queue-drain does.
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** empirically proven — KataGo is the dominant open-source Go engine and this is its production self-play/inference path; the mechanism is in shipping source, not a write-up.

### Ragged / variable-shape batch eval — kill the fixed-pad penalty

- **Mechanism:** Triton ragged batching and KataGo both run the forward at the ACTUAL row count, not padded to max_batch. KataGo calls getOutput with numRows = resultBufs.size(); Triton ragged batching 'avoids explicit padding by allowing inputs without shape checks,' batching different-size requests without padding to the largest. The general serving principle: dispatch a batch as soon as a preferred size is reached (Triton dispatches immediately at preferred_batch_size, only waiting up to max_queue_delay when a preferred size can't be formed) so partial batches are rare AND cheap.
- **Applies to us:** This directly addresses our stated regression: 'the server pads every forward to a fixed shape, so a PARTIAL batch pays the FULL pad (~2x waste).' That is a self-inflicted XLA constraint, not a law. Two fixes, in order: (1) BUCKETING — compile the JAX forward for a small fixed set of shapes (e.g. 8/16/32/64/128/256) and dispatch a drained batch of N to the smallest bucket >= N. This removes the 2x pad waste for partial batches while keeping XLA's fixed-shape requirement (XLA recompiles per shape, so a handful of cached buckets, not arbitrary N). (2) Set a preferred-batch target so the drain prefers a full bucket and only falls to a smaller bucket when starved. With buckets, the half-batch pipelining attempt that regressed becomes viable: a half-full drain pays the half bucket, not the max pad.
- **Source:** https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/ragged_batching.html
- **Proven / folklore:** empirically proven — ragged batching and shape-bucketing are standard production practice (Triton, plus XLA bucketing is the documented remedy for recompilation thrash).

### Dynamic batching with delay window (preferred_batch_size + max_queue_delay) — arrival/formation decoupling

- **Mechanism:** Triton's dynamic batcher is the textbook S-vs-E decoupler: requests arrive asynchronously and queue; the batcher forms a batch and dispatches IMMEDIATELY when a preferred_batch_size is reachable, otherwise waits up to max_queue_delay_microseconds for more arrivals to complete a preferred size, then sends whatever it has. Producers never block on batch formation — they just enqueue. Concurrency over one model comes from instance_group (multiple inference instances each pulling formed batches), NOT from making the producer wait.
- **Applies to us:** The max_queue_delay knob is the tunable that our strict-barrier adapter lacks and our greedy adapter sets implicitly to zero. For THROUGHPUT-ONLY (latency freely tradeable) we can set the delay generously: let the server accumulate until it hits a preferred bucket OR the recv queue drains empty. Because latency is irrelevant to us, we get the full benefit of the delay window with none of its cost — this is the regime Triton's own docs say to push the delay up for. ADR-0012 P7 note: the delay/batch policy lives entirely on the SERVER side; the wire frame stays a dumb transport envelope (correlation id only), so adding a queue-delay does not touch the codec — consistent with 'serialization is orthogonal to transport.'
- **Source:** https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/batcher.html
- **Proven / folklore:** empirically proven — Triton dynamic batching is one of the most widely deployed serving features; the throughput/latency tradeoff curve is documented and reproduced across many write-ups.

### Multiple server PROCESSES for cross-request parallelism (instance_group analog) under a single-threaded evaluator invariant

- **Mechanism:** Because XLA owns the forward and you cannot run it in a worker thread, the only way to overlap forwards is replicas: Triton's instance_group spins multiple model instances that pull from the same formed-batch stream; SEED RL tunes 'the number of TPU cores for inference' separately from training; KataGo exposes numNNServerThreadsPerModel to overlap host-side copy/launch with device compute even on one GPU. The shared idea: producers feed a queue; a small POOL of single-threaded evaluators drains it; the pool size is the cross-request parallelism.
- **Applies to us:** Our hard constraint says the JAX server is single-threaded by invariant and cross-request parallelism 'must come from more server PROCESSES.' On a 4-vCPU host with GEN scaling 4.0x linearly, the production answer is: run G generator cores feeding P server processes (P < 4), each server single-threaded, each draining its own ZMQ queue and coalescing. Since SERVE ceiling (190k+/s) hugely exceeds aggregate GEN (4x76k=304k/s but only ~3 cores free for gen if 1 is a server), ONE well-fed coalescing server process likely saturates all generator cores — start P=1 with queue-drain, add a 2nd process only if profiling shows the single server's recv+forward loop is the wall. Do NOT thread the forward (violates the invariant); scale by process, matching SEED RL / instance_group.
- **Source:** https://research.google/blog/massively-scaling-reinforcement-learning-with-seed-rl/
- **Proven / folklore:** empirically proven for the replica/process-pool shape (Triton instance_group, SEED RL); the specific P for our host is an empirical tuning question, not established.

### Continuous / iteration-level batching (Orca, vLLM) — admit new work without draining the in-flight batch

- **Mechanism:** Orca's contribution is iteration-level scheduling: instead of a batch running to completion before the next forms (static batching), the scheduler re-forms the batch EVERY iteration, admitting newly-arrived requests into open slots and retiring finished ones, so the device never idles waiting for a straggler. vLLM implements this in its per-iteration Scheduler. The win (up to 36.9x in Orca) comes from eliminating head-of-line blocking where short requests wait behind long ones.
- **Applies to us:** PARTIALLY applicable, and it is important to be honest here: continuous batching's headline win is for AUTOREGRESSIVE, multi-iteration, variable-length requests (LLM decode) where requests finish at different times. OUR forward is SINGLE-SHOT (one leaf -> one value/policy, fixed cost, fixed shape). There is no per-request multi-step loop, so iteration-level retirement buys us nothing — the relevant kernel of the idea for us is only the 'admit newly-arrived leaves into the next forward without waiting for a barrier,' which is exactly what the KataGo queue-drain already gives us. So adopt the queue-drain (the single-shot specialization of continuous batching) and do NOT build slot-level admission machinery — it solves a problem (ragged completion times) we do not have.
- **Source:** https://www.usenix.org/conference/osdi22/presentation/yu
- **Proven / folklore:** empirically proven for LLM serving (Orca OSDI'22, vLLM); but its applicability to our single-shot evaluator is LIMITED/folklore — citing it as our solution would be a mis-transfer.

### Adaptive batch sizing to a target (Clipper AIMD, Clockwork deterministic timing)

- **Mechanism:** Clipper amortizes RPC/framing overhead by transforming a concurrent query stream into batches, sizing them via AIMD against a LATENCY target (grow batch additively until batch latency exceeds the SLA, then cut multiplicatively). Clockwork goes further: because DNN inference time is deterministic, it predicts each batch-size's execution time and schedules to a deadline, choosing batch size from a known cost model rather than reacting.
- **Applies to us:** Clipper's CORE OBSERVATION matches our greedy-async failure exactly: framing/RPC overhead must be amortized by batching, and per-message framing is a wall (our greedy adapter is 'FRAME-OVERHEAD-bound, FLAT across depth' — Clipper would say it is paying RPC overhead per request). The remedy is Clipper's, not more overlap: COALESCE into batches so framing is amortized over E leaves. But the AIMD/SLA loop itself is IRRELEVANT to us — we have NO latency target, so there is no SLA to track AIMD against; we just want the largest bucket the drain can fill. Clockwork's deterministic-timing insight IS usable: our MLP forward cost per batch size is deterministic and measured (190k@B64 -> 264k@B512), so we can pick the eval-batch bucket from a static cost table (largest bucket whose per-leaf cost is near the B>=512 asymptote) rather than tuning reactively.
- **Source:** https://www.usenix.org/conference/osdi20/presentation/gujarati
- **Proven / folklore:** empirically proven (Clipper NSDI'17, Clockwork OSDI'20); but the adaptive/SLA control loop is NOT applicable to a throughput-only, no-latency-target workload — only the amortize-framing and deterministic-cost-table ideas transfer.

**Top recommendation:** Adopt the KataGo NNEvaluator queue-drain mechanism (cpp/neuralnet/nneval.cpp), which is the production solution to OUR EXACT problem (many search threads, one single-device batched NN), and which our two existing adapters are merely the two degenerate corners of. Concretely: (1) Producers (C++ search) submit leaves with HIGH in-flight depth and MANY leaves per wire frame (S>1 to amortize ZMQ framing — fixes the greedy-async frame wall; D>1 = many parked leaves outstanding — fixes the strict-barrier serialization). (2) The single-threaded JAX server runs a 'pop up to E_max' drain loop (waitPopUpToN analog): it grabs WHATEVER is currently queued, up to a cap, and runs ONE forward at that actual count — this decouples send-batch S from eval-batch E, the decoupling neither adapter has. (3) Eliminate the fixed-pad 2x penalty with SHAPE BUCKETING: pre-compile the JAX forward for a handful of fixed batch sizes (e.g. 16/32/64/128/256) and dispatch a drained batch of N to the smallest bucket >= N — this is the XLA-legal form of Triton ragged batching and is what makes partial-batch draining (and the half-batch pipelining that previously regressed) actually pay off. (4) Add a Triton-style queue-delay knob, set GENEROUSLY since latency is free for us, so the drain prefers a full bucket. (5) For cross-request parallelism under the single-threaded XLA invariant, scale by PROCESSES (Triton instance_group / SEED RL replica pattern), starting at one coalescing server and adding a second process only if profiling shows the single server's recv+forward loop is the wall. Per ADR-0012 P7 the entire batch/delay/bucket policy lives on the server side; the wire frame stays a dumb transport envelope (correlation id only), so none of this touches the codec.

## selfplay-mcts-eval-arch

**Systems surveyed:** KataGo NNEvaluator (cpp/neuralnet/nneval.cpp) — central GPU eval queue fed by many MCTS search threads, ELF / ELF OpenGo — producer-consumer M-of-N batching daemon (arXiv:1707.01067), Leela Chess Zero / Lc0 — single-iteration leaf-gathering minibatch with virtual loss / collisions (lczero.org), SEED RL — centralized accelerated inference with server-side batching module + streaming RPC (arXiv:1910.06591), KataGo paper — Accelerating Self-Play Learning in Go (arXiv:1902.10565)

### Decoupled enqueue + dynamic-size server drain (KataGo NNEvaluator)

- **Mechanism:** Search threads do NOT send a batch and block on a matching reply. Each thread enqueues ONE leaf pointer (NNResultBuf) onto a shared lock-free queryQueue via forcePush, then waits on its own per-leaf condition variable (clientWaitingForResult). A separate single server thread per GPU runs serve(), which calls queryQueue.waitPopUpToN(resultBufs, desiredBatchSize) where desiredBatchSize = min(maxBatchSize, currentBatchSize). It takes WHATEVER is currently queued up to the cap — it does NOT wait for a full batch and does NOT pad. The popped set becomes ONE forward (NeuralNet::getOutput over numRows actual rows); results are scattered back and each leaf's CV is notified. This fully decouples S (each producer contributes 1 leaf), E (server-chosen = whatever was drained, naturally large under load), and D (the queue depth IS the in-flight buffer; many leaves outstanding without any per-leaf wire round-trip). Single-device invariant is preserved exactly: exactly one server thread owns the forward; concurrency lives entirely in the producer-side queue, never in the evaluator.
- **Applies to us:** This is the design point our adapter is missing. Today both adapters fix one knob wrong: strict-barrier ties S=K to D=1 (serialization-bound, 49 dps); greedy-async sets S=1 with framing per leaf (frame-bound, 37 dps). KataGo's queue says: producers enqueue cheaply, a single drainer chooses E=drained-count. Map onto ZMQ: C++ search threads push leaves into a SINGLE in-process MPSC queue (not the wire); one C++ drain thread coalesces up-to-N queued leaves into ONE multi-leaf wire frame (so S = drain size > 1 AND amortizes framing), and keeps D>1 by issuing a new drained frame whenever the queue refills without blocking on the prior reply. The Python JAX server stays single-threaded (invariant holds) and runs E = however many leaves the frame carried. This is the 'batched sends AND overlap with E decoupled' sweet spot the prompt names neither adapter occupies. Cleanly ADR-0012-shaped: the queue/drain is imperative shell (P9), predict(features) stays the PORT (P2), the correlation id stitching reply->leaf is a transport-envelope frame not a codec field (P7).
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** empirically proven — production KataGo self-play, the strongest open Go engine; mechanism is in shipped source

### No-pad partial forward (run whatever is queued)

- **Mechanism:** KataGo's serve() runs getOutput on exactly numRows = the actual popped count; there is no pad-to-max. A burst of 3 leaves runs as a real batch of 3. The paper notes this 'favors latency over throughput' as a deliberate choice, but throughput is preserved because under load the queue is deep so drained batches are naturally large; the small-batch case only happens when there is nothing else to do anyway.
- **Applies to us:** Directly addresses our '~2x pad waste on a partial batch' finding that killed the half-batch pipelining attempt. Our server pads every forward to fixed max_batch, so a partial frame pays full cost — that is precisely why our greedy/half-batch attempts regressed. The fix is NOT bigger pads, it is multiple XLA-compiled shapes (a small bucket set, e.g. 32/64/128/256/512) selected per drained frame, so a partial drain pays close to its true size. Our serve ceiling already rises 190k->264k from B=64 to B>=512, so larger natural drains (from the queue coalescer above) move us up that curve for free; bucketing removes the pad penalty on the in-between sizes. This is the single change that makes batched-send pipelining net-positive instead of net-negative.
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** empirically proven (KataGo source); the multi-bucket XLA-shape refinement is folklore/standard practice (Lc0 minibatch caps, Triton dynamic batching) rather than copied verbatim from KataGo

### M-of-N blocked-producer batching (ELF producer-consumer daemon)

- **Mechanism:** ELF runs N games each in its own C++ thread; when M < N games have reached a state needing evaluation, the daemon blocks exactly those M games, ships the batch of M to the Python side, and the consumer returns replies that unblock the M games for their next action. M is intentionally < N so there are always more games than batch slots — overcommit guarantees a batch is always fillable. Producer and consumer can be same-process (their default, for low overhead) or separate.
- **Applies to us:** This is the throughput justification for OVERCOMMIT, which our single-tree setup currently lacks. Our gen ceiling (76k leaves/s/core) is ~2.5x below serve at B=64 and ~3.5x below at B>=512 — a single search tree cannot keep a large eval batch full from in-flight leaves alone (virtual-loss collisions cap it, exactly the Lc0 wall below). ELF's answer: run MORE concurrent games/trees than batch slots so the evaluator is never starved. Concretely for us, since gen scales 4.0x linearly across our 4 cores: run several independent self-play trees (more producer threads/processes) feeding ONE shared eval queue, so the drained batch is large and the single JAX server runs near its B>=512 ceiling. This is how you convert the linear gen scaling into evaluator saturation without violating the single-server invariant.
- **Source:** https://arxiv.org/pdf/1707.01067
- **Proven / folklore:** empirically proven — ELF OpenGo trained a superhuman Go agent with this exact daemon

### Single-iteration leaf gathering bounded by virtual-loss collisions (Lc0)

- **Mechanism:** Lc0 gathers a minibatch (default 256, max ~1200, hard ceiling ~1800 even at 10M allowed collisions) of leaves PER search iteration before one backend forward. Virtual loss / visits-in-flight (multivisit, is_collision flags on NodeToProcess) discourage re-picking an in-flight node, but crucially 'virtual loss doesn't help prevent collisions' once one move dominates — a SINGLE tree cannot gather arbitrarily large batches. crem's write-up concludes huge batches (10000+) need fundamentally more trees, not better collision hacks.
- **Applies to us:** This is the honest CEILING on the in-tree-only approach and the empirical proof you cannot solve our problem by simply gathering a bigger batch from one tree. It tells us our D (in-flight depth) from a single search will saturate well below the B>=512 region where serve is fastest — consistent with our greedy-async being FLAT across depth 64..256. Therefore the throughput win must come from coalescing across MULTIPLE producers/trees (ELF pattern) to feed the queue, not from pushing one tree's collision limit. Sets realistic expectations: tune our drain/bucket targets around what multi-tree gathering actually delivers, not toward an unreachable single-tree mega-batch.
- **Source:** https://lczero.org/dev/old/lc2/batching/
- **Proven / folklore:** empirically proven — measured, documented Lc0 ceilings (1200 typical, 1800 max); a negative result that bounds the design space

### Centralized accelerated inference with server-side batching + persistent stream (SEED RL)

- **Mechanism:** All model state stays on the central learner/inference core; actors send only observations over a PERSISTENT streaming gRPC connection (metadata sent once, connection kept open). A dedicated batching module on the server coalesces many actor inference calls into one accelerator forward. Co-located actors use unix domain sockets to cut syscall/CPU/latency overhead. Inference and training core counts are tuned to balance load.
- **Applies to us:** Two concrete transferable ideas. (1) Persistent connection + send-metadata-once is the antidote to our greedy-async frame-overhead wall: a DEALER stream where the only per-leaf bytes are the raw feature vector + a correlation id (P7 transport-envelope), with no per-message handshake/codec re-negotiation, is what makes S>1 framing cheap. (2) Co-location via UDS: since our JAX server is in-process / same-host, the wire should be the cheapest local IPC, not TCP loopback per leaf — this directly attacks the framing wall that pins greedy-async at 37 dps. SEED's server-side batching module IS the KataGo-style drain, validated at million-FPS scale, confirming the centralized-batcher pattern is the right macro-architecture for our throughput-only objective.
- **Source:** https://arxiv.org/pdf/1910.06591
- **Proven / folklore:** empirically proven — 2.4M FPS on DeepMind Lab with 64 TPU cores, 80x over IMPALA; published and open-sourced

**Top recommendation:** Adopt the KataGo NNEvaluator design point: producers enqueue single leaves into one in-process MPSC queue and block on a per-leaf correlation-keyed slot; a single C++ drain thread coalesces up-to-N queued leaves into ONE multi-leaf wire frame (S>1, amortized framing) and keeps multiple frames in flight (D>1) without blocking on prior replies; the single-threaded JAX server runs E = the drained count against a small set of XLA-compiled batch buckets (NO fixed-max pad), so partial drains stop paying ~2x. This is exactly the 'batched sends AND overlap, with E decoupled' sweet spot neither current adapter occupies — it makes S, D, E independent (the ADR-0012 P7 separation of transport from codec), keeps the single-device invariant (one drainer owns the forward), and is the shipped, superhuman-proven design in KataGo. Because a single tree's in-flight leaves cap out below serve's fast region (Lc0's measured ~1200-1800 collision ceiling), pair the queue with ELF-style overcommit — feed it from multiple concurrent trees across the 4 linearly-scaling gen cores — so drained batches reach the B>=512 region where serve hits 264k leaves/s.

## parallel-mcts

**Systems surveyed:** KataGo (lightvector/KataGo) — NNEvaluator queryQueue / waitPopUpToN serve loop, virtual loss, numSearchThreads vs nnMaxBatchSize, WU-UCT (Liu et al., ICLR 2020) — master/worker pool with 'unobserved samples' statistic, asynchronous task submit/collect, AlphaZero / AlphaGo Zero — leaf/tree parallelization with virtual loss, the canonical batch-via-many-concurrent-descents pattern, AlphaZero.jl (jonathan-laurent) — inference-server-per-process, num_workers vs batch_size relationship, partial-batch behavior, DeepMind mctx / Gumbel MuZero — fully vectorized single-device batched MCTS (the JAX-native extreme; contrast case), Batch Monte Carlo Tree Search (Cazenave, 2021) — explicit batching of NN queries per MCTS iteration

### Many-more-search-threads-than-batch-size + 'up to N' opportunistic serve (the KataGo NNEvaluator pattern)

- **Mechanism:** KataGo decouples all three of our knobs structurally. SEND: each search thread enqueues ONE leaf (NNResultBuf*) into a shared lock-free queryQueue via forcePush — so per-producer S=1, but the queue aggregates. EVAL-BATCH: a dedicated single NN server thread calls queryQueue.waitPopUpToN(bufs, desiredBatchSize) where desiredBatchSize = min(maxBatchSize, currentBatchSize). It does NOT wait for a full batch — it grabs whatever is queued up to the cap and forwards immediately, then calls getOutput(... numRows ...) with the ACTUAL count (no pad-to-max; the backend takes the real row count). IN-FLIGHT DEPTH: D is set by numSearchThreads, which is deliberately configured FAR larger than maxBatchSize (config guidance: nnMaxBatchSize ~= numSearchThreads/2; 'threads need not relate to cores and can easily exceed them'). Each thread blocks on a per-buf condition variable (clientWaitingForResult.wait) until the server notify_all's it. Virtual loss (child->virtualLosses.fetch_add(1) during descent, removed on backprop) lets the many threads diverge to DIFFERENT leaves so the queue fills with distinct work. Single-device invariant is honored: ONE server thread owns the forward; concurrency lives entirely in the producer threads + the queue.
- **Applies to us:** This is the missing sweet spot our finding identifies (batched sends S>1 AND overlap D>1, E decoupled), realized WITHOUT either of our adapters' extremes. Map it onto our adapter as: the C++ side runs M concurrent Gumbel descents (M >> eval-batch), each parking exactly one leaf into a shared queue with virtual-loss-style divergence; a drain thread pulls 'up to N' and emits ONE wire message per drained group (so wire S = the opportunistic group size, naturally >1 under load, and D = number of groups allowed outstanding). Crucially KataGo proves the server should size its forward to the ACTUAL drained count, not pad-to-max — which directly attacks our 'partial batch pays full pad (~2x waste)' regression. Since our SERVE ceiling is ~190k@B=64 rising to 264k@B>=512 and GEN is only ~76k leaves/s/core, the forward has enormous headroom: the win is purely (a) amortizing wire frame overhead across S>1 leaves per message and (b) keeping D>1 so the C++ search never stalls waiting on recv. Under ADR-0012: the drained-group correlation/count lives in the transport envelope (P7 — one authoritative frame home, both sides derive), the queue+drain is imperative-shell around the pure descent core (P9), and predict(features)->expected stays the PORT (P2) operating on a span of leaves.
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** empirically proven — KataGo is the strongest open-source Go engine and this exact NNEvaluator design is its production self-play and match path; the config tuning guidance (threads >> batch) is documented and battle-tested.

### Virtual loss for batch formation (diverging concurrent descents)

- **Mechanism:** When K threads/descents traverse one shared tree concurrently, each applies a transient penalty (a phantom loss / extra visit) to nodes along its path before the NN result returns, so subsequent descents are steered toward DIFFERENT leaves. This is what turns 'K concurrent searches' into 'K distinct leaves to batch' rather than K copies of the same leaf. The penalty is reverted during backprop once the real value arrives. AlphaGo Zero / AlphaZero use it; KataGo uses fetch_add atomics so it is lock-free.
- **Applies to us:** Virtual loss is the enabling precondition for batched sends S>1 to be USEFUL: without it, batching K parked leaves risks K near-identical features (wasted forward width). If our Gumbel search already keeps m=24 candidate actions and n_sims=256 produces many independent leaves per decision, we likely already have natural leaf diversity within a decision; virtual loss matters most if we widen S by overlapping leaves ACROSS in-flight depth from the same tree. For throughput-only with linear 4x core scaling already observed in GEN, the simplest path is independent trees per descent slot (root/tree parallelism) so we may not need virtual loss at all — note it explicitly as optional, not load-bearing, for our objective.
- **Source:** https://arxiv.org/pdf/1810.11755
- **Proven / folklore:** empirically proven (AlphaGo Zero, KataGo) — but its NECESSITY for our throughput-only single-machine case is folklore; literature notes pure cross-game batching can match it without the exploration bias virtual loss introduces.

### WU-UCT master/worker async submit-collect with 'unobserved samples' (decoupling send from result)

- **Mechanism:** A single master thread owns selection + backprop (the cheap, inherently-serial steps); it dispatches the expensive expansion/eval steps to a worker POOL and continues issuing more tasks WITHOUT blocking on any one result. Outstanding work is tracked per-node as O_s (initiated-but-incomplete count): the UCT score uses N_s + O_s in the exploration term so the master keeps selecting sensibly while D tasks are in flight. In-flight depth = pool size (paper uses up to 16+16). Results return asynchronously tagged with a task index tau; master applies the complete-update (O_s--, N_s++) on arrival. Near-linear speedup to 16 workers with negligible strength loss.
- **Applies to us:** This is the academically-clean justification for D>1 in-flight depth and answers 'how does a single-threaded selector stay busy while evals are outstanding'. Our greedy-async adapter already had D=K but stalled FLAT because S=1 made it frame-overhead-bound — WU-UCT says the fix is not more depth (we proved depth 64..256 buys nothing) but keeping the dispatcher non-blocking AND batching the dispatch. Adopt the master-owns-selection / pool-owns-eval split: the C++ search thread is the master, the ZMQ drain+wire is the 'pool boundary', and an O_s-style outstanding counter lets one C++ thread keep many leaves in flight across batched messages. Confirms our throughput interest: WU-UCT's whole contribution is throughput (linear speedup) with strength held constant — exactly our 'latency freely tradeable' regime.
- **Source:** https://ar5iv.labs.arxiv.org/html/1810.11755
- **Proven / folklore:** empirically proven — ICLR 2020, linear speedup to 16 workers on Atari + a production system, with reference implementation (liuanji/WU-UCT).

### Inference-server-per-process + cross-game batching (the AlphaZero.jl / SEED-RL-style scaling answer)

- **Mechanism:** Because the evaluator is single-threaded/single-device by invariant, cross-request parallelism is obtained by running MANY game/search workers feeding ONE inference server that batches across them (num_workers >= batch_size; batch_size is filled from distinct workers' pending requests). To use more than one device or break a single forward's serial ceiling, you run more SERVER PROCESSES, each with its own workers — not more threads inside one XLA forward. AlphaZero.jl observed that with too few active workers the batch never fills (12 of 128 threads active) — the batch is only as good as the number of genuinely-concurrent outstanding requests.
- **Applies to us:** Directly answers our HARD CONSTRAINT (server single-threaded; parallelism from more PROCESSES). With a 4-vCPU host and GEN scaling 4.0x linearly, the throughput-maximizing topology is N independent (C++ search + in-process JAX server) processes pinned to cores 0,1,2,3, each self-contained over its own ZMQ DEALER — NOT one server multiplexing 4 searches (that recreates the single-thread serialization wall we measured at 49->30 dps as threads rise). Our per-core SYSTEM number (~49 dps) is the unit to multiply; the literature says do not fight the single forward's serial ceiling, replicate it. The AlphaZero.jl 'only 12 of 128 active' caution maps to our finding that more in-flight depth alone buys nothing if requests aren't genuinely concurrent and batched.
- **Source:** https://github.com/jonathan-laurent/AlphaZero.jl/issues/71
- **Proven / folklore:** proven as engineering practice (SEED RL centralizes inference + batches across actors; AlphaZero.jl is a working implementation), though the specific per-process-pinned topology for our 4-vCPU box is a folklore extrapolation we'd validate by sweep.

### Fully-vectorized single-device batched MCTS (mctx / Gumbel MuZero) — the contrast / ceiling case

- **Mechanism:** mctx runs the ENTIRE search batched: B independent trees advance in lockstep, and every simulation step gathers exactly B leaves (one per tree) into a single forward, fully JIT-compiled, no wire, no async. Send-batch == eval-batch == B by construction; in-flight depth is irrelevant because there is no async — the whole step is one synchronous vectorized forward. Decoupling is achieved by making S=E=B a compile-time constant and never crossing a process boundary.
- **Applies to us:** This is the design that makes our 'forward is not the bottleneck' true and shows the theoretical ceiling: if the search itself were in JAX, batch alignment is free. We can't adopt it wholesale (our search is C++ over the wire, by constraint), but it gives the target invariant for ADR-0012 P7: make the eval-batch shape E a value DERIVED from how many leaves the search naturally produces per step (Gumbel's m and the simulation schedule), and align the wire send-batch S to that natural width so a forward is rarely partial. It also justifies why padding is the enemy: mctx never pads because B is fixed and full — our adapter should either pick a small fixed E that the search reliably fills, or (KataGo route) size the forward to the actual count and drop padding entirely.
- **Source:** https://github.com/google-deepmind/mctx
- **Proven / folklore:** empirically proven for the in-process JAX case (mctx is DeepMind's published, used Gumbel MuZero implementation); its applicability across our process/wire boundary is by-analogy, not direct.

**Top recommendation:** Adopt the KataGo NNEvaluator topology, dropped onto our wire: run M concurrent Gumbel descents per core (M chosen well above the eval-batch, KataGo's rule of thumb nnMaxBatchSize ~= M/2), each parking exactly one leaf; a single drain thread does an 'up to N' grab of all currently-parked leaves and emits ONE wire message per drained group — making wire send-batch S>1 opportunistically (amortizing the per-frame overhead that walls our greedy-async adapter) while in-flight depth D>1 keeps the C++ search from ever blocking on recv (curing the serialization that walls our strict-barrier adapter). Critically, follow KataGo and size the JAX forward to the ACTUAL drained count rather than pad-to-max, eliminating the ~2x partial-batch pad waste that killed the half-batch pipelining attempt. Because the JAX server is single-threaded by invariant, scale OUT to 4 independent (C++ search + in-process server) processes pinned one-per-vCPU (the AlphaZero.jl / SEED-RL lesson: replicate the single forward, do not multiplex it) — this multiplies our measured ~49 dps/core by the GEN-proven 4.0x linear core scaling instead of fighting the single-thread wall we already saw drop per-core throughput from 49 to 30 dps as threads rose. Under ADR-0012 the drained-group count/correlation-id lives in the transport envelope (P7), the queue+drain is the imperative shell around the pure descent core (P9), and predict over a span of leaves stays the PORT (P2).

## producer-to-accelerator-transport

**Systems surveyed:** KataGo NNEvaluator (lightvector/KataGo, cpp/neuralnet/nneval.cpp) — C++ MCTS threads feeding a single-GPU batched evaluator, SEED RL (Espeholt et al. 2019, arXiv:1910.06591) — centralized batched inference over streaming gRPC + unix domain sockets, vLLM continuous batching + Shared-Memory IPC cache (blog.vllm.ai 2025), NVIDIA Triton Inference Server dynamic batcher (max_queue_delay_microseconds, preferred_batch_size), Leela Zero / Leela Chess Zero batched leaf evaluation (leela-zero PR #526, lc0 MinibatchSize), Apache Arrow zero-copy/zero-serialization + SPSC shared-memory ring buffers (arXiv:2404.03030, ipc_ring)

### Decoupled server-side coalescing queue (KataGo NNEvaluator pattern)

- **Mechanism:** The canonical solution to EXACTLY our class problem. Search threads call evaluate(): they push their leaf into a single thread-safe MPSC queue (queryQueue.forcePush) and block on a PER-REQUEST condvar (NRResultBuf.clientWaitingForResult). A dedicated single inference thread runs serverLoop: it does NOT wait for a full batch — it calls waitPopUpToN(resultBufs, desiredBatchSize) which grabs WHATEVER is currently queued, up to a bound, runs one forward, then wakes each waiter via its own condvar. Crucially the bound is desiredBatchSize = min(maxBatchSize, currentBatchSize) — an ATOMIC, settable at runtime — so the eval-batch shape (E) is fully decoupled from how many leaves any producer submitted (S) and from how many are outstanding (D). Send-vs-eval decoupling is structural: producers submit ONE leaf each (their natural unit) but the eval batch is whatever the queue drains to. In-flight depth = total parked leaves across all search threads, naturally bounded by thread count * (1 + virtual-loss collisions). Single-threaded evaluator is the explicit design: ONE server thread owns the device; all cross-request parallelism comes from many producer threads filling the queue while it computes.
- **Applies to us:** This is the design point we should adopt, ported across the wire. The key insight that breaks our deadlock between the two adapters: S (send-batch), D (in-flight), E (eval-batch) must be three SEPARATE mechanisms. Producers should submit at their natural granularity and a server-side coalescer should form the eval batch from whatever has arrived — not the C++ side pre-deciding E. Concretely: keep D>1 (async, like greedy) so the server always has work, but do NOT pay per-leaf framing (our S=1 wall at 37 dps). Instead batch the SEND too — the C++ search collects parked leaves (virtual-loss style) into ONE wire message of S>1 leaves, the server appends them to its pending set and on each forward drains min(pending, E_target). E_target is a runtime atomic like KataGo's currentBatchSize, NOT pad-to-max_batch (our strict-barrier mistake). Because GEN ceiling is 76k leaves/s and SERVE is 190k+, the server will almost always have a healthy partial batch waiting; sizing E_target near the steady-state arrival rate (a few dozen, not 512) kills the partial-pad 2x waste. Per ADR-0012 P2 the predict() port stays predict(features)->expected{value,logits}; this pattern lives entirely behind it.
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** Empirically proven — KataGo is the strongest open Go engine; this exact nneval.cpp queue+condvar+waitPopUpToN drives all its self-play and analysis, and its docs state nnMaxBatchSize batching is the dominant throughput lever, not core count.

### Streaming RPC with metadata-once + unix domain sockets (SEED RL transport)

- **Mechanism:** SEED RL centralizes inference on the accelerator and keeps every actor->learner connection as a PERSISTENT streaming gRPC channel: handshake/metadata is sent ONCE and thereafter only raw observation tensors flow, so per-message overhead collapses. When actors are co-located with the learner it switches the gRPC transport to unix domain sockets, explicitly to cut latency, CPU and syscall overhead vs TCP. A learner-side batching module accumulates many in-flight actor calls into one forward; inference and training cores are split and tunable for max utilization. Decoupling: many actors each have one outstanding call, the batcher forms the device batch from whatever calls are pending. Reported: 260K-440K Atari FPS and up to ~1M queries/s on one machine.
- **Applies to us:** Two transferable facts. (1) Our ZMQ DEALER is already a persistent stream, so we are not paying connection setup — but SEED's lesson is that the WIRE ENVELOPE per message must be near-zero: our S=1 wall is exactly the per-message framing SEED designed away by sending only the tensor payload after a one-time handshake. The fix is to amortize the envelope over S>1 leaves per frame (a length-prefixed contiguous block of 241-float rows), so framing cost per leaf -> ~0. (2) SEED proves the in-process/co-located case wants the cheapest possible IPC; since our JAX server is IN-PROCESS, we could bypass ZMQ serialization entirely (see SHM ring pattern) — SEED only used UDS because actors were separate processes. ADR-0012 P7: the wire frame stays one authoritative home both sides derive; the correlation id is a transport-envelope field, never a codec field — consistent with SEED's metadata-once split of envelope vs payload.
- **Source:** https://ar5iv.labs.arxiv.org/html/1910.06591
- **Proven / folklore:** Empirically proven at large scale (published throughput numbers, widely reproduced architecture), but the specific gRPC/UDS choice is a separate-process design — only partially applicable since our server is in-process.

### Server-side dynamic batcher: queue-delay + preferred batch size (Triton)

- **Mechanism:** Triton's dynamic batcher holds arriving requests briefly and dispatches a forward the instant EITHER a preferred_batch_size is reached OR max_queue_delay_microseconds expires — whichever comes first. preferred_batch_size is a SET of shapes the model was tuned/compiled for; the batcher prefers forming those exact shapes (so XLA/TensorRT hits a cached, well-padded kernel) but will fire a smaller batch rather than starve. This is the explicit, tunable knob that decouples request arrival (S, D) from the forward shape (E): the delay is the single throughput<->latency dial.
- **Applies to us:** Directly names our E knob and our pad-waste problem. Our server pads EVERY forward to max_batch, so a partial batch pays full pad (~2x waste) — Triton's answer is preferred_batch_size: compile/pad to a SMALL set of shapes (e.g. 32, 64) and bucket the drained batch up to the nearest, instead of always max. Since latency is free for us (throughput-only), we can set the equivalent of max_queue_delay generously: let the coalescer wait a tiny bounded interval to grow the batch toward a preferred bucket before firing, which raises occupancy without the strict-barrier serialization. This is the principled cure for the half-batch regression: don't pad a 30-leaf batch to 512 — pad it to the 32-bucket. Fits ADR-0012 P2 (port stays predict-shaped) and P9 (bucket selection is functional-core logic).
- **Source:** https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/batcher.html
- **Proven / folklore:** Empirically proven — Triton's dynamic batcher is the industry-standard production serving path; the queue-delay/preferred-size tradeoff is documented and benchmarked widely.

### In-tree leaf coalescing via virtual loss (Leela Zero / AlphaGo minibatch)

- **Mechanism:** On the PRODUCER side, MCTS would normally descend, hit a leaf, and block on one eval. Virtual loss lets a search thread temporarily penalize the path it just took so the NEXT descent explores a DIFFERENT leaf without waiting — letting one thread (or a few) accumulate a minibatch of distinct leaves, then submit them as ONE batched evaluation. Leela's MinibatchSize is 64-512; PR #526 packs 2/4/8 leaves into a single GPU invocation (float2/4/8), selecting the width by current queue depth, for ~33% more evals/s at 8 threads. This is how the SEND batch S>1 is produced legitimately without stalling the search.
- **Applies to us:** This is the missing half of our sweet spot — it produces S>1 on the C++ side WITHOUT serializing the search (which the strict-barrier S=K,D=1 adapter failed to do, and which greedy S=1 refused to do). Our Gumbel search already parks leaves; coalesce the parked set into one wire frame of S>1 leaves (matching ADR-0012 P7 framing) and keep D>1 frames in flight. Combined with the KataGo server-side coalescer and Triton bucketed pad, this is the full design: producer batches sends (S>1, amortizing framing), keeps overlap (D>1, hiding the residual round-trip), and the server forms E from the drained queue padded to a preferred bucket. Note Gumbel root-parallelism gives us natural distinct leaves (m=24 considered actions, n_sims=256) so we may not even need full virtual loss to get S>1.
- **Source:** https://github.com/leela-zero/leela-zero/pull/526
- **Proven / folklore:** Empirically proven — standard in every batched-GPU MCTS engine (Leela, lc0, KataGo all do leaf coalescing); the PR reports concrete measured speedups.

### Shared-memory ring / zero-copy payload (vLLM SHM-IPC, Arrow, SPSC ring)

- **Mechanism:** When producer and consumer are co-located, skip socket serialization entirely: a memory-mapped ring buffer where the producer writes raw bytes and the consumer reads them in place via head/tail atomics — no copy, no encode/decode. vLLM's 2025 SHM-IPC cache uses a shared-memory object store (one writer, many readers) and reports +69.9% prefill throughput / -40.5% TTFT by removing redundant copies. Apache Arrow's zero-serialization work notes ~30% of RPC time is pure serialization of record batches; SPSC shm rings (e.g. ipc_ring) sustain 16M+ msgs/s lock-free.
- **Applies to us:** Our JAX server is IN-PROCESS with the C++ search, so ZMQ's serialize->socket->deserialize is pure overhead — this is the most aggressive cure for the S=1 framing wall: the C++ side writes the 241-float feature rows directly into an mmap'd ring (or a slab JAX can wrap as a DLPack/numpy view zero-copy), the server reads N contiguous rows in place as the eval batch, writes (value,logits) back into a result ring, and signals via one futex/eventfd per drain — not per leaf. This eliminates both per-leaf framing AND the per-message syscall. ADR-0012 P7/P9 fit cleanly: the ring's record layout is the ONE authoritative wire frame both sides derive (drift-tested), span/expected typed, correlation id is the ring slot index (a transport-envelope fact, not a codec field). Caveat: ZMQ inproc may already be much cheaper than TCP — measure ZMQ-inproc first before building a ring; the ring is the ceiling, not necessarily step one.
- **Source:** https://blog.vllm.ai/2025/11/13/shm-ipc-cache.html
- **Proven / folklore:** Mechanism empirically proven (vLLM/Arrow/SPSC all report measured gains), but the specific application to our adapter is folklore-grade extrapolation — no published MCTS engine uses an in-process shm ring to a JAX server; needs our own A/B vs ZMQ-inproc.

**Top recommendation:** Adopt the KataGo NNEvaluator design point, ported across the wire: a server-side coalescing queue that fully decouples our three conflated knobs. (1) PRODUCER side — the C++ Gumbel search coalesces its parked leaves (Leela virtual-loss / Gumbel-considered-action style) into ONE wire frame of S>1 leaves, amortizing the per-message envelope that is our 37 dps S=1 wall, and keeps D>1 such frames in flight to hide the round-trip. (2) SERVER side — the single JAX thread does NOT block per message and does NOT pad-to-max; it drains whatever leaves have accumulated up to a runtime-settable E_target (KataGo's currentBatchSize atomic) and pads only to the nearest Triton-style preferred bucket (e.g. 32/64), killing the partial-pad 2x waste that sank the half-batch attempt. Each leaf carries a correlation id in the transport envelope (ADR-0012 P7) so out-of-order completion is fine — latency is free. Because GEN (76k leaves/s) is far below SERVE (190k+), E_target sized to the steady arrival rate (tens of leaves) keeps the device busy at high occupancy with low pad waste; this is the only one of our points that simultaneously satisfies S>1 AND D>1 AND E-decoupled, which neither current adapter does. If after wiring this the ZMQ envelope still dominates, escalate to the in-process shared-memory ring (vLLM/Arrow pattern) to take serialization to zero — but measure ZMQ-inproc first.

## cpu-small-mlp-batching

**Systems surveyed:** KataGo NNEvaluator (lightvector) — central eval thread pulling a queue of leaves from many search threads onto one device, Leela Chess Zero (lc0) multiplexing backend — combines NN eval requests from several threads into one batch, routes to whichever child backend is free, SEED RL (Espeholt et al., 2019, arXiv:1910.06591) — centralized learner-side dynamic batched inference over streaming gRPC, Oracle 'Lessons from AlphaZero pt.5' (Anthony Young) — remote inference server fed by a bucket-size + timeout batcher from many worker threads + virtual loss, Batch Monte Carlo Tree Search (arXiv:2104.04278) — collect b leaves per forward, decoupled from rollout count, Packrat (arXiv:2311.18174) — CPU DNN serving: multiple smaller instances beat one all-cores instance, NVIDIA Triton dynamic batcher — preferred_batch_size buckets + max_queue_delay timeout, ragged batching to avoid padding

### Central eval thread + leaf queue (KataGo NNEvaluator pattern)

- **Mechanism:** A single dedicated server thread owns the device/forward. Search threads do NOT each block-and-recv; on reaching a leaf they enqueue(features) into a shared queue and park. The eval thread drains the queue, assembles whatever leaves are present up to nnMaxBatchSize into ONE forward, runs it, then wakes the waiters. This is the clean S>1 AND D>1 decoupling: the SEND unit (one enqueue per leaf) is divorced from the EVAL batch (whatever the eval thread scooped up this cycle). In-flight depth is governed by how many search threads/virtual-loss slots exist, NOT by the send granularity. Single-device is the design center: exactly one thread touches the net.
- **Applies to us:** This is the missing sweet-spot neither of our two adapters occupy. Today strict-barrier conflates S=K,D=1 (serializes) and greedy-async conflates S=1,D=K (pays per-leaf framing K times). Adopt KataGo's shape ACROSS THE WIRE: the C++ side runs a drain loop that pulls all currently-parked leaves and ships them as ONE multi-leaf wire frame (S>1), while keeping multiple such frames outstanding (D>1) so the search keeps descending under virtual loss. The Python server's eval-batch E is then whatever it pulls off the socket, decoupled from S. Per ADR-0012 P7 the correlation id binding leaf->result is a transport-envelope frame, not a codec field; the multi-leaf frame body is the typed std::span (P9). Our SERVE ceiling (190k@B64) is 4x our per-core GEN ceiling (76k leaves/s), so even modest batched frames leave the forward idle — framing/serialization, not compute, is the wall the per-leaf greedy adapter hit.
- **Source:** https://github.com/lightvector/KataGo/issues/252
- **Proven / folklore:** proven — KataGo's self-play and analysis engine is the dominant open-source MCTS+NN system and this is its production architecture; lightvector's config guidance (numGpus*nnMaxBatchSize >= numSearchThreads*numAnalysisThreads) is the explicit tuning rule.

### Bucket + timeout dynamic batcher (Oracle AlphaZero / SEED RL / Triton)

- **Mechanism:** The server-side batcher accumulates incoming requests into a bucket and fires a forward when EITHER the bucket reaches a preferred size OR a max-queue-delay timeout elapses — whichever comes first. SEED RL's learner runs a dedicated inference thread batching 'whatever requests arrived within a time window or up to a max'. Oracle explicitly 'combined inference requests from different worker threads into a bucket before sending them to our inference service' and notes 'choosing an appropriate inference bucket size and timeout was very important' because fill rate varies (late games stall the bucket). Triton generalizes this with preferred_batch_size (a small SET of shapes that dispatch immediately) + max_queue_delay_microseconds.
- **Applies to us:** Directly relevant: we ALSO use a remote/over-wire inference server fed by concurrent search. The timeout is the safety valve that makes S>1 batched sends safe under throughput-only objective — you never deadlock waiting for a bucket that won't fill, but you DO amortize framing when leaves are plentiful (which, at GEN 76k leaves/s/core vs SERVE 190k, they usually are). Concretely: set the C++ drain to flush on (bucket>=S_target) OR (short timeout), and on the server set E to dispatch at the arrived size. Since latency is freely tradeable for us, the timeout can be generous, biasing hard toward full frames. ADR-0012 P2: the batcher lives behind the predict(features)->expected<{value,logits}> PORT; it is an implementation detail of the evaluator boundary, not the search.
- **Source:** https://medium.com/oracledevs/lessons-from-alpha-zero-part-5-performance-optimization-664b38dc509e
- **Proven / folklore:** proven — SEED RL (1M QPS single machine), Triton (production serving), and Oracle's writeup (>95% utilization, training weeks->day) all report measured throughput from exactly this mechanism.

### Preferred-batch buckets / ragged batching to kill the fixed pad

- **Mechanism:** Instead of one fixed XLA shape padded to max_batch (so a half batch pays full cost), compile the forward at a SMALL SET of bucketed sizes (Triton preferred_batch_size, e.g. {16,32,64}) and dispatch at the smallest bucket >= arrived count. Ragged batching avoids padding entirely for backends that accept 1-D variable-length inputs. The point: the eval-batch shape E tracks demand rather than always paying max.
- **Applies to us:** This is the direct fix for our stated regression: 'the server pads every forward to a fixed shape, so a PARTIAL batch pays the FULL pad (~2x waste) — a half-batch pipelining attempt regressed.' Compile the JAX MLP at 2-4 fixed bucket shapes (XLA needs static shapes, so true ragged is out, but bucketing is the static-shape analog) and route each arrived batch to the nearest bucket >= count. This makes E genuinely decoupled from S and D: a partial frame dispatches at a cheaper shape instead of eating the max pad. With our tiny MLP (241->256->65) the buckets are cheap to JIT-cache. Caveat: each bucket is a separate XLA executable; keep the set small (e.g. {32,64,128,512}) to bound compile/cache cost. ADR-0012 P7: bucket boundaries are a server-side eval concern, fully orthogonal to the wire frame.
- **Source:** https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/batcher.html
- **Proven / folklore:** proven for the bucketing mechanism (Triton production); 'XLA must bucket because it needs static shapes' is well-established folklore-grade engineering practice (JAX/XLA recompiles per shape), but the specific 2x-pad-saving on a 241-dim MLP is unmeasured for us — verify empirically.

### Multiple server PROCESSES over one all-cores instance (Packrat)

- **Mechanism:** Packrat's central finding for CPU DNN serving: a single model instance using all CPU threads gives diminishing intra-op returns; deploying MULTIPLE instances (processes), each with FEWER threads and SMALLER batches, achieves better aggregate throughput/latency. Optimal config is a joint choice over (#instances i, threads/instance t, batch/instance b) depending on model and core count. For a tiny MLP, intra-op parallelism saturates almost immediately (the matmul is small relative to dispatch), so spare cores are better spent as independent instances each running a full batched forward.
- **Applies to us:** Decisive for our 4-vCPU box + single-threaded-XLA invariant. Our SERVE forward is single-threaded by design (no XLA in a worker thread), and a 241x256x65 MLP will NOT scale intra-op across 4 cores — it saturates near 1 core. So the right shape is NOT one fat server; it is N server PROCESSES, each single-threaded XLA, fed by a shard of the search. GEN scales 4.0x linearly across 4 cores, and SERVE per-core (190k) already dwarfs per-core GEN (76k) — meaning ~1 serve process can feed ~2 gen cores. Candidate allocation on 4 vCPU: 3 gen workers + 1 serve process, or 2 serve + 2 gen, sweep it. This also explains our 'per-core DROPS at 2 threads' symptom: contention on a single shared server/socket. Packrat says: give each shard its own instance. ADR-0012's process-level fan-out (az/parallel.py) is the existing home for this; the per-process inference PORT (P2) makes adding processes a config change, not a rewrite.
- **Source:** https://arxiv.org/abs/2311.18174
- **Proven / folklore:** proven — Packrat is a peer-reviewed CPU-serving system reporting 1.43x-1.83x improvement from multi-instance over single-instance across common DNNs; the 'tiny MLP saturates intra-op fast' premise is standard CPU-inference knowledge (oneDNN/AMX saturate by ~batch 16 and don't scale threads well on small GEMMs).

### Virtual loss as the in-flight-depth engine (decoupled from send unit)

- **Mechanism:** Virtual loss temporarily decrements a node's value/increments visits along the descended path so concurrent descents pick DIFFERENT leaves, letting many leaves be in flight to the NN without collapsing to one path. It is the mechanism that GENERATES batch fill: D (in-flight depth) is set by how many virtual-loss descents you allow, not by the wire. Batch-MCTS formalizes this as collecting b leaves per forward where b is decoupled from total rollouts; duplicates that still collide are merged via a transposition table so one forward serves them all.
- **Applies to us:** This is what lets our C++ search SUPPLY a deep enough in-flight queue to make S>1 batched frames fill quickly. Our greedy-async adapter already had D=K in flight but FLAT throughput across D=64..256 — because S=1 framing was the wall, not depth. Virtual loss tells us depth is already adequate; the lever is moving those K in-flight leaves into FEWER, FATTER frames (S>1). With n_sims=256, m=24, there is ample tree to descend under virtual loss to keep a bucket full between flushes. Add an in-process leaf-dedup/transposition check before framing so colliding leaves don't burn wire+forward twice (cheap throughput win, matches Oracle's '2x fewer requests' caching result).
- **Source:** https://arxiv.org/pdf/2104.04278
- **Proven / folklore:** proven — virtual loss is the standard AlphaZero/AlphaGo parallel-MCTS mechanism; the dedup/transposition throughput win is measured in both Oracle (>2x request reduction) and Batch-MCTS.

**Top recommendation:** Adopt the KataGo central-eval-thread shape over the wire, fixing E with a small set of XLA bucket shapes, and run it as MULTIPLE single-threaded server processes per Packrat. Concretely: (1) the C++ search drains all currently-parked leaves (under virtual loss + in-process dedup) into ONE multi-leaf wire frame — S>1 — and keeps several frames outstanding — D>1 — flushing on (bucket>=S_target OR short timeout); (2) the Python JAX server dispatches each arrived batch at the smallest precompiled bucket shape >= count, so a partial batch no longer pays the full pad; (3) because the 241->256->65 MLP saturates intra-op near one core and XLA is single-threaded by invariant, run N independent single-threaded server PROCESSES on the 4-vCPU box (sweep the gen:serve split, e.g. 3:1 or 2:2) rather than one shared server — this also removes the contention that makes per-core throughput DROP at 2 threads today. This occupies the sweet spot neither current adapter reaches (batched sends S>1 AND overlap D>1 with E decoupled), and every piece is the production design of KataGo / lc0-multiplexing / SEED RL / Triton / Packrat rather than a novel bet.

## centralized-inference-rl

**Systems surveyed:** KataGo (lightvector) — C++ MCTS feeding a single-device batched NN via NNEvaluator (closest published analogue), SEED RL (Espeholt et al. 2019, arXiv:1910.06591) — centralized batched inference for thousands of actors over streaming gRPC, Sample Factory (Petrenko et al. 2020, arXiv:2006.11751) — async sampling, double-buffered sampling, dedicated policy workers, Ape-X / IMPALA — distributed actor-learner (contrast: per-actor LOCAL NN copies, the anti-pattern under our 'forward must run in JAX over the wire' constraint), NVIDIA Triton dynamic batching (max_queue_delay) + BucketServe (arXiv:2507.17120) — batch-formation discipline and pad-waste mitigation

### KataGo NNEvaluator: oversubscribed synchronous threads + waitPopUpToN drain (the direct analogue)

- **Mechanism:** KataGo is exactly our problem: a C++ MCTS where every search thread is SYNCHRONOUS — on reaching a leaf it enqueues features and blocks on a per-request condvar (buf.clientWaitingForResult). A single NN server thread owns the device and runs a drain loop: desiredBatchSize = min(maxBatchSize, currentBatchSize); queryQueue.waitPopUpToN(resultBufs, desiredBatchSize) — it grabs WHATEVER IS AVAILABLE up to the cap rather than blocking for a full batch, runs ONE forward on exactly numRows present, then notify_all the waiters. This DECOUPLES the three knobs cleanly: send-batch S=1 per thread (each thread enqueues one leaf), eval-batch E = however many threads happen to be parked at the queue at drain time (a partial batch is processed immediately, no timeout), and in-flight depth D = numSearchThreads. The crux move is OVERSUBSCRIPTION: numSearchThreads is set FAR above core count (docs: 'optimal threads may be much higher than the number of CPU cores ... many threads are needed to feed efficient large batches'), and nnMaxBatchSize defaults to ~numSearchThreads. The batch fills not because senders pre-aggregate but because many independent in-flight requests pile up at the single server's queue between forwards.
- **Applies to us:** This is the design point our adapter is missing — neither of our two adapters occupies it. It says: don't make the C++ sender pre-batch (S>1) NOR block-after-send (D=1); instead make MANY in-flight single-leaf requests (D high) and let the SERVER form the eval batch by draining the queue (E = whatever arrived). Concretely for the zmq adapter: keep send-batch S small (even S=1), push D way up (our greedy-async already proved D=64..256 works without wedging), and move batch formation to the SERVER drain loop. Our greedy-async was FLAT across depth and stuck at 37 dps because the wall was per-leaf FRAMING, not lack of overlap — KataGo pays ~zero framing because it is in-process shared-memory enqueue, not a wire codec. So the KataGo lesson for us is two-fold: (1) adopt server-side drain (E decoupled, partial batch run immediately, NO pad-to-max), and (2) the only thing standing between us and it is the ZMQ frame cost, which forces S>1 to amortize. The synthesis: S moderate (amortize framing), D high (KataGo oversubscription), E = server-drained partial-OK batch. ADR-0012 P7 fits: the correlation id that lets D>1 responses return out-of-order is a transport-envelope frame, never a codec field — exactly KataGo's per-request resultBuf identity.
- **Source:** https://github.com/lightvector/KataGo/blob/master/cpp/neuralnet/nneval.cpp
- **Proven / folklore:** Empirically proven — production engine, this exact code path runs every KataGo self-play game; the 'oversubscribe threads to fill GPU batches' guidance is in the shipped config docs and reproduced by its benchmark tool.

### SEED RL centralized batched inference + multiple-envs-per-actor to hide RTT

- **Mechanism:** SEED moves ALL inference to a central learner-side accelerator (actors send observations, receive actions) — precisely our 'forward must run in JAX over the wire, never C++-local' constraint, made a deliberate design choice. Two mechanisms matter. (1) A batching MODULE on the learner aggregates many concurrent actor inference calls into one accelerator forward; it does NOT wait for a fixed full batch (no documented timeout) — it opportunistically batches whatever concurrent calls are present, same shape as KataGo's drain. (2) To keep actors from idling on the round-trip, each actor runs MANY environments, so there are always more outstanding requests than there are cores — this is the in-flight-depth knob D expressed at the actor side. Transport is STREAMING gRPC: the connection stays open and metadata is sent ONCE per stream, not per request — they explicitly engineered away per-message framing/handshake overhead, and on the same machine use unix domain sockets to cut syscall/CPU cost.
- **Applies to us:** SEED is the published vindication of our whole architecture (centralized single-device evaluator fed by concurrent search) AND a direct diagnosis of our wall. Our greedy-async adapter is SEED's shape (D=K in flight) but bottlenecked on per-message framing — SEED's headline transport optimization is EXACTLY removing that: 'streaming RPCs ... metadata sent only once.' Our ZMQ DEALER sends a fresh framed message per leaf; SEED would say amortize the envelope across many leaves on a persistent stream. Concrete adapter move: stop paying full envelope per leaf — either (a) coalesce multiple leaves per wire message (S>1, which our batched-frame result already shows beats latency-hiding at these sizes), or (b) strip the per-message metadata to a minimal correlation-id envelope (ADR-0012 P7: transport frame derived from one home, not re-sent per call). 'Multiple envs per actor' maps to: don't run one descent at a time per core — keep many parked leaves in flight per worker so the server always has a batch to drain. Throughput-only is SEED's own framing (they optimized FPS/QPS, latency was a free side-benefit). Note their numbers (260K–330K env FPS, ~1M QPS) are at TPU-pod scale; the ARCHITECTURE transfers, the magnitudes do not — our serve ceiling is 264K leaves/s on one core.
- **Source:** https://ar5iv.labs.arxiv.org/html/1910.06591
- **Proven / folklore:** Empirically proven at scale — published, open-sourced (google-research/seed_rl), benchmarked to 260K+ FPS on Atari-57 with 8 TPU v3 cores and 610 actors; the centralized-vs-local-copy win is the paper's central measured result.

### Sample Factory double-buffered sampling: overlap eval with generation by splitting the generator

- **Mechanism:** Sample Factory fully decouples three async stages (rollout workers / policy workers / learner) over SHARED MEMORY (no pickle, no socket serialization between them). The throughput-critical trick is DOUBLE-BUFFERED SAMPLING: each rollout worker simulates 2N environments split into two groups; while group A's observations are out at the policy worker for the GPU forward, the worker is already stepping group B. This 'practically eliminates idle time' — the generator never blocks on the evaluator. The policy worker is the single-device batcher: it collects observations from MANY rollout workers into one GPU forward (it 'balances waiting slightly for additional requests against processing immediately'). Send-batch vs eval-batch are decoupled: each worker sends its group; the policy worker forms the eval batch by union across workers. In-flight depth is the number of env-groups across all workers (= 2N x num_workers).
- **Applies to us:** Our GEN ceiling (76k leaves/s/core) and SERVE ceiling (190–264k leaves/s) sit far above our SYSTEM (≈49 dps). Double-buffering names the latency-hiding half of that gap: while a batch of parked leaves is out over zmq awaiting (value,logits), the C++ search should be DESCENDING the next batch of leaves, not blocking. Our strict-barrier adapter (D=1, submit-then-block) is the exact thing double-buffering forbids — it serializes gen and eval, which is why it tops out at 49. Concrete move: split each worker's tree-descent into two leaf-cohorts and alternate, so generation of cohort B overlaps the wire+forward of cohort A. This is the OVERLAP (D>1) axis combined with batched sends (S>1) — the 'sweet spot neither adapter occupies.' Caveat honestly: SF's overlap is cheap because it's shared-memory in-process; over our ZMQ wire the overlap only pays once framing is amortized (the SEED lesson), so double-buffering and frame-coalescing must land TOGETHER, not separately — a half-batch overlap attempt regressed for us precisely because partial batches paid full pad.
- **Source:** https://www.samplefactory.dev/07-advanced-topics/double-buffered/ (paper: arXiv:2006.11751)
- **Proven / folklore:** Empirically proven — Sample Factory reported ~100K+ FPS single-machine (Atari ≈130K FPS), with double-buffering an ablated, documented, default-available feature.

### Server-side dynamic batching with bucketing — kill the pad-to-max waste (Triton / BucketServe)

- **Mechanism:** Triton's dynamic batcher forms a batch from queued requests governed by max_queue_delay: at delay=0 it forms whatever-is-present (variable size, like KataGo's drain); a small delay lets more requests join to hit 'preferred batch sizes.' The key reported failure mode is PADDING WASTE — when a forward is padded to a fixed shape, a partial batch pays full compute for pad rows that 'do not contribute to results.' BucketServe's fix: group requests into a few size BUCKETS and run each at its bucket shape, cutting pad overhead at <1% bucketing cost. The discipline: a SMALL set of fixed eval shapes (so XLA/kernels stay compiled) rather than one giant pad-to-max shape.
- **Applies to us:** This targets our single sharpest stated regression: 'the server pads EVERY forward to a fixed shape, so a PARTIAL batch pays the FULL pad (~2x waste)' — which is why a half-batch pipelining attempt regressed. Our serve ceiling itself shows the shape sensitivity: 190k at B=64 rising to 264k at B>=512, so eval-batch shape E is a real throughput lever, not just a correctness detail. Two concrete moves, both ADR-0012-clean (E is orthogonal to S and D): (1) replace pad-to-max with a SMALL BUCKET SET of precompiled XLA shapes (e.g. 64/128/256/512) and round a drained partial batch UP to the nearest bucket — a 70-leaf batch runs at shape 128, not 512, killing most of the 2x waste while keeping XLA from recompiling per odd size. (2) Equivalently apply a tiny max_queue_delay on the server drain so partial batches fill toward a preferred bucket before firing — but since latency is free for us, prefer (1) bucketing, which adds zero stall. This is what makes the KataGo 'process partial batch immediately' lesson safe under a padding evaluator: drain immediately, but snap E to a bucket.
- **Source:** https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/batcher.html
- **Proven / folklore:** Mixed: Triton dynamic batching + preferred-batch-sizes is empirically proven and widely deployed; XLA-shape-bucketing to avoid recompilation is established JAX folklore/practice; BucketServe's specific RL-evaluator numbers are LLM-serving, so the magnitude transfer to a 241->256->65 MLP is unproven for us and should be measured against the existing B-sweep.

### Ape-X per-actor local NN copy (the rejected alternative — names why our constraint exists)

- **Mechanism:** Ape-X scales acting by giving EACH actor its own local copy of the DNN for action selection; only experience (and periodically parameters) crosses the network, and it 'batches all communications with the centralized replay ... at the cost of some latency.' Critically, inference is NOT centralized — there is no shared single-device evaluator on the acting path; the device-batching only happens on the learner's gradient step.
- **Applies to us:** This is the design we are explicitly FORBIDDEN from adopting ('a C++-local forward is off-limits; the forward MUST run in JAX over the wire'). Worth surfacing because it sharpens the trade: Ape-X's local-copy avoids the wire entirely (no framing wall, no single-server serialization) but duplicates the model per actor and is exactly what SEED RL was invented to replace (centralizing inference to recover accelerator utilization and avoid stale per-actor params). Our constraint puts us in SEED's regime, not Ape-X's — so our optimization budget must go into transport (frame amortization, persistent stream) + server-drain batching, NOT into eliminating the wire. It confirms there is no free lunch hiding in the literature that satisfies our constraints: given 'JAX over the wire' + 'single-threaded server,' KataGo+SEED is the target shape and the wire framing is a cost to amortize, not to delete.
- **Source:** https://ar5iv.labs.arxiv.org/html/1803.00933
- **Proven / folklore:** Empirically proven (DeepMind, published, strong Atari results) — but as a CONTRAST case; its mechanism is ruled out by our hard constraints, included to show the design space boundary honestly.

**Top recommendation:** Adopt the KataGo NNEvaluator shape, generalized to the wire: a single-threaded server DRAIN LOOP that pulls whatever requests are present up to a cap (waitPopUpToN-style, partial batch run immediately), fed by HIGH in-flight depth D from oversubscribed concurrent descents — and close the gap our two adapters leave open by combining the three knobs that the literature treats as orthogonal: moderate send-batch S (to amortize the ZMQ frame, the wall SEED RL removes via persistent-stream/metadata-once and our own batched-frame result confirms beats pure latency-hiding), high in-flight depth D (KataGo oversubscription + Sample Factory double-buffering, so generation of the next leaf cohort overlaps the forward of the current one), and a server-side eval-batch E that is DECOUPLED and SHAPE-BUCKETED (a small set of precompiled XLA shapes, drained partial batch snapped UP to the nearest bucket) so a partial batch no longer pays the full pad-to-max ~2x waste that sank the earlier half-batch attempt. This is the 'sweet spot neither adapter occupies' (S>1 AND D>1 with E decoupled), it is the empirically-proven architecture of the closest published analogue (KataGo) and its large-scale generalization (SEED RL), and every piece maps cleanly onto ADR-0012 (correlation-id as a transport-envelope frame enabling out-of-order D>1 returns under P7; predict-port partial-batch contract under P2; E orthogonal to transport).

# COMPLETENESS CRITIC: what the surveys missed

The five surveys converged hard on KataGo NNEvaluator + SEED RL + Triton dynamic-batching + Packrat multi-process, and they are right that this is the macro-shape. But that convergence left specific, directly-load-bearing gaps. Below, each miss is named with why it bites our exact S/D/E + single-threaded-server + pad-waste problem.

## 1. The lc0 **demux backend** (not just multiplexing) — MISSED, and it is the literal mechanism for the surveys' hand-waved "scale by processes"

Every survey cited lc0's *multiplexing* backend (gather threads -> one batch). None cited its sibling, the **demux backend**, which does the inverse and is the more relevant one for us: it "takes a large batch which comes from search, splits into smaller batches and sends them to children backends to compute in parallel," with a `minimum-split-size` parameter to stop sub-batches getting too small. There is also a **roundrobin** backend (1st request -> backend 1, 2nd -> backend 2, …, "sends them verbatim immediately").

Why it matters concretely: the surveys repeatedly recommend "run P single-threaded JAX server processes" (Packrat) but treat the *fan-out* as an unspecified detail. lc0 demux IS that fan-out as a shipped, named component — and `minimum-split-size` is precisely the knob that prevents our pad-waste failure mode (don't split a drained batch into sub-batches so small that each pays the per-forward fixed cost). roundrobin is the degenerate cheap version when all server processes are identical (our case: identical MLP replicas pinned one-per-vCPU). This directly informs our E-decoupling across P processes: one C++ drainer, demux to P identical servers, split no finer than `minimum-split-size`.
Source: https://lczero.org/blog/2019/04/backend-configuration/

## 2. The **framework-tax / small-batch CPU-overhead** result — MISSED, and it explains our pad-waste curve mechanistically

The surveys treat our "190k@B64 -> 264k@B512" curve and the "partial batch pays full pad ~2x" as two separate facts. The missing literature unifies them: at **small batch sizes, fixed per-forward host/dispatch overhead exceeds the GPU/compute work** ("at small batch sizes CPU overhead can become larger than GPU run time"; "the framework tax"). For our 241->256->65 MLP — a tiny GEMM — the forward is *dominated* by fixed dispatch cost, not the matmul, which is exactly why B64->B512 still gains 40%. This reframes the bucketing recommendation: the win from bigger natural drains is **not** avoiding pad FLOPs (the FLOPs are trivial), it's **amortizing the fixed per-forward dispatch cost** over more rows. That changes the bucket-set design: you want the *fewest, largest* buckets the drain can fill, because each distinct forward pays the fixed tax — not a fine-grained 16/32/64 ladder, which the surveys variously proposed.
Sources: https://arxiv.org/pdf/2302.06117 ("The Framework Tax"), https://pytorch.org/blog/accelerating-pytorch-with-cuda-graphs/

## 3. **CUDA-graph / persistent-compiled fixed-shape replay** — MISSED, and it is the *reason to prefer few fixed buckets over ragged*

The surveys cite Triton "ragged batching" as the pad-killer and then concede "XLA needs static shapes so true ragged is out." They missed the actual production resolution of that tension: **capture a fixed-shape forward once and replay it** (CUDA graphs; the JAX/XLA analog is an AOT-compiled executable per bucket shape). The measured result — replay "reduces per-iteration latency by eliminating ~28% overhead from kernel launch and stream synchronization" and "MLP doesn't have this problem [of grid dependence]… well-suited for CUDA graph capture" — says small fixed-shape MLP forwards are the *ideal* CUDA-graph case. The transferable lesson for our single-threaded JAX server: a handful of **AOT-compiled, fixed-shape** bucket executables (replay-cheap) beats any dynamic-shape scheme, and the fixed per-call overhead this removes is the same tax from miss #2. This is the substantiation for "bucket, don't pad-to-max" that the surveys asserted but never grounded in a measured number.
Sources: https://pytorch.org/blog/accelerating-pytorch-with-cuda-graphs/, https://docs.latentai.io/leip/deploy/2.1/content/guide/cuda_graph/

## 4. **ZMQ zero-copy + multipart framing** — MISSED the concrete mechanism behind "amortize the frame"

Every survey diagnosed greedy-async's 37 dps as "per-leaf framing" and prescribed S>1, but none named the ZMQ primitive that makes a multi-leaf frame actually cheap: **`zmq_msg_init_data()` zero-copy + multipart messages**, where "multipart messages work nicely together with zero-copy, allowing you to send multiple buffers from different sources as individual frames," and **inproc "is faster, especially on larger messages since there's no copying."** This is directly actionable and ADR-0012-shaped: a drained group of N leaves becomes one multipart message — one envelope frame (correlation ids, the P7 transport home) + N zero-copy body frames over the 241-float rows — with no serialization copy. It also gives the empirical ordering the producer-transport survey hedged on: **measure inproc first** — the guide states IPC ≈ local TCP but inproc avoids the copy, so for our in-process JAX server, `inproc://` is the cheap baseline before any shared-memory ring is built.
Sources: http://wiki.zeromq.org/blog:zero-copy, https://zguide.zeromq.org/docs/chapter2/

## 5. **Condition-variable wakeup latency** — MISSED a real failure mode of the KataGo per-leaf-condvar design we're being told to copy

All five surveys recommend KataGo's "block on a per-leaf condition variable, server notify_all" verbatim. The missed literature warns that at our throughput target this can itself become the wall: **"C++ condition variables proved insufficient in wake-up latency"**, futexes are ~30x faster than condvars on x64 for wakeup, and the concurrency-cost hierarchy shows blocking primitives cost on the order of microseconds. At 76k leaves/s/core a per-leaf condvar notify is in the hot path. The honest design note the surveys omit: prefer a **futex/eventfd signalled once per drained group** (not per leaf) — which composes with miss #4 (one wakeup per multipart message) — or a brief bounded spin, since latency is free for us and CPU is the constrained resource on a 4-vCPU box. This is a caveat on the survey's top recommendation, not a new architecture.
Sources: https://travisdowns.github.io/blog/2020/07/06/concurrency-costs.html, https://www.remlab.net/op/futex-condvar.shtml

## 6. **TensorRT-LLM / continuous-batching honesty check** — the surveys half-flagged this; the missing fact sharpens the boundary

The production-batching survey correctly cautioned that continuous batching is for autoregressive multi-iteration requests. The missing confirmation: TensorRT-LLM's "in-flight batching IS continuous/iteration-level batching" — admitting new requests *at each generation step of the token loop*. Our forward has **no generation loop** (one leaf -> one value/policy, single iteration), so the *entire* in-flight-batching machinery collapses to its trivial case for us: "form a batch from whatever arrived." This is worth stating sharply because it rules out a tempting mis-transfer — we should NOT reach for vLLM/TRT-LLM batch-manager schedulers; their value is iteration-level retirement we structurally cannot use. The surveys gestured at this; the explicit "in-flight == iteration-level == needs a token loop" definition closes it.
Source: https://nvidia.github.io/TensorRT-LLM/overview.html

## Net assessment

The surveys are **architecturally complete** (KataGo queue-drain + bucketed E + multi-process replicas is the right target; nothing overturns that). The misses are at the **mechanism layer** the implementation actually needs:

- **#1 lc0 demux/`minimum-split-size`** — the named fan-out primitive for the multi-process recommendation, and the floor on sub-batch size that prevents pad-waste.
- **#2 framework-tax** — reframes why bigger drains win (amortize fixed dispatch, not pad FLOPs) -> prefer **few large** buckets, not a fine ladder.
- **#3 CUDA-graph/AOT fixed-shape replay** — the measured (~28%) substantiation for "few fixed buckets beats ragged/pad-to-max" on a tiny MLP.
- **#4 ZMQ zero-copy multipart** — the concrete cheap-frame primitive; mandates `inproc://` baseline measurement first.
- **#5 condvar wakeup cost** — a real caveat on the copy-KataGo-verbatim advice; signal once per group via futex/eventfd, not per leaf.
- **#6 in-flight==iteration-level** — sharpens the boundary that rules out vLLM/TRT-LLM schedulers for our single-shot forward.

Misses #2 and #3 together change a survey conclusion: the bucket set should be **few and large** (amortize the fixed per-forward tax via AOT-compiled replay), not the 16/32/64/128 fine ladder several surveys proposed.
