# Phase 1 — The atomic feature surface

Each control tick the policy receives ONE marshalled observation: a flat dict produced by decode_features() (cpp/stage_a/issue_engine.py:34-49) from a single packed binary FEATURES frame. Wire layout (the one authoritative def in cpp/include/chocofarm/issue_control_bridge.hpp:38-39, codec lines 119-130): a 20-byte header [u32 magic=0x15C0F1A1 | u32 T=n_threads | u32 D=d_ceiling | f64 server_rows_per_forward] followed by T fixed 32-byte rows [i32 inflight; i32 ready; i64 msgs; i64 leaves; i64 rtt_us]. So the structure is 3 header scalars + 5 per-thread int/long vectors each of length T (one entry per PRODUCER thread, --pool-threads). The dict keys are exactly: n_threads, d_ceiling, server_rows_per_forward (scalars) and inflight, ready, msgs, leaves, rtt_us (length-T lists). NOTE the field rename across the boundary: the C++ struct names the latency channel mean_rtt_ms (double, milliseconds) but the wire re-encodes it as i64 microseconds (bridge line 129: mean_rtt_ms*1000.0) and the Python decoder exposes it raw under the key 'rtt_us' (integer microseconds) — it does NOT divide back to ms. msgs and leaves are CUMULATIVE per-thread counters (monotone non-decreasing over a run), so any rate the policy wants is a first difference across ticks that the policy must compute itself (no rate is supplied atomically). The snapshot is taken by IssueController::snapshot_features() (issue_controller.hpp:89-108) which copies the per-thread metric slots each worker last published; the slots are filled by the single live publish() site at cpp/src/runner_wire_batched.cpp:784, called once per received reply per thread inside run_episodes_wire_pipelined's post-recv drain loop (so a thread's row is as fresh as its last completed reply — staleness is a benign relaxed-atomic read, intentionally so). Geometry that bounds the ranges: per thread K = N x ceil(pool_batch/pool_threads) total slots (N = --trees-per-thread, runner_wire_batched.cpp:450-451; ceil() is RuntimeConfig::fibers_per_thread, runtime_config.hpp:29-32); D = max(1,--inflight-msgs) is the per-thread outstanding-message cap. On the production drain-all default per-thread message depth collapses to ~1 (the runner header / SYNTHESIS note: a non-forced issue drains ALL ready, so a second issue finds nothing ready), so inflight typically sits at 0 or 1 regardless of D unless chunk_floor is enabled.

## Atomic features

| feature | per-thread | unit | dtype | wired | measures | dynamics | source |
|---|---|---|---|---|---|---|---|
| `n_threads` | scalar | count (threads) | int (u32 on wire) | ✓ | number of producer worker threads in the run (the per-thread vector length T); each owns one DEALER socket to the server and its own slot set | constant for the whole run (a configuration constant, equals T = --pool-threads); typically 1-4 on the 4-vCPU host | constructor arg IssueController(n_threads,...) = wcfg.pool_threads (--pool-threads); echoed by snapshot_features (issue_controller.hpp:91) and bridge header (bridge:120) |
| `d_ceiling` | scalar | count (max outstanding messages per thread) | int (u32 on wire) | ✓ | the runner's fixed per-thread in-flight-message SAFETY ceiling D — read-only context the controller may NOT change (it only allows/denies the next discretionary issue; the runner's own `inflight_msgs < D` enforces it) | constant for the whole run (a configuration constant >= 1) | constructor arg d_ceiling = max(1,--inflight-msgs) (runner_wire_batched.cpp:452); echoed by snapshot_features (issue_controller.hpp:92) and bridge header (bridge:121) |
| `server_rows_per_forward` | scalar | rows per forward (float) | double (f64 on wire) | sentinel (0) | INTENDED: mean rows the single-threaded eval server concatenates into one batched forward (the cross-message/cross-thread coalescing degree the design's overcommit phase targets). Currently carries no value — the server->producer metrics channel does not exist. | identically 0.0 for the whole run (documented sentinel; the one acknowledged metrics-channel gap) | struct default 0.0 (issue_controller.hpp:50); never assigned in the controller path — flows straight to the wire (bridge:122). The same-named keys in overcommit_sweep.py / stage_b_*.py are a SEPARATE offline post-hoc aggregation of server logs, not this in-band channel. |
| `inflight` | yes | count (messages) | int (i32 on wire) | ✓ | per-thread count of outstanding (submitted-but-unanswered) MESSAGES at publish time — the live in-flight depth on that thread's DEALER socket (post-decrement of the reply just consumed) | ranges [0, D]; on the production drain-all default it sits at ~0-1 each tick (per-thread depth collapses to ~1 — SYNTHESIS note), so D is effectively a dead knob unless chunk_floor is on; rises toward D only on the depth>1 chunked path | publish() arg `inflight_msgs` (runner_wire_batched.cpp:784); slot s.inflight (issue_controller.hpp:81); decode key 'inflight' |
| `ready` | yes | count (slots / leaves) | int (i32 on wire) | ✓ | per-thread count of READY slots at publish time = slots parked at a leaf (search returned a leaf needing an eval) AND not already outstanding to the server; the un-submitted backlog available to coalesce into the next message | ranges [0, K] where K = trees_per_thread x ceil(pool_batch/pool_threads) is the per-thread slot count; fluctuates tick-to-tick with the search/eval pace — drains toward 0 after an issue, refills as replies resume slots; this is the only fast-moving instantaneous per-thread state besides inflight | publish() arg `ready_count()` (is_ready scan, runner_wire_batched.cpp:670-682, 784); slot s.ready (issue_controller.hpp:82); decode key 'ready' |
| `msgs` | yes | count (messages, cumulative) | long (i64 on wire) | ✓ | per-thread CUMULATIVE count of WIRE MESSAGES issued since the thread started (every successful submit_batch increments it) — a monotone progress/throughput counter | monotone non-decreasing from 0 across the run; absolute value grows unbounded with run length — a per-tick MESSAGE RATE requires a first difference across ticks (not supplied directly) | publish() arg `my_msgs` (incremented in issue(), runner_wire_batched.cpp:728, 784); slot s.msgs (issue_controller.hpp:83); decode key 'msgs' |
| `leaves` | yes | count (leaves/rows, cumulative) | long (i64 on wire) | ✓ | per-thread CUMULATIVE count of LEAVES (rows) sent into messages since the thread started (each issue adds gathered.size()); leaves/msgs is that thread's mean coalescing degree S (rows per wire message) | monotone non-decreasing from 0, growing faster than msgs (>=1 leaf per message); a per-tick LEAF RATE (the dps-aligned signal) and mean rows/msg both require differencing across ticks (the LEAF rate is the closest atomic proxy to the dps objective) | publish() arg `my_leaves` (incremented by gathered.size() in issue(), runner_wire_batched.cpp:727, 784); slot s.leaves (issue_controller.hpp:84); decode key 'leaves' |
| `rtt_us` | yes | microseconds (integer; the C++-side field mean_rtt_ms is milliseconds) | long (i64 on wire; double/ms in the C++ struct) | sentinel (0) | INTENDED: recent mean reply round-trip time per thread (submit->reply latency). Currently a sentinel: the live publish() site never passes the rtt argument so it defaults to 0.0. Surfaced to the policy under the key 'rtt_us' as integer microseconds (the C++ struct field is mean_rtt_ms in double ms; the wire and Python key are i64 microseconds). | identically 0 for the whole run (documented 0-until-wired sentinel); the storage slot and full ms<->us codec exist end-to-end, so it becomes live the moment the call site passes a measured RTT | publish()'s `mean_rtt_ms` arg defaults to 0.0 and is NOT passed at the call site (runner_wire_batched.cpp:784); slot s.rtt_us stores mean_rtt_ms*1000 (issue_controller.hpp:85); wire re-encodes ms*1000 (bridge:129); decode key 'rtt_us' (issue_engine.py:49) |

## Sentinel gaps (channels not yet wired)

- server_rows_per_forward (scalar f64): no server->producer metrics channel exists; struct default 0.0 flows straight to the wire — the server's true cross-message rows/forward coalescing degree, the signal the overcommit phase most wants, is unobservable in-band
- rtt_us / mean_rtt_ms (per-thread): the storage slot and the full ms<->microsecond codec are wired end-to-end, but the sole live publish() call (runner_wire_batched.cpp:784) omits the rtt argument so it defaults to 0.0 — the per-thread reply-latency channel reads 0 until that call site passes a measured RTT
