# Phase 2 — Derived features

| feature | online? | history | definition | captures |
|---|---|---|---|---|
| `msg_rate` | online | 1 prior tick (prev msgs[] + prev tick timestamp). | For each thread t: (msgs[t] - msgs_prev[t]) / dt_wall, where msgs_prev is the value of msgs[t] at the previous control tick and dt_wall is the wall-clock seconds between the two ticks (the policy must timestamp each tick itself with a monotonic clock, e.g. time.monotonic(); msgs is exposed atomically only as a cumulative counter). On the FIRST tick (no previous) it is undefined -> emit 0.0. | Per-thread WIRE-MESSAGE throughput (messages/sec). Differencing is mandatory because msgs (issue_controller.hpp:47, publish site runner_wire_batched.cpp:728) is monotone cumulative, not a rate. Note a thread only publishes inside its own drain loop (line 784), so over one bridge cadence interval a thread may have completed 0..many replies; the wall-time denominator makes this an honest average rate regardless. |
| `leaf_rate` | online | 1 prior tick (prev leaves[] + prev tick timestamp). | For each thread t: (leaves[t] - leaves_prev[t]) / dt_wall. dt_wall as in msg_rate. First tick -> 0.0. Cross-thread total leaf rate: sum_t leaf_rate[t]. | Per-thread (and pool-total) ROWS-EVALUATED-PER-SECOND. The cross-thread sum is the closest online proxy to the optimized objective (the bench's dps = leaves evaluated per wall-second), so leaf_rate is the signal a throughput-seeking policy should treat as its primary reward-correlated input. leaves is cumulative (issue_controller.hpp:48; += gathered.size() at runner line 727), so first-differencing is required. |
| `coalesce_degree_inst` | online | 1 prior tick (prev msgs[], prev leaves[]). | For each thread t: (leaves[t] - leaves_prev[t]) / max(1, msgs[t] - msgs_prev[t]) — the INTERVAL mean rows per wire message over the last tick window. If the message delta is 0 (thread issued nothing this interval) -> emit NaN/None sentinel (or carry-forward last valid), distinct from a true value of 1. | The instantaneous (windowed) coalescing degree S the thread is currently achieving — how many ready leaves it is packing per submit RIGHT NOW. This is the lever the issue gate ultimately moves: denying a thread's discretionary issue lets ready leaves accumulate so the NEXT issue coalesces more rows. The windowed (delta/delta) form tracks current behaviour, unlike the lifetime-average cumulative ratio. |
| `coalesce_degree_lifetime` | online | none (computed from the current cumulative snapshot alone). | For each thread t: leaves[t] / max(1, msgs[t]) — the run-to-date mean rows per message. Pool aggregate: (sum_t leaves[t]) / max(1, sum_t msgs[t]). | The thread's lifetime mean coalescing degree S (rows/message), and the pool-wide mean. A slowly-converging baseline/normalizer; useful as a denominator (e.g. to express the instantaneous degree as a fraction of historical) and as a low-variance summary, but it lags and washes out transients — pair it with coalesce_degree_inst for the live picture. |
| `inflight_saturation` | online | none. | For each thread t: inflight[t] / max(1, d_ceiling). Pool: max_t and mean_t of this ratio. | How close each thread sits to its outstanding-message ceiling D — i.e. whether the runner's own `inflight < D` guard (not the controller) is the thing currently blocking issue. On the production drain-all default this pins near 0 (depth collapses to ~1, inflight in {0,1}, so the ratio is ~0..1/D — D is effectively a dead knob); it only climbs toward 1.0 on the depth>1 chunked path (chunk_floor on). So a persistently-low value is itself the signal that D is not the binding constraint. |
| `ready_backlog_norm` | online | none for form (a); a running max across all prior ticks for the K_est lower-bound in form (b). | For each thread t: ready[t] / max(1, K_est[t]), where K_est is the per-thread slot count K = trees_per_thread * ceil(pool_batch/pool_threads) (runner line 451). K is NOT in the feature surface, so either (a) pass it to the policy as out-of-band config, or (b) approximate K_est = running max of ready[t] observed so far (an online lower-bound estimate of capacity). Fallback if neither available: ready[t] / max(1, max_t ready[t]) (cross-thread relative). | Fraction of a thread's slots currently parked-and-unsubmitted — the un-submitted leaf backlog available to coalesce into the next message. High backlog => denying issue would let it grow (more coalescing headroom); near-zero backlog => the thread is eval-starved and denial only stalls it. ready is the only fast-moving instantaneous per-thread state besides inflight, so its normalization is a core saturation indicator. |
| `ready_velocity` | online | 1 prior tick (prev ready[] + prev timestamp). | For each thread t: (ready[t] - ready_prev[t]) / dt_wall — signed rate of change of the ready backlog. | Whether a thread's un-submitted backlog is FILLING (positive: replies are resuming slots faster than issues drain them — coalescing headroom is opening) or DRAINING (negative: issues/forced-flushes are outpacing refill). A controller that wants larger batches denies issue while ready_velocity > 0 and releases as it plateaus. Because ready is instantaneous (not cumulative) this is a true derivative, sensitive to the publish jitter (a thread publishes only on its own replies), so smooth with ready_velocity_ewma if noisy. |
| `rate_ewma` | online | the prior EWMA value per series (O(1) state); plus the 1-tick history its underlying rate needs. | Exponentially-weighted moving average of any first-difference rate above (msg_rate, leaf_rate, ready_velocity, coalesce_degree_inst): r_ewma[t] <- alpha*r_now[t] + (1-alpha)*r_ewma_prev[t], alpha in (0,1] chosen for the bridge cadence (cadence_ms, default ~5ms). Time-aware variant: alpha = 1 - exp(-dt_wall/tau) for a fixed time-constant tau. | A denoised version of the inherently jittery per-tick rates. The publish cadence is irregular (each thread publishes once per completed reply at line 784, not once per bridge tick), so raw first-differences over a few-ms window are noisy; the EWMA gives the policy a stable trend signal at the cost of lag. The time-aware alpha keeps the smoothing constant in wall-time despite variable dt. |
| `cross_thread_spread` | online | none beyond whatever the underlying signal needs. | Over the per-thread vector of any chosen signal x (e.g. leaf_rate, ready, coalesce_degree_inst, inflight): report min_t x, max_t x, mean_t x, std_t x, and spread = max - min. Optionally normalized spread = (max-min)/max(eps, mean) (coefficient-of-range). | Cross-thread HETEROGENEITY / imbalance. With T<=4 on this host these aggregates are cheap and expose whether one producer is lagging the pool (a straggler with high ready + low leaf_rate) or whether load is even. An imbalance-aware policy can gate the fast threads (let them coalesce more / yield server attention) without starving slow ones; a balanced pool wants uniform gating. The forced-flush liveness carve-out means even a fully-denied straggler still drains, so spread is safe to act on. |
| `cross_thread_skew` | online | none beyond the underlying signal. | Over the per-thread vector x: skew = (mean_t x - median_t x) / max(eps, std_t x) (a robust skewness proxy), or the third standardized moment if preferred. Also report argmax_t x and argmin_t x (which thread is the outlier) and the gap of the top from the rest: max_t x - second_max_t x. | Whether imbalance is a single outlier (one hot or one stalled thread — large top-vs-second gap) versus a broad tilt. Tells the policy WHICH thread to gate rather than just that imbalance exists. Most informative on leaf_rate (an isolated low-throughput thread) and on ready (an isolated high-backlog thread that could be held to coalesce). Cheap at T<=4. |
| `idle_thread_flag` | online | none. | For each thread t: (ready[t] == 0) AND (inflight[t] == 0). Pool aggregate: count_t of idle threads, and a fully_idle_pool boolean = all threads idle. | A thread with nothing ready and nothing outstanding is momentarily IDLE (between a finalized episode's fill and the next park) — gating it is pointless (the gate only denies a discretionary issue it isn't about to make) and the forced-flush path won't even fire (it needs ready>0). The flag lets a policy avoid wasting an allow/deny decision on a thread that has no pending action, and signals pool-level under-utilization when many threads are idle at once. |
| `submit_pressure` | online | none. | For each thread t: ready[t] / max(1, d_ceiling - inflight[t]) — backlog relative to remaining issue headroom under the runner's D guard. (Guard against negative/zero denominator: when inflight>=D, headroom is 0 -> emit a large sentinel meaning 'D-blocked with backlog'.) | Combines the two fast instantaneous states (ready, inflight) into one 'is there backlog the runner could still legally issue?' signal. High value = lots of ready leaves AND headroom under D, so an issue is imminent and the gate is live/effective. Low/zero = either no backlog or D-blocked, where the controller's deny is moot. Distinguishes 'controller is the binding constraint' from 'D is the binding constraint' from 'nothing to do'. |
| `deny_effectiveness` | online | the policy's own prior emitted allow[] (1 tick) plus two O(1) running-mean accumulators per thread. | For each thread t, correlate the policy's OWN past allow/deny action with the realized change: e.g. delta_coalesce[t] = coalesce_degree_inst[t](tick k) given allow[t]=0 at tick k-1, versus given allow[t]=1. Maintained online as two running means per thread (mean inst-coalesce-after-deny, mean inst-coalesce-after-allow) and their difference. Requires the policy to log its own emitted gate vector each tick (the features do NOT report the current allow bits back). | A self-supervised estimate of whether denying THIS thread actually fattens its next batch (the intended mechanism) — closing the loop between action and outcome without any external label. Lets an adaptive policy learn per-thread gate sensitivity online. Caveat: it is observational (confounded by whatever else changed), not a clean causal effect; treat as a heuristic, not ground truth. |
| `realized_dps_window` | online | a ring buffer of the last W (sum-leaves, timestamp) pairs. | Pool-total leaf rate measured over a TRAILING WINDOW of W ticks rather than one inter-tick gap: (sum_t leaves[t] at tick k - sum_t leaves[t] at tick k-W) / (wall_time[k] - wall_time[k-W]). | A lower-variance online estimate of the achieved objective (dps) than the single-step leaf_rate, by averaging over a longer wall window. Useful as the reward signal for an online controller or as a stable normalizer (e.g. express per-thread rates as a fraction of pool dps). Trades responsiveness for stability via the window length W. |
| `server_coalesce_degree` | ? | none (the channel is absent — wiring the server->producer metrics path is the prerequisite, not a window). | INTENDED: server_rows_per_forward as published — mean rows the single-threaded server concatenates per batched forward. CURRENTLY a sentinel: the field is hardwired 0.0 (issue_controller.hpp:50, never assigned; flows straight to the wire at bridge:122), because the server->producer metrics channel does not exist. ANY derived feature over it (e.g. server_coalesce_degree / pool coalesce_degree_lifetime = cross-message amplification, or its trailing-window rate) is therefore identically 0/degenerate until that channel is wired. Listed to mark the gap, not as a usable input. | The cross-message coalescing the single-threaded server achieves when it drains multiple queued messages into ONE forward (the degree the overcommit phase most wants to observe). It is the natural target for the controller's effect on the server side, but is UNOBSERVABLE in-band right now. Flagged so a policy does not mistake the constant 0 for a real measurement. |
| `rtt_derived_block` | ? | for the trend/queueing variants once live: 1 prior tick; until then N/A (channel absent). | INTENDED family over rtt_us: per-thread latency level (rtt_us/1000 -> ms), its first-difference trend, queueing proxy = rtt_us * inflight[t] (Little's-law-style work-in-system), and cross-thread rtt spread. CURRENTLY degenerate: the sole live publish() call (runner_wire_batched.cpp:784) omits the rtt argument, so mean_rtt_ms defaults to 0.0 and rtt_us is identically 0 — every feature in this family is 0 until that call site passes a measured RTT. The ms<->us codec and storage slot are wired end-to-end (issue_controller.hpp:85, bridge:129, decoder key 'rtt_us'), so the family goes live the instant the call site supplies an RTT; NOTE the decoder exposes raw MICROSECONDS under key 'rtt_us' and does NOT divide back to ms (issue_engine.py:49). | Reply round-trip latency and its derivatives — the natural congestion/queueing signal for an issue controller (rising RTT = server saturating, a reason to throttle issues). Unobservable today (sentinel 0). Flagged so the policy treats rtt_us as not-yet-wired rather than as 'every thread has 0 latency'; also flags the ms/us unit trap at the boundary. |
| `label_terminal_dps` | offline-only (label) | the entire run (final totals + measure-window endpoints). | OFFLINE: the run's final reported objective, total_leaves / measured_window_seconds over the bench's [t0,t1] measure window (the dps the bench prints; runner accumulates total_leaves/total_msgs at lines 804-805 and captures measure_t0_ns/measure_t1_ns at 488-489). Attach this single scalar to EVERY tick of that run's feature trajectory as the regression target. | The supervised TARGET a policy is ultimately optimizing — the whole point of the fixture. Available only after the run completes (full-trajectory hindsight), so it is a LABEL, never an online input. Pairs each observed feature snapshot with the throughput the trajectory eventually achieved. |
| `label_future_leaf_gain` | offline-only (label) | forward window of H ticks (full-trajectory hindsight). | OFFLINE: for each tick k, the leaves the pool will evaluate over the NEXT H ticks (or next H_ms wall-ms): sum_t leaves[t](k+H) - sum_t leaves[t](k). A short-horizon look-ahead return computed by reading forward in the recorded trajectory. | A per-tick supervised target for the near-term throughput consequence of the state (and the action taken) at tick k — a value/return label for online RL or for fitting a one-step advantage. Requires reading the trajectory FORWARD past tick k, so it is offline-only (hindsight); the online counterpart is realized_dps_window, which only looks backward. |
| `label_counterfactual_gate_advantage` | offline-only (label) | two full paired runs (counterfactual hindsight), not a window within one run. | OFFLINE / experimental: difference in label_terminal_dps (or label_future_leaf_gain) between two runs that share seed/config but differ in the gate vector at a chosen tick — i.e. an A/B (or do-calculus) estimate of a gate decision's causal value. Requires running the deterministic fixture twice (default-all-allow reproduces the plain fixed-D runner byte-for-byte, giving a clean control arm). | A causally-clean (not merely observational) label for how much a specific gating decision helped or hurt throughput — the gold-standard supervision the online deny_effectiveness only approximates. Offline-only and EXPENSIVE: needs paired counterfactual runs, not a single trajectory. Listed as the rigorous label target the cheaper online/observational signals stand in for. |

## Notes

Grounding: I read the three control-interface files in full (cpp/stage_a/issue_engine.py, cpp/include/chocofarm/issue_controller.hpp, cpp/include/chocofarm/issue_control_bridge.hpp) and the relevant runner regions (cpp/src/runner_wire_batched.cpp lines ~430-490 for the K/D/S_min geometry and ~650-806 for is_ready/ready_count/issue/refill/publish). The step-1 atomic surface matches the source exactly.

Two facts that shape every rate-derived feature: (1) msgs and leaves are MONOTONE CUMULATIVE counters (issue_controller.hpp:47-48; incremented at runner 727-728), so NO rate is supplied atomically — every rate/ratio here is a first difference the policy must compute and store history for. (2) The per-thread row is published only inside that thread's own drain loop, once per received reply (runner_wire_batched.cpp:784), NOT once per bridge control tick (the bridge ticks on a fixed cadence_ms, issue_control_bridge.hpp:110-112). Consequence: between two control ticks a thread may have advanced its counters by zero, one, or many replies, and threads publish asynchronously w.r.t. each other. So every rate MUST be divided by an actual wall-clock interval the policy timestamps itself (time.monotonic at each tick) — dividing by 'ticks' would be wrong — and per-tick first-differences are inherently jittery, which is why I include the EWMA / trailing-window smoothers. The features dict carries no timestamp, so the policy is responsible for its own clock and its own prior-tick state.

Capacity normalizers (ready_backlog_norm, submit_pressure) want K = trees_per_thread * ceil(pool_batch/pool_threads), which is NOT in the feature surface; I give the out-of-band-config path as primary and an online running-max lower-bound estimate as fallback.

Sentinel honesty (ADR-0002 posture): server_rows_per_forward and rtt_us/mean_rtt_ms are BOTH hardwired 0 today — server_rows_per_forward because the server->producer metrics channel does not exist (issue_controller.hpp:50, flows straight to wire at bridge:122), and rtt_us because the live publish() call omits the rtt argument (the codec/slot are wired end-to-end but unfed). I include the derived families over them (server_coalesce_degree, rtt_derived_block) explicitly FLAGGED as currently degenerate/unobservable so a policy does not mistake the constant 0 for a measurement; they become live the instant their source channel is fed, with no wire change. Also flagged the unit trap: the decoder exposes raw MICROSECONDS under key 'rtt_us' and does not divide back to ms (issue_engine.py:49), while the C++ struct field is mean_rtt_ms in milliseconds.

Offline-only labels: the four offline_only features (label_terminal_dps, label_future_leaf_gain, label_counterfactual_gate_advantage, and the forward-looking leaf gain) require full-trajectory or paired-run hindsight and are SUPERVISED TARGETS, never online inputs. realized_dps_window is their backward-looking online counterpart. The all-allow default reproducing the plain fixed-D runner byte-for-byte (issue_controller.hpp:57, runner 746) gives a clean control arm for the counterfactual label.

deny_effectiveness needs the policy to log its OWN emitted allow[] vector — the feature surface does NOT echo the current gate bits back, so that is policy-side state, not an observation.

Scope: this is only the step-2 derived-feature catalog over the step-1 atomic surface; I did not diagnose the transport, identify a bottleneck, or characterize a failure mode.
