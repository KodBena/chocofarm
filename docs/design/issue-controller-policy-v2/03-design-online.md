# Phase 3 — Online learning design space

> Online learning — controllers that adapt DURING a single benchmark run, learning the per-thread issue-gate from the feature stream itself (no offline-trained weights; the only "training set" is the run in progress). Reward is NOT given: every scheme self-computes it by first-differencing the cumulative `leaves`/`msgs` counters against the policy's own `time.monotonic()` clock (the dps-aligned signal is pool-total `leaf_rate` = Δ(Σ leaves)/Δt = step-2 `realized_dps_window`). The defining constraints, grounded in the runner (cpp/src/runner_wire_batched.cpp run_episodes_wire_pipelined) and the bridge (cpp/include/chocofarm/issue_control_bridge.hpp): (1) the action is per-thread binary allow∈{0,1}, read once per refill() at the single gate `inflight<D && may_issue(tid)` (line 746-747), DENY-only; (2) the forced flush (line 748-749) is UNGATED — a permanent liveness floor that makes aggressive online exploration safe (no policy choice can deadlock a thread); (3) the bridge ticks on a slow cadence (~5ms, cadence_ms) so a run is only ~10^3–10^4 decision opportunities and the learner must converge within that budget or lose to a statically-tuned policy; (4) reward is a POOL-TOTAL scalar but the action is a PER-THREAD vector, so per-thread credit assignment is confounded (the step-2 deny_effectiveness caveat) — this is the family's central difficulty and it forks the design space three ways: shared/global-action, per-thread-independent, and homogeneous parameter-sharing. Default all-allow reproduces the fixed-D runner byte-for-byte (a clean within-run and A/B control arm). Honesty note (ADR-0009 / ADR-0011 Rule 3 measure-first / ADR-0012 P6): no candidate here carries a perf claim — each is mapped by its feedback signal, state, adaptation mechanism, and what it would COST to tune; any realized-speedup claim must be substantiated on the existing bench surface (stage_b_ab.py, overcommit_sweep.py) at the two-tier bar, against the byte-identical all-allow baseline, over ≥2 seeds.

## 1. C1 — Homogeneous threshold bandit (single shared rule, parameter-sharing across threads)

Learn ONE scalar gate threshold (or a tiny discrete set of thresholds) applied identically to every thread; treat the pool-total leaf_rate as the single reward of a small multi-armed bandit over candidate threshold values.

**Mechanism (features → per-thread {0,1}).** Pick one normalized per-thread saturation signal x[t] — recommended submit_pressure[t]=ready[t]/max(1,D-inflight[t]) or ready_backlog_norm[t] (both step-2, O(1), no history). Maintain a SMALL ARM SET A of candidate thresholds, e.g. {0 (=all-allow), 0.25, 0.5, 1.0, 2.0, +inf (=deny-until-forced)}. Each tick the bandit has a current chosen arm θ; the gate is allow[t] = 1 iff x[t] >= θ, ELSE 0 (deny → leaves accumulate → next non-forced issue coalesces more; the ungated forced-flush still drains a fully-denied thread). The SAME θ governs all T threads — this is the parameter-sharing that collapses the per-thread credit problem into a single-reward bandit and exploits the T<=4 thread symmetry.

**Hyperparameters.** The arm set A (values + cardinality, ~4–8 arms); bandit algorithm + its constant (UCB1 exploration c, or epsilon for epsilon-greedy, or a softmax/EXP3 temperature for the non-stationary variant); the reward-averaging WINDOW W (how many ticks an arm is held before its leaf_rate sample is scored — must exceed the pipeline's response lag, a few RTTs, or the sample attributes another arm's residual); a discount/sliding-window on arm stats (EXP3 or discounted-UCB) because the transport is non-stationary across the settle→steady ramp; which feature x is the gate variable.

**Data / training / tuning.** ZERO offline data and zero pre-trained weights — it learns entirely within the one run. State is O(|A|): a (count, mean-reward) pair per arm. The reward sample is the pool-total leaf_rate measured over the W-tick hold of each arm (realized_dps_window). Tuning need is light and one-dimensional: the arm set and exploration constant can be fixed by hand from the geometry (arms span all-allow → deny-until-forced) and, if desired, sanity-swept once on stage_b_ab.py against the all-allow baseline. No gradient, no model fit.

**Pros.** Cheapest learner that still adapts; sample-efficient enough to converge inside a short run (a handful of arms × a few W-windows each). Collapses the confounded per-thread credit problem into one clean pool-reward bandit by parameter-sharing. The forced-flush floor makes even the most aggressive arm (deny-until-forced) safe to try. all-allow is literally one of the arms, so the bandit can never do worse than baseline by more than its exploration regret, and it self-discovers whether gating helps at all. Pure-Python, swappable with no recompile. Honest, fully observable state — trivial to log and reason about.

**Cons.** One global θ cannot exploit cross-thread heterogeneity (a straggler vs a hot thread get the same threshold) — if imbalance is the lever, C1 leaves it on the table. Discrete arms only coarsely sample the threshold axis. Reward attribution still needs a hold window W long enough to clear pipeline lag, which slows adaptation; too-short W and the bandit chases noise. Non-stationarity demands the discounted/EXP3 variant or early-run samples mislead. It assumes a monotone 'higher threshold ⇒ more coalescing ⇒ (maybe) more dps' relationship that may be non-monotone (over-denial starves the server).

**Ergonomics.** Highest in the family. ~30 lines of Python in issue_engine.py: a dict of arm stats, an argmax-with-exploration pick each tick, a clock + prev-Σleaves for the reward. Trivial to unit-test (feed synthetic feature dicts, assert arm selection). Fully introspectable — you can print the per-arm leaf_rate table live. The natural FIRST thing to build because its all-allow arm makes it a strict superset of the baseline plus a tuned-threshold search.

## 2. C2 — Contextual bandit, homogeneous (LinUCB / linear-Thompson over a small derived-feature context, shared weights)

Per thread, choose allow/deny as the two arms of a contextual bandit whose context is a short vector of the step-2 derived features; weights are SHARED across threads (one model, pooled samples) and the per-tick pool-total leaf_rate is the shared reward.

**Mechanism (features → per-thread {0,1}).** Context φ[t] (length ~3–6, all step-2, all online): [submit_pressure[t], ready_backlog_norm[t], coalesce_degree_inst[t] (or its EWMA), inflight_saturation[t], ready_velocity_ewma[t], 1]. Two arms per thread: a0=deny(0), a1=allow(1). A LinUCB/linear-Thompson model maintains weight vectors (one per arm, or one over [φ ⊕ action]); each tick it scores allow vs deny for each thread from φ[t] and emits the per-thread argmax (with the UCB/posterior-variance exploration bonus). SHARED weights across threads = every thread's (context, action, reward) triple is one training sample, so T threads give T samples/tick — the parameter-sharing that makes a contextual model trainable inside one run. Reward credited to a thread's chosen arm is the pool-total leaf_rate over the next W ticks (the confound is acknowledged and partly mitigated by also crediting the per-thread leaf_rate[t] as an auxiliary signal).

**Hyperparameters.** Which context features and their normalization (capacity normalizers want K out-of-band, else the running-max K_est fallback from step-2 ready_backlog_norm); the regularization λ (ridge prior) and the exploration scale α (LinUCB) or posterior noise (Thompson); the reward window W and a forgetting factor for non-stationarity (sliding-window or discounted sufficient statistics); the reward definition (pool-total vs per-thread vs a blend).

**Data / training / tuning.** No offline data required (cold-start from the ridge prior), but it WANTS more samples than C1 to estimate the linear weights — feasible in-run only because thread-sharing multiplies the sample rate by T and the feature dimension is kept tiny. State is O(d^2) per arm (the A matrix + b vector for the closed-form ridge update). Optionally warm-startable from a prior run's fitted weights (then it straddles into the offline-trained family — note that overlap honestly). Tuning is moderate: α/λ and the context set benefit from a small offline sweep on the A/B harness.

**Pros.** Can exploit per-thread STATE — denies a thread that is in a high-backlog/filling regime while allowing an eval-starved one, which C1's single threshold cannot. Closed-form online updates (no SGD) keep it cheap and stable. Thompson/LinUCB give principled exploration that anneals as confidence grows, fitting the short-run budget better than fixed-epsilon. Degrades gracefully to near-all-allow if the weights stay near prior (low regret). Still pure-Python, swappable.

**Cons.** Linear model may underfit a non-linear gate boundary; needs feature engineering to linearize (the step-2 catalog gives candidates but choosing them is work). The pool-reward confound is sharper here than C1 because the model tries to attribute a shared scalar to individual per-thread contexts — auxiliary per-thread leaf_rate helps but is itself noisy. More state and more hyperparameters than C1, so more ways to misconfigure within a run that is too short to recover. Normalization needs K (out-of-band or estimated), an extra coupling.

**Ergonomics.** Moderate. ~80–120 lines: ridge sufficient-statistics update + per-arm scoring. Still introspectable (you can dump weights and per-feature contributions) but harder to unit-test deterministically than C1. A good SECOND step once C1 establishes that gating helps and you want per-thread differentiation — but only if a single run yields enough samples to beat C1's regret.

## 3. C3 — Per-thread independent bandits (T tiny decoupled allow/deny learners sharing the global reward)

Run one independent 2-arm (allow/deny) bandit PER THREAD; each is updated from the pool-total leaf_rate (or, better, that thread's own leaf_rate[t]) as its reward — the explicitly-confounded but maximally-decoupled path.

**Mechanism (features → per-thread {0,1}).** T separate bandit instances. Thread t's bandit holds (count,mean) for arm0=deny and arm1=allow; each tick it picks thread t's arm by epsilon-greedy/UCB and that IS allow[t]. Optionally contextualize each per-thread bandit on a SCALAR of that thread (e.g. discretize submit_pressure[t] into 2–3 bins → a tiny tabular contextual bandit per thread). Reward per thread: primary = per-thread leaf_rate[t] (the only per-thread reward available, less confounded than pool-total but noisier and not exactly the objective); optional shaping = + a small weight on pool-total leaf_rate to keep threads cooperating rather than each greedily maximizing its own rate at the server's expense.

**Hyperparameters.** Per-bandit exploration constant; the per-thread reward choice (own leaf_rate vs pool-total vs blend weight β); optional context binning of the per-thread scalar; reward window W; non-stationarity discount. (All T bandits typically share the same hyperparameter values — they differ only in their learned stats.)

**Data / training / tuning.** No offline data. State O(T × arms) — tiny. BUT each per-thread bandit sees only ~(1/T) of the sample-attribution clarity of C1 (its reward is either confounded pool-total or noisy own-rate), so each converges SLOWER than the single shared bandit — in a short run this is the key risk. No model fit; pure counting.

**Pros.** Maximally simple per learner and fully decoupled — captures heterogeneity (each thread learns its own allow/deny tendency) without a shared model. Using per-thread leaf_rate[t] as reward sidesteps the pool-total credit confound at the per-thread granularity. Trivial to implement and reason about per thread. Safe (forced-flush floor).

**Cons.** THE WORST sample efficiency in the family: T independent learners, each on a noisy/confounded reward, dividing the run's evidence T ways — likely too slow to converge inside one short run, the exact failure the family's budget punishes. Per-thread own-rate reward is myopic: a thread maximizing its OWN leaf_rate may flood the single server and HURT pool dps (a tragedy-of-the-commons the pool-reward blend β must counter, adding a tuned knob). No cross-thread information sharing wastes the T<=4 symmetry that C1/C2 exploit. Joint-action effects (one thread's denial changing another's optimal) are invisible to independent learners.

**Ergonomics.** Simple to write (a list of C1-style bandits) but HARDER to make work than C1 because of the sample-split and the commons hazard. Introspectable per thread. Mainly valuable as a contrast/ablation against C1's parameter-sharing — if C3 underperforms C1 (likely), that itself substantiates the homogeneity choice. Not the first thing to try.

## 4. C4 — Adaptive single-threshold control via online stochastic optimization (SPSA / 1-D gradient-free hill-climb on a shared gate threshold)

Hold the gate rule fixed in FORM (allow iff x[t] >= θ, θ shared) but treat θ as a CONTINUOUS parameter optimized online by a gradient-free stochastic scheme (SPSA, simultaneous perturbation, or a simple sign-of-finite-difference hill-climb) using pool-total leaf_rate as the objective.

**Mechanism (features → per-thread {0,1}).** Same gate mapping as C1 (one shared threshold θ on submit_pressure or ready_backlog_norm) but θ ∈ ℝ continuous. Adaptation: each measurement period, perturb θ by ±Δ (SPSA uses a random sign), measure pool-total leaf_rate over a W-tick window at each perturbed value, estimate the finite-difference gradient g ≈ [J(θ+Δ)−J(θ−Δ)]/(2Δ), and step θ ← θ + a·g with decaying gains a_k, Δ_k (the standard SPSA gain sequences). This is the ‘PID-gains-tuned-online’ overlap the task flags: it is the static-family threshold controller with its single tunable adapted live by a stochastic optimizer rather than fixed by an offline sweep.

**Hyperparameters.** SPSA gain sequences a_k = a/(k+1+A)^α and c_k = c/(k+1)^γ (the classic a, c, A, α, γ); the per-evaluation window W (must clear pipeline lag for a low-variance J estimate); the gate feature x; bounds/clamp on θ (e.g. [0, x_max] so it spans all-allow → deny-until-forced).

**Data / training / tuning.** No offline data; no model. State is O(1): current θ plus the SPSA iterate counters. It needs 2 measurement windows per gradient step, so its convergence speed is set by W — the same lag constraint as the bandits but it pursues a CONTINUOUS optimum rather than picking among discrete arms. Light tuning: SPSA constants have standard defaults; W from the observed RTT/pipeline response.

**Pros.** Optimizes the threshold on a CONTINUOUS axis (finer than C1's discrete arms) while keeping O(1) state and one tuned dimension. Gradient-free — needs no model and no differentiable reward, ideal for a noisy black-box throughput signal. Naturally tracks a slowly-drifting optimum (decaying-but-nonzero gains or a fixed small gain for continual tracking). The mechanism is well-understood and provably convergent under mild noise assumptions, which fits the project's proof-honesty posture.

**Cons.** SPSA on a NOISY objective with a short horizon can be slow and can wander if W is too small (gradient estimate variance) or the step gains are mis-scaled — the same short-run risk as the bandits, with an extra sensitivity to gain tuning. Single shared θ ⇒ no heterogeneity exploitation (same limitation as C1). Two-sided perturbation spends half the run off the current best θ (exploration cost). Assumes a locally-smooth, roughly-unimodal J(θ); a flat or multi-modal landscape stalls it. Needs careful clamping so a perturbation cannot leave the meaningful [all-allow, deny-until-forced] range.

**Ergonomics.** Moderate–simple. ~40–60 lines: a small SPSA loop with a 2-phase (perturb-up/perturb-down) measurement state machine over ticks. Slightly trickier to test than C1 because of the stochastic perturbation schedule. A natural ALTERNATIVE to C1 when you want a continuous threshold rather than discrete arms; shares C1's all-allow-reachable safety if θ’s range includes the no-gate point.

## 5. C5 — Dual/Lagrangian online controller (drive an observable proxy to a setpoint via an adapted price; the gate is a thresholded price)

Frame the gate as enforcing a soft constraint — keep each thread's achieved coalescing degree (coalesce_degree_inst) at or above a target S* — and adapt a single Lagrange-multiplier 'price' online by dual ascent; deny issues when the price says the thread is under-coalescing.

**Mechanism (features → per-thread {0,1}).** Choose a controllable proxy with a clear monotone link to the gate: per-thread coalesce_degree_inst[t] (rows/msg achieved) or its EWMA. Maintain a scalar price p (one shared, or per-thread p[t]). Dual update each tick: p ← max(0, p + η·(S* − coalesce_degree_inst[t])) — price rises when coalescing falls short. Gate: allow[t] = 1 iff p below a fixed cutoff OR coalesce_degree_inst[t] already >= S* (i.e. deny precisely the threads currently under-coalescing, holding their ready leaves to fatten the next batch); release as soon as the proxy recovers. The OBJECTIVE’s role: S* itself is adapted on a slow outer loop by pool-total leaf_rate (raise S* while dps improves, back off when it stops) — a two-timescale scheme: fast dual ascent on the price, slow search on the setpoint.

**Hyperparameters.** The dual step size η; the coalescing setpoint S* (and its slow-loop adaptation rule/step, or a small candidate set tied to the runner's S_min geometry); the price cutoff; the proxy smoothing (EWMA α); shared-vs-per-thread price.

**Data / training / tuning.** No offline data. State O(1) (one price) or O(T) (per-thread prices) plus the slow-loop S*. The inner dual loop is a pure recurrence; the outer S* search is a tiny bandit/hill-climb (reuses C1/C4 machinery on a single scalar). Tuning: η and S* benefit from one offline calibration pass, but the scheme is designed to self-correct the price online. Note the overlap with the static family: with S* fixed and η fixed this is a static price controller; adapting S* online is what places it here.

**Pros.** Directly couples the action to the MECHANISM the gate physically moves (coalescing degree), so its control variable is interpretable and the link from action→proxy→objective is the clearest in the family — closest to a 'honest, mechanistic' design (matches the codebase's authoring posture). Dual ascent is stable and O(1). Per-thread prices give heterogeneity if wanted. The setpoint S* connects directly to the runner's existing S_min knob, so the controller and the static floor share vocabulary. Naturally constraint-shaped (keep coalescing up without over-denying).

**Cons.** Two-timescale tuning (fast price + slow setpoint) is the most conceptually involved scheme here and the easiest to mis-separate (timescales too close → oscillation). It optimizes a PROXY (coalescing) not the objective directly; if higher coalescing does not monotonically raise dps (e.g. server-side already saturated, or over-denial starves throughput) the proxy target is wrong and only the slow S* loop catches it — slowly. Requires the link 'deny → coalesce_degree_inst rises' to actually hold, which is the very mechanism the fixture is meant to test (a chicken-and-egg if used as the FIRST probe). More moving parts than C1/C4.

**Ergonomics.** Moderate. ~60–90 lines: a price recurrence + a thresholded gate + a slow S* updater. Interpretable state (you can watch the price and the achieved coalescing track the setpoint). Harder to tune than C1/C4 because of the two timescales. Best deployed AFTER the simpler schemes have shown that gating moves coalescing as expected — then C5 is the principled, mechanism-aligned refinement.

## 6. C6 — Tabular online RL on a coarse-discretized per-thread state (Q-learning / SARSA with parameter-sharing) — boundary/contrast candidate

Cast the per-thread gate as a small Markov decision: discretize each thread's state into a few bins, action allow/deny, reward = leaf_rate; learn a SHARED tabular Q(s,a) online by Q-learning/SARSA, acting epsilon-greedily. The most RL-flavored scheme that is still plausibly trainable in-run.

**Mechanism (features → per-thread {0,1}).** Per-thread state s[t] = coarse discretization of a 1–2 feature tuple, e.g. (bin of submit_pressure[t] ∈ {low,med,high}) × (sign of ready_velocity_ewma[t] ∈ {filling,draining}) → ~6 states. Actions {deny,allow}. ONE shared Q-table (parameter-sharing) updated from every thread's transition each tick with reward = per-thread leaf_rate[t] (or pool-total). Gate: allow[t] = argmax_a Q(s[t],a) with epsilon exploration. Bootstrapped TD update Q(s,a) ← Q(s,a) + α[r + γ·max_a' Q(s',a') − Q(s,a)] credits future return, addressing the DELAYED effect of a denial (its payoff is the fatter NEXT batch, an explicitly temporal credit the bandits approximate with a window).

**Hyperparameters.** State discretization (which features, how many bins — the dominant design choice); learning rate α; discount γ; exploration epsilon (and its decay); reward choice and window; the eligibility/n-step horizon if SARSA(λ). Shared table size = |S|×2 kept small on purpose.

**Data / training / tuning.** No offline data required, but RL is the LEAST sample-efficient here: even a ~6-state shared table needs many visited transitions per (s,a) to stabilize Q, and the run’s tick budget may not supply them — so in practice C6 either needs warm-starting from prior runs (→ crosses into offline-trained, note the overlap), a very coarse state, or it underperforms the bandits. State O(|S|×2). No function approximation (kept tabular precisely to be trainable in-run; deep-RL/DQN is OUT of scope for a single short run — it would be an offline-trained policy, a different family).

**Pros.** Only scheme that models the TEMPORAL structure (a denial’s payoff is delayed to the next batch) via bootstrapping, rather than approximating it with a reward window. Parameter-sharing + tiny state keeps it the trainable end of RL. Captures simple state-dependent policies (deny when high-pressure-and-filling, allow when draining) that a fixed threshold cannot. Fully introspectable Q-table.

**Cons.** Sample efficiency is the worst of all candidates for the in-run budget — high risk it spends the whole short run still exploring (the family's cardinal failure). Discretization is brittle and is itself a tuning burden equivalent to designing the controller by hand. Reward confound + delayed credit + bootstrap bias compound. Non-stationarity breaks the stationary-MDP assumption Q-learning rests on. Strong temptation to grow it into deep RL, which leaves this family entirely (becomes an offline-trained policy needing many runs).

**Ergonomics.** Moderate to write (~70 lines tabular TD) but the HARDEST to get to actually beat a simpler scheme inside one run; discretization tuning dominates. Best treated as a CONTRAST/ablation that demonstrates whether temporal credit buys anything over the window-based bandits — not a first or second choice given the run-length budget.

## Recommended first

C1 — the homogeneous threshold bandit. It is the unique design that satisfies all of the family's binding constraints simultaneously: (a) it converges inside a single short run because it has only a handful of arms and one pool-total reward (the bandits/RL that split evidence per-thread — C3, C6 — likely cannot); (b) it dissolves the pool-reward-vs-per-thread-action credit confound by parameter-sharing (one threshold, one reward) rather than fighting it; (c) all-allow is literally one of its arms, so it is a strict superset of the byte-identical baseline and bounds its own downside to exploration regret — it self-discovers whether gating helps AT ALL, which is exactly what a first online probe should establish before any heavier machinery is justified; (d) it is ~30 lines of swappable Python with fully observable state, the best ergonomics and testability in the family. Concretely: gate on submit_pressure[t] (or ready_backlog_norm[t]) with arms spanning {all-allow, …, deny-until-forced}, a discounted-UCB or EXP3 rule for non-stationarity, and a reward window W set a few RTTs long so each arm's leaf_rate sample clears pipeline lag. Sequence the rest by what C1 reveals: if C1 shows gating raises dps but a single threshold leaves cross-thread heterogeneity on the table, go to C2 (homogeneous contextual, per-thread differentiation, same parameter-sharing); if you want a continuous rather than discrete threshold, C4 (SPSA on the shared θ) is the drop-in continuous twin; C5 (dual/Lagrangian on coalescing) is the principled mechanism-aligned refinement once it is confirmed that denial actually fattens batches as intended; and C3/C6 are best kept as contrast/ablation arms (they substantiate the parameter-sharing and the no-temporal-credit-needed choices respectively) rather than primary controllers, because each divides the short run's evidence in a way the tick budget punishes. Every realized-throughput comparison among these must be run on the existing A/B surface (stage_b_ab.py / overcommit_sweep.py) against the all-allow control arm at ADR-0009's behavioral two-tier bar over ≥2 seeds — none of the above asserts a speedup, only a mechanism.

Three facts shape EVERY candidate and are worth isolating: (1) THERE IS NO REWARD IN THE OBSERVATION — the dps-aligned signal is pool-total leaf_rate, which the policy must self-compute by first-differencing the cumulative leaves counter against its own time.monotonic() clock (step-2 realized_dps_window); this makes reward delayed (≥1 tick), noisy (publish is once-per-reply at line 784, async w.r.t. the bridge tick), and pool-scalar. (2) THE CREDIT-ASSIGNMENT FORK is the family's defining axis: reward is pool-total but the action is per-thread, so the design space splits into shared/global-action (C1/C4/C5 with one shared parameter), homogeneous parameter-sharing with per-thread CONTEXT (C2/C6 — one model, T samples/tick), and fully independent per-thread learners (C3 — the confounded, evidence-splitting path). Parameter-sharing is the lever that makes online learning tractable in a short run, which is why my recommendation centers on it. (3) THE RUN IS SHORT (decision-budget capped, settle ramp excluded) and the bridge is slow (~5ms), so the family's cardinal failure mode is a learner that is still exploring when the measurement window closes — this is the single criterion that most separates the candidates and is why I rank the low-dimensional adapters (C1, C4) above the per-thread/RL schemes (C3, C6).

Honest overlaps flagged per the task: C4 (SPSA on a threshold) and C5 (fixed-setpoint dual control) are the STATIC family's threshold/price controllers with their tunable adapted online — the task explicitly allows this overlap. C2 warm-started from a prior run's weights, and C6 warm-started from a prior Q-table, would CROSS INTO the offline-trained family (many runs → a fixed policy); I kept both framed as cold-start in-run learners and named the crossover rather than smuggling it. Two further honesty points: the per-thread reward leaf_rate[t] (used by C3/optionally C6) is myopic and risks a tragedy-of-the-commons against the single shared server (a thread maximizing its own rate can flood the server and lower pool dps) — every scheme that uses it needs a pool-reward shaping term, a named tuned knob; and any rtt_us-based congestion signal is UNAVAILABLE today (the step-1/step-2 sentinel: the live publish() call omits the rtt argument, so rtt_us is identically 0 and server_rows_per_forward is identically 0), so NO candidate here may depend on latency or server-coalescing feedback until those channels are wired — the schemes are deliberately built only on the live signals (inflight, ready, msgs, leaves and their derived rates). Scope: this is only the step-3 design-space map for the online-learning family; I did not diagnose the transport, identify a bottleneck, characterize a failure mode, implement any controller, or assert any speedup.
