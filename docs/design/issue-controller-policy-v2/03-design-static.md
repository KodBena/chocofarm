# Phase 3 — Static / engineering-informed design space

> STATIC, LITERATURE-/ENGINEERING-INFORMED POLICIES — hand-designed feedback / setpoint / threshold-hysteresis / rate-limiter / AIMD / discrete-event controllers whose gains and thresholds are set by analysis or offline tuning, never learned from a labelled dataset. Each maps the (atomic + derived) feature dict to the per-thread DENY-only binary issue gate allow[tid] in {0,1}, consulted by the runner at the single non-forced-issue point (runner_wire_batched.cpp:746-747, gate = inflight<D && allow[tid]); the ungated forced flush (lines 748-749) is the liveness floor every candidate here relies on for never-deadlock.

## 1. C0 — All-allow identity (the null controller / control arm)

Return [1]*T every tick; the gate reduces to the runner's own inflight<D, byte-identical to the plain fixed-D runner.

**Mechanism (features → per-thread {0,1}).** allow[t] = 1 for all t, unconditionally — ignores the feature dict entirely. This is exactly issue_engine.identity_policy (issue_engine.py:59-60) and reproduces IssueController's default actuation (issue_controller.hpp:63, all slots store 1).

**Hyperparameters.** None.

**Data / training / tuning.** None. No tuning, no history, no clock.

**Pros.** Zero risk, zero cost, deterministic. It is the byte-for-byte CONTROL ARM the whole fixture is built around (the all-allow default reproduces the fixed-D runner exactly), so it is the mandatory A/B baseline every other candidate is measured against and the clean arm for label_counterfactual_gate_advantage. Cannot deadlock, cannot regress.

**Cons.** By construction it never improves throughput — it is the do-nothing reference, not a contender to win. Listed because a static-controller study that omits the identity baseline cannot honestly claim any other controller helped.

**Ergonomics.** Trivial: already implemented and shipped as the default. Use it as the fixed comparator in every sweep.

## 2. C1 — Per-thread coalescing-floor hysteresis (bang-bang deadband) gate

Deny a thread's next discretionary issue while its un-submitted backlog sits below a target band, so withheld leaves accumulate and the next allowed issue packs more rows; release (allow) once backlog is high enough or the thread risks starving.

**Mechanism (features → per-thread {0,1}).** Per thread t, maintain a 1-bit hysteresis state g[t] (start 1=allow). Compute backlog b[t]=ready[t] and capacity-normalized r[t]=ready_backlog_norm[t] (ready[t]/K_est[t]; K passed out-of-band as config — it is N*ceil(pool_batch/pool_threads), runner_wire_batched.cpp:451 — or, if unavailable in-band, K_est[t]=running max of ready[t] seen so far, the step-2 fallback). Two thresholds with a deadband, r_lo < r_hi (e.g. 0.25 and 0.60): if r[t] >= r_hi set g[t]=1 (enough backlog banked => let it fire a fat batch); if r[t] <= r_lo set g[t]=0 (too little banked => hold and let it fill); in the deadband keep g[t] unchanged (hysteresis kills chatter). LIVENESS/anti-starvation overrides, applied AFTER the hysteresis, all using only live features: (i) if idle_thread_flag[t] (ready==0 && inflight==0) force g[t]=1 (gate is moot anyway — deny would do nothing but waste a decision, and we never want to be the reason a just-refilled thread waits); (ii) if inflight[t]==0 force g[t]=1 — at depth-1 a deny here is overridden by the ungated forced flush (runner_wire_batched.cpp:748) so denying is pointless AND emitting allow keeps the intent honest; (iii) optional hold-timeout: if g[t] has been 0 for more than H consecutive ticks, force g[t]=1 for one tick (bounded worst-case hold even if backlog never reaches r_hi because replies are slow). emit allow[t]=g[t].

**Hyperparameters.** r_lo, r_hi (the deadband edges as fractions of K), or equivalently absolute backlog floors B_lo,B_hi in rows; H (max-consecutive-deny ticks, the safety release); the K source (out-of-band config vs running-max estimate). Optionally smooth r[t] with rate_ewma(alpha) if the raw ready scan is jittery.

**Data / training / tuning.** No dataset. Tuning is a small offline grid/1-D sweep of (r_lo,r_hi) against the bench's reported dps on a few seeds — exactly the warm-pool HPO bench already in the repo (stage_b_poolbatch_sweep.py is the analogous sweep harness). State per thread: 1 hysteresis bit + (for K_est fallback) a running max + (for option iii) a deny-streak counter; all O(1), all policy-side.

**Pros.** Directly and minimally actuates the ONE real lever (hold backlog to fatten the next coalesced batch) using ONLY live non-sentinel features (ready, inflight). Bang-bang + deadband is the textbook-correct controller for a binary actuator: no gain to detune, provably no limit-cycle chatter inside the band, robust to the publish-jitter the step-2 catalog warns about. It is the per-thread, slow-loop, software mirror of the existing static in-tree S_min/chunk_floor producer floor (runner_wire_batched.cpp:458,717) — so it tests 'dynamic per-thread floor-holding' against 'static compile-time floor' cleanly. Reduces to identity (C0) at B*=0/r_hi=0, so the A/B is honest. Cheap, transparent, easy to reason about and debug from the feature log.

**Cons.** At the drain-all default, per-thread depth is ~1 and inflight sits in {0,1}, so the override (ii) makes the gate a no-op for most of each thread's cycle — the deny only bites in the brief inflight>0 window, which limits how much extra coalescing a per-thread gate can manufacture without the chunk_floor path on. Single global (r_lo,r_hi) ignores cross-thread heterogeneity (a fast and a slow thread share thresholds). Needs K (out-of-band) or a warm-up before the running-max K_est is trustworthy. Acting to fatten batches trades a little latency for batch size — only a net win if the server-side coalescing actually rewards bigger messages (an empirical question the bench answers).

**Ergonomics.** Excellent. ~30 lines of pure Python in the policy fn; one obvious knob; deterministic; logs cleanly (g[t] and r[t] per tick). Fastest path from 'identity' to 'a real static controller' and the natural first rung.

## 3. C2 — Pool-level PI controller on mean coalescing degree (setpoint on S)

A single slow PI loop drives the pool's measured rows-per-message S toward a setpoint S*, converting the scalar PI output into per-thread allow bits by a threshold.

**Mechanism (features → per-thread {0,1}).** Each tick compute the pool's instantaneous coalescing degree S_now = coalesce_degree_inst aggregated over threads (sum of leaf deltas / sum of msg deltas across the tick, the step-2 windowed form), or the lower-variance realized_dps_window-style trailing mean. Error e = S* - S_now. PI: u = Kp*e + Ki*integral(e dt) with integral anti-windup clamped to [u_min,u_max] and to the actuator's reachable range. Map the scalar u to gates by a 'deny pressure': higher u (we want MORE coalescing) => deny more threads. Concretely rank threads by submit_pressure or ready (descending) and DENY the top floor(p*T) where p = clamp(sigmoid(u) or a linear map, 0,1); or simpler, deny thread t iff its r[t]=ready_backlog_norm[t] < theta(u) where the single threshold theta is slid up/down by the PI output (so the PI loop tunes C1's threshold automatically). Liveness overrides from C1 (idle/inflight==0/hold-timeout) applied last; never deny so many that a thread starves.

**Hyperparameters.** S* (target rows/message), Kp, Ki, integral clamp, the u->p (or u->theta) map, the same K source as C1, plus the liveness H. EWMA alpha on S_now.

**Data / training / tuning.** No dataset, but PI gains want tuning: a small offline sweep / step-response on the bench (push S* and watch dps and settling), classic Ziegler-Nichols-style or just a coarse grid. State: integrator (1 scalar) + prev-tick counters/timestamp + EWMA. Choosing S* sensibly may want one offline characterization pass relating message size to server throughput (which today is only observable post-hoc via server logs, NOT in-band, since server_rows_per_forward is sentinel-0).

**Pros.** A principled setpoint controller: the integral term removes steady-state offset so the pool actually converges to S* rather than hovering near a threshold (C1's bang-bang has no integral action). One global loop is simple to reason about and only T deny-decisions to emit. Naturally auto-tunes C1's threshold instead of hand-setting it. Smooth, well-understood theory (anti-windup, settling time).

**Cons.** Two structural mismatches: (1) the actuator is per-thread BINARY and quantized, so a continuous PI output must be discretized (the rank-and-deny-top-p map) — the loop gain through a {0,1}*T actuator is coarse and the effective resolution is 1/T (T<=4 here), inviting hunting unless heavily damped. (2) Choosing S* well presumes you know the message size the server rewards; that signal (server_rows_per_forward) is sentinel-0 in-band today, so S* is set blind or from offline logs. Integral windup is a real hazard when the forced-flush keeps firing (the actuator saturates: you deny but flushes still issue, so e never closes and the integrator runs away) — anti-windup must be tuned against exactly that saturation. More moving parts than C1 for an uncertain gain at T<=4.

**Ergonomics.** Moderate. ~60-80 lines; needs the differenced-rate plumbing (prev counters + monotonic clock) the step-2 catalog specifies, plus an integrator and anti-windup. Tunable but the gain-through-a-binary-actuator makes the tuning fiddly. Worth it only if C1's single static threshold proves too blunt.

## 4. C3 — Per-thread token-bucket / leaky-bucket issue rate limiter

Each thread gets a token bucket refilled at a chosen issue rate; allow=1 only when a token is available, so the gate caps each thread's discretionary issue rate (forcing leaves to accumulate between permitted issues => bigger batches).

**Mechanism (features → per-thread {0,1}).** Per thread t hold a bucket level tok[t] (float, cap C_burst) refilled at rate rho[t] tokens/sec: each tick add rho[t]*dt_wall (dt from the policy's own monotonic clock, since features carry no timestamp), clamp to C_burst. Decide: allow[t]=1 iff tok[t]>=1; on observing that the thread actually issued (detect via msgs[t] increment vs prev tick — msg_rate>0) consume one token (tok[t]-=1). If tok[t]<1, allow[t]=0 (rate-limited; backlog builds). Refill rate can be static (rho is a constant knob, a pure leaky-bucket rate cap) OR mildly adaptive: scale rho[t] by a target — e.g. set rho[t] so the thread's implied rows/sec (rho*target_batch) tracks its leaf_rate, throttling fast threads harder. Liveness overrides from C1 last (idle/inflight==0/hold-timeout => allow), and because the forced flush is ungated a token-starved thread still drains, never deadlocks.

**Hyperparameters.** rho[t] (refill rate, the rate cap — the primary knob, can be uniform rho or per-thread), C_burst (bucket depth = burst tolerance), the issue-detection rule (msgs delta), liveness H. Optionally a target batch size that sets rho.

**Data / training / tuning.** No dataset. rho and C_burst tuned by a small offline sweep against bench dps. State per thread: 1 float bucket + prev msgs[] + prev timestamp; O(1). Needs the policy's own monotonic clock (step-2 requirement).

**Pros.** A clean, well-understood rate-limiting primitive that decouples 'how often a thread may issue' from instantaneous backlog: by spacing out permitted issues it deterministically lets leaves pile up to a controllable degree, which is precisely the coalescing lever. Burst depth C_burst gives explicit, tunable tolerance for short idle/refill transients (smoother than bang-bang). Per-thread rho naturally throttles a hot thread without starving a slow one. Maps onto standard traffic-shaping intuition and is trivial to reason about (tokens in, issues out).

**Cons.** Rate-based control is an indirect handle on what we actually care about (batch SIZE): a fixed rho assumes a roughly steady leaf arrival, but leaf arrival is bursty and reply-driven, so a static rho can either over-throttle (starve, hand work to forced-flush at depth-1) or under-throttle (no effect). It needs the self-clock and reliable issue-detection (msgs delta can miss/merge issues since a thread publishes per-reply, not per-tick — step-2 note). At the drain-all default the same inflight==0 forced-flush override that limits C1 limits this too. More state/tuning than C1 for a less direct grip on batch size.

**Ergonomics.** Good. ~40 lines; one or two intuitive knobs; the bucket is easy to log and reason about. Slightly more bookkeeping than C1 (clock + per-thread float + issue detection). A reasonable second experiment if framing the lever as a rate cap is preferred to framing it as a backlog floor.

## 5. C4 — AIMD-on-throughput per-thread gate (congestion-control analogue, RTT-free variant)

Treat each thread's permitted issue cadence as a congestion window: additively relax (allow more) while pool throughput rises, multiplicatively back off (deny harder) when throughput stalls or drops — TCP-style, but using leaf_rate as the signal because RTT is sentinel-0.

**Mechanism (features → per-thread {0,1}).** Maintain a per-thread continuous 'issue permit level' w[t] in [0,1] (or a per-thread deny-probability). Each tick measure the pool throughput signal R = realized_dps_window (trailing pool leaf_rate, the dps proxy). Compare to the best-seen R_max (or last tick's R): CONGESTION/STALL EVENT = R dropped by more than epsilon below R_max (a static threshold) OR a per-thread stall proxy fires (e.g. ready[t] high AND leaf_rate[t] ~ 0 => that thread's leaves aren't draining). On no event: additive increase w[t] += a (relax — allow). On event: multiplicative decrease w[t] *= b, b in (0,1) (back off — deny pressure up), applied preferentially to the threads flagged by cross_thread_skew/submit_pressure as the congestion contributors. Convert w[t] to a bit: allow[t] = 1 iff w[t] >= 0.5 (or sample allow[t]~Bernoulli(w[t]) for a finer effective rate at T<=4). Liveness overrides from C1 last.

**Hyperparameters.** a (additive step), b (multiplicative backoff), epsilon (stall-detection band on R), the trailing window W for R, the w->bit map (threshold vs Bernoulli), liveness H.

**Data / training / tuning.** No dataset. a,b,epsilon,W tuned by offline sweep on bench dps (AIMD is notoriously sensitive to a/b — wants a careful but small grid). State: per-thread w + R_max + the trailing-window ring buffer (step-2 realized_dps_window) + prev counters/timestamp.

**Pros.** A self-tuning hill-climber that needs NO target setpoint (unlike C2/C3): it searches for the throughput-maximizing operating point online by probing (increase) and retreating (decrease), which is attractive when the right batch size / issue cadence is unknown and possibly drifts. Well-studied stability/fairness properties. Backs off exactly the threads implicated by the cross-thread skew features. Could in principle find a better operating point than any single static threshold.

**Cons.** AIMD's natural signal is a congestion/loss/RTT event; here that channel (rtt_us) is sentinel-0, so the variant must synthesize a stall signal from a NOISY differenced pool rate (realized_dps_window) — and a hill-climb driven by a noisy reward over a slow (~5ms) cadence with a 1/T-resolution binary actuator is prone to thrash and to chasing measurement noise (it can interpret jitter as congestion and back off spuriously). Sawtooth operating point means it never settles, adding throughput variance. Most complex in this family and the least transparent to debug; the empirical edge over the simpler C1/C2 is speculative at T<=4. Strictly better once RTT is wired (see C5).

**Ergonomics.** Moderate-to-involved. ~80-100 lines; the noisy-reward sweep is the slow part. Harder to reason about a sawtooth than a deadband. Recommend only after C1/C2 are characterized, and prefer the C5 RTT-driven form once the latency channel exists.

## 6. C5 — RTT-driven congestion controller (AIMD / Vegas-style) — GATED ON the rtt channel being wired

The proper congestion-control formulation: use per-thread reply RTT as the back-pressure signal (rising RTT => server saturating => deny/throttle issues), in the standard AIMD or delay-based (Vegas/BBR-lite) shape.

**Mechanism (features → per-thread {0,1}).** Identical control skeleton to C4, but the STALL/CONGESTION signal is the principled one: per-thread mean_rtt_ms (rtt_us/1000) and its first-difference trend, plus the Little's-law work-in-system proxy rtt_us*inflight[t] from the step-2 rtt_derived_block. Delay-based form (Vegas-like): keep each thread's issue permit so that estimated in-server queueing (RTT above a measured baseline RTT_min) stays within [alpha,beta]; if RTT rises above baseline+beta, deny (back off); if it sits near baseline, allow (relax). Convert the per-thread permit to allow[t] by the same threshold/Bernoulli map as C4. Liveness overrides from C1 last.

**Hyperparameters.** RTT_min baseline (measured online), the [alpha,beta] queueing band (or AIMD a,b,epsilon on RTT), the permit->bit map, liveness H.

**Data / training / tuning.** No dataset. Band/gains tuned offline. PREREQUISITE (not a hyperparameter): the rtt channel must actually carry data — TODAY mean_rtt_ms is sentinel-0 because the sole live publish() call omits the rtt argument (runner_wire_batched.cpp:784 passes no rtt; the slot/codec exist end-to-end, issue_controller.hpp:85 / bridge:129, so feeding it is a one-line call-site change, NOT a wire change). State: RTT_min tracker + per-thread permit + prev tick.

**Pros.** This is the RIGHT static controller for the actual physics: a single shared server saturating IS a queueing/congestion phenomenon, and RTT is its canonical observable. Delay-based control reacts BEFORE loss/overflow and converges to low queueing with high utilization (the textbook win over rate/threshold heuristics). Once RTT is live this dominates C4 (real congestion signal instead of a noisy throughput-difference proxy) and complements C1 (C1 fattens batches; C5 prevents over-driving the server).

**Cons.** UN-RUNNABLE today: the signal it depends on is identically 0 (sentinel), so it would silently behave as 'every thread has 0 latency => never congested => always allow' (== identity) — which is the exact silent-failure trap the step-2 catalog flags. It is therefore a DESIGN-READY candidate contingent on a small upstream enablement (have publish() pass a measured RTT), not something to run now. Delay-based control also needs a clean RTT_min estimate, which the per-reply (not per-tick) publish jitter complicates.

**Ergonomics.** Good once enabled (the control code is C4's skeleton with a cleaner signal), but blocked on the rtt wiring. Flag it explicitly as gated so no one runs it against the dead channel and mistakes the resulting all-allow for a controller that 'did nothing wrong'.

## 7. C6 — Discrete-event / queueing-network (Petri-net-style) threshold scheduler — GATED ON the server channel being wired

Model the producer pool + single server as an explicit queueing/Petri net and gate issues so the modelled server batch occupancy stays in a target band — a model-based static scheduler rather than a black-box feedback loop.

**Mechanism (features → per-thread {0,1}).** Maintain a lightweight discrete-event model: per-thread places for ready (tokens=ready[t]) and inflight (tokens=inflight[t]), and a server place whose marking is the cross-message batch the server is assembling. The issue 'transition' for thread t is ENABLED (allow[t]=1) iff firing it keeps the modelled server batch within a target occupancy band B_srv* (Kingman/M/M/1-batch reasoning sets the band that maximizes server forward utilization without over-queueing); DISABLED (allow[t]=0) otherwise. In practice this reduces to a coordinated multi-thread threshold: allow the set of threads whose combined ready would bring the server's next-forward row count to B_srv*, deny the rest, prioritizing by submit_pressure/skew. Liveness overrides from C1 last.

**Hyperparameters.** B_srv* (target server rows/forward band), the per-thread priority/tie-break rule, the model's queueing assumptions, liveness H.

**Data / training / tuning.** No dataset, but the model needs the server's coalescing observable to be CALIBRATED and CLOSED-LOOP: the target is the server's rows/forward, exposed in-band as server_rows_per_forward — which is sentinel-0 today (no server->producer metrics channel; issue_controller.hpp:50 flows straight to the wire). Without it the server place is unobservable and the model runs open-loop (blind). Offline calibration of service-rate parameters from server logs (the post-hoc aggregation in overcommit_sweep.py / stage_b_*.py) is the substitute until the channel exists.

**Pros.** The most model-faithful: it reasons about the actual structure (many producers, one batching server) and targets the server-side coalescing the design's overcommit phase most cares about, coordinating threads jointly rather than gating each in isolation (the only candidate here that is genuinely multi-thread-aware in its objective). Given a good model it can place the operating point analytically rather than by sweep.

**Cons.** Heaviest to build and the most assumption-laden; a queueing model is only as good as its calibration, and the key observable (server_rows_per_forward, and ideally rtt) is sentinel-0 in-band right now, so it runs open-loop/blind today — high risk of model-vs-reality drift with no in-band signal to correct it. Overkill for T<=4 where the simpler feedback controllers already cover the space. Strong scope/complexity cost for an uncertain gain; the step-2 catalog explicitly flags server_rows_per_forward as a gap, not a measurement.

**Ergonomics.** Heavyweight. Most code, most modelling assumptions, and blocked on the server-metrics channel for closed-loop operation. A worthwhile design-on-paper target for the overcommit phase, but not a near-term runnable candidate; list it as gated and de-prioritized behind C1/C2.

## Recommended first

C1 (per-thread coalescing-floor hysteresis gate). Try it first because it is the minimal static controller that (a) directly actuates the one real lever — deny while a thread's un-submitted backlog is below a target so the next issue coalesces more rows — using ONLY live, non-sentinel features (ready, inflight, plus an optional differenced ready_velocity); (b) has exactly one primary knob with an obvious physical meaning (the per-thread ready-backlog target B*, expressed as a fraction of K or, where K is unavailable in-band, of a running-max estimate), tunable by a 1-D sweep on the bench's own dps; (c) is bang-bang/hysteretic so it cannot oscillate the way a P-gain on a quantized {0,1} actuator would, and the deadband makes it robust to the publish jitter the step-2 catalog flagged; (d) degrades to the exact identity policy (all-allow => byte-unchanged fixed-D runner, the clean control arm) at B*=0, so the A/B against default-allow is honest by construction; and (e) is the in-band, hand-tuned static mirror of the EXISTING in-tree S_min producer floor (runner_wire_batched.cpp:458, chunk_floor path) — it does in the slow Python policy, adaptively and per-thread, what min_coalesce does statically in C++, so it is the lowest-risk way to learn whether dynamic per-thread floor-holding beats the static compile-time floor before spending effort on anything richer. C0 (all-allow identity) is the mandatory baseline to run alongside it, not a competitor. Escalate to C2 (PI on pool coalescing degree) only if C1's single global B* proves too blunt across heterogeneous threads; escalate to C5/C6 (RTT-driven AIMD / queue control) only AFTER the rtt_us / server_rows_per_forward channels are fed, since today they are identically 0 and those controllers would be acting on a dead signal.

FOUR SOURCE FACTS that shape the whole family and that I verified, not assumed: (1) The action is DENY-ONLY and consulted at exactly ONE point — refill()'s non-forced loop, gate = `inflight_msgs < D && may_issue` (runner_wire_batched.cpp:746-747). The controller cannot force an issue, cannot change D, cannot touch the forced flush. (2) The forced flush (lines 748-749) is UNGATED and fires when inflight_msgs==0 && ready>0, so every candidate here gets never-deadlock for free — but the SAME fact means a deny is a NO-OP whenever a thread has nothing outstanding (the flush overrides it), which is why every candidate carries the inflight==0 / idle_thread_flag => force-allow override and why a per-thread gate has limited grip at the drain-all default (depth ~1, inflight in {0,1}). (3) The cadence is slow (cadence_ms_ default 5.0, bridge:51) vs the hot relaxed-atomic gate read — the textbook slow-outer-loop / fast-inner-plant regime these controllers assume, and a stale gate is explicitly benign (issue_controller.hpp:74-76). (4) Two of the most controller-relevant signals are SENTINEL-0 today: mean_rtt_ms/rtt_us (the sole live publish() omits the rtt arg, runner_wire_batched.cpp:784; codec/slot exist end-to-end so it is a one-line call-site enablement) and server_rows_per_forward (no server->producer channel; issue_controller.hpp:50). I therefore split the family into RUNNABLE-NOW candidates that use only live features (C0 identity, C1 backlog-hysteresis, C2 PI-on-coalescing, C3 token-bucket, C4 RTT-free throughput-AIMD) and GATED candidates that are design-ready but would act on a dead channel today (C5 RTT-driven AIMD/Vegas, C6 queueing/Petri-net server-occupancy scheduler) — flagged explicitly so no one runs them against the constant 0 and mistakes the resulting all-allow behaviour for 'a controller that did nothing wrong' (the exact silent-failure trap the step-2 catalog warns about).

Why hysteresis/bang-bang over a continuous PID as the lead: the actuator is a quantized per-thread BIT with effective resolution 1/T at T<=4, so a proportional gain on it cannot do smooth control and tends to chatter; a deadband relay is the control-theory-correct choice for a binary actuator and is inherently robust to the per-tick publish jitter (threads publish per-reply, not per-tick — runner_wire_batched.cpp:783-784 — so raw first-differences are noisy, which is also why every rate knob pairs with the EWMA/trailing-window smoothers from step-2). C1 is also the in-band, per-thread, hand-tuned mirror of the EXISTING static C++ producer floor (the S_min/min_coalesce + chunk_floor path, runner_wire_batched.cpp:458,717), which makes it the most directly interpretable first experiment: it asks 'does an adaptive per-thread coalescing floor in the slow policy beat the static compile-time one?' All candidates are hand-designed (gains/thresholds by analysis or a small dps sweep), consistent with this family's remit; none is learned from a labelled dataset (that is a different family). The four step-2 offline labels (label_terminal_dps, label_future_leaf_gain, label_counterfactual_gate_advantage, forward leaf gain) are NOT inputs to anything here — they are the OFFLINE measuring stick the sweeps optimize against, with C0/all-allow as the clean counterfactual control arm.
