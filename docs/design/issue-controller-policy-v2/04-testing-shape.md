# Phase 4 — ADR-0012-compliant testing shape

## Recommended shape

A single throwaway Python harness — call it cpp/stage_a/issue_control_ab.py, built in the established stage_b_ab.py / stage_b_poolbatch_sweep.py mold (subprocess-drives chocofarm-wire-ab-bench against an in-process StageAServer, parses its result line, writes under ~/w/vdc, hard subprocess timeout, no background watcher) — that scores ANY candidate issue-gate policy against the byte-identical all-allow baseline on the SAME faithful warm pool, with ZERO changes to the C++ bench, the bridge, the controller, or issue_engine.py. End to end:

(1) ONE in-process StageAServer is stood up over ONE published net (the build_and_publish + start_server pair lifted verbatim from stage_b_ab.py: 241->H->65, seed=17, residual=False, bucket+group, warmed), server pinned to core 0; the C++ bench runs under taskset -c 1,2,3 with --pool-threads 3 (the M3 1:3 split the existing harnesses already use). The server is the SAME for the baseline arm and every candidate arm — the only thing that varies across arms is the policy, so a dps delta is attributable to the controller and nothing else (the P6/ADR-0009 single-difference discipline the Stage B harness already states for the transport-schedule axis, here applied to the policy axis).

(2) The bench is invoked ONCE PER ARM in --sweep-configs mode — `--sweep-configs cf:S:D,...` builds ONE warm pool (snapshotted at --warmup-decisions, fixed --pool-seed) and REPLAYS it per producer config, each config measuring --measure-decisions after --settle-decisions over that one reproducible population, emitting one `SWEEP_RESULT cf=.. S=.. D=.. dps=.. wire_rows_per_msg=..` line per config (wire_ab_bench.cpp:209-296). This is the ergonomic core: the expensive warmup (reaching a faithful staggered representative-belief pool) is amortized once, and every arm/config measures the identical population, so the numbers are directly comparable. The per-config measure seed is fixed (7919ull, wire_ab_bench.cpp:257) so a given (arm, config) is reproducible.

(3) THE POLICY ARM is selected by whether --control-endpoint is passed to the bench (wire_ab_bench.cpp:258-268: empty endpoint => controller nullptr => byte-unchanged fixed-D runner; non-empty => a per-config IssueController + an IssueControlBridge that ZMQ-REQs the Python engine bound at that endpoint every --controller-cadence-ms). The harness therefore runs each arm as: stand up an IssueEngine(endpoint, policy=candidate, on_features=optional_recorder).start() bound to an ipc:// endpoint, run the bench with --control-endpoint <that endpoint>, engine.stop() (which re-raises any policy exception — ADR-0002), collect the SWEEP_RESULT lines. The candidate policy is just a `Callable[[dict], list[int]]` (the existing `Policy` type alias, issue_engine.py:63) — features dict in, per-thread [0/1] out — so dropping in a new controller is: write the function (or a factory that returns one), name it on the CLI, get a comparable dps. The engine, the wire codec, and the bench are untouched.

(4) BASELINE arms (two, both reported): (a) the BYTE-IDENTITY floor — the same --sweep-configs run with NO --control-endpoint (controller nullptr, the fixed-D runner byte-for-byte, issue_controller.hpp:57); (b) the WITHIN-ENGINE control — the same run WITH --control-endpoint but serving issue_engine.identity_policy (all-allow over the live bridge), which holds the bridge/snapshot/cadence overhead constant so the candidate's delta isolates the POLICY rather than the cost of having a control loop at all. A candidate that does not beat (b) on held-out configs/seeds is a negative result to record, not to ship (P6/ADR-0009 posture).

(5) SCORING: for each arm run >=2 seeds (vary --pool-seed and the per-arm res-token) x the iters the operator picks, aggregate dps per (arm, config) as mean +/- pstdev AND [min-max] using the agg() helper already in both stage_b harnesses, and report the candidate-vs-baseline(b) delta with the same non-overlapping-bands check stage_b_ab.py already prints (a3 min > a1 max). dps comes straight off SWEEP_RESULT (decisions/measure_wall); the harness never recomputes it.

The whole thing is one file of ~200 lines reusing build_and_publish/start_server/agg/run_bench-shaped subprocess calls from the two existing harnesses; the ONLY new mechanism over stage_b_poolbatch_sweep.py is (i) constructing an IssueEngine around the candidate and tearing it down per arm, and (ii) a tiny CLI registry mapping a policy name to its factory.

## Policy interface

The seam a candidate plugs into ALREADY EXISTS and is exactly `Policy = Callable[[dict], list]` (issue_engine.py:63): a function from the decoded observation dict (keys: n_threads, d_ceiling, server_rows_per_forward, and the length-T lists inflight, ready, msgs, leaves, rtt_us — decode_features, issue_engine.py:34-49) to a list of T ints in {0,1} (1=allow the next discretionary issue, 0=deny). A candidate is dropped in by constructing IssueEngine(endpoint, policy=candidate, on_features=optional_recorder) — no edit to the engine, the bridge, the controller hub, or the runner (P2: a new capability is a new impl behind the seam, the env<->Policy inversion mirrored here). To accommodate the three candidate families under ONE seam without widening it, the harness layers a thin, OPTIONAL factory protocol over the bare Callable (it does not replace it):

  class PolicyFactory(Protocol):
      def build(self, n_threads: int, d_ceiling: int, K_per_thread: int) -> Policy: ...

build() receives the out-of-band geometry the feature surface does not carry (notably K = trees_per_thread * ceil(pool_batch/pool_threads), the capacity normalizer the derived features need — runner_wire_batched.cpp:451) and RETURNS a fresh Policy (a closure) for one arm/run, so per-policy state (prev-tick counters + the policy's own time.monotonic clock for first-differencing the cumulative msgs/leaves; bandit arm stats; a loaded weight blob) is constructed once per run and lives in the closure, not in globals (P2: derived/owned state lives with the thing whose lifetime it shares, never a module global; P3: one owner per concern). A STATIC stateless policy is just `build = lambda *_: my_fn`; the all-allow baseline IS issue_engine.identity_policy unwrapped; a SUPERVISED policy's build() loads the JSON/params and returns the per-thread scorer; an ONLINE learner's build() returns a closure holding its mutable arm/Q state. The harness's CLI maps a --policy NAME to its PolicyFactory via a small dict registry, so the test author adds one entry and one function. Typed throughout (P8: the signature is the contract — Policy and PolicyFactory are annotated and honored). The candidate NEVER imports or re-implements the wire codec (P7: decode/encode have one derived home in issue_engine).

## Evaluation protocol

Same warm pool, mean and spread, against the all-allow baseline. Concretely:

WARM POOL: one StageAServer + one published net for the entire sweep; the C++ bench in --sweep-configs mode builds ONE warm pool (--warmup-decisions, fixed --pool-seed) and replays the SAME snapshotted population for every config of every arm (wire_ab_bench.cpp:200-232). So baseline and candidate are measured on byte-identical belief populations and the only variable is the policy.

ARMS (all run with the IDENTICAL --sweep-configs cf:S:D list, --measure-decisions, --settle-decisions, --pool-threads 3, server on core 0 / producers on 1,2,3):
  - BASE0 (byte-identity floor): no --control-endpoint => controller nullptr => the fixed-D runner byte-for-byte (issue_controller.hpp:57). Establishes the unmodified reference dps.
  - BASE1 (within-engine control, the PRIMARY comparator): --control-endpoint set, engine serving identity_policy (all-allow over the live bridge at the same --controller-cadence-ms). Holds the bridge/snapshot/cadence cost constant so a candidate's delta isolates the POLICY, not the existence of a control loop. (BASE0 == BASE1 within noise is itself a useful check that the loop overhead is negligible.)
  - CANDIDATE(s): --control-endpoint set, engine serving the candidate Policy.

REPLICATION & METRIC: each arm is run over >=2 seeds (vary --pool-seed; vary the per-arm --res-token so result keys never collide) x N iters (operator-chosen, default ~5, matching stage_b). dps is read straight off each `SWEEP_RESULT ... dps=...` line (wire_ab_bench.cpp:292-294) — the harness NEVER recomputes it. For each (arm, config) aggregate dps as mean, pstdev, min, max via the existing agg() helper. Optionally also aggregate wire_rows_per_msg (same line) as the mechanism read-out (did denial actually fatten batches), but dps is the figure of merit.

SCORING (ADR-0009 two-tier, P6): the headline is the candidate-minus-BASE1 dps delta per config, reported as mean +/- std [min-max] for both arms side by side, plus the NON-OVERLAPPING-BANDS verdict stage_b_ab.py already prints (candidate min > BASE1 max => a real win; else inconclusive/negative). Because dps under the live Gumbel search + float32 forward is a float-sensitive aggregate (not a logic invariant), the bar is statistical separation of the bands over >=2 seeds, NOT bit-identity (P6's calibration: hold a float-sensitive numeric to aggregate behavioral equivalence/separation, claim bit-identity only where free — here the only bit-exact claim is that BASE0 reproduces the fixed-D runner, which the nullptr path guarantees by construction). A candidate that fails to separate from BASE1 on held-out configs/seeds is recorded as a negative result, not shipped.

FAIL-LOUD: a non-zero bench rc (including a bridge error, surfaced via bridge->failed() at wire_ab_bench.cpp:274-277) raises in run_bench; engine.stop() re-raises any policy-side exception (issue_engine.py:115-116) — no arm silently degrades to all-allow and gets mistaken for a controller that 'did nothing wrong' (the sentinel-0/dead-signal trap the feature catalog warns about). Output JSON + the per-record SWEEP_RESULT trail are written under ~/w/vdc/chocobo/runs/ (never /tmp), preserving every arm.

## ADR-0012 compliance

| principle | how it is honored |
|---|---|
| **P2 — seam/port discipline (dependency inversion)** | The candidate plugs into the ALREADY-EXISTING injected port: Policy = Callable[[dict], list] served by IssueEngine. A new controller is a new Callable (optionally produced by a PolicyFactory.build) with ZERO edits to the engine, the bridge, the IssueController hub, or the runner — the env<->Policy inversion of control mirrored at the issue-gate seam (the runner depends on the hub's may_issue + publish, never on the policy; issue_controller.hpp:19-21). The engine is the translate-and-validate ACL: it decodes the foreign binary frame into the native dict and rejects a bad magic/length (issue_engine.py:38-41) rather than coercing. Per-run policy state lives in the build() closure (owned with the run's lifetime), never a module global keyed by identity. |
| **P3 — no god-objects / one owner** | Each component owns exactly one axis: the harness owns orchestration+scoring ONLY; StageAServer owns serving; IssueEngine owns the wire/REP service and the observation tap; the candidate owns the gate logic; build_and_publish/start_server/agg each own one sub-task (publish / serve / aggregate). The harness does NOT absorb the server, the codec, or the policy — naming its responsibility needs no 'and'. The optional data recorder is a separate callback, not folded into the policy or the orchestrator. |
| **P7 — cross-language wire discipline** | The packed-binary control frame has ONE authoritative definition (issue_control_bridge.hpp:35-42); the Python side DERIVES it once in issue_engine.decode_features/encode_gates with a magic+length runtime parity floor (issue_engine.py:26-54) — and the harness and every candidate REUSE that single derived codec, authoring NO second hand codec. The control loop correctly rides the ZeroMQ MESSAGING FABRIC (coordination/streaming), not redis-key-polling (the bytes-store-as-sync smell P7 forbids) — the harness adds nothing on the wire, it only binds the endpoint the engine already serves. Serialization contract (the frame) is kept distinct from the coordination mechanism (ZMQ), exactly as P7 mandates. |
| **P8 — typed signatures are the contract's SSOT** | The seam's contract is its typed signature: Policy = Callable[[dict], list] is already declared (issue_engine.py:63) and honored by every candidate body; the added PolicyFactory is a typed Protocol (build(n_threads:int, d_ceiling:int, K_per_thread:int) -> Policy). The harness functions carry param+return annotations (mirroring stage_b_ab.py's typed run_bench/agg), so a candidate's input/output contract lives in one checkable place and the body honors it — no lying signature (a policy must return exactly length-T 0/1, which encode_gates enforces structurally). |
| **P9 — functional core / modern C++** | No new C++ is written — the design's P9 compliance is that it BUILDS ON the already-P9-clean compiled fixture and keeps it clean: the IssueController hub is single-writer-per-cell relaxed atomics (race-free by construction), RAII unique_ptr owns ctl/bridge per config (wire_ab_bench.cpp:262-266), the hot gate is a const [[nodiscard]] may_issue (one relaxed load), and the bench returns std::expected and checks bridge->failed() to fail loud (wire_ab_bench.cpp:270-280). The harness actuates this purely through flags + the Python engine, so the functional-core/imperative-shell split (pure actuation cells vs the ZMQ-coordinating bridge shell) is preserved, not perturbed. |
| **P6 — substantiate perf/equivalence claims (composes with ADR-0009)** | The eval protocol IS the substantiation: same warm pool, candidate-vs-BASE1(all-allow-over-engine) dps as mean +/- std [min-max] over >=2 seeds, with the non-overlapping-bands verdict, at the float-sensitive aggregate bar (NOT bit-identity — dps under live Gumbel+float32 is a behavioral numeric). The only bit-exact claim is BASE0 reproducing the fixed-D runner, true by construction via the nullptr path. A candidate that does not separate from baseline is recorded as a negative result. This imports ADR-0009's two-tier bar wholesale, as P6 requires. |
| **P5 / ADR-0002 — fail loud** | Every failure surfaces loudly and is never silently swallowed into an all-allow degrade: a non-zero bench rc or a bridge error (bridge->failed(), wire_ab_bench.cpp:274-277) raises in run_bench; engine.stop() re-raises any policy exception (issue_engine.py:100-116); a wire-contract drift raises on decode (issue_engine.py:39,41). This is the specific guard against the sentinel-0 trap — a candidate acting on a dead rtt_us/server_rows_per_forward channel that silently behaves as all-allow would otherwise be mistaken for a controller that did no harm; the loud arm-labeling + the BASE1 comparator make that visible. |
| **P4 — live, not frozen** | The harness honors the live-cell discipline already in the fixture: the per-thread allow bits are read hot on the runner gate each refill (a live cell, stale-tolerant by design), D and the warm pool are the read-only INSTANCE/RESTART context, and the candidate's own rates are first-differenced live from the cumulative counters each tick. The harness bakes only true per-run invariants (the net, the config list) and leaves the gate live — it does not freeze a swept knob at construction. |
| **P1 — single source of truth / derive-don't-duplicate** | The harness reuses build_and_publish/start_server/agg and the redis keys/net geometry from their existing homes (transport.py, inference_server.py, StageAServer, stage_b_ab.py) rather than re-encoding them; dps is read off SWEEP_RESULT, never recomputed; the wire layout is derived from the one authority via issue_engine, not re-typed. Every fact the harness needs has one home it reads from. |

## Open questions

- The same Python IssueEngine is bound for the whole multi-config sweep, but the C++ side rebuilds a fresh IssueController + IssueControlBridge PER config (wire_ab_bench.cpp:262-268), so the REQ socket reconnects each config. A stateful candidate (online learner, EWMA/prev-tick state) therefore sees a per-config discontinuity in its own feature stream — is the intended unit-of-adaptation one config (reset policy state on the per-config bridge reconnect) or the whole run? The harness should let a PolicyFactory opt into per-config rebuild; needs the test author to declare which, and ideally the engine/bridge to signal a config boundary (today there is no in-band 'new config' marker — the policy would have to infer it from a counter reset, which the cumulative msgs/leaves do NOT provide since they restart per config).
- Two of the most controller-relevant channels are sentinel-0 today (server_rows_per_forward; rtt_us — the live publish() omits the rtt arg, runner_wire_batched.cpp:784, and no server->producer metrics channel exists). Any candidate that depends on them silently degrades to all-allow. Should the harness add an explicit GUARD (e.g. assert the chosen feature subset is non-constant across a probe run, or have the candidate declare which features it reads and the harness refuse a candidate that reads a known-dead channel) so a dead-signal policy cannot be mistaken for a working one that scored a null delta?
- dps in --sweep-configs mode is decisions/measure_wall (RECORDED GUMBEL DECISIONS, wire_ab_bench.cpp:286-288), not leaves/sec; the online/SL candidates' reward proxy is leaf_rate. The two are tightly coupled but not identical. The protocol scores on dps (the right objective) but should the harness ALSO surface a leaves-based number (e.g. from the wire_summary or a server counter) so a candidate tuned on leaf_rate can be checked for the objective<->proxy gap, and so a divergence between them is visible rather than hidden?
- Statistical bar: with T<=4 and a short --measure-decisions window the per-config dps has real variance, and the existing harnesses use pstdev + a min/max non-overlap heuristic rather than a formal test. For a controller whose true effect may be a few percent, is the non-overlapping-bands check enough, or should the harness compute a proper two-sample statistic (and how many seeds/iters does the operator budget for separation at the expected effect size)?
- K (trees_per_thread * ceil(pool_batch/pool_threads)) is needed by the capacity-normalized derived features but is not on the wire. The PolicyFactory.build() receives it out-of-band from the harness (which knows the config), but in --sweep-configs mode pool_batch/threads are fixed across configs while cf:S:D vary — confirm K is constant across the swept configs (it is, since the sweep varies only chunk_floor/S_min/D, not pool_batch/threads), so a single K passed at build() is correct; flag if a future sweep also varies pool_batch.
