# Online learning — formal supplement & prior art

> Family scope: online

## C1 — Homogeneous threshold bandit (single shared rule, parameter-sharing across threads)

**Formal object.** Under SYNTHESIS §6, the plant is a self-clocking batch-service queue whose state is the server's batch-size fixed point B* (the absorbing-ceiling recurrence min(T·K, max_batch+K-1) reached in one drain cycle) plus the per-thread ready/inflight backlog of §3.1. In the LIVE chunk_floor regime (caveat: depth>1, D live, DOF-depth un-pinned) the action `allow∈{0,1}`, read ONCE per refill() and gating the WHOLE `while(inflight<D && may_issue && issue(false))` chunk-burst (runner :746-747), throttles how many S_min-row chunks (DOF-B = S_min, §4) a thread injects into the ROUTER queue during the server's current forward (DOF-F self-batching, §3.3/§4). With ONE shared threshold θ on a per-thread saturation signal x[t], C1 is formally a single-state (context-free) MULTI-ARMED BANDIT whose arms are quantized values of θ and whose stochastic reward is the pool-total leaf_rate ≈ the steady-state throughput B*/S(B*) of the §6 fixed point that θ induces. State = the §6 plant; action = the shared θ that shapes the arrival process feeding §3.3; objective = maximize the §6 throughput; the per-tick decision is the arm pull held for W ticks (one fixed-point relaxation time).

**Reduction / algorithm fit.** Karp-style: the gate-tuning problem reduces to STOCHASTIC MULTI-ARMED BANDIT (Lai-Robbins / UCB1 / EXP3) by the transformation 'discretize the policy-parameter axis θ into the arm set A, treat one W-tick hold as one bandit round, and define the reward as the pool-total leaf_rate the §6 fixed point yields at that θ.' This is valid IFF the map θ↦B*↦throughput is (i) well-defined per θ and (ii) approximately stationary over the W-window. §6 supplies (i): the fixed point is unconditionally non-divergent and reached in one cycle (the two S-independent clamps), so each held θ DOES induce a single well-defined throughput — the reduction's reward is a real number, not an ill-posed limit. §6 also supplies the WARNING the v2 'non-monotone' Cons gestures at but cannot formalize: throughput = B*/S(B*), and beyond saturation (R2) B* is pinned at the ceiling while S can jump on an overshoot cold-compile (DOF-overshoot, one-time per width once T·K>max_batch) — so the reward landscape over θ is NOT monotone and has a flat saturated plateau, which is exactly the regime where Lai-Robbins gap-dependent regret degrades (arms with near-equal means). Non-stationarity (the settle→steady ramp, R3→R1→R2) forces the ADVERSARIAL-bandit form (EXP3) or discounted-UCB rather than stationary UCB1; the model makes this non-stationarity STRUCTURAL (the fixed point CLIMBS as offered load rises, §6 effect (3)), not incidental.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D and DOF-B primarily: a shared deny collapses every thread's chunk-burst, so per-thread depth drops from ⌊W/S_min⌋-toward-D back toward 1 and fewer S_min-row messages exist (approximate reading — the chunk_floor='on' regime the caveat names; under chunk_floor='off' the action is inert per the SYNTHESIS :686-688 / 02-derived inflight_saturation note and C1 degenerates to all-allow). Secondarily DOF-F (§3.3): changing how many chunks arrive during a forward moves the server's next-drain batch B and thus the §6 fixed point. DOF-S only indirectly (a different B changes the padded forward shape, hence S, only across the max_batch / overshoot boundary, §3.2). Flag: all four readings are the approximate depth>1 substrate, since the model's proofs are for depth==1.

**What the model adds vs v2.** The model converts C1's hand-waved 'monotone higher-threshold⇒more-coalescing⇒maybe-more-dps (may be non-monotone)' assumption into a DERIVED reward shape: throughput = B*/S(B*) with a negative-feedback plateau (R1→R2) and an overshoot cliff (DOF-overshoot, reachable exactly at T·K>max_batch — i.e. at the large-N the controller is FOR). This (a) tells C1 the reward axis is flat-then-cliffed, not smoothly unimodal, so EXP3/discounted-UCB (not UCB1) is mandated and W must exceed ONE fixed-point relaxation, not merely 'a few RTTs'; (b) gives the all-allow arm a precise meaning — it is the θ that sends the system to the §6 fixed point with the maximal injection rate, i.e. the right edge of the arm set, which §6 says is bounded/safe (never diverges), formally justifying v2's 'cannot do worse than baseline by more than exploration regret'; (c) warns that C1's single global θ cannot move the ONE knob the model says actually sets steady-state width — the server-side drain — so its ceiling on achievable gain is the cross-thread arrival SHAPE, not the per-thread degree, which the heterogeneity-blind C1 leaves untouched.

## C2 — Contextual bandit, homogeneous (LinUCB / linear-Thompson over a small derived-feature context, shared weights)

**Formal object.** The per-thread gate becomes a CONTEXTUAL bandit whose context φ[t] is the per-thread projection of the §3.1 source state (ready backlog, the offered δ_park superposition's instantaneous eligible-set size, coalesce_degree_inst as the realized DOF-B) and arms {deny,allow} choose whether thread t's chunk-burst feeds the §3.3 self-batching window this tick. Formally this is admission control into the batch-service queue with PER-THREAD CONTEXT but a SHARED reward (the §6 pool throughput): state = (B* fixed point, per-thread backlog vector); action = the per-thread allow vector chosen by argmax over a context-linear value; dynamics = §3.3 closed loop (a denied thread's held rows pool into its next chunk, :694, and its absence fattens or thins the server's next drain depending on whether the server is arrival-bound R3 or service-bound R2); objective = §6 throughput. depth>1/D-live is the operative regime (caveat).

**Reduction / algorithm fit.** Reduction to LINEAR CONTEXTUAL BANDIT (LinUCB / linear-Thompson, Abbasi-Yadkori disjoint-arm) by 'featurize each thread's §3.1/§3.2 state as φ[t], make (φ[t],a[t],r) one ridge-regression sample, pool the T samples/tick under shared weights.' The transformation is sound where the value of allow vs deny is approximately LINEAR in φ — but the model exposes precisely why linearity is fragile here: the §6 reward is throughput = B*/S(B*) which is NONLINEAR and REGIME-PIECEWISE (arrival-bound R3 ⇒ deny HURTS, the linear coefficient on backlog should be negative; service-bound R2 ⇒ deny is neutral-to-helpful, coefficient ≈ 0 or positive; overshoot ⇒ a cliff). A single shared linear model therefore needs a REGIME INDICATOR in φ (e.g. a coarse 'is the server arrival- or service-bound' proxy from coalesce_degree_inst vs its ceiling) or it averages opposite-signed coefficients across R3/R2 and underfits — this is the model's concrete instruction for the v2 'linear may underfit / needs feature engineering' Con. The credit confound is the model's §6 'pool-scalar reward' fact: T threads share the SAME B*, so the per-thread context cannot be cleanly attributed — a structural (not merely statistical) confound, mitigated only by the regime split that makes the shared scalar's sign predictable from context.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D and DOF-B per thread (a per-thread deny selectively collapses ONE thread's chunk-burst, leaving others' depth>1 — the heterogeneity C1 cannot express), and through that DOF-F (the server's next drain now coalesces a DIFFERENT subset of threads' chunks, §3.3). The approximate reading (caveat): in the depth>1 regime a per-thread allow vector shapes WHICH threads' S_min-chunks populate the ROUTER fair-queue during a forward — i.e. it moves the cross-thread arrival pattern that §6/DOF-F turns into the next batch. DOF-S only across the shape boundary.

**What the model adds vs v2.** Two corrections the unscaffolded v2 could not make. (1) It identifies the MISSING context feature: a regime indicator (R3 arrival-bound vs R2 service-bound, derivable from coalesce_degree_inst relative to its lifetime ceiling) is REQUIRED for the linear model to have a stable sign, because §6 makes the optimal action sign FLIP across regimes — v2 listed candidate context features but had no formal reason to prefer any; the model says the regime axis is the load-bearing one. (2) It reframes the credit confound from 'a statistical nuisance the auxiliary per-thread leaf_rate partly fixes' to a STRUCTURAL consequence of the shared §6 fixed point — all T threads' rewards are reads of the same B*/S(B*), so per-thread attribution is identifiable only THROUGH the regime context, which is why the auxiliary per-thread leaf_rate[t] (a thread's OWN rows/s) is itself confounded by the shared server and cannot disambiguate. This sharpens v2's hedge into a design constraint.

## C3 — Per-thread independent bandits (T tiny decoupled allow/deny learners sharing the global reward)

**Formal object.** T independent 2-arm bandits, one per thread — formally T parallel admission controllers acting on a SHARED, COUPLED plant (the single §6 fixed point + the single ROUTER fair-queue, §3.3). Each thread's bandit sees state = its own §3.1 backlog, action = its own allow bit moving its own chunk-burst (DOF-B/depth), but reward = either the shared §6 pool throughput (fully confounded — all T bandits credit the same scalar) or its own leaf_rate[t] (the rows the SERVER chose to return to it, itself a function of the joint arrival pattern via §3.3). This is the textbook DECENTRALIZED / GAME-THEORETIC control of a shared queue: T selfish admission controllers, one shared service resource. depth>1/D-live regime (caveat).

**Reduction / algorithm fit.** Two competing reductions, and the model adjudicates. (a) As INDEPENDENT bandits the reduction to per-thread UCB/ε-greedy is trivial but VACUOUS-to-harmful, because the plant is coupled: §3.3 'no recv during a forward' means thread t's optimal action depends on whether OTHER threads are also injecting (their chunks share the drain), so the per-thread reward is non-stationary w.r.t. the other learners — the stationary-bandit premise fails. (b) The correct formal object is therefore a STOCHASTIC GAME / decentralized POMDP on the shared §6 queue, and the model exposes the failure mode the v2 'tragedy-of-the-commons' Con only names: a thread maximizing its OWN leaf_rate[t] floods the single-threaded server (§3.3 serialization), pushing the system toward R2 saturation / R6 RCVTIMEO (§5: firing probability RISES with offered load) — i.e. the selfish equilibrium of the game is a CONGESTED one, the Braess/price-of-anarchy outcome. The pool-reward shaping weight β the v2 names is, formally, the mechanism that turns the stochastic game into a POTENTIAL GAME (aligning each thread's objective with the shared §6 throughput so the Nash equilibrium coincides with the social optimum) — that is the model's name for v2's 'keep threads cooperating' knob.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D and DOF-B per thread, fully decoupled in ACTION but coupled in EFFECT through DOF-F: each thread's deny independently collapses its own chunk-burst, but because the server's drain (§3.3) pulls across all threads' queued chunks, the joint effect on the §6 fixed point is non-separable. Approximate reading (caveat): the depth>1 regime is exactly where this coupling bites — at depth==1 (drain-all) each thread emits one message and the cross-thread coupling is the cleaner cross-thread coalescing of §3 SAT example; at depth>1 each thread emits ⌊W/S_min⌋ chunks and the ROUTER fair-queue interleaving (DOF-R cross-DEALER) makes the joint arrival order, hence B, genuinely a function of the joint action.

**What the model adds vs v2.** The model upgrades the v2 'tragedy-of-the-commons' from an analogy to a NAMED game-theoretic structure: T selfish admission controllers on one shared batch server have a congested Nash equilibrium (price-of-anarchy > 1), and the §5 R6 analysis says the congested equilibrium is not merely lower-throughput but LIVENESS-ADJACENT (the RCVTIMEO firing probability rises with the offered load the uncoordinated threads create). This (a) formally explains why C3 is the worst candidate — it is not just sample-split, it is solving a HARDER (game-theoretic, non-stationary) problem than C1's single-agent bandit; (b) gives the β shaping weight a precise target (make the game a potential game whose potential is §6 throughput); (c) confirms v2's 'best as a contrast/ablation' — the model says the gap between C3 (decentralized, congested) and C1 (centralized shared θ, which CANNOT congest because one θ moves all threads coherently) directly measures the price of anarchy, which is itself a substantiation of the parameter-sharing choice.

## C4 — Adaptive single-threshold control via online stochastic optimization (SPSA / 1-D gradient-free hill-climb on a shared gate threshold)

**Formal object.** Identical plant to C1 but the action is a CONTINUOUS shared θ∈ℝ optimized as a stochastic-approximation iterate. Formally: minimize -J(θ) where J(θ) = the §6 steady-state throughput B*(θ)/S(B*(θ)) induced by threshold θ, observed only through the noisy pool-total leaf_rate estimator (a Monte-Carlo read of J over a W-window). State = θ_k + the §6 plant; action = the ±Δ perturbation and the gradient step; dynamics = §3.3 closed loop relaxing to B*(θ) within ~one fixed-point time; objective = the smooth-ish J(θ) surface §6 guarantees is bounded. depth>1/D-live (caveat).

**Reduction / algorithm fit.** Reduction to KIEFER-WOLFOWITZ / SPSA stochastic approximation by 'treat J(θ) as a noisy black-box objective and estimate ∇J by simultaneous perturbation.' The model supplies the convergence PRECONDITION the v2 'assumes locally-smooth roughly-unimodal J' Con states without justification: §6's non-divergence + single-cycle fixed point means J(θ) is WELL-DEFINED and BOUNDED for every θ (no θ sends throughput to a pole), so the SPSA objective is a real bounded function — the reduction is not ill-posed. BUT §6 also says J has a FLAT SATURATED PLATEAU (R2: B* pinned at the ceiling, dJ/dθ≈0 over the whole saturated region) and an overshoot DISCONTINUITY (DOF-overshoot cold-compile cliff). On the plateau the SPSA gradient estimate is ~0+noise, so the iterate RANDOM-WALKS (the v2 'wander if W too small' risk, now explained: it is not noise sensitivity per se, it is a genuinely flat true gradient on the §6 plateau). The correct transformation is therefore SPSA WITH A SATURATION GUARD — clamp θ to the steepest-ascent edge of the plateau (the all-allow side §6 marks as bounded) — or accept that SPSA degenerates to C1's right-edge arm in saturation. Provable convergence holds only on the sub-saturation (R3→R1) ramp where J is genuinely sloped.

**DOF moved (depth>1 live; approximate per caveat).** Same as C1 — DOF-depth/D and DOF-B via the shared continuous θ, indirectly DOF-F through the §6 fixed point — but on a CONTINUOUS axis, so it can land BETWEEN the discrete chunk-burst counts C1's arms quantize (e.g. a θ that lets a thread issue 2 of its ⌊W/S_min⌋ chunks before denying). Approximate reading (caveat): in the depth>1 regime the continuous θ sets, in expectation, the fraction of each thread's chunk-burst that flies, hence a real-valued average outstanding depth in [1,D] — the one place the CONTINUOUS twin genuinely buys resolution the model can see (depth is integer-quantized per thread but its time-average over W is continuous).

**What the model adds vs v2.** The model turns SPSA's abstract 'smooth unimodal' assumption into the CONCRETE §6 landscape: bounded everywhere (good — SPSA well-posed), flat on the R2 plateau (bad — zero true gradient, the iterate stalls/wanders not from noise but from a flat truth), discontinuous at the overshoot width (bad — a cliff SPSA's finite difference cannot see until it straddles it). This (a) reframes v2's 'wander if W too small' as 'wander because the §6 plateau gradient IS ~0', a fundamentally different and unfixable-by-W diagnosis on the plateau; (b) mandates a θ-clamp to the §6-bounded all-allow edge so a perturbation cannot fall off the overshoot cliff into a one-time S_cold spike (DOF-overshoot, §3.2/R2); (c) confirms SPSA's edge over C1 is real only on the R3→R1 ramp (the under-saturated regime), and tells the designer to expect SPSA≈C1 once the run reaches the §6 saturated fixed point the large-N controller targets.

## C5 — Dual/Lagrangian online controller (drive an observable proxy to a setpoint via an adapted price; the gate is a thresholded price)

**Formal object.** This is the candidate the model fits BEST. Frame the gate as enforcing a soft constraint 'per-thread coalescing degree (the realized DOF-B = coalesce_degree_inst, which IS the per-thread analogue of S_min) ≥ S*' via a dual price p adapted by ascent, with S* searched on a slow outer loop by §6 throughput. Formally: a CONSTRAINED MDP on the §6 queue solved by Lagrangian relaxation — state = per-thread realized coalescing + the §6 B* fixed point; action = deny[t] = 1{p > cutoff AND coalesce_degree_inst[t] < S*}; fast dynamics = dual ascent p ← max(0, p + η(S* − coalesce_degree_inst[t])); slow dynamics = the outer S* hill-climb on pool throughput; objective = §6 throughput subject to the coalescing-floor constraint. The two timescales map exactly onto the model's two clocks: the FAST inner price tracks the per-thread DOF-B, the SLOW outer S* tracks the §6 fixed point B*. depth>1/D-live (caveat).

**Reduction / algorithm fit.** Reduction to TWO-TIMESCALE STOCHASTIC APPROXIMATION / primal-dual (Borkar) on a constrained MDP. The model makes this reduction UNUSUALLY tight because the constraint variable IS a model DOF: S* is the producer-side coalescing floor that, in the code, is literally S_min (runner :458, the chunk_floor binds each message to S_min rows). So C5's dual ascent is online auto-tuning of the SAME S_min the chunk_floor regime fixes statically — the controller and the plant share one variable (the model's DOF-B). The transformation 'soft-constrain coalescing ≥ S*, price the violation, search S* on throughput' is valid IFF (i) deny→coalescing-rises monotonically (the chicken-and-egg v2 Con) and (ii) the timescales separate. The model adjudicates BOTH: for (i), §3.3 + :694 say a denied thread's held rows DO pool into its next chunk, so deny→more-rows-per-next-chunk holds MECHANICALLY at the producer (DOF-B) — but the model's §6 warns the OBJECTIVE link 'more per-thread coalescing ⇒ more THROUGHPUT' fails in R2 (server already at the ceiling) — so the outer S* loop is steering a proxy whose correlation with the true objective is regime-dependent, exactly the §6 plateau again. For (ii), the model gives the fast timescale a physical floor: the inner price must equilibrate within ONE §6 fixed-point relaxation, and the outer S* must move slower than that — Borkar's timescale-separation condition grounded in the model's relaxation time.

**DOF moved (depth>1 live; approximate per caveat).** DOF-B DIRECTLY and BY NAME — S* is the model's coalescing-degree DOF, and the dual price drives the realized per-thread coalescing toward it. Through DOF-B it moves DOF-depth/D (holding sub-S* threads' issues changes their outstanding chunk count) and DOF-F (the server re-coalesces the held-then-released chunks, §3.3 — and §1's note that the chunk_floor design INTENT is producer supplies depth, server controls width, exactly the DOF-B↔DOF-F handoff). Approximate reading (caveat): C5's price acts on the per-thread coalescing degree that only EXISTS as a free quantity in the depth>1 regime — at depth==1 every message is already drain-all and coalesce_degree_inst is pinned by the arrival process, so the price has nothing to push; chunk_floor un-pins it (the caveat's exact mechanism) and gives C5 its control authority.

**What the model adds vs v2.** The model gives C5 the cleanest scaffolding of the family because its control variable S* IS a model DOF (DOF-B / the code's S_min). It (a) confirms the v2 'most mechanism-aligned' intuition formally — the action→proxy→objective chain is action→DOF-B→§6-throughput, the shortest in the family; (b) resolves the v2 chicken-and-egg ('does deny actually fatten batches?') at the PRODUCER level: §3.3+:694 say YES mechanically (held rows pool forward), so the inner loop is well-founded; (c) but RELOCATES the real risk to the OBJECTIVE level via §6 — fatter per-thread chunks raise throughput only while the server is arrival-bound (R3→R1); in R2 saturation the §6 ceiling caps throughput and a higher S* just makes messages fatter without raising B*/S, so the SLOW S* loop is chasing a proxy that decorrelates from dps exactly where the large-N controller operates — the model says this is what the slow loop must detect, and it is the §6 plateau once more; (d) reveals the deepest design opportunity: since S* = the code's S_min, C5 is the online controller that CO-DESIGNS with the server-side floor θ — the two-sided handoff §1 names (producer depth via S_min, server width via θ) — pointing past the single-sided gate toward the joint controller below.

## C6 — Tabular online RL on a coarse-discretized per-thread state (Q-learning / SARSA with parameter-sharing)

**Formal object.** The per-thread gate as a small MARKOV DECISION PROCESS: state s[t] = coarse bins of (per-thread backlog, ready_velocity sign) — a discretization of the §3.1 source state; action {deny,allow} moving the chunk-burst; reward = leaf_rate; a SHARED tabular Q(s,a) bootstrapped by TD. Formally this is the ONLY candidate that models the §3.3 DELAYED-CREDIT structure as a real MDP transition: a denial's payoff is the FATTER NEXT DRAIN (the held rows pool forward, :694, and the server coalesces them next cycle, §3.3 'no recv during a forward'), which is a genuine state→next-state→reward bootstrap, not a windowed approximation. State = §3.1 backlog bins × the §6 plant; action = per-thread allow; dynamics = §3.3 closed loop with the one-step delay the §6 fixed point relaxes over; objective = discounted §6 throughput. depth>1/D-live (caveat).

**Reduction / algorithm fit.** Reduction to tabular Q-LEARNING/SARSA on a finite MDP by discretizing the §3.1 continuous source state. The model both JUSTIFIES and INDICTS this. Justification: §3.3's self-batching feedback IS a one-step temporal coupling (a forward's duration sets the next drain's batch), so there genuinely is a Markov transition with delayed reward that TD bootstrapping is built for — the v2 'only scheme that models temporal structure' claim is model-correct. Indictment: the model shows the temporal horizon is SHORT and SELF-CORRECTING — §6 says the fixed point is reached in ONE cycle via S-independent clamps, so the 'delayed' credit is delayed by ~one drain, not a long return; a γ-discounted multi-step bootstrap is overkill for a horizon-1 self-clocking plant, and the v2 'window-based bandit approximates the delay' is, per §6, an ADEQUATE approximation precisely because the delay is one cycle. Worse, §6's non-stationarity (the fixed point CLIMBS along R3→R1→R2) violates Q-learning's stationary-MDP premise — the model makes the v2 'non-stationarity breaks the MDP' Con structural, not incidental. So the reduction is formally valid but the model says the MDP it targets is a thinly-disguised one-step (contextual-bandit-depth) problem, which is why C1/C2 dominate it on sample efficiency.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D and DOF-B per thread via the shared Q-policy, and uniquely DOF-F as an EXPLICIT transition: the bootstrap Q(s',·) term IS the model's self-batching feedback (the next state's batch size, §3.3/§4 DOF-F) entering the value of the current denial. Approximate reading (caveat): the depth>1 regime is what makes the 'denial fattens the next batch' transition non-trivial — at depth==1 a denial just delays one drain-all message (no real next-batch fattening to bootstrap on); chunk_floor's held-remainder (:694) is the state variable whose evolution the Q-table would have to capture, and it exists only in the live regime.

**What the model adds vs v2.** The model resolves whether temporal credit BUYS anything (the v2 open question C6 is meant to ablate): §6 says the plant self-corrects to its fixed point in ONE cycle, so the delayed reward C6 bootstraps is a one-step delay — making C6's multi-step TD machinery formally REDUNDANT over a contextual bandit (C2) that uses a one-fixed-point-time reward window. This is a sharper verdict than v2's 'worst sample efficiency': it is not merely that C6 explores too slowly, it is that the §6 self-clocking structure means there is NO long-horizon return to learn, so the RL-flavored value of C6 (temporal credit) is illusory for THIS plant. The model thereby substantiates the v2 ranking (C6 as contrast/ablation) from first principles: the ablation will show ≈0 gain from temporal credit because §6 makes the horizon 1.

## NEW: Whittle-index / restless-bandit per-thread gating (one shared index policy, indexability from the §6 plant)  🆕 (new — surfaced by the formal lens)

**Formal object.** Cast each thread as one ARM of a RESTLESS MULTI-ARMED BANDIT (RMAB): arm t's private state = its §3.1 backlog / realized DOF-B (coalesce_degree_inst, inflight depth in the live regime); 'active' = allow (let its chunk-burst fly into the §3.3 window), 'passive' = deny (hold rows, :694). Both actions evolve the state (restless: a denied thread's backlog still RISES as replies resume slots, §3.1) — exactly the RMAB defining feature. The shared resource is the single-threaded server's §6 batch-service capacity; the planner wants to choose, each tick, WHICH subset of the T arms to activate to maximize §6 pool throughput. State = T private backlogs + the shared §6 fixed point; action = the binary activation vector (= the allow vector); dynamics = §3.1 source + §3.3 service coupling; objective = long-run §6 throughput. depth>1/D-live (caveat).

**Reduction / algorithm fit.** Reduction to the WHITTLE INDEX policy via Lagrangian relaxation of the per-tick activation constraint: relax 'choose any subset' to 'choose subsets costing ≤ a price λ per active arm', decouple the T arms, and for each compute its Whittle index W_t(state) = the λ at which activate and not-activate are equin-value; then activate the highest-index arms. This is the FORMAL upgrade of v2's C2/C5 'per-thread differentiation' from heuristic to the provably near-optimal RMAB policy (Weber-Weiss asymptotic optimality as T→∞ under indexability). The transformation needs INDEXABILITY (the passive set monotone in λ); the model gives the structural reason to expect it: §6's negative-feedback monotonicity (more backlog ⇒ fatter drain ⇒ higher throughput, until the ceiling) means a thread's marginal value of activation is MONOTONE DECREASING in how saturated the server already is — the single-crossing property indexability requires. The index ORDERS threads by 'whose chunk-burst most fattens the §6 drain right now' — heterogeneity-aware (unlike C1) but with a SINGLE shared index FUNCTION (parameter-sharing, the family's tractability lever), so it inherits C1's one-reward learnability while expressing C2's per-thread action. Caveat-flag: indexability is asserted from the §6 monotonicity, which is proven for depth==1; in the depth>1 regime it is the approximate substrate, to be checked, not assumed.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D and DOF-B jointly across threads: the index policy selects which threads run their chunk-bursts (setting their depth toward D) vs hold (DOF-B fattening), and the activation SUBSET is chosen to shape DOF-F (the §6 next-drain batch) optimally. Approximate reading (caveat): the RMAB framing is most natural in the depth>1 regime because each arm's state genuinely includes its outstanding chunk count (1..D), which only varies when chunk_floor un-pins DOF-depth; at depth==1 every arm's 'active' is the same single drain-all message and the RMAB collapses to a trivial 1-state-per-arm problem.

**What the model adds vs v2.** This candidate exists BECAUSE the model reframes the gate as resource allocation across coupled arms (the §6 shared fixed point + T private backlogs) — a structure the unscaffolded v2 saw only as 'per-thread credit assignment confound'. The model converts that confound into a SOLVED problem class: RMAB with a shared index dissolves the pool-reward-vs-per-thread-action tension the v2 family is built around (the index is learned from ONE pooled reward but ACTS per-thread), giving the heterogeneity of C2/C3 with the learnability of C1 and a near-optimality guarantee neither has. It also supplies the indexability hypothesis from §6 monotonicity — a falsifiable, model-grounded claim the v2 pass had no vocabulary to state.

## NEW: Lyapunov-drift / backpressure admission control (deny when the drift bound says holding lowers the §6-coupled queue)  🆕 (new — surfaced by the formal lens)

**Formal object.** Treat each thread's ready backlog Q_t (parked-unsubmitted leaves, the §3.1 offered backlog bounded by ≤K) as a virtual QUEUE, and the server's drain as the shared service. The gate is then a MAXWEIGHT/BACKPRESSURE admission decision: define a Lyapunov function L = ½ Σ_t Q_t² and choose the allow vector each tick to minimize the drift-plus-penalty ΔL − V·(pool leaf_rate). Formally a network-utility-maximization (NUM) over the §6 queue: state = the per-thread virtual queues + the §6 B* fixed point; action = the allow vector; dynamics = §3.1 arrivals (replies resuming slots) minus §3.3 service (the drained chunks); objective = maximize throughput (the V-weighted penalty) subject to queue stability. depth>1/D-live (caveat).

**Reduction / algorithm fit.** Reduction to LYAPUNOV DRIFT-PLUS-PENALTY / backpressure (Tassiulas-Ephremides, Neely's NUM) by 'model the parked-leaf backlog as a queue, the chunk-burst as the admission/service action, and pick the action greedily minimizing the one-step drift bound plus a throughput penalty.' The model makes this reduction APT in a way v2 could not see: §6 ALREADY proves the closed loop is a bounded, self-clocking negative-feedback system whose batch size is Lyapunov-bounded by the absorbing ceiling min(T·K, max_batch+K−1) — i.e. the model has ALREADY exhibited a Lyapunov-type drift argument (the divergent loop foreclosed link-by-link), so casting the controller in the same drift framework is structurally consonant, not a forced fit. The transformation: the §6 absorbing-ceiling proof IS a drift bound with V=0 (pure stability); the controller adds V>0 to trade a little stability slack for throughput. Backpressure's classic result (throughput-optimal admission without knowing arrival rates) maps to the v2 family's 'no reward in observation, must self-compute' constraint — backpressure needs only queue LENGTHS (the live `ready` feature), not rates or the absent server_rows_per_forward/rtt sentinels. The drift-plus-penalty V parameter is the SAME stability↔throughput knob as C5's S* and the RMAB's λ, unifying the three.

**DOF moved (depth>1 live; approximate per caveat).** DOF-B and DOF-depth/D: the drift rule denies a thread whose holding lowers the weighted queue drift (letting its backlog build = fatter next chunk, DOF-B), which in the live regime reduces its outstanding chunk count (DOF-depth). Through the shared service it moves DOF-F (the §6 drain batch). Approximate reading (caveat): backpressure's queue Q_t is the `ready` backlog that, in the depth>1 regime, the chunk_floor leaves partly un-drained (:694) — the held remainder IS the queue the Lyapunov function penalizes; at depth==1 drain-all empties Q_t every issue and the queue never builds, so the backpressure signal is degenerate — the live regime is what gives this controller a non-trivial queue to manage.

**What the model adds vs v2.** The model reveals that §6's own non-divergence proof is ALREADY a Lyapunov/drift argument (the absorbing-ceiling, two-S-independent-clamps result), so the natural controller for this plant is the one cast in the SAME mathematics — drift-plus-penalty — which the unscaffolded v2 never reached (it stayed in bandit/RL vocabulary). This gives a controller that (a) needs only the LIVE `ready` queue lengths, sidestepping every sentinel-0 channel (server_rows_per_forward, rtt_us) the v2 candidates wished for; (b) inherits backpressure's throughput-optimality-without-arrival-knowledge, matching the 'reward is not given' constraint; (c) unifies with C5 (S*) and the RMAB (λ) under one stability↔throughput parameter V, suggesting these are three reductions of ONE constrained-optimization object — a structural insight the model's §6 makes visible and the v2 family's analogy-only treatment could not.

## NEW: Joint producer-depth × server-width controller (two-sided DOF-B↔DOF-F co-design; the gate as one half of a Stackelberg/team problem)  🆕 (new — surfaced by the formal lens)

**Formal object.** The model's §1 states the chunk_floor DESIGN INTENT explicitly: the producer supplies overcommit DEPTH (DOF-B / S_min) and the server controls forward WIDTH (the server-side floor θ that re-assembles the D×T chunks into one wide forward). The issue gate is therefore only HALF a controller; the full formal object is a TEAM (or Stackelberg) control problem over TWO coupled actuators: the per-thread gate (moving DOF-depth/DOF-B) and the server's min_forward_rows θ (moving the §6 drain WIDTH / DOF-F directly). State = the §6 plant; actions = (allow vector, θ); dynamics = §3.3 with BOTH the producer arrival shaping AND the server drain shaping live; objective = §6 throughput. depth>1/D-live (caveat — and this candidate is the one that takes the chunk_floor regime's stated purpose at face value).

**Reduction / algorithm fit.** Reduction to a TWO-PLAYER TEAM / STACKELBERG game (or, if both are one designer's knobs, a bilevel optimization) on the §6 queue: the server (leader) sets θ committing to a drain width; each producer thread (follower) gates to supply chunks that the committed θ will coalesce well; solve by the leader optimizing θ anticipating the followers' gate response. The model MOTIVATES this uniquely: §1 says the gate ALONE 'floods the single-threaded server with small messages it under-coalesces (the revert's convoy)' — i.e. the single-sided gate's Nash outcome is the CONGESTED convoy the producer-floor negative result (the recent commit '3928c21 docs(stall): producer-floor negative result') recorded; the model's §6 + §3.3 say throughput is set by the SERVER's drain width, which the gate cannot touch — so the gate is provably insufficient ALONE and must be co-designed with θ. The reduction's payoff: the v2 family optimizes one actuator against a fixed plant; the model shows the plant's throughput-determining knob (server width, DOF-F) is a SECOND actuator, and the right formal object is joint. This is the model's strongest 'broadening' contribution — it says the entire v2 online family is solving a sub-problem of a 2-actuator team problem.

**DOF moved (depth>1 live; approximate per caveat).** ALL of DOF-depth/D, DOF-B (producer side, via the gate) AND DOF-F directly (server side, via θ) — the only candidate that moves DOF-F by a DIRECT actuator rather than indirectly through arrival shaping, plus DOF-S (a wider committed θ changes the padded forward shape, §3.2). Approximate reading (caveat): the depth>1 regime is the PRECONDITION for this candidate to be meaningful — the producer half (chunk_floor depth) and the server half (re-coalescing width) only have a handoff to co-design when depth>1 un-pins DOF-B; at depth==1 the producer already drain-alls and θ has nothing to re-assemble, so the two-sided problem collapses to the one-sided §6 self-clocking the model proves.

**What the model adds vs v2.** This candidate is INVISIBLE without the model: the v2 family treats the server as a fixed plant and optimizes only the gate, but SYNTHESIS §1 + §6 + §3.3 prove the throughput-determining knob is the SERVER drain width (DOF-F), making the gate a single follower in a two-actuator team problem. The model thereby (a) explains the producer-floor negative result the branch recorded — the gate alone produces the under-coalesced convoy because it cannot move the width knob §6 says sets throughput; (b) reframes the controller goal from 'optimal gate' to 'optimal (gate, θ) pair', the chunk_floor's own stated intent (§1); (c) connects to the live `server-floor-ab` work in MEMORY (the leaf-eval server coalescing floor) — the model says the online issue-gate controller and the server-coalescing floor are the TWO HALVES of one team problem and should be co-tuned, not designed in isolation. This is the broadening the lens asks for: from a one-actuator bandit family to a two-actuator team-control formulation grounded in the model's §1 design-intent statement.
