# Static / engineering-informed — formal supplement & prior art

> Family scope: STATIC / engineering-informed (Phase 3, docs/design/issue-controller-policy-v2/03-design-static.md): hand-designed feedback / setpoint / threshold-hysteresis / rate-limiter / AIMD / discrete-event controllers; each maps the IssueFeatures dict to the per-thread DENY-only binary issue gate allow[tid] in {0,1}, consulted at runner_wire_batched.cpp:746-747 (gate = inflight_msgs < D && may_issue), with the ungated forced flush (line 748-749) as the liveness floor.

## C0 — All-allow identity (null controller / control arm)

**Formal object.** Under the model, C0 is the UNCONTROLLED open-loop plant itself, not a controller: action a_t = 1^T fixed, so the gate predicate inflight_msgs<D && may_issue collapses to inflight_msgs<D and the system runs the bare pipelined arm. Formally it realizes regimes R0/R1/R2/R3 (SYNTHESIS §5) verbatim with no exogenous input on the action channel. State = the model's free joint process (per-slot delta_park source nondeterminism §3.1 x server service-shape band S §3.2) under the depth-1-OR-(depth>1 under chunk_floor) closed loop §3.3; dynamics = the self-clocking negative-feedback batch-size map of §6; objective = whatever throughput that uncontrolled fixed point delivers. It is the reference trajectory every other candidate is a perturbation of.

**Reduction / algorithm fit.** No algorithm to fit — C0 IS the plant's free evolution, so the 'reduction' is the identification a_t=1 => the controlled MDP degenerates to the autonomous Markov chain of §5/§6. Its load-bearing formal role is as the BASELINE POLICY in the policy-improvement / regret sense: it is the b(s)=1 behavior policy against which label_counterfactual_gate_advantage (02-derived-features) measures any other policy's advantage A^pi = J(pi) - J(C0). Because §6 proves the autonomous fixed point is bounded and non-divergent for all N (absorbing ceiling min(T*K, max_batch+K-1)), C0's J is a finite, well-defined operating point — the model GUARANTEES the comparator exists and is stable, which an unscaffolded treatment asserts only empirically.

**DOF moved (depth>1 live; approximate per caveat).** None — C0 moves no DOF; it lets all of DOF-park, DOF-B, DOF-depth/D, DOF-F, DOF-S evolve freely at their autonomous fixed point. (Approximate/heuristic per the caveat: even 'moves nothing' is regime-conditional — under chunk_floor OFF this is the §0 depth-1 fixed point the model proves; under chunk_floor ON it is the depth>1 autonomous fixed point the model does NOT re-derive, so C0's own baseline shifts with chunk_floor and that shift is outside the model's settled scope.)

**What the model adds vs v2.** The model upgrades C0 from 'do-nothing reference' to 'a PROVABLY bounded, well-defined fixed point' (§6: divergence UNSAT at N in {1,8,33,75,200}; absorbing ceiling reached in one cycle). It also names the one way the baseline itself can move: R6 (RCVTIMEO abort) firing-probability rises with offered load (§5 R6, §6 point 4), so a fair A/B must hold the operating point below the S~timeout_ms knee or C0 and its challenger are compared across a liveness cliff. v2 treats C0 as trivially safe; the model says it is safe AND quantifies the one regime (R6) where 'safe' has an N/load-dependent edge — the same edge any deny-only controller inherits and can only ever IMPROVE (denying reduces offered load, pushing R6 further away).

## C1 — Per-thread coalescing-floor hysteresis (bang-bang deadband) gate

**Formal object.** The per-thread gate becomes a THRESHOLD POLICY on a per-thread controlled birth-death chain whose state is the in-flight message depth j_t in {0,...,D} (DOF-depth, reactivated and LIVE under chunk_floor per the caveat) coupled to the un-submitted backlog W_t=ready[t]. Action allow[t]=g[t] in {0,1}; dynamics (chunk_floor ON, the inquiry regime): on allow at a refill, j jumps to min(D, j + floor(W/S_min)) emitting floor(.) chunks of S_min=32 rows each (issue() line 717 break; the burst runs because may_issue is read once per refill, line 746, then the while-loop fires repeatedly); on deny, j holds and W accumulates via the post-recv resume loop (§3.1 C2 reply-causality); births are recv completions (j-=1, line 765). Backlog W is fed by the superposition of up-to-K delta_park streams (§3.1). Objective: maximize pool leaf_rate (the §6 throughput). The deadband on r[t]=ready/K_est is exactly a hysteresis band on W_t, i.e. on the model's DOF-B coalescing degree the next allowed burst will achieve.

**Reduction / algorithm fit.** This is a textbook RELAY/bang-bang controller for a binary actuator, and the model says WHY it is the right primitive: §3.3 + §6 make the plant a negative-feedback, monotone (B up => S up => throughput up to the cap) SISO-per-thread system, and for a monotone plant with a binary actuator the optimal stationary policy IS a threshold/switching policy (a hysteresis band removes chatter on a quantized actuator — Tsitsiklis-style monotone-MDP threshold-optimality applies once the per-thread chain is monotone in W). The Karp-style transformation: cast each thread as an M/G/1-type batch-admission control where admitted batch size = S_min*chunks; C1's r_lo/r_hi map onto the classic two-threshold (s,S)-style hysteretic admission band. The reduction is exact in the inquiry regime; it FAILS to be interesting only under chunk_floor OFF, where §0 (depth-1, D-dead) makes the burst length always 1 and the forced flush overrides every deny — i.e. the model's superseded spine is exactly the condition under which C1's grip vanishes, so the caveat REVERSES v2's lead 'Con'.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D (PRIMARY, now live): the deadband directly sets how high j_t is allowed to climb — deny below r_lo truncates the per-thread burst at low depth, allow above r_hi lets it climb toward D. DOF-B (coalescing degree): holding W below r_hi banks ready rows so the next allowed issue's chunk-count floor(W/S_min) is larger (more S_min-chunks => more DOF-B per refill window). DOF-F (self-batching, approximate): more outstanding messages from this thread arriving during a server forward => fatter next server drain (§3.3 'no recv during a forward'). [All flagged approximate per the caveat: the model does not re-derive the depth>1 batch-size fixed point, so the exact j-trajectory C1 induces is heuristic, not proven-bounded the way §6 proves the depth-1 one.]

**What the model adds vs v2.** Three corrections to v2. (1) v2's headline Con — 'at drain-all depth~1 the override (ii) makes the gate a no-op' — is the model's §0 superseded spine; in the ACTUAL inquiry regime (chunk_floor ON, DOF-depth reactivated) C1's deny is NOT a no-op, it is the burst-depth truncator, so C1 is materially stronger than v2 concedes. (2) The model names the correct state variable: not 'backlog' loosely but the (j, W) pair, and tells C1 to read inflight_saturation (j/D) alongside ready_backlog_norm — the v2 mechanism uses only W. (3) §6's monotone negative-feedback structure is the formal LICENSE for bang-bang-over-PID that v2 argues only by 'binary actuator => relay': monotone plant => threshold policy is optimal, not merely convenient. (4) The override (ii) 'inflight==0 => force allow' is revealed as the forced-flush boundary of the chain (the j=0 reflecting barrier), not an ad-hoc safety bolt.

## C2 — Pool-level PI controller on mean coalescing degree (setpoint on S)

**Formal object.** A single-input single-output regulation problem: track the pool's realized coalescing degree S_now (rows/message, = coalesce_degree_inst aggregated) to a setpoint S*. Under the model the controlled output S_now is exactly the model's mean_rows_per_msg telemetry (SYNTHESIS §6 falsifiable prediction, closed form P_total/(Sigma ceil(P_t/(N*base)) + Sigma r_t)) — but the caveat MOVES it: that closed form is the depth-1 drain-all form; under chunk_floor ON each message is pinned at S_min=32 by construction (issue() line 717), so per-MESSAGE S is clamped to ~S_min and the genuine free variable is rows-per-FORWARD (the server's cross-message coalescing, DOF-F) not rows-per-message. State = PI integrator (1 scalar) + plant backlog; action = scalar u discretized to a deny-count across T threads; dynamics = §6 self-clocking feedback; objective = drive S (or, corrected, rows/forward) to S*.

**Reduction / algorithm fit.** Standard PI regulation of a first-order-plus-deadtime plant — applicable AFTER a transformation, but the model exposes a structural mismatch the v2 pass only half-saw. The transformation that WOULD make PI fit: treat the pool as one aggregate queue whose 'fill rate to a forward' is the manipulated variable; PI on the integrated error is the I-controller that removes the steady-state offset a pure threshold (C1) leaves. BUT the model supplies the disqualifier concretely: (a) under chunk_floor the per-message S is pinned at S_min, so PI-on-rows-per-message is regulating an output the producer floor already clamps — the loop has near-zero plant gain on its nominal output; the only live output is server rows/forward, which is server_rows_per_forward = SENTINEL-0 (01-atomic-features), so the PI loop is OPEN (blind) exactly as v2's Con (2) says, and the model explains WHY (the §3.3 single-thread serialization is the coalescing engine, and its degree is the unobserved channel). (b) §6 point: the actuator saturates against the UNGATED forced flush (line 748) — the model's R3->R1 forced-flush firing means e never closes when the system is arrival-bound, the classic integrator-windup-under-saturation C2's Con names; the model pins the saturation source precisely (the liveness floor, not a generic actuator limit). Verdict: PI fits the formal object only with rows/forward observable; on today's signals the reduction degrades to C1-with-windup.

**DOF moved (depth>1 live; approximate per caveat).** DOF-B (coalescing degree, nominal target — but largely pre-empted by the S_min producer floor under chunk_floor). DOF-depth/D (the actual lever the discretized deny-count moves: ranking-and-denying top-p threads truncates their bursts, same mechanism as C1 but pool-coordinated). DOF-F (the TRUE objective output — server rows/forward — but unobservable). [Approximate per caveat: with S_min pinning per-message size, the model says C2's nominal DOF-B target is the wrong knob in the inquiry regime; the live one is DOF-F, which it cannot see.]

**What the model adds vs v2.** The model SHARPENS C2's two Cons into a structural verdict. v2 says 'S* is set blind because server_rows_per_forward is sentinel-0'; the model says more: under chunk_floor the per-message degree is CLAMPED at S_min by the producer floor (§ caveat / issue() line 717), so C2 is regulating a near-constant — the integral action it sells over C1 has almost no output to integrate. The genuinely free, throughput-relevant degree is server rows/FORWARD (DOF-F, the §3.3 cross-message coalescing), which is exactly the sentinel-0 channel. So the model REFRAMES C2: it is the right shape of controller aimed at the wrong (clamped) output; its value is contingent on wiring server_rows_per_forward, at which point it becomes a regulator on DOF-F — and then it is essentially C6's observable, making C2-on-rows/forward and C6 converge. v2 missed that the producer floor pre-empts C2's nominal target.

## C3 — Per-thread token-bucket / leaky-bucket issue rate limiter

**Formal object.** Per-thread admission control: the gate is a (rho, C_burst) token bucket regulating the ISSUE-instant process, i.e. it shapes the point process of allowed refill-bursts. Under the model the controlled object is the per-thread arrival process INTO the depth-chain (the rate at which thread t is permitted to push its floor(W/S_min) chunks). State = token level tok[t] (continuous) + depth j_t; action = allow iff tok>=1; dynamics = bucket ODE (refill rho, drain on detected issue via msgs-delta) composed with the §3 depth-1-or-depth>1 closed loop; objective = throughput. This is precisely the classic (sigma, rho) leaky-bucket regulator of a fluid source, mapped onto DOF-park's offered process (§3.1, the superposition of K delta_park streams) as the thing being rate-shaped.

**Reduction / algorithm fit.** Direct isomorphism to network-calculus (sigma,rho)-regulation / GPS-style traffic shaping — the named algorithm fits the FORM but the model exposes a units mismatch the v2 Con states and the model now grounds. The transformation: a token bucket bounds the cumulative issues A(t) <= rho*t + C_burst (an arrival-curve constraint). For this to control THROUGHPUT (the objective) you need a deterministic relationship issues->rows->forward-utilization; the model says that relationship is NOT a function of issue-rate alone — §3.2/§3.3 make service depend on the forward SHAPE (padded), and §6 makes batch size a negative-feedback fixed point of offered LOAD, not of issue-rate. So rate-shaping is an indirect handle on batch SIZE (v2's Con), and the model says why precisely: under production padmax, S is ~row-count-INVARIANT below max_batch (DOF-S, §3.2), so spacing issues changes WHEN rows arrive but the per-forward cost is flat until the cap — the bucket's rho buys batch size only through the §6 feedback (slower issues => more accumulation per allowed burst => fatter chunks-count), a second-order effect. Cleaner reduction available: re-cast the bucket to meter ROWS not MESSAGES (refill rho in rows/sec, consume gathered.size()) — then it is a (sigma,rho) regulator on the rows-offered process, directly on DOF-B, and the model's monotone-throughput-in-B (§6) gives it a target. That row-metered transform is the model's contribution; message-metered (v2's form) is the looser fit.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D (spacing permitted bursts caps how fast j climbs to D — a rate-limit on the birth process of the depth chain). DOF-B (a slower rho banks more W between issues => larger floor(W/S_min) chunk-count per allowed burst). DOF-park (the bucket directly shapes the §3.1 offered process — it is the only candidate that acts on DOF-park's TIMING rather than its threshold). [Approximate per caveat: the depth>1 accumulation the bucket controls is the un-pinned-DOF-depth regime the model does not re-derive; the bucket's effect on the unproven depth>1 fixed point is heuristic.]

**What the model adds vs v2.** The model REPOSITIONS C3 from 'rate cap on messages' to 'arrival-curve regulator on the offered ROW process (DOF-park/DOF-B),' and tells it to meter rows not messages — because §3.2 (padmax S row-invariant below cap) means message-rate is nearly decoupled from per-forward cost, so a message-metered bucket has weak throughput leverage exactly as v2's Con guesses, while a row-metered bucket lands on the monotone DOF-B the §6 feedback rewards. It also confirms v2's issue-detection worry with a mechanism: a thread publishes per-reply not per-tick (§ the publish site line 784, 01-atomic-features), so msgs-delta under-counts a multi-chunk burst (chunk_floor emits several messages between publishes) — the model says the consume event should be leaves-delta/S_min (chunk count), not msgs-delta, to meter the actual burst. v2 named the detection hazard; the model gives the corrected estimator.

## C4 — AIMD-on-throughput per-thread gate (congestion-control analogue, RTT-free)

**Formal object.** A model-free stochastic-approximation HILL-CLIMBER on the per-thread permit level w[t], driving the pool to the throughput-maximizing operating point of the §6 fixed point WITHOUT a model of it. State = per-thread w[t] (a continuous congestion-window proxy) + R_max best-seen throughput + trailing-window estimate; action = threshold/Bernoulli(w) -> allow bit; dynamics = AIMD recurrence (w += a on no-event; w *= b on a stall event) layered on the §6 negative-feedback plant; objective = argmax over operating points of realized_dps_window. Formally it treats the unknown map (gate-aggressiveness -> throughput) as a noisy black box and does derivative-free ascent — a bandit/zeroth-order-optimization view of selecting where on the §6 batch-size curve to sit.

**Reduction / algorithm fit.** Two candidate reductions, and the model picks the loser for the RTT-free variant. (a) AIMD-as-distributed-optimization (Kelly/Low NUM): AIMD provably converges to the maximizer of a concave utility MINUS a congestion-price integral — this fits IFF there is a genuine congestion PRICE signal that rises with over-driving. The model says the true price is server queueing delay (§3.3 single-thread serialization => rising latency under load, the R6 approach), whose observable is rtt_us = SENTINEL-0 (01-atomic-features). So the NUM reduction's price channel is dead, and C4 must synthesize the event from a differenced throughput proxy (realized_dps_window), which §3.2's nondeterministic S band (S>0 free in a shape-determined band) makes intrinsically noisy — the model PROVES the signal C4 climbs is noisy by construction (S is free latitude, not a constant), grounding v2's 'chasing measurement noise' Con as a structural fact, not a guess. (b) Zeroth-order ascent / 1-D bandit on a 1/T-resolution arm: with T<=4 the action grid is coarse and the reward variance is the §3.2 service-band variance — classic bad regime for stochastic ascent. Verdict: AIMD's FORM fits a NUM problem but its required price signal is exactly the model's sentinel-0 channel, so the RTT-free variant is climbing a §3.2-noisy surrogate — formally underdetermined. The reduction succeeds only as C5 (next), where the price signal is real.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D (w sets per-thread burst-depth aggressiveness, sawtoothing j's ceiling up and down). DOF-F (the throughput signal R it climbs IS the server self-batching output, §3.3 — C4 is implicitly hill-climbing on DOF-F's realized effect, just via a noisy proxy). DOF-S (the stall events it reacts to are §3.2 service-time excursions). [Approximate per caveat: the sawtooth rides the depth>1 dynamics the model leaves free; AIMD stability on the depth>1 fixed point is not derived.]

**What the model adds vs v2.** The model converts v2's 'noisy-reward' hand-waving into a structural impossibility-ish statement: the reward surface IS noisy because §3.2 leaves S as bounded nondeterminism in a shape band (Open Question 2: the interior fixed point's exact location/approach is unpinned), so a differenced-throughput AIMD is ascending a surface the model itself declines to pin — it cannot converge tighter than that latitude. It also identifies the missing price: the NUM/Kelly reading wants the congestion price = server queueing delay (§3.3 serialization, the R6 latency ramp), which is the rtt channel — so the model REDIRECTS C4 to C5 (the price exists once rtt is wired) rather than the throughput-difference proxy. v2 says 'strictly better once RTT is wired'; the model explains the formal reason (the price channel the NUM convergence proof requires is rtt, not dps).

## C5 — RTT-driven congestion controller (AIMD / Vegas-style) — GATED on rtt wiring

**Formal object.** Delay-based congestion control regulating per-thread in-server queueing to a band [alpha,beta] above a measured baseline RTT_min. Under the model this is the CANONICAL formal object for this plant: the single-threaded server (§3.3 'drain k+1 cannot begin until scatter k completes; forwards never overlap — THIS IS THE COALESCING ENGINE') IS an M/G/1-type queue, and per-thread RTT = sojourn time = the Little's-law work-in-system proxy rtt_us*inflight[t] (02-derived-features rtt_derived_block). State = per-thread permit + RTT_min estimate + queueing estimate (RTT - RTT_min); action = Vegas band test -> allow/deny; dynamics = the §3.3 closed loop where the server's output timing feeds back into its input timing (depth-1-or-depth>1); objective = keep utilization high (full forwards) while sojourn stays below the R6 timeout knee (§5 R6, §6 point 4).

**Reduction / algorithm fit.** Near-ISOMORPHISM to TCP-Vegas / BBR-lite delay control of an M/G/1 queue — the cleanest fit in the family, and the model certifies the queue structure that makes it apply. Transformation: identify (server) with the bottleneck link, (per-thread reply RTT) with the link's standing-queue delay, (RTT_min) with the propagation delay = the zero-queue forward cost S_warm at the prevailing shape (§3.2). Then Vegas's invariant 'keep alpha < (RTT-RTT_min)*rate < beta' is exactly 'keep this thread's contribution to the server backlog in a band' — and §6's negative-feedback fixed point is the queue stability the delay-controller's convergence proof assumes. The model ADDS the precise hazard to the reduction: RTT_min is not constant — §3.2's compilation discontinuity (S_cold >> S_warm on first sight of a shape) and the R2 overshoot one-time S_cold per width (§5 R2, §6 point 2) mean the 'propagation delay' baseline SHIFTS when N/load crosses T*K>max_batch; a naive RTT_min tracker will mistake a cold-compile spike for congestion and back off spuriously. So the reduction is exact for the steady shape and needs a cold-compile-aware RTT_min (e.g. percentile-floor that ignores one-time spikes) — a concrete correction the model's DOF-S/R2 structure dictates.

**DOF moved (depth>1 live; approximate per caveat).** DOF-depth/D (the Vegas permit caps per-thread burst depth to hold sojourn in band — the principled version of C1's threshold). DOF-S (RTT IS the service-time observable; C5 closes the loop the model leaves open between source and sink, §3.3). DOF-F (holding per-thread queueing in band shapes how many messages coalesce into each forward — directly the server self-batching degree). DOF-depth is the actuated one; DOF-S/DOF-F are the observed/affected ones. [Approximate per caveat: depth>1 is the regime where multiple per-thread messages are in flight, so rtt_us*inflight as work-in-system is only meaningful once depth>1 — the model's §0 depth-1 made inflight in {0,1} and the Little's-law proxy near-degenerate; the caveat's reactivation of DOF-depth is precisely what makes C5's queueing estimate non-trivial.]

**What the model adds vs v2.** The model PROMOTES C5 from 'the right congestion controller' (v2's intuition) to 'the controller whose formal object — delay control of the §3.3 single-thread M/G/1 coalescing queue — is exactly the model's central mechanism,' and supplies the two things v2 lacks: (1) the work-in-system estimator rtt_us*inflight is only informative under depth>1 (the caveat's live regime), which is WHY the v2-era depth-1 spine made RTT-control look optional and the inquiry's depth>1 regime makes it central; (2) the RTT_min baseline must be cold-compile-aware because §3.2/R2 inject one-time S_cold spikes that are NOT congestion (the overshoot bundle of O(N) one-time compiles, §6 point 2) — a Vegas baseline that floors over those avoids the spurious backoff v2 doesn't anticipate. It also confirms the gating is real and explains the silent-failure shape: on the dead channel C5 reads RTT=0 => queueing=0 => always-allow == C0, the §5 R6 'never congested' degenerate the v2 Con flags.

## C6 — Discrete-event / queueing-network (Petri-net-style) threshold scheduler — GATED on server channel

**Formal object.** A model-based scheduler that maintains an explicit queueing/Petri-net replica of the plant and gates issues to hold the modelled server batch occupancy in a band B_srv*. This is LITERALLY the model: SYNTHESIS §2 IS the Petri-net/assume-guarantee structure (per-thread ready/inflight places, the server drain transition), §4 is the marking-evolution DOF table, §6 is the reachability/boundedness analysis. State = the marking (ready[t], inflight[t] per thread = j_t, plus the modelled server assembling-batch place); action = enable thread-t issue transition iff firing keeps the server batch in B_srv*; dynamics = the §3.3 token-flow with the §3.2 service-shape timing; objective = max server forward utilization (rows/forward toward max_batch without crossing the R6 latency knee). C6 is the only candidate whose formal object the model GIVES wholesale rather than one the model must be transformed into.

**Reduction / algorithm fit.** This is not 'reduce C6 to a known algorithm' — C6 instantiates the model as a controller, so the reduction is the OTHER direction: the model (§2 places/transitions, §4 DOFs, §6 absorbing ceiling min(T*K, max_batch+K-1)) supplies C6's net for free, and the known algorithm that then applies is OPTIMAL CONTROL OF A CONTROLLED MARKOV/PETRI NET — specifically, because §6 proves the marking is bounded (the reachable set is finite: depth in {0..D}, server batch in {1..max_batch+K-1}), the gated reachability graph is FINITE, so the operating-point placement is a finite MDP solvable exactly by value/policy iteration or by an LP over the stationary marking distribution (the LP-of-an-MDP / Kingman-bound reasoning v2 gestures at). The transformation that makes it tractable: §6's two S-independent clamps collapse the state to (per-thread depth vector, server-batch occupancy), and §0's depth structure (under chunk_floor: depth in {0..D} with S_min-quantized jumps) discretizes the server place to multiples of S_min — a small finite chain at T<=4, D=8, S_min=32. The disqualifier the model makes concrete: the controlled output (server_rows_per_forward) is the marking of the server place, which is SENTINEL-0 in-band (01-atomic-features, §2.4 A-side has no server->producer channel) — so the net runs OPEN-LOOP (the model's own Open Question 1: which arm runs is a file outside its world; here, the server marking is outside the producer's observation), exactly v2's Con. C6 is the model-as-controller, executable closed-loop only once the server place is observable.

**DOF moved (depth>1 live; approximate per caveat).** ALL of them jointly, by construction — C6 is the only candidate whose objective is multi-thread/whole-marking: DOF-depth/D (per-thread issue transitions set the depth marking), DOF-B (chunk count per issue), DOF-F (the server-place occupancy IS the self-batching degree it targets), DOF-S (the service-shape band sets the forward cost it optimizes against), DOF-drain (it reasons about the §3.3 greedy drain boundary directly). [Approximate per caveat: the model's §6 boundedness proof that makes the net FINITE was derived for depth-1; under chunk_floor depth>1 the reachable marking set is larger (depth up to D with S_min jumps) and the model does NOT re-prove its boundedness — so C6's claim 'the net is finite and exactly solvable' is heuristic in the inquiry regime until the depth>1 ceiling is re-derived.]

**What the model adds vs v2.** The model collapses C6's biggest v2 cost — 'build a queueing model, it's assumption-laden' — because the model IS that queueing model, validated (Z3-confirmed §2/§3, forward-derived from code): C6 does not need to be invented, it needs to be INSTANTIATED from SYNTHESIS §2/§4/§6. The model also tells C6 its state is FINITE and BOUNDED (§6 absorbing ceiling), upgrading 'model-based scheduler' to 'finite-MDP exactly solvable by value iteration / stationary-LP' — a much stronger claim than v2's 'place the operating point analytically.' And it pins the exact blocker: the server-place marking = server_rows_per_forward is the one §2.4 channel that does not exist, so C6 is the model running open-loop on the producer side — the same gap C2-on-rows/forward and C5's price hit, all three pointing at wiring the server->producer metrics channel as the single enabling step. v2 lists C6 as heaviest/most-assumption-laden; the model says it is the LEAST assumption-laden (the assumptions are already discharged) but the MOST observation-starved.

## NEW: C7 — Per-thread Whittle-index restless-bandit gate (indexable burst-depth scheduler)  🆕 (new — surfaced by the formal lens)

**Formal object.** Cast the T-thread pool as T coupled RESTLESS BANDITS competing for one shared server (the §3.3 single-threaded forward, the bottleneck resource). Each thread t is an arm with internal state s_t = (j_t in-flight depth, W_t backlog) evolving as the per-thread controlled birth-death chain of C1's formal object; the binary gate allow[t] is the bandit's 'activate' action (issue) vs 'rest' (deny); the shared constraint is the single server that can productively absorb only one fat forward at a time (§3.3 serialization). Reward = marginal throughput contribution of activating thread t this tick (its leaf_rate increment). Objective = maximize pool leaf_rate subject to the implicit one-server coupling — a restless multi-armed bandit with an activation constraint set by server occupancy.

**Reduction / algorithm fit.** Karp-style reduction to the WHITTLE-INDEX heuristic for restless bandits. Transformation: relax the hard 'one server' coupling to a LAGRANGIAN per-issue PRICE lambda on offered rows (the model already threads lambda as a live per-call argument elsewhere in the codebase — the env/Policy seam's lambda — so a price-per-issue is idiomatic here); under the relaxation each thread DECOUPLES into an independent per-thread MDP (the C1 birth-death chain priced by lambda), and IF that per-thread chain is INDEXABLE (the deny-set monotonically grows as the price rises — plausible because §6 makes per-thread throughput monotone/concave in burst-depth, the standard indexability sufficient condition), the Whittle index W_t(s_t) exists and the gate is: allow the threads with the highest indices up to what the server can absorb, deny the rest. This is the principled, provably-near-optimal version of C2's ad-hoc 'rank threads by submit_pressure and deny top-p' — the rank key BECOMES the Whittle index instead of a heuristic pressure. ## NEW: C8 — Cold-compile-shielding gate (one-time-spike-aware burst guard, DOF-S/R2 derived)  🆕 (new — surfaced by the formal lens)

**Formal object.** A guard whose state is a learned set of 'seen' forward shapes (widths) and whose action denies discretionary issue precisely when an allowed burst would push the server's drained batch across a NOT-YET-COMPILED width boundary (the R2 overshoot, §5 R2 / §6 point 2). Formal object under the model: the controlled variable is the server's drained row count B, which the §3.2 service map S(B) penalizes with a ONE-TIME S_cold(B) per distinct unseen width (the compilation discontinuity, §3.2; the O(N)-sized bundle of one-time compiles smeared across a run, §6 point 2). State = the empirical set of widths whose S_cold has been paid (a small learned set, |.| <= K-1 distinct overshoot widths per §5 R2); action = deny the marginal issue that would cross into an unpaid width during the transient, allow once that width is amortized; objective = avoid re-triggering avoidable cold compiles that, while one-time, raise per-reply latency toward the R6 timeout knee (§5 R6).

**Reduction / algorithm fit.** Reduction to OPTIMAL STOPPING / a deterministic-then-stochastic admission guard, NOT a continuous controller — and the model is the SOLE source of this candidate. The transformation: §3.2 says the forward cost is piecewise-flat in B under padmax UNTIL the soft-cap overshoot, where crossing max_batch by one whole message yields an unpadded fresh shape B in (max_batch, max_batch+K-1] paying S_cold once (DOF-overshoot, §4; reachable exactly when T*K>max_batch, §5 R2). The gate's decision 'allow the burst that overshoots NOW vs hold' is a stopping problem: pay the one-time S_cold now (and amortize it over the rest of the run) vs defer and keep the batch under the cap. Because the spike is ONE-TIME PER WIDTH (§6: a single cached jax.jit, cannot lift its own ceiling, no positive-feedback divergence), the optimal policy is a simple threshold on 'have I paid this width yet': allow overshoot freely once paid (it's now S_warm), guard only the first crossing during a latency-sensitive window. The model PROVES this guard is bounded-benefit and cannot backfire (the spike is non-recurring and the §6 feedback stays negative), so the reduction is to a finite, provably-terminating amortization schedule — there is no analogous algorithm in the v2 pass because v2 had no model of the compile discontinuity at all.

DOF-overshoot (§4: it gates exactly the whole-last-message overshoot past max_batch). DOF-depth/D (secondarily: truncating a burst keeps T*K offered rows from crossing the overshoot threshold). [Approximate per caveat: the overshoot threshold T*K>max_batch and the K-1 overshoot magnitude were derived under §0's depth-1 / drain-all rows-per-message; under chunk_floor each message is S_min-pinned, so the relevant overshoot is the SERVER drained batch across multiple S_min messages, not one drain-all message — the model's R2 must be re-read with the server coalescing S_min-chunks, which the model does not do; flag C8's threshold arithmetic as heuristic in the depth>1 regime.]

**What the model adds vs v2.** This candidate is ENTIRELY a model contribution — the v2 static pass has no concept of the forward-shape compilation discontinuity because the unscaffolded treatment never modeled the server's §3.2 service map.
