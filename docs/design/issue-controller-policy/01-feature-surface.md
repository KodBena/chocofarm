# Phase 1 — Atomic feature surface

FEATURE SURFACE for the IssueController policy (Phase 1). Files: producer C++ at cpp/src/runner_wire_batched.cpp (run_episodes_wire_pipelined, the worker lambda, the publish call at line 783-784); the controller/marshalling at cpp/include/chocofarm/issue_controller.hpp (IssueFeatures struct lines 46-56, Slot atomics lines 131-137, publish() lines 91-98, tick_() marshalling lines 141-165); the wire transport at cpp/include/chocofarm/wire_leaf_pool.hpp (no submit-time tracking in inflight_); the eval harness at cpp/src/wire_ab_bench.cpp (--controller-cadence-ms wiring lines 256-264, identity policy today); the empirical motivation at docs/design/stall-investigation/server-gen-floor-result.md.

THREE STRUCTURAL FACTS that frame everything:

(1) THE PUBLISH POINT. Exactly one call site: runner_wire_batched.cpp:783-784, inside the pipelined drain's recv loop, AFTER each recv_batch()+refill(), guarded by `if (controller != nullptr)`. Signature today: publish(tid, inflight_msgs, ready_count(), my_msgs, my_leaves) — the 6th param mean_rtt_ms defaults to 0.0 and is never passed. Cadence = per recv_batch reply (one publish per resolved message), which is event-driven and FASTER than the controller's own cadence_ms tick; the control thread samples the latest published slot values on its slow cadence (issue_controller.hpp:108-115). All producer-published features are therefore "free" (one relaxed atomic store each) on a path already paid; the only non-trivial cost is ready_count()'s O(K) scan, already incurred unconditionally by refill() each iteration anyway.

(2) THE SCOPE MODEL. publish() is single-writer-per-tid (P9). The policy sees per-thread VECTORS (size T) + a couple aggregate/const scalars. "Aggregate" features (e.g. cross-thread sums, rows/forward) are NOT published per-slot; the controller's tick_() could derive simple aggregates (sum/mean over the per-thread vectors) for free, but anything the producer doesn't already compute per-thread is a gap.

(3) THE SERVER-SIDE GAP IS REAL AND HAS NO CARRIER. server_rows_per_forward (the in-flight DEPTH metric the whole investigation cares about, per server-gen-floor-result.md) is computed ONLY in Python, post-hoc, end-of-run: StageAServer keeps n_forwards/n_real_rows counters (stage_a_server.py:99-124) and prints SERVER_STATS mean_real_rows_per_fwd at subprocess exit (line 182). The wire reply frame is [ident, *envelope, resp] where resp = encode_response [ver][B][n_actions][B×(value,logits)] (inference_wire.hpp:19) — a pure value codec with NO telemetry field — and the corr-id envelope is round-tripped opaquely (frames[1:-1]). So there is NO existing per-reply carrier for a server metric; wiring it live needs EITHER a codec/frame extension (P7 wire-discipline change) OR a side-channel (a second ZMQ socket / a redis key the server writes and the producer reads on the control tick). This is the one true "gap_needs_channel" in the surface.

RTT IS ALSO A GAP (smaller). mean_rtt_ms has a sentinel slot in IssueFeatures but the producer cannot fill it without adding a submit-timestamp store: WireLeafPool::inflight_ maps corr-id -> vector<int> (slots only, no time); the only timestamps present are the bench-window measure_t0/t1_ns and the default-OFF CHOCO_EV SUBMIT/RECV event-log lines (event_log.hpp). Adding per-corr submit-time (a corr-id -> mono_ns map or a parallel field) is trivial-to-add and purely producer-local (no cross-process channel), so RTT is "trivial_to_add", distinct from the server-rows gap.

## Features

### `inflight (per-thread inflight_msgs)` — int · per_thread · **wired**
- **source:** runner_wire_batched.cpp local `int inflight_msgs` (line 522), incremented in issue() (line 726), decremented per recv (line 765); published as arg 2 at line 784. Marshalled into IssueFeatures.inflight (issue_controller.hpp:50,153).
- **publish cadence:** per recv_batch reply (one store per resolved message), sampled on the control tick
- **cost:** one relaxed atomic store; value already in a hot local
- **notes:** The count of outstanding (submitted, unanswered) coalesced messages this thread holds, in [0,D]. THE central control signal: the policy denies issue when inflight is healthy and allows when a thread is about to go idle. NOTE the publish at line 784 happens AFTER refill(), so inflight reflects post-refill depth (already topped back toward D), not the pre-reply trough — relevant to what the policy can infer about idleness.

### `ready (ready_count)` — int · per_thread · **wired**
- **source:** runner_wire_batched.cpp ready_count() lambda (lines 677-682), an O(K) scan over is_ready(s) = active && ts && ts->running && !submitted[s]; published as arg 3 at line 784. Marshalled into IssueFeatures.ready (issue_controller.hpp:51,154).
- **publish cadence:** per recv_batch reply
- **cost:** O(K) scan (K = N*ceil(pool_batch/threads), e.g. ~198 at N=9) — but refill() ALSO calls ready_count() at line 748 unconditionally, so under a controller the scan is paid once more per reply; cheap relative to a forward
- **notes:** Slots parked at a leaf but not yet submitted — the backlog the next issue would coalesce. Read jointly with inflight: high ready + low inflight = a thread starving the issue path (the convoy precursor); ready near 0 = the thread is keeping up. This is the natural per-thread signal a coalescing-floor-style policy thresholds on.

### `msgs (my_msgs, cumulative messages)` — long · per_thread · **wired**
- **source:** runner_wire_batched.cpp local `long my_msgs` (line 523), ++ in issue() (line 728); published as arg 4 at line 784. Marshalled into IssueFeatures.msgs (issue_controller.hpp:52,155).
- **publish cadence:** per recv_batch reply
- **cost:** one relaxed atomic store; value in a hot local
- **notes:** Cumulative monotone counter. Per se not a control signal; its value is as a RATE source — the policy (or tick_) differences successive samples over the control interval to get messages/s per thread, an issue-throughput estimate. Differencing must be done by the consumer (tick_ holds no history today).

### `leaves (my_leaves, cumulative leaves/rows sent)` — long · per_thread · **wired**
- **source:** runner_wire_batched.cpp local `long my_leaves` (line 523), += gathered.size() in issue() (line 727); published as arg 5 at line 784. Marshalled into IssueFeatures.leaves (issue_controller.hpp:53,156).
- **publish cadence:** per recv_batch reply
- **cost:** one relaxed atomic store; value in a hot local
- **notes:** Cumulative leaves coalesced into messages. leaves/msgs over an interval = mean rows-per-WIRE-MESSAGE (S) this thread is producing — the producer-side analog of the wire_summary mean_rows_per_msg (line 832). NOT the server's rows/forward (the server coalesces across messages+threads). Differenced with msgs by the consumer it yields per-thread mean coalescing degree, the closest in-process proxy for the depth the investigation tracks.

### `n_threads (T)` — int · global_const · **wired**
- **source:** IssueController ctor arg T_ (issue_controller.hpp:69-70), set from wcm.pool_threads at wire_ab_bench.cpp:258; written into IssueFeatures.n_threads each tick (issue_controller.hpp:143).
- **publish cadence:** constant for the run (set at construction, re-stamped each tick)
- **cost:** zero (a const member)
- **notes:** Read-only context: the vector length and the normalizer for any aggregate the policy forms over the per-thread vectors.

### `d_ceiling (D)` — int · global_const · **wired**
- **source:** IssueController ctor arg d_ceiling_ (issue_controller.hpp:71), set from max(1, wcm.max_inflight_msgs) at wire_ab_bench.cpp:258-259; written into IssueFeatures.d_ceiling each tick (issue_controller.hpp:144).
- **publish cadence:** constant for the run
- **cost:** zero (a const member)
- **notes:** The runner's FIXED safety ceiling, passed as a READ-ONLY feature (the controller never sets D). The policy uses it as the headroom reference: inflight/d_ceiling is the per-thread fractional fill the allow/deny decision modulates. Per the control contract D is the cap, the action is allow/deny.

### `mean_rtt_ms (per-thread reply RTT)` — double · per_thread · **trivial_to_add**
- **source:** GAP — sentinel today. IssueFeatures.mean_rtt_ms (issue_controller.hpp:54) and Slot.rtt_us (line 136) exist; publish()'s 6th arg (line 91) defaults 0.0 and is NEVER passed at the call site (line 784). To fill: store per-corr submit mono_ns at issue() (WireLeafPool::inflight_ holds only slots — wire_leaf_pool.hpp:147,242 — so add a corr->time map or a parallel store) and compute now-submit at recv_batch resolution (runner_wire_batched.cpp:763-765), EWMA into my_rtt, pass as arg 6.
- **publish cadence:** per recv_batch reply (RTT closes when the reply lands)
- **cost:** one extra map insert per submit + one lookup/erase + a subtract per reply; one relaxed store to publish. Purely producer-local — no cross-process channel. The CHOCO_EV SUBMIT/RECV pair already brackets exactly this interval (event_log.hpp), but is default-OFF and file-only, not a live feature.
- **notes:** Reply latency is the direct observable of server load/starvation: rising mean_rtt under low ready = the server is the bottleneck (good, keep depth); rising rtt WITH rising ready = a forming convoy. The single best in-process latency signal and trivial to wire — distinct from and easier than the server-rows gap.

### `server_rows_per_forward` — double · aggregate · **gap_needs_channel**
- **source:** GAP — the one true new-channel item. Computed ONLY in Python, post-hoc: StageAServer.n_forwards/n_real_rows counters (stage_a_server.py:99,122-124) printed as SERVER_STATS mean_real_rows_per_fwd at subprocess EXIT (line 182), aggregated by the Python harnesses (stage_b_poolbatch_sweep.py:261, overcommit_sweep.py:315). NO in-process C++ carrier: the reply frame is [ident,*envelope,resp] with resp = encode_response [ver][B][n_actions][B×(value,logits)] (inference_wire.hpp:19) — a pure value codec — and the corr-id envelope is round-tripped opaquely.
- **publish cadence:** would be whatever a new channel delivers (e.g. per-N-forwards, or polled each control tick)
- **cost:** NOT free. Needs EITHER a wire/codec extension (a server-stats field on the reply frame — a P7 cross-language-wire change, every side re-derives one authoritative layout) OR a side-channel (a 2nd ZMQ socket the server publishes rows/forward on, or a redis key the server writes and the producer reads on the slow control tick — cheap, off the hot path, and the control tick is already slow). The redis-key side-channel is the lowest-touch (no P7 codec change, server already has redis).
- **notes:** This is the depth metric server-gen-floor-result.md is ABOUT (the win is variance/idle-time, tracked via srv rows/fwd 108 vs 195). A policy that targets the server's amortization directly needs it live, but it is GLOBAL/aggregate (one server scalar, not per-thread) and the gappiest to wire. The sentinel (0.0, issue_controller.hpp:55) is honest until a channel exists.

### `messages-per-s per thread (issue rate)` — double · per_thread · **trivial_to_add**
- **source:** DERIVED, not a new producer field: difference the already-wired cumulative msgs across two control ticks / the elapsed cadence. Needs the consumer (tick_, issue_controller.hpp:141) to retain the previous sample + a tick timestamp (it holds neither today).
- **publish cadence:** per control tick (the differencing interval)
- **cost:** zero new producer cost; a few subtractions + storing prior msgs[]/leaves[] and a steady_clock::now() in the controller per tick
- **notes:** Listed because the policy almost certainly wants a RATE, and the raw cumulative msgs/leaves are wired but a level, not a flow. Trivial, but it is a CONTROLLER-side derivation (retain history), not a producer publish — calling it out so the policy design doesn't assume rates arrive pre-differenced.

### `mean rows-per-message per thread (S, coalescing degree)` — double · per_thread · **trivial_to_add**
- **source:** DERIVED: leaves/msgs per thread (both wired), as a level (cumulative ratio) for free, or as an interval ratio (Δleaves/Δmsgs) with consumer-side history. Mirror of the run-level wire_summary mean_rows_per_msg (runner_wire_batched.cpp:832).
- **publish cadence:** per control tick
- **cost:** zero new producer cost; one divide in tick_ (level) or two subtracts + a divide (interval)
- **notes:** The producer-side coalescing degree — the in-process proxy for batch width when the true server rows/forward is unavailable. At depth-1 drain-all this tracks the natural ~74; under chunk_floor it tracks ~S_min. Cheap and useful as a fallback target while server_rows_per_forward stays a gap.

### `aggregate inflight / ready / fractional fill (cross-thread)` — double/int · aggregate · **trivial_to_add**
- **source:** DERIVED in tick_ by summing/averaging the already-marshalled per-thread vectors (IssueFeatures.inflight/ready, issue_controller.hpp:153-154). No new producer field.
- **publish cadence:** per control tick
- **cost:** zero new producer cost; O(T) reduction in tick_ (T is tiny, ~3-4)
- **notes:** Total in-flight messages across threads (Σinflight) and total ready backlog (Σready) approximate the aggregate the single server sees. Because the server batches across ALL producers, an aggregate view is arguably the right altitude for an issue policy aiming to keep the server fed; it is free to compute from the per-thread vectors the surface already carries.
