# Phase 2 — Derived features

Read end to end (ADR-0002 discipline): docs/design/stall-investigation/server-gen-floor-result.md, docs/adr-synopsis.md, cpp/include/chocofarm/issue_controller.hpp, cpp/src/runner_wire_batched.cpp (the full run_episodes_wire_pipelined drain), and the controller wiring in cpp/src/wire_ab_bench.cpp (lines 230-306). The two Phase-1 gap claims were verified directly at source: cpp/include/chocofarm/wire_leaf_pool.hpp inflight_ is std::unordered_map<uint64_t, std::vector<int>> (slot lists only, NO timestamps) -> RTT needs a producer-local per-corr submit-time store (trivial_to_add, no cross-process channel); and cpp/stage_a/stage_a_server.py keeps n_forwards/n_real_rows Python-side, printed only at subprocess exit as SERVER_STATS mean_real_rows_per_fwd, with the reply frame [ident, *envelope, resp] a pure value codec -> server_rows_per_forward has NO live carrier (gap_needs_channel; redis-key side-channel is the lowest-touch).

Online vs offline separation (kept clean per the task): every static_derived feature is computable inside tick_() from the marshalled IssueFeatures on the slow cadence, costing at most an O(T) reduction or a controller-side EWMA/difference (T ~ 3-4). Three of them (issue_rate, coalescing_degree interval form, allow_history, rss_pressure, rtt EWMA) require the controller to retain HISTORY that tick_() does not hold today (prev_msgs[]/prev_leaves[]/prev_rtt[] + a steady_clock tick timestamp + the prior allow vector) — flagged inline as consumer-side derivations, NOT producer publishes, exactly so the policy design does not assume rates/EWMAs arrive pre-differenced. rtt_load is gated on the mean_rtt_ms field actually being filled (sentinel today); server_fed_proxy is an explicit in-process SURROGATE for the gap-channel server_rows_per_forward, named as such so it is not mistaken for the measured scalar.

ADR-0008 vocabulary respected throughout: the action is ISSUE allow/deny; overcommit/depth>1 is the downstream emergent property, surfaced ONLY as a label (realized_inflight_depth_trajectory) and an aggregate observation (agg_inflight), never as an action. ADR-0009: every offline label is anchored to the reproducible warm-pool sweep (--pool-seed) so counterfactual pairs and regime tags are joinable; the SERVER_STATS mean_real_rows_per_fwd at run exit is the offline ground-truth the online surrogate approximates, and regime_annotation is how that gap-channel signal is distilled into a deployable in-process policy without a P7 codec change.

These are a DESIGN analysis only — no code was written or proposed for commit; the IssueController seam (the IssuePolicy ctor argument, empty/identity today) is where any policy consuming these features would plug in, with all history/EWMA/differencing state living controller-side in tick_(), not on the hot producer path."

## Static-derived (online, cheap)

### inflight_fill (per-thread fractional in-flight fill)
- **definition:** fill[t] = inflight[t] / max(1, d_ceiling). A clamped ratio in [0,1] (clamp upper at 1.0; inflight never exceeds D by the runner's gate, but clamp defensively).
- **inputs:** IssueFeatures.inflight[t] (wired, per_thread), IssueFeatures.d_ceiling (wired, global_const). No history; pure point-in-time transform in tick_.
- **rationale:** The single most natural issue-gate signal and the one the controller header itself names: the allow/deny decision modulates how close a thread sits to its safety ceiling. fill near 0 = the thread is about to idle on recv (the depth-1 starvation the gen-floor exists to cure -> ALLOW); fill near 1 = the thread is healthily saturated and a further issue only risks RSS growth -> the regime where DENY is cheap. Because D is read-only context (the controller never sets it), the *fraction* is the dimensionless control variable that a single threshold generalizes across D in {8,16,32} (the doc shows the operating point is D-insensitive once D>=16, so a fill-fraction policy transfers across that flat region without re-tuning).

### ready_pressure (per-thread unsubmitted backlog, normalized)
- **definition:** ready_frac[t] = ready[t] / max(1, S_target), where S_target is the coalescing target (use the run's S_min when chunk_floor is on, else the natural drain-all width ~ the wire mean_rows_per_msg the run reports). A saturating indicator: ready_pressure[t] = min(1.0, ready_frac[t]).
- **inputs:** IssueFeatures.ready[t] (wired, per_thread). S_target is a run constant the controller can be handed (it already knows D; S_min/chunk_floor live in the same WireRunnerConfig). No history.
- **rationale:** ready is the backlog the next issue would coalesce; read alone it is a level, normalized against the coalescing width it becomes 'is there enough parked work to form a full message right now'. High ready_pressure WITH low inflight_fill is the convoy precursor the Phase-1 surface flags (a thread starving the issue path) -> ALLOW. ready_pressure near 0 means a forced issue now would be a thin under-coalesced message (the producer-floor-negative regime: tiny messages the single-threaded server under-coalesces) -> DENY and let the backlog fatten. This directly encodes the doc's finding that S_min=16 under-coalesces (173 dps) while S_min=32 is the peak: the policy should not allow an issue that would emit a sub-target-width message.

### convoy_score (joint starvation-vs-saturation indicator)
- **definition:** convoy[t] = ready_pressure[t] * (1 - inflight_fill[t]). High only when backlog is high AND in-flight is low simultaneously — the exact 'thread starving the issue path while parked work piles up' state.
- **inputs:** Derived from inflight_fill and ready_pressure above (both from wired inflight[t]/ready[t]/d_ceiling). No history.
- **rationale:** The two convoy DOFs (ready, inflight) are individually ambiguous — high ready with high inflight is fine (work queued behind healthy depth), low inflight with low ready is fine (a thread keeping up). The collapse this whole investigation targets is the *conjunction*: parked rows accumulating while the pipe drains to empty. A single product term gives the policy one scalar whose top-decile is the metastable-collapse onset, so the allow rule can be a single threshold on convoy[t] rather than a 2D region the policy must learn. Cheap (one multiply per thread).

### issue_rate (per-thread messages/s)
- **definition:** rate_msgs[t] = (msgs[t] - prev_msgs[t]) / dt_tick, where dt_tick is the wall interval between this control tick and the last (steady_clock::now() differenced). Same for rate_leaves[t] from leaves[t].
- **inputs:** IssueFeatures.msgs[t], IssueFeatures.leaves[t] (both wired cumulative). REQUIRES controller-side history: tick_ must retain prev_msgs[]/prev_leaves[] and a prev_tick timestamp (it holds neither today — this is a consumer-side derivation, NOT a producer publish).
- **rationale:** The raw cumulative msgs/leaves are levels; the policy almost certainly wants a flow. messages/s per thread is the issue-throughput estimate that detects a thread that has gone quiet (rate -> 0 with ready > 0 = stalled, the wedge precursor) independent of absolute counts. Explicitly flagged because the Phase-1 surface warns the policy must not assume rates arrive pre-differenced: differencing is the controller's job and needs the tick to be timestamped.

### coalescing_degree (per-thread mean rows-per-message, S)
- **definition:** Interval form: S[t] = (leaves[t]-prev_leaves[t]) / max(1, msgs[t]-prev_msgs[t]). Level fallback (no history): S_level[t] = leaves[t] / max(1, msgs[t]).
- **inputs:** IssueFeatures.leaves[t], IssueFeatures.msgs[t] (wired). Interval form needs the same controller-side history as issue_rate; level form is free.
- **rationale:** The in-process proxy for batch width when the true server rows/forward is a gap — the producer-side analog of the run-level wire_summary mean_rows_per_msg. The doc's whole parameter-shape result is a coalescing-width story (wire rows/msg 28 at the peak vs 165 greedy; srv rows/fwd 108 vs 195), so the policy's best available local handle on 'am I emitting messages the server can amortize' is this ratio. A thread whose S has collapsed toward 1 is in the producer-floor-negative regime (tiny-message flood) and should be DENIED until its backlog refattens.

### rtt_load (per-thread reply-latency regime, EWMA + trend)
- **definition:** rtt_ewma[t] = a*mean_rtt_ms[t] + (1-a)*rtt_ewma_prev[t] (a ~ 0.3); rtt_trend[t] = rtt_ewma[t] - rtt_ewma_prev[t]. Pair with ready: starve_vs_convoy[t] = sign(rtt_trend[t]) interpreted jointly with ready_pressure[t].
- **inputs:** IssueFeatures.mean_rtt_ms[t] (status: trivial_to_add — sentinel 0.0 until the producer stores per-corr submit mono_ns and EWMAs now-submit at recv). Controller-side history for the EWMA/trend.
- **rationale:** Reply latency is the direct observable of server load/starvation and disambiguates the convoy_score sign that ready+inflight alone cannot: rising rtt UNDER LOW ready = the server is the genuine bottleneck and depth is doing its job (keep ALLOW — do not back off, the server amortizing is the goal); rising rtt WITH rising ready = a forming convoy (back off / DENY before RSS climbs). This is the single best in-process latency signal and, unlike server_rows_per_forward, needs no cross-process channel — so a latency-aware policy is wirable without the P7 codec change. Gated on the RTT field actually being filled.

### agg_inflight / agg_ready (cross-thread totals the single server sees)
- **definition:** sum_inflight = sum_t inflight[t]; sum_ready = sum_t ready[t]; mean_fill = sum_inflight / max(1, T*d_ceiling). Also agg_S = sum_t leaves[t] / max(1, sum_t msgs[t]).
- **inputs:** IssueFeatures.inflight[], ready[], msgs[], leaves[], n_threads (all wired). O(T) reduction in tick_ (T ~ 3-4, trivial). No producer change.
- **rationale:** There is exactly ONE server, batching across ALL producer threads' in-flight messages into one forward. So the right altitude for a policy that aims to keep the server fed is the aggregate: sum_inflight approximates the depth the server's group-wakeup drain coalesces over. A per-thread gate that only saw its own thread could starve the server even while individually healthy; the aggregate lets the policy reason 'total in-flight is low -> the server will under-coalesce -> allow broadly' vs 'total in-flight is plentiful -> the server is fed -> I can deny the threads nearest their RSS ceiling'. Free from the per-thread vectors the surface already carries.

### server_fed_proxy (aggregate amortization estimate, in-process surrogate)
- **definition:** While server_rows_per_forward stays a gap, estimate the server's effective batch as fed_proxy = sum_t (leaves[t]-prev_leaves[t]) over the tick, i.e. total rows the producers PUT ON THE WIRE per control interval, optionally / by an estimate of forwards (unavailable in-process -> leave as a rows/interval level). Saturation form: min(1.0, fed_proxy / max_batch).
- **inputs:** IssueFeatures.leaves[] (wired), controller-side history, max_batch (run const). NB this is a SURROGATE for the true server_rows_per_forward (status: gap_needs_channel — only a redis-key side-channel or a P7 reply-frame extension delivers the real one); flagged as such so the policy does not mistake the proxy for the server's actual rows/forward.
- **rationale:** The objective is the server amortizing its fixed per-forward cost over rows/forward, but that metric has no live carrier. The closest free surrogate is total producer egress per interval: if the producers are collectively putting plenty of rows on the wire, the single-threaded server's group drain has material to coalesce. This is a deliberately-honest fallback target (the Phase-1 surface's 'mean rows-per-message' raised to the aggregate) usable NOW; when the real channel exists the policy swaps the surrogate for the measured scalar with no structural change.

### allow_history (per-thread recent deny duty-cycle)
- **definition:** deny_frac[t] = EWMA over recent ticks of (1 - allow_bit_written[t]). Equivalently a short ring of the last K allow decisions per thread, summarized as the fraction denied.
- **inputs:** The controller's OWN written allow bits (out[] in tick_, the actuation surface) plus a per-thread EWMA state. No producer feature; entirely controller-internal.
- **rationale:** Per-thread gating aggregated over threads x time emulates a dynamic FRACTIONAL in-flight depth (the contract's own framing). To realize a *target* fractional depth, the policy must know what fraction it has recently been denying — otherwise it has no closed loop and will chatter. deny_frac is the integral term: it lets the policy hold a steady aggregate overcommit (e.g. 'keep ~2 of 3 threads allowed') rather than re-deciding statelessly each tick, which both stabilizes the emergent depth and is the natural anti-windup guard against an oscillating allow pattern that would itself induce the convoy.

### rss_pressure_proxy (cumulative in-flight saturation guard)
- **definition:** A per-thread and aggregate high-water signal: hwm_inflight[t] tracked as max over recent ticks, and over_target = max(0, sum_inflight - D_target_total) where D_target_total is the aggregate in-flight the policy wants to hold. A monotone-pressure scalar that rises only when in-flight sits pinned at the ceiling across consecutive ticks.
- **inputs:** IssueFeatures.inflight[] (wired), d_ceiling, controller-side history (recent-tick max). No producer change.
- **rationale:** The objective explicitly forbids producer-RSS runaway, and the doc records a 6 GiB kill-ceiling guarding the sweep (the harmful prior regime had unbounded RSS). The runner's fixed D caps per-thread in-flight messages, but the controller can pre-empt the ceiling: a thread pinned at fill=1 across many ticks is the RSS-growth signature. rss_pressure gives the deny rule a saturation term so the policy stops ALLOWING into a thread that is already maximally committed — turning the hard D ceiling into a soft, earlier, controller-side backoff. Cheap (a running max + one subtract).

## Offline-derived (for supervised learning)

### future_window_dps (the primary regression label)
- **definition:** For each control tick at time t, the realized throughput over the lookahead window [t, t+W]: label_dps(t) = (decisions completed in [t, t+W]) / W, for one or a few fixed horizons W (e.g. 100ms, 500ms, 1s). The supervised target the per-tick feature vector maps to.
- **requires logging:** A timestamped decision counter (total_decisions is already maintained in the runner — log it with a steady_clock timestamp at each control tick, or log the per-tick measure_t0/t1-style window endpoints). Plus the full IssueFeatures snapshot and the written allow vector at every tick (the feature row the label attaches to).
- **supervised use:** Train a value/regression head that, given the online static-derived features at a tick, predicts the throughput the next window will deliver. Thresholded, it becomes the issue-gate classifier's score (the header's 'softmax regression thresholded into bits'). This is the direct objective-aligned label: maximize sustained dps.

### collapse_onset_label (the variance-tail classification target)
- **definition:** A binary/graded label marking ticks that PRECEDE a metastable collapse: label_collapse(t) = 1 iff window dps in [t+d, t+d+W] drops below a regime threshold (e.g. < greedy's collapse floor, the 169 vs 183 gap the doc quantifies) for some short lead d. The 'about to convoy' label.
- **requires logging:** Per-tick IssueFeatures + allow vector + the timestamped decision counter (to compute forward windowed dps), across MANY warm-pool-seeded runs so both collapse and non-collapse windows are sampled (the doc: ~2-4% of windows collapse, so the positive class is rare -> log enough runs/seeds for class balance).
- **supervised use:** The empirical headline is that the win is VARIANCE reduction (removing the collapse tail), not mean dps. A model trained to predict future dps mean will be nearly flat (the tail is 2-4% of windows); a model trained on collapse_onset directly targets the actual lever. This is the label that lets a policy ALLOW pre-emptively at the onset signature (high convoy_score, rising rtt-with-ready) before the collapse, which the mean-dps label would under-weight.

### counterfactual_allow_delta (per-tick action-value advantage)
- **definition:** For a tick, the difference in subsequent windowed dps between the allow decision taken and the counterfactual: delta(t) = E[dps | allow flipped] - E[dps | allow as-taken], estimated by matching ticks with near-identical feature vectors but opposite allow bits (or by paired A/B runs that diverge only in the policy's decision at controlled points).
- **requires logging:** Per-tick features + allow vector + window dps from PAIRED runs sharing a pool-seed (--pool-seed makes the warm pool reproducible per the eval harness) but differing in the injected policy, so the only divergence is the action. Log the pool-seed alongside every trace so pairs are joinable.
- **supervised use:** Turns the trace into an advantage-style supervised signal: which allow/deny decisions actually moved throughput, controlling for the regime the thread was already in. Lets training distinguish 'denied and dps held' (correct) from 'denied and dps would have held anyway' (no-op) — the counterfactual the online surface can never see. Directly leverages the reproducible warm-pool sweep (--pool-seed) as the matched-pair generator (ADR-0009 reproducibility).

### regime_annotation (operating-point / parameter-regime tag)
- **definition:** Each trace window annotated with its regime: (D, S_min, chunk_floor, N-overcommit, theta_server) plus a derived coarse class — e.g. {under-coalesced (S_min=16), peak (S_min=32), greedy-depth1, server-floor-on}. The doc's own taxonomy (S_min=16 under-coalesces -> 114 rows/fwd; S_min=32 peak; depth-1 collapse-prone).
- **requires logging:** The full WireRunnerConfig per run (D, min_coalesce, chunk_floor, trees_per_thread, the server's min_forward_rows theta) — already on the SWEEP_RESULT line and the wire_summary — joined to the per-tick trace by run id. Also the end-of-run SERVER_STATS mean_real_rows_per_fwd (the post-hoc server depth) for the ground-truth amortization the online surrogate approximates.
- **supervised use:** Lets the model condition on / be evaluated per regime, and lets training avoid learning a spurious policy that only works at one operating point. It is also how the offline label supplies the ONE metric the online surface lacks: SERVER_STATS mean_real_rows_per_fwd is logged at run exit, so an offline model can be trained with the TRUE server rows/forward as an auxiliary target/feature even though the online policy must run on the in-process surrogate (server_fed_proxy) — distilling the gap-channel signal into a deployable in-process policy.

### realized_inflight_depth_trajectory (the emergent-depth label the action does not directly set)
- **definition:** The downstream in-flight depth that a sequence of allow/deny decisions actually produced: depth(t) reconstructed from the logged inflight[] trajectory and the allow history — the emergent fractional overcommit (sum_inflight over time) that the contract says is downstream of the issue decisions, not the action itself.
- **requires logging:** Per-tick inflight[] vector + the written allow vector + tick timestamps over the whole run (a contiguous trace, not just sampled ticks, so the trajectory is continuous).
- **supervised use:** The control contract is explicit that overcommit (depth>1) is a DOWNSTREAM emergent property of allowed issues + in-flight history, not the action. To train a policy whose allow/deny choices steer toward a target emergent depth, you need the realized depth each decision sequence produced — available only by replaying the logged trajectory offline. This is the label that closes the gap between the action (binary issue) and the objective (a sustained fractional overcommit), and lets supervised training credit-assign a deny to the depth reduction it caused several ticks later (the RTT-lagged effect the online tick cannot observe).
