# Phase 3 — Design exploration: online

**Frame:** ONLINE-LEARNING DESIGN SPACE for the IssueController policy (Phase 3). The control task, restated precisely against the fixture: each cadence_ms tick the controller marshals IssueFeatures and must emit a per-thread binary issue gate may_issue(tid) in {allow, deny}, read once per refill (one relaxed atomic). The action is ISSUE allow/deny (ADR-0008): overcommit/depth>1 is the DOWNSTREAM emergent effect of allowed issues x in-flight history, surfaced only as the label realized_inflight_depth_trajectory and the observation agg_inflight, never decided. The objective is sustained dps (= leaves/s) AND collapse-tail variance reduction, without producer-RSS runaway. The empirical result (server-gen-floor-result.md, read end to end) is load-bearing: the win is VARIANCE (greedy [169-183] -> gen-floor [183-184]), mean only +3.7% with MIN merely tying greedy's MAX, and the mechanism is producer IDLE-TIME, not batch width (gen-floor wins with SMALLER forwards 108 vs 195 by keeping otherwise-depth-1 threads from blocking idle on a single outstanding reply). This reframes the learning problem: it is NOT a mean-dps maximizer (the mean surface is nearly flat, so a mean-reward learner sees ~no gradient and will overfit noise); it is a RARE-EVENT / TAIL-RISK controller whose job is to detect the convoy onset (high convoy_score = ready_pressure*(1-inflight_fill), rising rtt-with-ready) and pre-emptively ALLOW the starving thread before the collapse, while DENYING threads pinned at fill=1 to cap RSS.\n\nThree structural facts constrain ALL online approaches. (1) CREDIT-ASSIGNMENT LAG. The reward (windowed dps, or the per-tick decision-rate delta) is observable only AFTER the action propagates through the RTT-lagged pipe (the gen-floor's effect manifests several ticks later as depth>1 prevents an idle stall). The action is per-thread but the dps reward is GLOBAL (one server, batching across all threads) -- per-thread credit assignment is the central difficulty. (2) THE REWARD HAS A CHEAP IN-PROCESS PROXY AND A TRUE GAP. The global dps over a control interval is reconstructable in-process from sum_t (leaves[t] - prev_leaves[t]) / dt_tick (server_fed_proxy raised to a rate; all inputs wired) -- so the learner needs NO new channel for its reward. The TRUE amortization metric server_rows_per_forward is the gap_needs_channel item (Python-only, post-hoc), needed only if a learner wants to reward batch-width directly (it should not: the doc says width is NOT the lever). (3) THE SAFETY ENVELOPE IS PRE-BUILT AND TIGHT. The forced-flush floor (ungated, guarantees depth-1 liveness for any denied thread -- a denied thread can NEVER deadlock) and the fixed D ceiling (caps per-thread in-flight) mean the policy's action space is provably safe at BOTH extremes: all-deny degrades to the depth-1 greedy baseline (still live, ~177 dps), all-allow is the fixed-D runner. So exploration is FREE of catastrophic risk -- the worst an exploratory deny does is forgo one overcommit (a transient idle), the worst an exploratory allow does is one extra in-flight message (bounded by D, RSS-bounded). This is the rare RL setting where unconstrained exploration is bounded by construction, which makes online learning unusually safe here and is the strongest argument for it over a frozen offline-trained policy.\n\nThe design axes I sweep below: (A) what closes the loop -- bandit (per-tick reward, no state-transition model) vs full RL (value bootstrapping across the lagged horizon) vs adaptive/self-tuning control (online-estimated setpoint, no explicit reward model); (B) the credit-assignment altitude -- per-thread independent learners vs one shared learner over the aggregate vs a hybrid (shared body, per-thread context); (C) exploration mechanism vs the safety envelope; (D) per-tick update + inference cost on the slow tick (must be cheap -- T~3-4, cadence_ms slow, so even O(T*d) GLM updates are free; the constraint is statistical sample-efficiency, NOT compute). ADR-0012 fit is uniform: the learner is injected behind IssuePolicy (P2 -- the runner depends only on the allow read + publish, never the learner), all learner state (weights, eligibility traces, reward-estimator history, RNG) lives controller-side in tick_ (P4 live-not-frozen), the forced-flush+D envelope is the P5 fail-loud/liveness backstop the learner cannot breach, and any dps-improvement claim is substantiated on the reproducible --pool-seed warm-pool sweep (ADR-0009 / P6). I rank the candidates with a strong recommendation toward the lowest-variance, most-sample-efficient ones because the eval budget (3 iters/config, ~minutes/run) and the rare-event nature (2-4% collapse windows) make sample efficiency the binding constraint, not asymptotic optimality.

## Candidates

### C1: Per-thread contextual bandit (LinUCB / linear Thompson sampling) over the binary gate
- **maps features to gate:** Each thread t runs an independent contextual bandit with two arms {allow, deny}. Context x_t = the static-derived feature vector for that thread at the tick: [inflight_fill, ready_pressure, convoy_score, coalescing_degree, issue_rate (if history kept), mean_fill (aggregate, shared), rtt_load (if RTT wired)]. The arm value is a linear function theta_a . x_t; pick argmax of UCB (theta_a.x + alpha*sqrt(x' A_a^-1 x)) for LinUCB, or sample theta_a ~ N(mu_a, A_a^-1) for Thompson and pick argmax. The chosen arm IS the gate bit. d~7 features, 2 arms.
- **objective or signal:** Per-tick scalar reward observed one tick later (the bandit is stateless across ticks so a 1-tick delay is the natural lag): r = Delta(global decisions)/dt_tick over [last tick, this tick], i.e. the in-process dps proxy from sum_t Delta leaves / dt -- minus a small RSS penalty beta*max(0, sum_inflight - D_target_total). Because reward is GLOBAL but the action is per-thread, each thread's bandit is credited with the SAME global reward delta (a shared-reward multi-agent bandit). Optionally shape with the convoy term: add a bonus for having allowed a thread whose convoy_score was top-decile (rewards pre-emptive anti-collapse allows the flat mean reward under-weights).
- **tuning or training:** Online from cold start (theta=0, A=lambda*I). One hyperparameter: exploration alpha (UCB) or prior variance (Thompson). Warm-start optional from an offline LinUCB fit on logged --pool-seed traces (the offline_derived future_window_dps label). Tune alpha and the RSS penalty beta on the reproducible warm-pool sweep.
- **stability and liveness:** Liveness: unconditional (forced-flush floor; a deny is always safe). Stability: LinUCB/Thompson are no-regret under stationary rewards; the reward here is quasi-stationary within a run (the warm pool fixes the regime). The risk is oscillation if adjacent ticks' allow flips chatter -- damp by including allow_history (deny duty-cycle) in the context so the bandit sees its own recent action (turns it into a crude stateful learner) and/or by holding the gate for K ticks (action persistence).
- **inference cost:** Per tick: per thread, 2 arms x (one d-dim dot + one quadratic form x'A^-1x) = O(d^2) ~ 100 flops/thread, x T~4 = trivial. Inference is one argmax. Update: one rank-1 Sherman-Morrison per pulled arm, O(d^2). Total per tick << microseconds, dwarfed by cadence_ms. Zero hot-path cost (gate read is the relaxed atomic).
- **pros:** Simplest online learner that still adapts; closed-form incremental update (rank-1 A_a inverse via Sherman-Morrison, O(d^2)~50 flops per thread per tick -- free). No discounting / no bootstrapping / no convergence subtleties of TD. Principled exploration (UCB/Thompson) that the safety envelope makes risk-free. Linear model is interpretable (theta tells you which feature drives allow) -- ADR-0008/0009 honest. Thompson variant is especially robust to the delayed/noisy reward. Transfers across D because features are fractions (inflight_fill) per the doc's D-insensitivity.
- **cons:** Treats the 1-tick-lagged reward as immediate -- mis-credits when the real effect (idle-stall avoidance) lands several ticks later (RTT-lagged). Shared global reward gives weak per-thread credit (a thread that correctly allowed gets the same reward as one that wrongly denied if the net dps was flat). The mean-dps reward is nearly flat (the doc's headline) so the bandit's gradient is dominated by the rare collapse windows -- needs the convoy-bonus shaping or it learns almost nothing. No state transitions: cannot learn 'deny now to enable a better allow later'.
- **adr fit:** Strong P2 (injected, linear weights are controller-side state). P4 (theta is a live cell). ADR-0009: reward and improvement both measured on --pool-seed. ADR-0008: action stays 'issue', the bandit never names overcommit. Risk register: the shared-global-reward credit weakness is the honest limitation to declare (ADR-0002 no-bluff).
- **risks:** Flat-mean-reward starvation of the gradient (mitigate with convoy-shaped reward). Mis-credit from the 1-tick lag (mitigate with a short reward-attribution window or eligibility, which pushes toward C2). Non-stationarity if the regime shifts mid-run (the warm pool mostly prevents this).

### C2: Per-thread / shared tabular-or-linear RL with eligibility traces (Sarsa(lambda) / Q(lambda) over the binary action)
- **maps features to gate:** Model the tick as an MDP step: state s_t = the per-thread feature vector PLUS aggregate context (mean_fill, agg_ready, server_fed_proxy rate), action a_t in {allow, deny}, reward r_t = the windowed dps proxy minus RSS penalty. Q(s,a) is linear in tile-coded or raw features (function approximation -- the state is continuous). Gate bit = epsilon-greedy / softmax over Q(s, allow) vs Q(s, deny). Eligibility traces (lambda~0.8-0.9) propagate the lagged reward back over the ~RTT-many ticks it took to manifest, which is the precise fix for C1's credit-lag weakness.
- **objective or signal:** Same in-process dps-proxy reward, but now bootstrapped: the TD target r_t + gamma*Q(s_{t+1}, a_{t+1}) credits an action for the value of the state it led to, and the eligibility trace distributes a late reward back to the deny-now-that-enabled-the-allow-later decisions. This is the candidate that can learn the doc's actual mechanism: 'maintain depth>1 so the thread does not idle-stall N ticks from now' -- a temporally-extended consequence a bandit cannot represent. gamma modest (0.9-0.95) because the horizon of interest is a few RTTs.
- **tuning or training:** Can run fully online (tabula rasa) but realistically warm-started: pretrain Q offline by fitted-Q / LSTD on the logged --pool-seed traces (offline_derived counterfactual_allow_delta gives the advantage signal, realized_inflight_depth_trajectory gives the state-transition replay), then fine-tune online. Tune gamma/lambda/alpha/temp on the reproducible sweep; report MIN-beats-MAX per ADR-0009.
- **stability and liveness:** Liveness floor unchanged (forced-flush). Stability: use on-policy Sarsa(lambda) (convergent with linear FA under standard step-size conditions) NOT off-policy Q-learning (deadly-triad divergence risk). Add the allow_history/rss_pressure terms to the state so the value function sees the integral, damping oscillation. The safety envelope means even a transiently-diverged Q only mis-gates within [all-deny=greedy, all-allow=fixed-D] -- bounded blast radius.
- **inference cost:** Per tick: Q(s, allow), Q(s, deny) = two d-dim dots O(d), argmax/softmax, one TD update with trace vector O(d) (or O(active tiles) for tile coding). Per thread x T = trivial. Trace vector is O(d) state per thread. All << cadence_ms.
- **pros:** Directly models the lagged credit assignment the convoy story demands (eligibility traces are the textbook tool for delayed reward). Can learn deny-to-enable-future-allow. Still cheap with linear FA. Shared-learner variant (one Q over [thread-features ++ thread-id-embedding or just thread-features]) pools data across T threads for sample efficiency, which matters given the rare-event regime. Softmax action selection gives smooth, tunable exploration the safety envelope bounds.
- **cons:** More moving parts: gamma, lambda, learning rate alpha, exploration temperature -- four hyperparameters vs the bandit's one, and TD with FA can diverge (the deadly triad: bootstrapping + off-policy + FA). On-policy Sarsa(lambda) sidesteps divergence but is slower to converge. The per-thread reward is still the GLOBAL dps (per-thread credit unsolved -- traces help with TIME credit, not THREAD credit). Sample efficiency: an RL learner needs more ticks to converge than a bandit, and the eval runs are minutes-long -- convergence within a single warm-pool run is the open question.
- **adr fit:** P2/P4 as C1. ADR-0009: the offline pretraining uses the --pool-seed counterfactual pairs as the matched-pair advantage source. ADR-0008: action vocabulary clean. The honest declaration (ADR-0002): TD+FA convergence is conditional -- name the on-policy choice as the guard, do not claim unconditional convergence.
- **risks:** Deadly-triad divergence (mitigate: on-policy Sarsa). Slow convergence vs run length (mitigate: offline warm-start). Per-THREAD credit still global (shared by all RL variants here -- the structural limit). Over-engineering risk: if the bandit (C1) already captures the win, the RL machinery is unjustified complexity (ADR-0007 density).

### C3: Adaptive / self-tuning control -- online setpoint-tracking PI on aggregate depth, threshold realized per-thread (the control<->learning intersection)
- **maps features to gate:** Frame it as classical regulation, not reward maximization. The controlled variable is the aggregate emergent depth proxy D_eff = sum_inflight (or mean_fill). The controller holds a setpoint D* (the target fractional overcommit the doc identifies: ~depth>1, e.g. mean_fill ~ 0.5-0.7 corresponding to S_min=32/D=16). A PI loop computes a single scalar control u = Kp*(D* - D_eff) + Ki*integral(error) each tick; u is the FRACTION of threads to allow. Realize the fraction as a per-thread gate by RANKING threads by convoy_score (allow the top ceil(u*T) most-starved, deny the rest) -- this is the per-thread binary gate the contract requires, with the fraction emerging across threads x time exactly as the contract's framing says. The ADAPTIVE/learning part: auto-tune (Kp, Ki) online (relay-feedback / MIT-rule / recursive-least-squares self-tuning regulator) and ADAPT the setpoint D* online by hill-climbing the in-process dps proxy (extremum-seeking: perturb D* by +-delta, keep the direction that raised the windowed dps proxy).
- **objective or signal:** Two-level. Inner loop: regulation error (D* - D_eff), no reward model -- just track the setpoint (cheap, stable, fast). Outer loop: extremum-seeking on the dps proxy to LEARN where D* should sit (this is the online-learning content: the system discovers the throughput-optimal depth setpoint by slow gradient ascent on measured dps, which is exactly the doc's empirical S_min/D sweep done online and continuously instead of offline and discretely). The RSS guard is a hard clamp: D* never exceeds D_target_total (rss_pressure_proxy).
- **tuning or training:** (Kp,Ki) auto-tuned online by relay feedback (induce a limit cycle, read the ultimate gain/period, Ziegler-Nichols) OR fixed from one offline tuning pass on the --pool-seed sweep. D* initialized at the doc's known-good operating point (mean_fill for S_min=32/D=16) so the extremum-seeking starts AT the optimum and only tracks drift -- this is the sample-efficiency win: it does not have to discover the optimum from scratch, just hold and refine it.
- **stability and liveness:** Strongest stability story: PI with anti-windup (the rss_pressure clamp IS the anti-windup) is unconditionally stable for well-chosen gains; extremum-seeking is stable under timescale separation (outer loop slower than inner). Liveness floor unchanged. The integral term is the natural anti-chatter (allow_history's role) -- the regulator inherently avoids the oscillating-allow convoy the Phase-2 notes warn about. Blast radius bounded by the same envelope.
- **inference cost:** Per tick: one O(T) reduction for D_eff, one PI update (2 mults + 1 add), one O(T log T) sort of convoy_score to rank, threshold to bits. Outer loop: one perturbation bookkeeping step every M ticks. Cheapest of all candidates. State: the integral accumulator + (Kp,Ki) + D* + the extremum-seeking phase -- a handful of scalars.
- **pros:** Most STABLE and sample-efficient: PI regulation converges in a few ticks, needs no per-action reward credit (sidesteps the global-reward/per-thread-credit problem entirely -- it regulates an AGGREGATE the single server actually sees, which is the right altitude per the Phase-2 agg_inflight rationale). The convoy-score ranking puts the allows exactly where the collapse forms -- directly targets the variance lever, not the flat mean. Extremum-seeking is a well-understood, provably-convergent (under slow-perturbation) online optimizer. Intersects cleanly with a static PID (the task explicitly blesses an auto-tuned PID). Lowest risk of the three -- a control engineer's default and the best variance-reduction story.
- **cons:** Not a 'learner' in the RL sense -- the policy class is a regulator with two tuned gains + an adapted setpoint, less expressive than a Q-function (cannot learn arbitrary feature->action maps, only setpoint tracking). Extremum-seeking is SLOW (the outer loop needs many perturbation periods to climb, and the dps proxy is noisy -- the flat mean means the gradient it climbs is tiny, the same flat-reward problem as C1's). Ranking by convoy_score assumes that score is the right collapse predictor (it is the Phase-2 hypothesis, not yet validated). The setpoint framing presumes a single good D* exists (the doc supports this: S_min=32/D>=16 is a flat optimum, so a setpoint regulator landing anywhere in that plateau is near-optimal -- a strong point FOR this candidate).
- **adr fit:** Strongest ADR-0008 fit: it regulates AGGREGATE depth and ranks per-thread, never deciding 'overcommit' -- the emergent fractional depth is literally what it tracks as a downstream CV, matching the contract's framing verbatim. P2/P4/P5 clean (gains+setpoint are live cells; the rss clamp is the fail-loud guard). ADR-0009: extremum-seeking IS continuous online reproduction of the offline sweep -- the same dps measurement, made live; substantiate on --pool-seed.
- **risks:** Flat-mean dps gradient starves the extremum-seeking outer loop (mitigate: seed at the known optimum and rely on regulation, treat outer loop as drift-tracking only). convoy_score's validity as the ranking key is the load-bearing assumption (validate offline on collapse_onset_label before trusting the ranking). Over-conservatism if D* is set too low (regulator holds depth-1, forgoes the variance win).

### C4: Online logistic-regression gate trained by streaming SGD on a collapse-onset surrogate (the discriminative middle ground)
- **maps features to gate:** A single shared logistic model p_allow(x) = sigmoid(w . x) over the per-thread feature vector (++ aggregate context broadcast). Gate bit = 1[p_allow(x) > 0.5] (or sample for exploration). This is the 'softmax regression thresholded into bits' the controller header itself names. The model is trained ONLINE by SGD against a self-supervised target derived from the lagged outcome: label a past tick's allow decision GOOD (target 1) if the thread did NOT idle-stall in the next window (ready stayed fed, no convoy), BAD (target 0) if a convoy formed -- i.e. learn to predict collapse_onset and allow exactly the threads about to starve.
- **objective or signal:** Self-labeled streaming target from the lagged trace: after W ticks, look back at tick t's state+action and assign a binary label from the realized outcome (did dps hold / did the thread stall). This is a delayed-label online classification problem (the label for tick t arrives at t+W). The reward is implicit in the label = the collapse_onset_label computed online. Sample weighting upweights the rare collapse windows (class imbalance handling -- the doc's 2-4% positive rate).
- **tuning or training:** Online SGD with a small step size; warm-start strongly recommended (offline classifier on --pool-seed traces with the collapse_onset_label). Hyperparameters: step size, class-imbalance weight, the stall-label threshold, the lookahead W. Tune on the reproducible sweep.
- **stability and liveness:** Liveness floor unchanged. Logistic SGD is stable (convex loss). The delayed-label loop can lag a regime change but the warm pool fixes the regime. Add allow_history to the features to damp chatter. Bounded blast radius via the envelope.
- **inference cost:** Per tick: one sigmoid(w.x) per thread = O(d) x T, threshold. Update: one SGD step per now-labeled past row, O(d). Ring buffer of W pending rows x T. Trivial compute; the only real cost is the W-deep pending-label buffer (tiny). << cadence_ms.
- **pros:** Directly trains the variance-lever predictor (collapse onset) rather than the flat mean -- aligns with the doc's headline better than any reward-maximizer. Cheapest possible parametric learner (one logistic, O(d) update). Naturally produces a calibrated allow PROBABILITY -> smooth exploration by sampling. Warm-startable from an offline-trained classifier on the logged traces (offline_derived collapse_onset_label is exactly this target). Shared model = max sample efficiency.
- **cons:** Self-supervised label is heuristic (what counts as 'stalled' is a threshold the designer picks -- a modeling choice that could be wrong, the formal-stall-faithful-model caution from memory: over-abstraction -> false labels). Delayed labels (W-tick lag) complicate the online update (need a ring buffer of pending feature rows awaiting their label). Discriminative, not decision-theoretic -- predicts collapse but does not weigh the dps cost of a wrong allow vs wrong deny (no explicit reward), so it can be confidently-wrong about the action even when the prediction is good. Pure classification ignores the action's effect on future state (no closed loop -- like the bandit but without even the reward).
- **adr fit:** P2/P4 clean (w is a live cell). The header literally names 'softmax regression thresholded into bits' as a sanctioned form, so this is the most fixture-aligned policy class. ADR-0008: action clean. ADR-0002 honesty: the self-supervised stall-label is a MODELING CHOICE that must be declared and validated (the memory's faithful-model caution applies -- the label's threshold is the DOF whose mis-setting makes the predictor wrong).
- **risks:** Mis-specified stall label (the central risk -- validate the label against actual collapse windows offline before deploying). Delayed-label bookkeeping bugs. No action-cost awareness (a deny that the predictor 'correctly' makes might still cost dps -- the classifier cannot see that; only a reward-based method (C1-C3) can). Confidently-wrong failure mode if the offline regime and online regime diverge.

## Recommendation
Try C3 (adaptive self-tuning PI on aggregate depth, per-thread realized by convoy_score ranking) FIRST, then C1 (per-thread contextual bandit) as the learning-register comparison. Rationale, tied to the evidence: (1) The empirical result is a VARIANCE/idle-time story with a FLAT mean and a known, FLAT optimum (S_min=32, D>=16 all ~182-184). A flat optimum is the ideal target for a setpoint REGULATOR and the worst target for a reward-MAXIMIZER (no gradient to climb) -- so C3, which regulates an aggregate toward a known-good setpoint and only adapts it for drift, is matched to the problem shape, while C1/C2/C4 all fight the flat-mean gradient. (2) C3 regulates the AGGREGATE depth the SINGLE server actually sees (agg_inflight), which sidesteps the per-thread-credit-assignment problem that is the structural weakness of every reward-based candidate here (the reward is global, the action per-thread). (3) C3's convoy_score ranking puts the allows exactly at the collapse onset -- it targets the variance lever directly. (4) It is the cheapest and most stable, and the task explicitly blesses an auto-tuned PID. (5) The safety envelope (forced-flush floor + D ceiling) makes even C3's extremum-seeking perturbations risk-free, so the one online-learning element (drift-tracking the setpoint) is safe to run live. Run C1 alongside as the honest contrastive baseline: if a simple per-thread bandit with the convoy-shaped reward matches C3, the regulator's extra structure is unjustified (ADR-0007 density); if C1 underperforms (likely, due to the flat reward), that is the empirical confirmation that the problem is regulation-not-maximization. DEFER C2 (full RL with traces) unless C1/C3 both leave temporally-extended credit on the table (the deny-to-enable-later-allow pattern) -- its convergence-within-a-run and deadly-triad risks are unjustified until a simpler method demonstrably fails. C4 is the best PURE-PREDICTION fallback and the most fixture-aligned by the header's own 'softmax-regression' framing, but its self-supervised stall-label is an unvalidated modeling DOF (the memory's faithful-model caution) -- promote it only after the label is validated offline against collapse_onset_label on logged --pool-seed traces. Across all: warm-start from an offline fit on the logged traces (so the online learner refines rather than discovers), keep ALL learner state controller-side in tick_ behind IssuePolicy (P2), and substantiate every dps claim on the reproducible --pool-seed warm-pool sweep with the MIN-beats-MAX bar (ADR-0009), reporting the VARIANCE (min-max spread) as the primary metric since that is the documented lever, not the mean.

## What the fixture must provide
The fixture must provide, per online-learning candidate: (A) AN IN-PROCESS REWARD/CV SIGNAL ON THE TICK. The global dps-proxy reward (and C3's CV) needs sum_t Delta leaves over the tick interval divided by dt_tick. This requires the controller-side history the Phase-2 analysis flagged: tick_ must retain prev_leaves[]/prev_msgs[] and a steady_clock tick timestamp (it holds NEITHER today). This is a CONTROLLER-side addition (not a producer publish) -- the single most important missing piece for ANY reward-based learner. Trivial (a few scalars of state in IssueController). (B) RTT, if a latency-aware reward/feature is wanted (rtt_load, C1/C2 context): mean_rtt_ms is a sentinel today; filling it is trivial_to_add and purely producer-local (a corr->submit-mono_ns store in WireLeafPool + an EWMA at recv, pass as publish's 6th arg). Not required for the recommended C3/C1 (they can run on the wired inflight/ready/leaves alone) but it is the single best disambiguator of server-bottleneck-vs-convoy, so wire it for the contrastive runs. (C) NO new cross-process channel is needed for the recommended path: server_rows_per_forward (gap_needs_channel) is the TRUE amortization metric but the doc says width is NOT the lever, so the in-process server_fed_proxy (leaves-egress rate) suffices online; the true scalar is needed ONLY as an offline auxiliary target (regime_annotation distillation) if a later policy wants to learn against measured rows/forward. (D) OFFLINE TRACE LOGGING for warm-start and counterfactual eval (ADR-0009): the eval harness must log, per control tick, the full IssueFeatures snapshot + the written allow vector + a timestamped global decision counter + the run's WireRunnerConfig (D, S_min, chunk_floor, N) + the pool_seed, so (i) future_window_dps / collapse_onset_label / counterfactual_allow_delta labels are computable, (ii) paired --pool-seed runs differing only in policy yield the matched-pair advantage, and (iii) the end-of-run SERVER_STATS mean_real_rows_per_fwd is joinable as the offline ground-truth. The trace must be CONTIGUOUS (every tick, not sampled) for realized_inflight_depth_trajectory replay. (E) THE EVAL LOOP: extend wire_ab_bench's --sweep-configs warm-pool path to accept an injected non-identity IssuePolicy (the seam is the IssueController ctor's IssuePolicy arg, empty today) and to run paired policy-vs-identity under a fixed --pool-seed so the policy's dps and variance are measured against the byte-identical baseline (the regression baseline of the seam, the way RandomPolicy validates env<->Policy). Report MIN/MAX/spread, not just mean, since variance is the documented win. (F) ACTION PERSISTENCE / HISTORY STATE for anti-chatter: every candidate benefits from allow_history (per-thread deny duty-cycle) and rss_pressure_proxy (recent-tick max inflight) -- both are controller-side derivations needing the prior allow vector + a recent-tick ring, which the controller must retain. None of (A)-(F) touches the hot path or requires a P7 wire change except the optional offline-only true-rows-per-forward channel.

## Key uncertainties
(1) WHETHER THE MEAN IS LEARNABLE AT ALL ONLINE. The doc's headline is that the mean is nearly flat (+3.7%, MIN ties greedy MAX) and the win is variance. If a single warm-pool run contains too few collapse windows (2-4%) to estimate the rare-event reward within a run's length, NO online learner converges within one run -- it must be warm-started offline and only TRACK online. This is the binding uncertainty and the reason every recommendation is warm-start-then-refine, not learn-from-scratch. (2) PER-THREAD CREDIT ASSIGNMENT. The reward is global (one server) but the action is per-thread -- can any reward-based method (C1/C2/C4) attribute a dps change to the RIGHT thread's allow/deny? C3 sidesteps this by regulating the aggregate, which is WHY it is recommended first; but if per-thread heterogeneity matters (one thread chronically starves), the aggregate regulator misses it. Open: is per-thread credit actually needed, or is the aggregate altitude sufficient (the Phase-2 agg_inflight rationale argues sufficient -- the single server only sees the sum). (3) CONVOY_SCORE VALIDITY. C3's ranking and C1/C4's most-useful feature both lean on convoy_score = ready_pressure*(1-inflight_fill) being the true collapse predictor. It is the Phase-2 HYPOTHESIS, not yet validated against actual collapse windows (collapse_onset_label).
