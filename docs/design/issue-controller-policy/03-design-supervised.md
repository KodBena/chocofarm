# Phase 3 — Design exploration: supervised

**Frame:** SUPERVISED LEARNING for the per-thread binary issue gate. The crux is the LABEL: the action is `may_issue(tid)` (allow/deny), and "good" must be defined from logged OUTCOMES, but the objective (server amortizing its fixed per-forward cost; the empirically-confirmed win is collapse-tail VARIANCE reduction, NOT mean dps — server-gen-floor-result.md headline) is a GLOBAL, LAGGED, EMERGENT property, while the gate is PER-THREAD and INSTANTANEOUS. The whole design space partitions on how it bridges that gap. I separate four data-collection/labeling families and pick a model family for each.

DATA COLLECTION (shared across all candidates): run wire_ab_bench --sweep-configs with --controller-cadence-ms > 0 but swap the identity policy for an EXPLORATORY/RANDOM gate (a Bernoulli-per-thread coin re-flipped each tick, with the deny-probability itself swept across runs, e.g. p_deny in {0, 0.15, 0.35, 0.6, 0.85} and a few epsilon-greedy-around-a-heuristic runs so the state distribution covers both healthy and starving regimes). This is the only honest way to get (features, action, outcome) tuples whose action varies independently of the state — a fixed heuristic gate confounds action with state and the model learns the heuristic, not the value. Log at EVERY control tick: the full IssueFeatures snapshot, the per-thread written allow vector, a steady_clock timestamp, AND the running total_decisions counter (already maintained in the runner) so forward windowed dps is computable. Stamp every trace with its pool-seed and full WireRunnerConfig (D, S_min, chunk_floor, N, theta_server) — this is the regime_annotation join key AND the matched-pair key. At subprocess exit log SERVER_STATS mean_real_rows_per_fwd (the offline ground truth for the gap-channel signal the online surface lacks). This is all off the hot path (the tick is already slow; logging is one appended line per tick) and reproducible per ADR-0009 (--pool-seed).

THE LABEL PROBLEM (the crux, stated precisely): a per-tick, per-thread row needs a scalar/binary target. Three labeling strategies, in increasing fidelity and cost: (A) DIRECT-WINDOW value regression — label each tick with realized forward dps over [t,t+W]; trivial to compute but it labels the STATE not the action (every thread in a tick gets the same global dps), so a classifier on it learns 'what states precede good throughput', and the gate is then 'allow toward good-throughput states'. (B) COUNTERFACTUAL ADVANTAGE — pair ticks (or paired pool-seed runs) with near-identical feature vectors but OPPOSITE allow bits on thread t, and label the action by the dps DELTA; this is the only label that credits the ACTION rather than the state, and it is exactly what the reproducible warm-pool sweep was built to generate (--pool-seed matched pairs). (C) COLLAPSE-ONSET classification — a rare-positive binary label (does a metastable collapse, forward dps < the regime's collapse floor e.g. greedy's 169 vs 183, occur in [t+d, t+d+W]?); this is the objective-ALIGNED label because the headline is variance/tail removal, and a mean-dps regressor (strategy A) is nearly flat over the 96-98% non-collapse windows and under-weights the 2-4% that are the entire lever. I treat the deployable policy as a fusion: a CHEAP online gate whose default is allow, that DENIES only when the model flags RSS/over-saturation, and ALLOWS pre-emptively (forces the gate open) at the collapse-onset signature — i.e. the asymmetry of the contract (deny is the only discretionary risk; the forced-flush floor guarantees liveness) means the model only needs to decide deny-vs-default-allow plus pre-emptive-allow, a 1-D thresholded score, not a rich classifier.

INFERENCE CONTRACT: whatever model is fit, the per-tick evaluation must be a handful of dot-products / comparisons over the marshalled IssueFeatures (T~3-4 threads), producing T bits written to the allow atomics. The hot gate read stays one relaxed atomic load (unchanged). So the model family is constrained to: a per-thread linear/logistic score, a tiny GBDT (depth<=4, <100 trees) evaluated per thread, or at most a 1-hidden-layer MLP on the per-thread feature vector + a few aggregates — anything whose per-tick cost is microseconds and carries no XLA/device dependency (it runs in the C++ control thread or a trivial hand-coded scorer; the trained weights are exported as a constexpr table, P9/P1 derive-from-one-source).

## Candidates

### C1 — Per-thread logistic deny-gate on static-derived features (collapse-onset label)
- **maps features to gate:** A single per-thread logistic score s[t] = sigmoid(w . x[t] + b) over the static_derived vector for thread t: inflight_fill, ready_pressure, convoy_score, coalescing_degree (level form, history-free), plus the two aggregates agg_inflight/mean_fill and (when wired) rtt_load. Gate = deny iff s[t] > tau_deny AND the thread is in the saturation regime (high inflight_fill), allow otherwise; a SECOND threshold tau_allow on convoy_score forces allow at the collapse-onset signature. Same weight vector for all threads (per-thread is just the input row), so T evaluations/tick.
- **objective or signal:** Strategy-C collapse-onset binary label: positive iff forward dps in [t+d, t+d+W] drops below the regime's collapse floor (the 169 vs 183 gap the doc quantifies). Trained as imbalanced binary classification (class-weight or focal loss for the 2-4% positive rate). The label is computed offline from the logged timestamped decision counter.
- **tuning or training:** Fit with L2-regularized logistic regression (or a tiny LBFGS) on the pooled multi-seed multi-p_deny traces; tau_deny/tau_allow chosen on a held-out seed to maximize the MIN of windowed dps (ADR-0009's MIN-beats-MAX bar is the variance objective). Class-weight ~ inverse positive frequency.
- **stability and liveness:** Liveness guaranteed by the runner's ungated forced-flush floor regardless of gate (a denied thread still drains at depth-1). To avoid allow/deny CHATTER (which itself induces convoy), add the allow_history (deny duty-cycle) feature as a hysteresis input or band the thresholds — the integral term the Phase-2 surface flags. Stable because the score is monotone-ish in convoy_score.
- **inference cost:** One sigmoid dot-product over ~6-8 features per thread, two threshold compares, T writes. Sub-microsecond; runs entirely in the C++ control thread. Zero hot-path cost (gate read unchanged).
- **pros:** Cheapest possible inference (one dot product + two compares per thread, weights as a constexpr table — P9-clean, no XLA). Directly targets the variance/tail lever the doc identifies as the actual win, not the near-flat mean. Interpretable weights survive ADR-0009 substantiation (you can read off which feature drives a deny). Needs NO server-rows channel and NO RTT if you drop rtt_load (degrades gracefully to a no-gap policy). The asymmetric allow-default + deny-only-on-flag matches the control contract's risk structure (forced-flush floor bounds the downside).
- **cons:** Labels STATE-correlation, not action effect — it learns 'these features precede collapse' and gates against entering those states, but cannot distinguish a deny that PREVENTED collapse from a deny that was irrelevant (no counterfactual). Linear boundary may miss the convoy conjunction unless convoy_score (the product feature) is pre-engineered in — which it is in Phase 2, so this is mitigated by feature engineering rather than model capacity. Rare-positive label needs many seeds to get class balance.
- **adr fit:** Strong ADR-0012 fit: P2 (plugs into IssuePolicy seam, weights exported from one source — P1/P7-style single authoritative table), P9 (pure function of the typed IssueFeatures, no untyped effect), P4 (thresholds are live tunables not ctor invariants). ADR-0008: action stays 'issue', overcommit is downstream. ADR-0009: reproducible warm-pool sweep is the substantiation.
- **risks:** If the collapse signature is not linearly separable in the engineered features, the gate underfits and reverts to ~identity (safe but useless). The d (lead) horizon and W must be tuned or the onset label is mis-timed relative to the RTT-lagged effect.

### C2 — Counterfactual-advantage regression (matched-pair paired-run label), tiny GBDT
- **maps features to gate:** A small gradient-boosted tree (depth<=4, <100 trees) predicts the per-tick, per-thread ADVANTAGE of allow vs deny: adv[t] = E[future dps | allow] - E[future dps | deny] given the feature row. Gate = allow iff predicted adv[t] > 0 (with a deadband to avoid chatter); equivalently deny iff the model predicts denying this thread now does not cost (or improves) future dps. Features: the full static_derived set INCLUDING the aggregates and rss_pressure_proxy (so the model can learn 'deny the threads nearest the RSS ceiling when the server is already fed').
- **objective or signal:** Strategy-B counterfactual advantage: estimated from PAIRED runs sharing a --pool-seed but differing only in the injected gate decision at controlled divergence points (or by matching ticks with near-identical features and opposite allow bits within the exploratory traces). The label is the realized forward-windowed dps delta between the as-taken action and its counterfactual. This is the only label that credits the ACTION, not the state.
- **tuning or training:** Train the GBDT (e.g. a depth-capped histogram booster) on the advantage labels; early-stop on a held-out seed by the windowed-dps-MIN metric. The deadband around adv=0 is the live tunable. Requires the paired-run generator as a data-collection harness extension.
- **stability and liveness:** Same forced-flush liveness floor. Advantage-thresholding with a deadband is naturally less chattery than a raw classifier. Risk of instability if the advantage model is over-confident in a regime under-sampled by the pairs — mitigate by conservative deadband (default allow when |adv| small).
- **inference cost:** Per thread: walk <100 shallow trees (each a few compares) — tens of microseconds for T threads, still vastly cheaper than a forward and off the hot path. Exportable as a flat (feature_idx, threshold, left, right, leaf) array the C++ control thread walks (P9-clean, no library dependency).
- **pros:** Directly answers the control question — 'does denying THIS thread NOW help?' — which strategy A/C cannot. GBDT captures the convoy CONJUNCTION (ready high AND inflight low) without hand-engineering the product, and handles the regime non-linearity (S_min=16 under-coalesce vs S_min=32 peak) via splits. Leverages exactly the asset the harness was built for (--pool-seed reproducible matched pairs, ADR-0009). Distinguishes 'denied and dps held' (correct) from 'denied and dps would have held anyway' (no-op).
- **cons:** Counterfactual data is EXPENSIVE: paired runs that diverge only at a controlled point require either deterministic replay to a divergence tick then a forced flip (engineering on the warm-pool replay), or heavy reliance on feature-matching within exploratory traces (high variance for the rare collapse region — exactly where matched pairs are scarcest). GBDT inference is more expensive than a dot product (must export the tree ensemble as a constexpr decision-table or a flattened threshold array). Advantage estimates are noisy because the outcome is global dps attributed to one thread's bit.
- **adr fit:** ADR-0012 P2/P9 fit (injected policy, tree table from one source). ADR-0009 fit is the STRONGEST of the candidates — the counterfactual is anchored to the reproducible matched-pair sweep, which is the project's substantiation mechanism. ADR-0008: the label is about the issue action, overcommit stays emergent.
- **risks:** The paired-run data generation is the biggest engineering lift and the biggest source of bias (if divergence points are not representative). Tree ensembles are harder to audit than a linear gate (ADR-0009 substantiation cost). If matched pairs in the collapse region are too few, the model is blind exactly where the win lives.

### C3 — Sequence/state model (small GRU or temporal-window MLP) on per-thread feature trajectories, distilled to a cheap online scorer
- **maps features to gate:** A tiny recurrent or windowed model consumes the last L control ticks of the per-thread feature trajectory (inflight_fill, ready_pressure, convoy_score, rtt_trend, issue_rate, coalescing_degree — the HISTORY-bearing derived features) plus aggregates, and emits a per-thread allow/deny logit. Because the collapse is a metastable TRAJECTORY (the convoy forms over several ticks as the RTT-lagged effect propagates), a model that sees the trend disambiguates 'rising rtt under low ready = keep allowing' from 'rising rtt with rising ready = back off' — the exact sign-disambiguation the rtt_load feature is designed for. CRITICAL: the recurrent model is trained offline; for inference it is DISTILLED into C2/C1-shaped cheap scorer (a small EWMA-state linear gate) so the control tick stays microsecond-cheap.
- **objective or signal:** Strategy A or C label over the trajectory: future_window_dps regression and/or collapse_onset classification, with the realized_inflight_depth_trajectory as an auxiliary target so the model learns the action->emergent-depth->dps causal chain the contract names (overcommit is downstream of the issue sequence + in-flight history). Multi-task: predict next-window dps AND collapse-onset AND the depth the current allow-sequence will produce.
- **tuning or training:** Train the small GRU/temporal-MLP with the multi-task losses; distill (match the distilled linear-EWMA gate's outputs to the GRU's allow logits on the trace distribution). Tune L (window length) and the EWMA alphas. Held-out-seed windowed-dps-MIN as the model-selection metric.
- **stability and liveness:** Forced-flush floor as always. The EWMA smoothing in the distilled gate is inherently anti-chatter (it IS the hysteresis). Risk: a distilled gate that diverges from the GRU in the rare collapse region (where distillation data is sparse) — guard by defaulting to allow when the distilled features are out of the training support.
- **inference cost:** DISTILLED form only: a per-thread linear/threshold gate over a handful of EWMA features the control thread maintains incrementally (the EWMA update is O(1) per thread per tick). Comparable to C1. The undistilled GRU is offline-only. So online cost is C1-class IF distillation succeeds; otherwise this candidate is not deployable under the contract.
- **pros:** Only family that natively models the TEMPORAL/LAGGED structure — the convoy is a several-tick metastable transition, and a point-in-time classifier (C1) sees the onset only at the last instant. Can use the rtt_trend / issue_rate / coalescing_degree interval features as designed. The auxiliary depth-trajectory target closes the action->emergent-overcommit gap the contract is explicit about. Distillation keeps online inference cheap.
- **cons:** Heaviest to TRAIN and the easiest to over-fit the rare collapse tail; needs the most data (contiguous traces, many seeds). The recurrent model cannot run on the control tick directly (state + matmul) — it MUST be distilled, adding a distillation step and a fidelity gap. The controller must retain per-thread history (prev_msgs/prev_leaves/prev_rtt/EWMA state + tick timestamp) which tick_() does not hold today — a real consumer-side state addition (flagged in Phase 2 as the rate/EWMA history requirement).
- **adr fit:** ADR-0012 P9: the DISTILLED gate is the deployable artifact (pure function over EWMA state, exported weights) — the GRU is an offline training tool, never on the hot path, so the seam stays clean. P4: EWMA alphas + window L are live tunables. ADR-0008: action stays issue; the depth-trajectory aux target is the contract's downstream-emergent quantity used ONLY as a label, never as an action — exactly the Phase-2 discipline.
- **risks:** Distillation may fail to preserve the temporal signal that justified the model (if the win really needs trajectory state, a memoryless distilled gate loses it — then you are back at C1 with extra steps). Highest train cost and highest over-fit risk on a 2-4% positive class. Only pursue if C1's history-free features demonstrably underfit the onset.

## Recommendation
Try C1 FIRST. It is the cheapest to build, cheapest to run (one dot product per thread, constexpr weights — exactly the P9 functional-core artifact the seam wants), and it targets the doc's actual lever (collapse-tail variance via the collapse-onset label) rather than the near-flat mean. The Phase-2 feature engineering already supplies convoy_score (the conjunction) and rss_pressure_proxy (the saturation guard) as pre-computed inputs, so a LINEAR boundary is plausible without model capacity — and if C1's held-out windowed-dps-MIN ties or beats the gen-floor's [183-184] band while a no-controller identity run swings [169-183], that is the whole result, measured per ADR-0009 on --pool-seed-matched configs. Establish the IDENTITY-policy baseline (already the default) and a random-gate baseline first so any C1 gain is attributable.

Escalate to C2 (counterfactual-advantage GBDT) ONLY if C1 underfits — specifically if its denies cannot be shown to PREVENT collapse vs merely correlate with non-collapse states. C2 is the right tool the moment you need to credit the ACTION, and it is the candidate most tightly bound to the project's substantiation asset (the --pool-seed matched-pair sweep), but it costs a paired-run data-generation harness, so it is not the first move. C3 (sequence model) is a research escalation reserved for the case where C1 AND C2 both show the onset is only separable WITH trajectory history (rising-rtt-with-ready over several ticks) — and even then the deployable artifact is its DISTILLED memoryless gate, so C3's payoff must beat C1 by enough to justify the distillation fidelity risk. Net: C1 to ship, C2 to credit-assign, C3 only if the temporal structure proves load-bearing.

One design invariant across all three (ADR-0008 / the control contract): the model output is ALWAYS the per-thread issue allow/deny bit; the emergent overcommit depth appears only as an offline LABEL (realized_inflight_depth_trajectory) and an online aggregate OBSERVATION (agg_inflight), never as a model action. And the default is allow — the model's discretionary power is to DENY (and to force-allow at onset); the runner's ungated forced-flush floor bounds every policy's worst case to depth-1 liveness.

## What the fixture must provide
The fixture/features must provide, to make this supervised program runnable:

(1) AN EXPLORATORY GATE POLICY for data collection — the IssuePolicy seam must be fed a Bernoulli/epsilon-greedy random gate (deny-prob swept across runs) instead of identity, so the logged action varies independently of state. This is a new IssuePolicy lambda, not a fixture change; the seam already accepts it. Without action variation the labels confound action with state and every model learns the heuristic, not the value.

(2) A PER-TICK TRACE LOGGER — the controller's tick_() must, under a logging flag, emit one line per tick carrying: the full IssueFeatures snapshot (all per-thread vectors + aggregates), the written allow vector, a steady_clock timestamp, and the current total_decisions counter (so forward windowed dps is computable offline). This requires the controller to read total_decisions (currently a runner-local atomic — it must be exposed to the control tick, a small plumbing add) and to retain a tick timestamp. Off the hot path; one appended line per slow tick.

(3) CONTROLLER-SIDE HISTORY STATE for the derived features the Phase-2 surface flags as consumer-side derivations: prev_msgs[]/prev_leaves[]/prev_rtt[] + a prev-tick timestamp + the prior allow vector, so issue_rate, coalescing_degree (interval form), rtt EWMA, allow_history (deny duty-cycle), and rss_pressure_proxy (recent-tick max) are computable. tick_() holds NONE of this today. C1 can run on the history-FREE subset (level-form coalescing_degree, point-in-time fills, convoy_score) and degrade gracefully; C2/C3 need the history.

(4) THE RTT FIELD ACTUALLY FILLED (trivial_to_add, producer-local) — to use rtt_load / rtt_trend, the producer must store per-corr submit mono_ns (a corr-id->time map alongside WireLeafPool::inflight_, which holds only slot lists today), compute now-submit at recv_batch resolution, EWMA into my_rtt, and pass it as publish()'s 6th arg (defaulted 0.0 and never passed at runner_wire_batched.cpp:783-784 today). No cross-process channel. All three candidates IMPROVE with RTT but C1 can run without it.

(5) THE OFFLINE GROUND-TRUTH JOIN for regime_annotation and the server-rows distillation — every trace stamped with its --pool-seed and full WireRunnerConfig (D, S_min, chunk_floor, N, theta_server), and the subprocess-exit SERVER_STATS mean_real_rows_per_fwd captured per run. This is the ONLY source of the true server_rows_per_forward (no live carrier — gap_needs_channel, the reply frame is a pure value codec). An offline model can train with the TRUE rows/forward as an auxiliary target/feature and distill into a deployable policy that runs on the in-process surrogate (server_fed_proxy / agg_inflight), so the gap-channel signal reaches the online gate WITHOUT a P7 codec change.

(6) THE EVAL HARNESS for substantiation (ADR-0009) — wire_ab_bench --sweep-configs with --pool-seed must be the train/eval generator: paired runs (identity vs trained policy; and for C2, paired runs differing only at controlled divergence points) sharing a pool-seed so the only variation is the policy. The success metric is the held-out-seed windowed-dps MIN (the variance/tail bar), reported on the same config grid as the doc's [169-183] vs [183-184] table.

NOT NEEDED: any new live server->producer ZMQ/redis channel (the offline SERVER_STATS + distillation substitutes for it); any P7 wire/codec extension; any change to the hot gate read (one relaxed atomic, unchanged) or the runner's fixed D ceiling (the controller never sets D — action stays allow/deny).

## Key uncertainties
(1) Is the collapse-onset signature linearly separable in the engineered Phase-2 features (convoy_score, rss_pressure, rtt_trend)? If yes, C1 ships; if no, C2/C3 are needed. This is the pivotal unknown and is cheap to test (fit C1, inspect held-out separation) before committing to the expensive candidates.

(2) Class balance: the collapse positive is 2-4% of windows. How many seeds/p_deny settings does the exploratory sweep need to get enough positives for a stable boundary AND enough matched counterfactual pairs IN THE COLLAPSE REGION (where C2's pairs are scarcest)? The rare-positive region is simultaneously the highest-value and the lowest-data region.

(3) The action->outcome lag: the convoy forms over several ticks (RTT-lagged), so the lead horizon d and window W in the onset label must align with the actual metastable-transition timescale, which is not yet measured. A mis-set d times the onset label wrong and any model trained on it gates too late.

(4) Attribution noise: the outcome (forward dps) is GLOBAL but the action is PER-THREAD. Even the counterfactual (C2) attributes a global dps delta to one thread's bit; with T~3-4 threads and a single server the per-thread credit is intrinsically noisy. It is an open question whether per-thread gating gives a cleaner signal than a single global allow-fraction action (which would change the action space — out of scope for this contract, but a real altitude question for whether per-thread is the right granularity).

(5) Whether the in-process server_fed_proxy surrogate tracks the true SERVER_STATS rows/forward well enough that a policy distilled with the true signal generalizes when deployed on the surrogate — the doc shows wire rows/msg (28) and srv rows/fwd (108) diverge sharply, so the surrogate's fidelity to the objective is unverified.

(6) Cadence_ms vs the metastable timescale: the control tick is SLOW and publish is event-driven-faster-but-sampled. If the collapse onset-to-collapse interval is shorter than a few cadence_ms ticks, no tick-rate policy can react in time and the lever has to move to the gen-chunk-floor (structural) rather than the controller (reactive) — i.e. the supervised policy may be fundamentally rate-limited by the tick cadence relative to the phenomenon it targets.
