# Issue-controller policy design — synthesis

Six-agent design exploration for the `IssueController` per-thread **issue gate** policy (`cpp/include/chocofarm/issue_controller.hpp`). Phases: [feature surface](01-feature-surface.md) → [derived features](02-derived-features.md) → design space ([static control](03-design-static-control.md) · [supervised](03-design-supervised.md) · [online](03-design-online.md)) → this synthesis.

## Ranked recommendations

**1. C8/identity + single-threshold affine baseline (Exploration-1 C8) as the mandatory ADR-0009 regression anchor, run FIRST and unconditionally. allow[t]=1 (the seam's documented default, byte-unchanged) plus a one-constant ablation allow[t]=(inflight_fill[t]<c). Zero new code beyond the seam (it is already the identity default at issue_controller.hpp:159), zero hot-path cost, zero risk. It establishes the A/B control the whole method requires and reproduces greedy's [169-183] swing so any candidate's variance reduction is attributable.**

> The empirical headline (server-gen-floor-result.md, read end to end) is that the win is VARIANCE not mean: the best gen-floor's MIN (183) only TIES greedy's MAX (183), so by the strict MIN-beats-MAX bar (ADR-0009, line 41) it does not raise the ceiling — it removes the [169] downside. Any controller claim is therefore a tail-variance claim measured against a baseline, and ADR-0009/ADR-0012-P6 make the baseline non-optional. It costs nothing (the seam defaults to it) and it is the only thing that makes every subsequent number honest. The single-threshold affine variant is the ablation that isolates whether hysteresis (rank 2) buys anything over a bare threshold.

**2. BUT FIRST run a decisive empirical pre-check before committing to ANY level-based policy: log inflight[t] on the post-refill tick across a warm-pool run at chunk_floor OFF and ON. The runner header (runner_wire_batched.cpp:732-738) and SYNTHESIS §0 state that at chunk_floor OFF, drain-all empties the ready set in one issue, so depth IS IDENTICALLY 1 and D is unused — the gate has nothing to deny. publish() also fires AFTER refill() (line 780-784), so the policy sees the topped-up depth, not the pre-reply idle trough. If inflight_fill has no dynamic range on that tick at chunk_floor OFF, every level-based candidate (hysteresis, PI, logistic-on-fill) degenerates to identity and the only observable carrying the idle signal is RTT.**

> This is the pivotal, cheapest-to-run unknown shared by all three explorations (Exploration-1 key-uncertainty 1-2, Exploration-2 uncertainty 1, Exploration-3 uncertainty 3-4). It determines the entire downstream design: it tells you whether the controller can act at all at the operating point the gen-floor result was measured at, and whether RTT must be wired before anything else. It is one logging flag + one warm-pool run. Doing it before building any policy avoids investing in a level-based controller that the post-refill/drain-all structure renders inert. It also resolves the deeper question the contract raises: the gate can only RAISE depth by not-denying, and at depth-1 there is nothing to deny — so the controller's realistic role is to REGULATE around whatever depth chunk_floor+D already create, trimming RSS/variance, not to CREATE depth (Exploration-1 key-uncertainty 2).

**3. C3 — adaptive/self-tuning regulation (Exploration-3 C3): hold an aggregate emergent-depth setpoint via a cheap PI/integral loop on mean_fill (= sum_inflight/(T*d_ceiling)), realize the continuous control fraction per-thread by ALLOWING the ceil(u*T) threads with the highest convoy_score and DENYING the rest, seed the setpoint at the documented flat optimum (S_min=32/D>=16 -> mean_fill plateau), and adapt it only for drift by slow extremum-seeking on the in-process dps proxy. Equivalently expressible as Exploration-1 C5 (Lyapunov drift-plus-penalty), which is the same aggregate-regulation idea with a provable backlog bound (one tunable V) and is more idiomatic to the project's dual-bound culture.**

> The doc's optimum is a FLAT plateau (S_min in {32,64}, D in {8,16,32} all ~182-184). A flat optimum is the ideal target for a setpoint REGULATOR and the worst target for a reward MAXIMIZER (no mean gradient to climb) — this single observation is why three independent explorations all converge here and away from mean-dps RL. C3/C5 regulate the AGGREGATE depth the SINGLE server actually sees (one server batches across all T threads), which structurally sidesteps the per-thread-credit-assignment problem that is the common weakness of every reward-based candidate (the reward is global, the action per-thread). The convoy_score ranking puts the allows exactly at the collapse onset (the variance lever), the integral/V term is the natural anti-chatter (an oscillating allow pattern itself induces the convoy), and it is the cheapest and most stable candidate. ADR-0008 fit is the strongest of all candidates: it regulates aggregate depth and ranks per-thread, never deciding 'overcommit' — the emergent fractional depth is literally the downstream CV it tracks, matching the control contract verbatim.

**4. C1 — per-thread hysteresis (Schmitt-trigger) on inflight_fill with a convoy_score force-allow override (Exploration-1 C1), run as the SIMPLEST contrastive learner alongside rank 3. Two-threshold band [f_lo, f_hi] on fill with one latch bit per thread; convoy_score = ready_pressure*(1-fill) overrides to force-allow at the starvation onset regardless of the latch. Equivalently, the Exploration-3 C1 contextual bandit (LinUCB/Thompson) with a convoy-shaped reward serves the same contrastive role in the learning register.**

> If a trivial per-thread hysteresis (or a one-feature bandit) matches the aggregate regulator, the regulator's extra structure is unjustified (ADR-0007 density); if it underperforms (likely, because the post-refill smoothing flattens fill and the flat mean starves any per-thread reward gradient), that IS the empirical confirmation that the problem is regulation-not-maximization and the aggregate altitude is the right one. Hysteresis is the textbook cure for the relay-chatter that an oscillating allow pattern would itself induce, needs NO new channel (runs on already-wired inflight/ready/d_ceiling), and its convoy_score override is the one pre-emptive DOF the variance tail needs. Cheap to build and run; the honest contrastive baseline that tells you whether rank 3 earns its complexity.

**5. WIRE THE RTT FIELD (trivial_to_add, producer-local, no cross-process channel) as a parallel enabling task, NOT a policy itself: add a corr-id -> submit mono_ns store alongside WireLeafPool::inflight_ (which holds only slot vectors today), subtract at recv_batch resolution (runner line 763-765), EWMA into a per-thread my_rtt, and pass it as publish()'s 6th arg (currently defaulted 0.0 and never passed at line 784). This unlocks the latency-aware candidates (Exploration-1 C2 AIMD, C5 Lyapunov utility, C6 Little's-law CONWIP) and the rtt_load disambiguator.**

> RTT is the single best in-process disambiguator of the two convoy regimes that inflight+ready alone cannot separate: rising rtt UNDER LOW ready = the server is the genuine bottleneck (keep allowing — amortization is working), rising rtt WITH rising ready = a forming convoy (back off). If the rank-2 pre-check shows the post-refill/drain-all structure flattens inflight_fill, RTT becomes the ONLY observable still carrying the idle signal, making the level-based ranks 3-4 inert and the latency-based candidates mandatory. It is purely producer-local (no P7 codec change), so it is cheap insurance worth wiring early. C6 (Little's-law L*=lambda*W CONWIP) cannot run at all without it; C5/C2/C3 run on proxies without it but are sharply better with it.

**6. C2/C4 SUPERVISED gates (Exploration-2): per-thread logistic on the collapse-onset label (Exploration-2 C1), escalating to a counterfactual-advantage GBDT (Exploration-2 C2) only if the logistic underfits. DEFER as a second wave, gated on the rank-2 pre-check and on enough logged warm-pool seeds to balance the 2-4% rare-positive collapse class.**

> The collapse-onset label directly targets the variance lever (a mean-dps regressor is near-flat over the 96-98% non-collapse windows and under-weights the 2-4% that ARE the lever). The deployable artifact is a constexpr weight table evaluated as one dot-product per thread in the C++ control thread (P9-clean, no XLA) — fully compatible with the cheap-tick constraint. But it requires an exploratory (Bernoulli/epsilon-greedy) gate for data collection so action varies independently of state, a per-tick trace logger, and many seeds for class balance — strictly more machinery than ranks 3-4, justified only after the simple regulators are shown to leave the onset-prediction win on the table. The counterfactual GBDT (Exploration-2 C2) is the tool the moment you need to credit the ACTION not the state, and it is the candidate most tightly bound to the project's substantiation asset (--pool-seed matched pairs), but it costs a paired-run data-generation harness.

**7. C7 MPC (Exploration-1) and C2/C3 full TD-RL with eligibility traces / sequence models (Exploration-3 C2, Exploration-2 C3): HOLD as offline upper-bound references the simple controllers are SCORED against, not first ships.**

> These border on learned-policy-in-disguise (MPC optimizes online over an offline-identified plant model; TD+FA carries the deadly-triad divergence risk; the sequence model must be distilled to a memoryless gate to meet the cheap-tick constraint anyway). They are the most data-hungry (the rare-positive collapse class is exactly where their training data is scarcest), the most regime-overfit-prone (regime_annotation matters), and the within-a-run convergence is the open question given minutes-long runs and a flat-mean reward. The honest use is as the offline upper bound: if the simple regulator (rank 3) ties the MPC/RL ceiling on the warm-pool sweep, the simple thing ships and the heavy machinery is unjustified (ADR-0007 density). The +3.7%-mean / variance-only / 2-4%-tail reality makes the asymptotic-optimality advantage of these methods unmeasurable within the eval budget.

## The testing shape (ADR-0012-compliant)

### policy seam usage
A candidate policy plugs in ENTIRELY behind the existing IssuePolicy seam (issue_controller.hpp:62) with ZERO runner edits — this is already P2-correct and must stay so. The typed contract is `using IssuePolicy = std::function<void(const IssueFeatures&, std::vector<char>& allow_out)>` (features in, per-thread allow bits out). The runner depends only on the actuation surface (controller->may_issue(tid), one relaxed atomic load at runner line 746) and the metrics sink (controller->publish, line 783-784); it never names the policy. To make a candidate testable, change exactly ONE line at the injection site wire_ab_bench.cpp:257-259 — the IssueController is currently constructed with a defaulted-empty (identity) IssuePolicy; pass the candidate as the ctor's 4th argument instead. Every candidate is a closure capturing its own controller-side state (PI integrator + setpoint for C3; hysteresis latch bits for C1; logistic weight table for the supervised gate; LinUCB A_a/theta_a for the bandit), so the seam needs no signature change for any of ranks 1-7. P8 cleanliness: the signature is already honest (no lying signature — the policy genuinely consumes IssueFeatures and writes allow_out, nothing hidden). P9 cleanliness REQUIRED of every candidate: the policy is a pure-ish function of the typed IssueFeatures returning bits through the typed out-vector; its only mutation is its own controller-side state captured by-reference in the closure (the integrator/latch/weights), which is the sanctioned 'explicitly-typed Context&'-style state, not an untyped-effectful void. The trained-weights artifact for supervised/bandit candidates must be a constexpr table derived from ONE authoritative source (P1/P7 — the exported model file), not re-authored in C++. CRITICAL CONTRACT FACTS the policy must respect (verified at source): (a) the action is ISSUE allow/deny, never 'overcommit' (ADR-0008; depth>1 is the downstream emergent effect of allowed-issues x in-flight history — runner header lines 6-12); (b) the policy NEVER sets D (read-only feature d_ceiling); (c) the forced-flush backstop (runner line 748-749) stays ungated — a denied thread always drains at depth-1, so the policy's worst case is bounded to suboptimal throughput, NEVER a deadlock or RSS-past-D (P5 liveness floor); (d) the gate read is one relaxed atomic and stale values are benign-by-design (advisory, re-read every refill), so the policy may be arbitrarily slow on the cadence tick without a hot-path hazard.

### evaluation protocol
Reuse wire_ab_bench's reproducible warm-pool sweep (--sweep-configs + a per-pool --pool-seed) AS THE EVAL HARNESS — it already builds one warm pool, then runs each cf:S:D config to a budget-sized measure pass and prints SWEEP_RESULT dps + wire_rows_per_msg (wire_ab_bench.cpp:278-280). The candidate-policy A/B is: for a FIXED --pool-seed (so the warm population is byte-reproducible) and a fixed cf:S:D config, run TWO sweeps that differ ONLY in the injected IssuePolicy — identity (the rank-1 baseline, byte-unchanged) vs the candidate — so the only divergence is the policy decision. THREE baselines per config grid point, exactly as the explorations converge on: (1) IDENTITY (the seam default, reproduces greedy's [169-183]); (2) the BEST FIXED (S,D) config from the doc (gen-floor ON, S_min=32, D=16, theta_server=0 — the [183-184] target the controller must match or beat); (3) the candidate policy. Run the grid at BOTH chunk_floor OFF (the depth-1 regime where the gate has little authority — the honest stress test) and chunk_floor ON (where depth>1 exists to regulate). The success bar is ADR-0009's MIN-beats-MAX applied to the VARIANCE: report MIN, MAX, and the [min-max] SPREAD across >=N pool-seeds (the doc uses 3 iters/config in runs/server_gen_floor_grid/refine-*; power UP to many more seeds because the lever is a 2-4% rare-event tail — a mean-powered N is under-powered for a tail effect, Exploration-1 uncertainty 3). The PRIMARY metric is the spread/min, NOT the mean: a candidate wins iff it shrinks the [169-183] swing toward [183-184] (removes the collapse tail) without lowering the MAX and without producer-RSS runaway. Guard every run with the doc's 6 GiB producer-RSS kill-ceiling + a 300s timeout (the sweep already does) so an RSS-runaway candidate fails loud rather than wedging.

### metrics
PRIMARY (the documented lever): the dps [min-max] spread and the MIN over pool-seeds — variance/collapse-tail, not mean. A candidate is reported as a variance result the way the doc reports it: a per-config row (config | dps mean | [min-max] | srv rows/fwd | wire rows/msg) so it slots directly into the doc's existing table format. SECONDARY: mean dps (expect near-flat, +3.7%-class; report it honestly as ceiling-tied if MIN only ties greedy's MAX). DIAGNOSTIC (to explain WHY a candidate wins or fails, per ADR-0009 capturing the investigation reproducibly): wire mean_rows_per_msg (already on SWEEP_RESULT, line 280 — the producer-side coalescing degree S), the end-of-run SERVER_STATS mean_real_rows_per_fwd (stage_a_server.py:182-183 — the true server amortization the in-process surrogate approximates), producer peak RSS (against the 6 GiB ceiling), and — once RTT is wired — the per-thread mean_rtt distribution. PROCESS/SAFETY metrics: fraction of windows below the regime's collapse floor (the 169-vs-183 gap quantified as a tail-frequency), and the allow/deny duty cycle the policy realized (to confirm it is not chattering — an oscillating allow pattern is itself a convoy inducer). The honest framing every metric carries: per the doc the win is variance-elimination iff the metastable collapse is a real production cost (~2-4% of windows, occasionally wedging) — so the metric that justifies SHIPPING is the tail-frequency reduction, and the mean is explicitly NOT the justification.

### reproducibility
ADR-0009 reproducible measurement is anchored on the --pool-seed warm pool: a fixed pool-seed makes the warm population byte-reproducible, so an identity-vs-candidate A/B at the same seed + config differs ONLY in the policy decision — the matched-pair generator the counterfactual labels need (counterfactual_allow_delta is estimated from paired pool-seed runs that diverge only in the injected policy). The two-tier bar (ADR-0009 / P6) applies as: the action-gate logic is a bit-exact invariant (identity policy is byte-identical to no-controller — assert ==), while the dps/variance is float-and-scheduling-sensitive and held to AGGREGATE behavioral terms (the [min-max] spread over >=N seeds within run-to-run noise, never bit-equal — the metastable collapse is inherently stochastic). Every dps/variance claim is captured the way the doc captures it: a dated point-in-time result file (ADR-0005 Rule 9, like server-gen-floor-result.md) with the runs/ path (runs/<grid>/<dated-run>/), the config grid, the seed set, and the [min-max] table — never an author-intuition claim. The eval is kept CHEAP and REPEATABLE by reusing the existing one-pool-many-configs structure (one warm pool amortized across the whole grid, wire_ab_bench.cpp:220-281) and the honest time-box (warmup separated from measurement, the measure pass sized to ~budget seconds floored at total_slots, lines 308-327) — so adding a policy is one ctor arg and re-running the same sweep, and the per-config cost is unchanged. Because the lever is a 2-4% rare-event tail, power the seed count UP from the doc's 3-iters confirmation set to enough seeds that the tail-frequency estimate is stable — the honest caveat (ADR-0002 no-bluff) is that a controller's measured effect may fall within run-to-run noise, so the evaluation must be powered for a tail effect not a mean effect, and a null result reported loudly is a legitimate outcome.

### data collection for learning
Supervised/online data is collected THROUGH THE SAME FIXTURE, off the hot path (the control tick is already slow; the producer publish is one relaxed atomic store on a path already paid). Two additions, both controller-side (NOT producer publishes), gated behind a logging flag so the production path stays byte-unchanged: (A) AN EXPLORATORY GATE for unconfounded data — inject a Bernoulli/epsilon-greedy IssuePolicy whose per-thread deny-probability is SWEPT across runs (e.g. p_deny in {0, 0.15, 0.35, 0.6, 0.85} + a few epsilon-greedy-around-a-heuristic runs), so the logged action varies independently of state. A fixed heuristic gate confounds action with state and the model learns the heuristic, not the value — this exploratory policy is itself just another closure behind the same seam, no fixture change. The safety envelope (forced-flush floor + D ceiling) makes this exploration RISK-FREE by construction (the worst an exploratory deny does is forgo one overcommit; the worst an exploratory allow does is one extra in-flight message, bounded by D) — the rare RL setting where unconstrained exploration is bounded by the substrate. (B) A PER-TICK TRACE LOGGER in tick_() that, under the flag, appends ONE line per slow control tick carrying: the full IssueFeatures snapshot (all per-thread vectors + d_ceiling/n_threads), the written allow vector, a steady_clock timestamp, and the running total_decisions counter (already maintained in the runner — expose it to the control tick, a small plumbing add) so forward windowed dps is computable offline. Stamp EVERY trace line with its --pool-seed and full WireRunnerConfig (cf, S_min, D, N, theta_server) — this is the regime_annotation join key AND the matched-pair key for counterfactual advantage. At subprocess exit, capture the existing SERVER_STATS mean_real_rows_per_fwd line (stage_a_server.py:182) and join it by run id — this is the OFFLINE GROUND-TRUTH for the server_rows_per_forward gap, the auxiliary target an offline model trains against and DISTILLS into a deployable policy that runs on the in-process surrogate (server_fed_proxy = leaves-egress rate), so the gap-channel signal reaches the online gate WITHOUT a live channel. The labels (future_window_dps, collapse_onset, counterfactual_allow_delta, realized_inflight_depth_trajectory) are all computed OFFLINE from these contiguous --pool-seed-stamped traces; the trace must be contiguous (every tick, not sampled) for the realized-depth-trajectory replay. The controller-side history tick_() does not hold today (prev_msgs[]/prev_leaves[]/prev_rtt[] + a prev-tick steady_clock timestamp + the prior allow vector) must be added to compute the rate/EWMA/duty-cycle derived features — flagged as consumer-side derivations, not producer publishes; rates do NOT arrive pre-differenced.

### adr0012 compliance
P2 (seam/port): the policy is injected behind IssuePolicy and the runner depends ONLY on the may_issue actuation + publish metrics sink, never the policy implementation — verified at runner lines 746 and 783-784 and at the wire_ab_bench injection site 257-259. No runner edit is needed for any candidate, which is the seam working as designed. P8 (typed-signature SSOT): the IssuePolicy signature is already honest and is the single contract — no candidate may smuggle state or effects outside (features in, allow_out bits out); a candidate's trained weights are exported from ONE source (the model artifact) and read as a constexpr table, never re-authored in C++ (this is the P1/P7 derive-don't-duplicate twin). P9 (functional-core/typed C++): each policy is a total function of typed IssueFeatures writing the typed out-vector; its controller-side state (PI integrator/setpoint, hysteresis latch, logistic weights, bandit A_a inverse) is captured by-reference as the sanctioned explicitly-typed Context-style mutation, NOT an untyped-effectful void / raw out-pointer / sentinel — the marshalling already passes std::vector<char>& not a raw char*, and IssueFeatures uses std::vector members not raw pointers, so the existing surface is P9-clean and candidates must keep it so. P4 (live-not-frozen): every tunable — PI gains, the fill setpoint, hysteresis [f_lo,f_hi], the RSS-penalty V, the exploration alpha, the logistic thresholds — is a LIVE swept cell (a ctor arg / config, tuned on the warm-pool sweep), never a frozen ctor invariant; a swept tunable's heat is decided by where it lives. P5 (fail-loud + liveness): the policy's risk is BOUNDED by construction — the ungated forced-flush floor (runner 748-749) guarantees a denied thread drains at depth-1 (never deadlocks) and the fixed D ceiling caps in-flight, so all-deny degrades to the live depth-1 greedy baseline and all-allow is the fixed-D runner; the 6 GiB RSS kill-ceiling is the fail-loud backstop against an RSS-runaway candidate. ADR-0008 throughout: the action is ISSUE allow/deny; overcommit/depth>1 is surfaced ONLY as a downstream observation (agg_inflight) and an offline label (realized_inflight_depth_trajectory), never as an action — the recommended aggregate-regulator (rank 3) is the strongest ADR-0008 fit because it tracks emergent depth as a downstream CV and ranks per-thread, never deciding overcommit.

### ergonomics notes
The whole loop is buildable with MINIMAL fixture change because the seam is already correct: a candidate is one closure + one ctor-arg change at wire_ab_bench.cpp:257-259, and the eval is the existing --sweep-configs + --pool-seed sweep re-run with that closure. Ergonomic build order: (1) run the rank-1 identity baseline + the rank-2 pre-check (log inflight_fill range on the post-refill tick at cf OFF and ON) — this is pure logging, no policy, and it tells you whether level-based policies have any authority at the operating point; (2) if fill has range, ship the rank-3 aggregate regulator (one PI integrator + convoy_score ranking, ~30 lines of closure) and the rank-4 hysteresis contrastive; (3) in parallel wire the trivial producer-local RTT field (rank 5) as insurance, since the pre-check may show the depth-1 regime flattens fill and forces the latency-based path; (4) only if the simple regulators leave a measured win on the table, stand up the exploratory-gate + trace-logger for the supervised/bandit second wave (rank 6). Keep ALL learner/controller state controller-side in tick_() behind IssuePolicy so the runner stays unaware (P2) and the production path stays byte-unchanged (the controller is default-OFF; controller_cadence_ms=0 -> no controller -> the production-shaped path, wire_ab_bench.cpp:255).

## Feature-channel gaps
THREE channel changes, in increasing cost and decreasing priority; the recommended path needs only the first two, both cheap and producer-local-or-controller-local. (1) CONTROLLER-SIDE HISTORY in tick_() (highest priority, zero producer cost, no wire change): tick_() holds no history today — it must retain prev_msgs[]/prev_leaves[]/prev_rtt[], a prev-tick steady_clock timestamp, and the prior allow vector. This unlocks every rate/EWMA/duty-cycle derived feature (issue_rate = delta-msgs/dt, coalescing_degree interval form, rtt EWMA/trend, allow_history deny-duty-cycle, rss_pressure recent-tick-max) AND the in-process dps-proxy reward (sum_t delta-leaves/dt) that the aggregate regulator (rank 3) and any reward-based learner consume. These are consumer-side derivations, NOT producer publishes — the surface explicitly warns rates do not arrive pre-differenced. It is a handful of scalar/vector members in IssueController + a steady_clock::now() capture per tick. (2) THE RTT FIELD (trivial_to_add, producer-local, NO cross-process channel): mean_rtt_ms is a sentinel today (issue_controller.hpp:54, defaulted 0.0 at publish() line 91, never passed at runner line 784). WireLeafPool::inflight_ is unordered_map<uint64_t, vector<int>> (slot lists only, no timestamps — verified). Add a parallel corr-id -> submit mono_ns store, record now at issue()/submit_batch, subtract at recv_batch resolution (runner line 763-765), EWMA into a per-thread my_rtt, and pass as publish()'s 6th arg. This is the single best in-process disambiguator (server-bottleneck vs forming-convoy) and becomes MANDATORY if the rank-2 pre-check shows the post-refill/drain-all structure flattens inflight_fill (then latency is the only observable carrying the idle signal). (3) THE server_rows_per_forward GAP (lowest priority, OPTIONAL for shipping, NOT needed on the recommended path): this is the one true gap_needs_channel item — the true server amortization metric is computed ONLY in Python, post-hoc, printed at subprocess exit as SERVER_STATS mean_real_rows_per_fwd (stage_a_server.py:182), and the reply frame [ident, *envelope, resp] is a pure value codec (encode_response, inference_wire.hpp) with NO telemetry field, so there is NO live carrier. THE DOC SAYS WIDTH IS NOT THE LEVER (the gen-floor wins with SMALLER forwards, 108 vs 195 rows/fwd — the mechanism is producer idle-time, not batch width), so the in-process server_fed_proxy (producer leaves-egress rate per interval) SUFFICES for the online policy, and the true scalar is needed only as an OFFLINE auxiliary target (distill the gap-channel signal into a deployable surrogate-driven policy via regime_annotation). IF a residual latency/variance gap proves the surrogate inadequate, close it with the LOWEST-TOUCH live channel: a redis key the server writes (it already has redis at 127.0.0.1:6380/6379 per config.py) and the controller reads on its SLOW tick — off the hot path, no P7 codec change. Do NOT extend the reply codec for it: a P7 cross-language-wire change (every side re-deriving one authoritative layout) for a slow-cadence diagnostic scalar is disproportionate; the side-channel is the proportionate mechanism. NO change is needed to the hot gate read (one relaxed atomic, unchanged), the runner's fixed D ceiling (the controller never sets D), or the production path (controller default-OFF).

## Open questions

- THE PIVOTAL ONE (rank-2 pre-check): does inflight_fill have usable dynamic range on the POST-REFILL tick? publish() fires after refill() (runner 780-784) so the policy sees topped-up depth not the pre-reply trough; and at chunk_floor OFF the runner header (732-738) + SYNTHESIS §0 state drain-all empties the ready set in one issue, so depth IS IDENTICALLY 1 and D is unused — the gate has nothing to deny. If fill is ~1 on every tick at cf OFF, all level-based candidates (ranks 3-4, PI/hysteresis/logistic-on-fill) degenerate to identity and the RTT-based path (rank 5) becomes mandatory. Cheapest to test (one logging flag + one warm-pool run); resolve BEFORE building any policy.
- Does the binary per-thread gate have enough AUTHORITY over emergent depth at all? The contract says depth is downstream of allowed-issues x in-flight history, but at the depth-1 drain-all operating point the gen-floor was measured at, the only way the gate raises depth is if denying a thread one tick lets ready accumulate so the next allowed issue coalesces more — a weak, indirect, lagged lever. chunk_floor creates depth by CHUNKING within issue() (runner 713-717), a mechanism the gate cannot reach. So the controller's realistic role may be narrower than 'emulate the gen-floor': REGULATE around whatever depth chunk_floor+D already produce (trim RSS/variance), not CREATE the depth. This bounds what ANY candidate can claim and should be stated honestly in the result.
- Is the win MEASURABLE within run-to-run noise? The doc's effect is +3.7% mean / variance-only / a 2-4% rare-positive tail. How many --pool-seeds does the sweep need to estimate the tail-frequency reduction stably (the positive class is rarest exactly where it matters most)? There is real risk a controller's measured effect falls within noise; the eval must be powered for a TAIL effect not a mean effect, and a loud null result is a legitimate outcome (ADR-0002 no-bluff).
- PER-THREAD CREDIT vs AGGREGATE ALTITUDE: the reward/dps is GLOBAL (one server batching across all T threads) but the action is per-thread.
