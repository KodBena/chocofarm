## Appendix A — Component Deep-Reads (verbatim, 11 subsystems)

Each subsystem was read in full by one independent `general-purpose` subagent under the shared audit charter. Severity vocabulary as emitted by the agent: `critical` / `major` / `minor`. Health vocabulary: `elegant` / `sound` / `messy` / `rotten` / `abomination`.

### A.1 — model (simulation Environment + belief mechanics: env.py, arrangement.py, facemodel.py, instance.json, faces.json)

**Health:** `messy`

**Summary.** The core env/Policy seam is genuinely clean and well-conceived: Environment owns dynamics/belief/simulation, Policy is a thin injected `decide` interface, and `lam` (the rate target) is correctly threaded per-call rather than frozen. But the Environment is a build-time monolith — instance, value vector, K, entry, teleport overhead, and the whole distance table are baked in `__init__` and every one of the ~30 call sites constructs `Environment()` with zero args, so the parameterization that exists is vestigial. The deeper rot is SSOT erosion: the belief/dynamics math is copy-pasted into MiniEnv, info_relaxation, and analyzer; the C(N,K) world construction lives in three files; the "honest" SenseAction abstraction in facemodel.py is fully built but dead (env reimplements it inline); and instance.json still carries the superseded 16-region detector data.

**Praise (earned, with citations):**

- Genuinely clean env/Policy boundary: chocofarm/model/env.py:8-10 docstring states the contract ('It knows nothing about HOW a decision is made — that is a Policy, passed in') and the code honors it — `simulate` (env.py:138-146) calls `policy.decide(self, ...)` and nothing in env.py imports a solver. Adding NMCS/ISMCTS/AZ is genuinely a new Policy subclass with no env edit. This is the right inversion of control.
- `lam` (the Dinkelbach rate target) is correctly threaded as a per-call argument through `decide`/`simulate`/`apply` (env.py:138, base.py:18) rather than frozen at construction — exactly the hot-where-it-belongs treatment that `value`/`K`/`entry` should also get. The fixed-point search (dinkelbach_rate, env.py:159-165) varies it freely.
- The distance memoization (env.py:66-82) is honest and well-documented: the table is built from the SAME `math.hypot` inputs as the live path, with a total fallback in `d()` for keys not in the table, so it is bit-identical rather than an approximation — a structural memo, correctly labeled as such.
- arrangement.py is a clean, effect-free value module: `Face` is a frozen dataclass (arrangement.py:32-42), the cover-at-representative-point logic correctly handles non-convex faces (arrangement.py:51-54, 71), and persist/load round-trips through faces.json. It 'knows nothing about solvers, beliefs, or travel' (arrangement.py:16-17) and lives up to it.

**Findings:**

1. **[major] Environment is a build-time monolith: instance, value, K, entry, teleport_overhead all frozen in __init__ and never varied**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* Every experimental axis the research needs to vary — the value vector (the attic het_values_eval.py experiment literally rebinds `M.value = [10.0 if i in HIGH else 1.0 ...]`), the present-count K, the entry teleport, the teleport overhead — is captured once in __init__ and closed over. There is no setter, no `with_value()`, no reconfigure. The `value=`/`entry=`/`teleport_overhead=` kwargs exist (env.py:24) but are dead: a grep of all ~30 `Environment(` call sites shows every single one is zero-arg (eval/*.py, az/*.py, bounds/*.py, tests/*.py). To run a heterogeneous-value or alternate-K sweep you must construct a fresh Environment (rebuilding the 4.5k-entry distance table and the faces) per config, and any cached object keyed to the env (id(env) tables in az/actions.py, FeatureBuilder) silently belongs to the old config. The parameterization is theatre; the real config seam is 'edit the default or rebuild from scratch'.  
   *Evidence:* env.py:24 `def __init__(self, instance_path=None, value=None, teleport_overhead=12.0, entry="CSNE"):`; env.py:32-33 `self.value = list(value) if value is not None else [1.0] * self.N` / `self.entry, self.tp = entry, float(teleport_overhead)`; every caller is zero-arg, e.g. eval/harness.py:54 `env = Environment()                      # unit values`, az/parallel.py:190 `env = Environment()`.  
   *Right abstraction:* Separate the immutable geometry/instance (treasures, teleports, faces, distance table) from the mutable scenario knobs (value vector, K, entry, lambda regime). Build the expensive geometry once; expose value/K/entry as either a lightweight Scenario passed alongside it or via copy-on-write `env.with_value(v)` that shares the distance table. Then a value sweep is a list comprehension over Scenarios, not 30 rebuilds, and no downstream id(env)-keyed cache goes stale.

2. **[major] K=5 hardcoded in env.__init__, divorced from instance.json which carries no K**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The exactly-K-of-N present prior is THE central modeling assumption (it defines the C(N,K) world set, the belief, every marginal). Yet K is a bare literal `5` wedged into a tuple-assignment in __init__, while N is read from the instance file. instance.json — the supposed single source of instance truth — has no `K` key at all. So the instance's defining parameter is split: N lives in data, K lives in code. analyzer.real_instance (analyzer.py:88) hardcodes `K: int = 5` AGAIN as a default, and minienv reduces it to `k_local`. Anyone editing instance.json to a different map cannot change K without finding the literal in env.py — and will likely miss the duplicate in analyzer.py.  
   *Evidence:* env.py:31 `self.N, self.K = len(self.treasures), 5`; instance.json has keys treasures/teleports/regions_wkt/delta_treasures/overlaps but no K; analyzer.py:88 `def real_instance(..., N: int = 20, K: int = 5)`.  
   *Right abstraction:* K belongs in instance.json next to `treasures` (it is instance data, not a tuning knob), read once in env.__init__ as `data["K"]`. N should likewise be `len(treasures)` derived (it already is in env, but analyzer.py:88 hardcodes N=20 as a default too). One source: the instance file.

3. **[critical] MiniEnv copy-pastes the entire belief/dynamics surface of Environment instead of subclassing or sharing it**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* MiniEnv re-implements marginals, filter_treasure, filter_detector, sample_world, legal_actions, and apply (minienv.py:81-115) as verbatim copies of Environment's (env.py:99-135). The bodies are character-identical down to `(1 if present else 0)`. This is the belief mechanics — the most correctness-critical, most-likely-to-be-revised code in the model. Any fix to the filtering/observation semantics (and the file's own docstrings show this code HAS been revised once, the cover_mask→face migration) must now be made in two places that have already silently drifted: MiniEnv.legal_actions iterates `self.keep` (minienv.py:99) while Environment iterates `range(self.N)` (env.py:118). A future change to apply's reward logic will be applied to env.py and forgotten in minienv.py, and the dual-bound validation (whose whole purpose is to certify correctness) will validate against stale dynamics.  
   *Evidence:* minienv.py:106-115 `def apply(self, loc, bw, collected, action, world): ... pres = bool((world >> i) & 1) ... return r, (kind, i), self.filter_treasure(bw, i, pres), nc, dt` is line-for-line env.py:125-135. minienv.py:81-84 marginals duplicates env.py:99-102 exactly.  
   *Right abstraction:* MiniEnv differs from Environment only in three data members (worlds, detectors/cover_mask subset, the keep set used in legal_actions). It should subclass Environment (or share a BeliefMechanics mixin) and override ONLY those three, inheriting apply/filter_*/marginals/sample_world unchanged. The geometry delegation it already does correctly (minienv.py:69-79); the belief math should be inherited the same way, not retyped.

4. **[major] facemodel.SenseAction is a fully-built but dead abstraction; env.py reimplements its logic inline and only references it in a comment**  
   *Pattern:* `accepted-but-ignored-param`  
   *Why cancerous:* facemodel.py defines SenseAction with cost/filter/observe/informative (facemodel.py:29-56) — the clean, cohesive 'a sense action is a face: where to stand + what it tells you' object. It is never instantiated by the live model. env.__init__ instead explodes each face into three parallel dicts (det_pt, cover_mask, and coord entries: env.py:45-46, 51-52) and reimplements observe/filter/informative inline (filter_detector env.py:108-110, the informative test in legal_actions env.py:120-123, the observe in apply env.py:134). facemodel.py even carries a 25-line `ENV_ADOPTION` comment (facemodel.py:67-92) describing how Environment 'would' consume it — adoption that never happened. So the model has TWO encodings of face-sense semantics: the live inline one in env.py and the abstracted dead one in facemodel.py, kept in sync by nobody. The `('f', k)` action shape the comment prescribes is also drifted: the live env uses `('d', i)`. The only thing that exercises SenseAction is one smoke test (test_smoke.py:53).  
   *Evidence:* facemodel.py:42-44 `def filter(self, bw, positive): hit = (bw & self.bitmask) != 0; return bw[hit if positive else ~hit]` vs the live duplicate env.py:108-110 `def filter_detector(self, bw, i, pos): hit = (bw & self.cover_mask[i]) != 0; return bw[hit if pos else ~hit]`. env.py:35 only *mentions* facemodel in a comment ('facemodel.ENV_ADOPTION'); there is no `import facemodel` in env.py.  
   *Right abstraction:* Either env.__init__ builds `self.senses = facemodel.sense_actions(faces)` and apply/legal_actions/filter_detector delegate to the SenseAction methods (the documented ENV_ADOPTION), making the face the single carrier of position+cover+semantics it claims to be — or facemodel.py is deleted and its docstring's insight folded into env. Keeping a parallel ideal-but-unused implementation alongside the real inline one is the worst of both.

5. **[major] The C(N,K) world-set construction is duplicated verbatim in three files**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The equiprobable-worlds bitmask array — `[sum(1 << t for t in c) for c in itertools.combinations(range(N), K)]` — is the foundational belief object. It appears identically in env.py:56-58, minienv.py:51-54 (over `self.keep`), and analyzer.world_array (analyzer.py:105-107). Three copies of the prior's definition. If the prior ever changes (e.g. non-uniform world weights, a different combinatorial structure), three independent edits are required and the analyzer/bounds will certify against a different prior than the env simulates.  
   *Evidence:* env.py:56-58 `self.worlds = np.array([sum(1 << t for t in c) for c in itertools.combinations(range(self.N), self.K)], dtype=np.int64)`; analyzer.py:105-107 `return np.array([sum(1 << t for t in c) for c in itertools.combinations(range(inst.N), inst.K)], dtype=np.int64)` (docstring even says 'it is the prior, ports verbatim'); minienv.py:51-54 same shape over `self.keep`.  
   *Right abstraction:* One `world_array(N, K, support=None)` helper in the model package; env, minienv, and analyzer all call it. The 'ports verbatim' comment in analyzer is an admission that this should be shared, not re-typed.

6. **[minor] instance.json carries superseded detector model data (overlaps, delta_treasures) that the live env ignores**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* instance.json is the instance SSOT, but two of its top-level arrays — `overlaps` (17 pairs) and `delta_treasures` — encode the OLD 16-region cover_mask detector model that the face arrangement replaced (the whole point of consult-002, per env.py:36-42). The live Environment never reads them (grep: consumed only by attic/chocobo_stage2_approx.py and scripts/verify_faces.py). A reader of instance.json cannot tell which fields are live (treasures, teleports, regions_wkt) and which are fossils; an edit to overlaps will silently do nothing, and delta_treasures is recomputed from scratch by analyzer.delta anyway. Stale data in the canonical input file is a latent trap.  
   *Evidence:* instance.json:116-191 `"delta_treasures": [3,4,16,19]` and `"overlaps": [[0,1],[0,2],...]`; no `.py` under chocofarm/ reads `data["overlaps"]` or `data["delta_treasures"]` (only attic/ and scripts/). env.py reads only `data["treasures"]`/`data["teleports"]`; arrangement.py reads only `data["regions_wkt"]`.  
   *Right abstraction:* Drop overlaps/delta_treasures from instance.json (they are derivable and superseded), or move them to a clearly-marked `legacy`/`derived` section. The instance file should contain only the live, non-derivable instance facts.

7. **[minor] The episode-horizon constant 40 is hardcoded independently in at least four places**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The max-steps truncation that bounds an episode is a model-level invariant (it must exceed the longest sensible plan, K collects + a few senses). It appears as a bare literal `40` in env.simulate's default (env.py:138 `max_steps=40`), in solvers/base.py:125 `for _ in range(40)` (the base-policy rollout), in bounds/info_relaxation.py:476 `for _step in range(40)`, and in az/exit_loop.py:60 `max_steps=40`. These four loops must agree for the rollout value to match the simulated value, but nothing ties them together. Raising the horizon for a longer instance means hunting four literals; missing one biases a value estimate silently.  
   *Evidence:* env.py:138 `def simulate(self, policy, world, lam, rng, max_steps=40):`; solvers/base.py:125 `for _ in range(40):` (in `_base_value`); bounds/info_relaxation.py:476 `for _step in range(40):`; az/exit_loop.py:60 `max_steps=40`.  
   *Right abstraction:* A single model-level constant (e.g. `Environment.MAX_STEPS` or derived as a function of N/K) referenced everywhere. Better: most of these rollouts terminate naturally on TERMINATE, so the cap is a safety net — it should be one named guard, not four magic 40s.

8. **[minor] instance loading is reimplemented in three places, each re-opening instance.json with the same parsing boilerplate**  
   *Pattern:* `leaky-boundary`  
   *Why cancerous:* Loading the instance (open file, parse treasures with `{int(i): tuple(xy) ...}`, parse teleports) is done identically in env.__init__ (env.py:28-30), arrangement.detection_regions (arrangement.py:45-48, reads regions_wkt), and analyzer.real_instance (analyzer.py:90-93). The path is computed by the same `os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", "data", ...)` dance in env.py:26-27 and arrangement.py:28-29 — coupling instance discovery to the module's position in the file tree (move the file and the default path breaks). There is no single `load_instance()` returning a parsed Instance object; the treasure-dict comprehension `{int(i): tuple(xy) for i, xy in data["treasures"].items()}` is typed three times.  
   *Evidence:* env.py:26-30 `instance_path = os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", "data", "instance.json"); data = json.load(open(instance_path)); self.treasures = {int(i): tuple(xy) for i, xy in data["treasures"].items()}`; arrangement.py:28 `INSTANCE = os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", "data", "instance.json")`; analyzer.py:90-93 reopens and reparses the same way.  
   *Right abstraction:* A single `load_instance(path=DEFAULT) -> Instance` in the model package that owns the path-resolution and parsing; env, arrangement, and analyzer consume its output. The data directory location should be resolved once (e.g. importlib.resources), not via three independent `__file__`-relative joins.

9. **[minor] az/actions.py caches slot tables keyed by id(env) — hidden global state with a lifetime/aliasing hazard tied to env identity**  
   *Pattern:* `hidden-global-state`  
   *Why cancerous:* Because the env freezes its action space at construction, az/actions.py memoizes the slot<->action bijection in a module-global dict `_SLOT_TABLES` keyed by `id(env)` (actions.py:48,55-65). id() reuse after garbage collection means a NEW env that happens to reuse a freed id silently inherits the old env's slot table; conversely the cache never evicts, so long-lived multi-env runs leak. This is a direct downstream symptom of the frozen-env design: because there is no first-class 'action space' object owned by the env, a global keyed on object identity is bolted on. FeatureBuilder (features.py:96-107) similarly snapshots env.N/K/detectors/cover_mask at its own construction — if the env's value/K could change live (it can't, see the frozen-monolith finding), these snapshots would silently diverge.  
   *Evidence:* actions.py:48 `_SLOT_TABLES = {}`; actions.py:55-57 `key = id(env); tabs = _SLOT_TABLES.get(key); if tabs is None:`; features.py:99-106 `self.K = env.K; self.detectors = list(env.detectors); ... self.cover = np.array([env.cover_mask[i] for i in self.detectors], ...)`.  
   *Right abstraction:* The env should own its action-space mapping (an attribute computed in __init__), so consumers read `env.slot_tables` directly — no module global, no id() keying, lifetime tied to the env object itself. id()-keyed caches are an anti-pattern precisely because identity is not value-stable across GC.

**Frozen-at-construction (config that should be live):**

- **value (the per-treasure reward vector)** — `env.py:32 `self.value = list(value) if value is not None else [1.0] * self.N`` — Heterogeneous-value sweeps are a first-class experiment (attic/het_values_eval.py rebinds the value vector globally to test exactly this: `M.value = [10.0 if i in HIGH else 1.0 ...]`). value does not enter the distance table or the world set, so it is cheap to vary — yet it is frozen in __init__ and every live caller takes the [1.0]*N default. It should be a live attribute or a copy-on-write `with_value()` so a value sweep doesn't rebuild the env.
- **K (the exactly-K-of-N present count)** — `env.py:31 `self.N, self.K = len(self.treasures), 5`` — K defines the world set and the whole belief; MiniEnv already demonstrates that varying K (to k_local) is a wanted operation (minienv.py:43). Hardcoding it as a literal 5 in a tuple-assignment, absent from instance.json, makes a K-sweep require editing source in two files (env.py and analyzer.py:88).
- **entry (the starting teleport)** — `env.py:33 `self.entry, self.tp = entry, float(teleport_overhead)`` — Which teleport you start from is a natural what-if (the instance has three: CSNE/CSCE/tau_4). It only affects the first leg of every episode, not any precomputed structure, so it is trivially safe to vary per-run — but it is closed over at construction and defaulted to 'CSNE' at every call site.
- **teleport_overhead (the fixed exit/teleport toll, 12.0)** — `env.py:24,33 `teleport_overhead=12.0` ... `self.tp = float(teleport_overhead)`` — The teleport overhead is a sensitivity knob (how the optimal policy trades exits against extra digging depends on it) and a bare magic constant 12.0 with no documented provenance. It feeds only exit_cost (env.py:85), so it is cheap to vary live for a sensitivity analysis — but it is frozen at construction and never passed.

### A.2 — solvers

**Health:** `messy`

**Summary.** The Policy ABC is a genuinely good seam (single `decide(env, loc, bw, collected, lam, rng)` contract, env owns all dynamics) and most solvers honor it cleanly. But the abstraction is undermined at the edges: DecompPolicy silently duck-types instead of subclassing, every hyperparameter (iterations, c, width, depth, level, horizon, sample budgets, candidate widths) is frozen in __init__ so sweeping requires re-instantiation, and shared machinery (candidate pruning, rollout horizons, the c=0.7 constant, the entry-anchor computation) is copy-pasted across files rather than living in one place. decomp.py is not really a god object — it is three honest layers — but it re-derives state the env already owns and freezes λ inside lru_cache closures.

**Praise (earned, with citations):**

- The Policy ABC (solvers/base.py:16-19) is the right seam and is documented as such: `def decide(self, env, loc, bw, collected, lam, rng)` with env passed in so a policy 'may freely query dynamics/belief primitives' — the env owns dynamics, the policy owns only the decision. This is clean inversion: solvers/base.py:7-9 'Everything else (dynamics, simulation, evaluation) lives in env.py'.
- lam is correctly threaded as a LIVE argument to decide() rather than frozen at construction (solvers/base.py:18, env.py:141) — the one piece of config that genuinely must be hot (the Dinkelbach penalty changes every fixed-point iteration) is the one piece that is not frozen. That is the right call and the rest of the audit's 'frozen config' findings are about everything that was NOT given this treatment.
- _base_value (solvers/base.py:121-131) is a real shared abstraction: the λ-penalised base-policy rollout-to-end is written once and reused by RolloutPolicy, SparseSamplingPolicy, ISMCTS, UCT, and NMCS — exactly the kind of common machinery that SHOULD be one function, and is.
- The UCT vs ISMCTS module docstrings (solvers/uct.py:11-62, solvers/ismcts.py:9-37) are exemplary: they precisely state the single controlled difference (information-set aggregation / determinization) and explicitly hold c, the rollout base, and the iteration knob fixed so the comparison is fair. This is research-grade discipline about what is and isn't held constant.
- GreedyStopBase (solvers/ismcts.py:45-70) earns its existence with a documented, specific correction (nets exit-relocation into the step value to stop over-collection under renewal reward) rather than being a vague 'better greedy'.

**Findings:**

1. **[major] DecompPolicy duck-types the Policy interface instead of subclassing the ABC**  
   *Pattern:* `leaky-boundary`  
   *Why cancerous:* Every other solver is `class XPolicy(Policy)` (uct.py:96, ismcts.py:103, nmcs.py:49, base.py:22/37/60/89), so the ABC's abstractmethod check and `isinstance(x, Policy)` are meaningful guards. DecompPolicy declares `class DecompPolicy:` with no base, so it is a Policy by accident of having a `decide` method with the right name. UCTPolicy's constructor literally does `isinstance(rollout, Policy)` (uct.py:118) to decide how to treat a passed-in base — pass it a DecompPolicy and that check silently fails. The one solver most likely to be used as an oracle/base (it IS used as the AZ teacher, az/dataset.py:75) is the one outside the type system. Any future `register_solver(cls)` that filters on `issubclass(cls, Policy)` drops decomp.  
   *Evidence:* decomp.py:505 `class DecompPolicy:` (no `(Policy)`), vs ismcts.py:103 `class ISMCTSPolicy(Policy):` and uct.py:118 `if isinstance(rollout, Policy):`  
   *Right abstraction:* `class DecompPolicy(Policy):`. It already satisfies the contract; the only thing missing is the declaration so the ABC and every isinstance check downstream stay honest.

2. **[major] Every solver hyperparameter is frozen in __init__ and closed over for the object's life**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* iterations, c, horizon, n_samples, near_det/near_tre, depth, width, level, playout/step samples, cand_det/cand_tre, max_steps — the entire experimental knob set — is captured once in __init__ and never re-readable per-decision. To sweep iterations you must build a fresh policy per budget (tb_runner.py:66 `pols = {c: make_policy(...) for c in configs}`, eval_uct.py:76 `pol = UCTPolicy(iterations=iters)` inside the loop). There is no way to anneal c, grow search width as belief sharpens, or scale budget with remaining horizon, because the value is `self.c`/`self.S`/`self.iterations`, not a function of live state. lam was correctly made a live decide() arg; nothing else was, even though several knobs (search width vs belief size, budget vs episode depth) are exactly the kind of thing that should adapt to live state.  
   *Evidence:* uct.py:114-117 `def __init__(self, iterations=300, c=0.7, ... ): self.iterations = int(iterations); self.c = float(c); self.horizon = int(horizon)`; base.py:64-65 `def __init__(self, base, n_samples=10, near_det=3, near_tre=3): self.base, self.S, self.nd, self.nt = base, n_samples, near_det, near_tre`; nmcs.py:78-86  
   *Right abstraction:* Either accept an optional per-decision override dict/SearchConfig in decide(), or make the budget a callable of live state (e.g. `budget(bw, depth)`), so experimentation and adaptive search don't require object re-construction. At minimum, hold these in one frozen `@dataclass SearchConfig` passed to __init__ so a sweep mutates one object instead of threading scalars through every call site.

3. **[major] decomp freezes λ inside lru_cache closures and never evicts the per-λ table cache**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* build_cluster_micro closes `lam` into the memoised `solve` (decomp.py:248-296) and `expectation` closures; the MicroSolution stores that closure (`solve=solve`, decomp.py:347) and DecompPolicy calls `sol.solve(loc, sup, col_local)` at runtime (decomp.py:634). The cached Q-values are only valid for the λ that built them. Correctness is preserved ONLY because _build keys a separate table per rounded λ (decomp.py:546 `key = round(lam, 6)`) and rebuilds the whole stack on a λ change (decide() at decomp.py:590 `self._ep['lam'] != round(lam, 6)`). But that means: (a) the Dinkelbach loop rebuilds every micro table from scratch each iteration — the expensive exact backward induction is thrown away and recomputed; (b) `self._cache` (decomp.py:540) and every `lru_cache(maxsize=None)` (decomp.py:248) grow without bound across a λ sweep — there is no eviction, so a long sweep accumulates one full solved MDP per distinct λ forever. The λ-rounding (1e-6) is also a silent SSOT for 'what counts as the same penalty' duplicated against the env's Dinkelbach precision with no shared constant.  
   *Evidence:* decomp.py:248 `@lru_cache(maxsize=None)` then decomp.py:274 `q = (wp * vp + wa * va) / W - lam * cost` (lam closed over); decomp.py:540 `self._cache = {}` with decomp.py:546 `key = round(lam, 6); if key in self._cache: return`  
   *Right abstraction:* Make λ an explicit argument to the value recursion (or store value functions in a λ-indexed structure with an LRU bound), so the table is not silently coupled to a closed-over scalar, and add a cap/eviction to _cache so a sweep does not leak one solved MDP per λ. The 1e-6 rounding should be a single named tolerance shared with the Dinkelbach driver.

4. **[major] Rollout-horizon / step-cap magic numbers (40, 24) duplicated across solvers and the env with no shared source**  
   *Pattern:* `SSOT-violation`

**Frozen-at-construction (config that should be live):**

- **Search budget (iterations / width / depth / sample counts) is frozen in __init__ and cannot scale with live state** — `uct.py:114-117, ismcts.py:110-114, base.py:64-65 (RolloutPolicy n_samples/near_det/near_tre), base.py:93-94 (SparseSampling depth/width/leaf), nmcs.py:78-86` — Good anytime/adaptive search spends MORE budget when the belief is wide and the decision matters, LESS when the belief has collapsed. Here `self.iterations`/`self.width`/`self.S` are constants, so the search cannot widen as `len(bw)` grows or deepen near a high-VoI detector chain. lam was correctly threaded live into decide(); budget deserves the same treatment (a callable of bw/depth) so experimentation and adaptive allocation don't require rebuilding the object.
- **Exploration constant c (UCB1) frozen at construction with no annealing hook** — `uct.py:115 `self.c = float(c)`, ismcts.py:111 `self.c = float(c)`` — UCB exploration is commonly annealed (high early in the budget, low as visit counts grow) or scaled to the observed reward range. Frozen `self.c` precludes any schedule; to study c-sensitivity you must construct a new policy per value (the eval scripts do exactly this), and the 'hold c fixed across UCT/ISMCTS' invariant is asserted in prose rather than shared in code.
- **λ-conditioned exact value functions are frozen inside lru_cache closures and a whole micro/macro stack is rebuilt on every λ change** — `decomp.py:248 `@lru_cache(maxsize=None)` closing over `lam` (decomp.py:274), cached as `solve=solve` decomp.py:347, gated by `round(lam,6)` table cache decomp.py:540/546` — The Dinkelbach driver changes λ every fixed-point iteration (env.py:159-165). Because the value function is welded to a closed-over λ and keyed per rounded λ, each Dinkelbach step discards and recomputes the expensive exact backward induction, and the unbounded `_cache`/`lru_cache(maxsize=None)` accumulate one full solved MDP per distinct λ for the life of the process. λ should be an explicit argument to the value recursion with a bounded cache, so it can vary live without rebuild-and-leak.
- **Episode horizon is a frozen literal in three different places instead of one live instance property** — `base.py:125 `for _ in range(40)`, env.py:138 `max_steps=40`, uct.py:114 `horizon=24` / ismcts.py:110 `max_depth=24` / nmcs.py:79 `max_steps=24`` — The horizon is a property of the problem instance (how many steps an excursion can take), not of each solver's constructor. It should be read live from the env so changing the instance's episode length updates rollout depth, search depth, and simulation depth together; today they are independent frozen numbers that already disagree (rollouts run to 40 inside depth-24 trees).

### A.3 — az-net (the value+policy MLP: numpy inference, JAX/optax training, dtypes, kernels, dataset)

**Health:** `messy`

**Summary.** The net was split into a numpy inference object (ValueMLP) and a JAX/optax trainer (JaxTrainer), which is a defensible boundary — but the weight-layout knowledge (param registry, residual toggle, masked softmax, the forward graph) is now copy-pasted across FOUR forward implementations that must be kept bit-compatible by hand, and the equivalence test only pins three of them. The fourth (mlp_jax.MlpJaxForward, falsely documented as "rejected") is live-selectable in the search and silently drops the residual block, a latent correctness bug. lr/l2 are frozen at JaxTrainer construction while the ExIt loop still threads them per-call to a function that ignores them, and the value standardization is the one piece of state mutated live through a side-channel.

**Praise (earned, with citations):**

- dtypes.py is a genuinely clean SSOT for the one switchable precision: a single DTYPE object + is_float32() helper, fail-loud on an unrecognized value (dtypes.py:35-40 'CHOCO_AZ_DTYPE={_NAME!r} is not a recognised precision'), with the integer bit-mechanics explicitly carved out as always-int64 (dtypes.py:23-24). This is the right shape for a cross-cutting knob.
- kernels.py is exemplary: the fused belief reduction (belief_marg_cover, kernels.py:27-51) does both reductions in one pass with no (nb×*) temporary, is documented as bit-exact with the numpy reduction it replaces (kernels.py:13-15), and carefully stays integer-only so the float32/float64 knob cannot perturb it. The standalone marginals_kernel + warmup() round it out. No frozen config, no hidden state — a pure kernel module.
- The f32 inference cache coherence design in mlp.py:86-101/153-201 is the rare case of a maintainer choosing an INVARIANT over per-writer gates: validity is checked by source-array identity (`is`) so any rebinding writer (load, warm-start, JaxTrainer write-back) auto-invalidates with zero cooperation. The reasoning ('not a per-writer gate') is exactly right for avoiding stale-serve bugs across multiple writers.
- test_jax_equivalence.py is a serious, load-bearing safeguard: it pins numpy-f64 _forward AND the production numpy-f32 _predict_both_f32 against the JIT'd training forward (test_production_f32_forward_matches_jax_jit, lines 132-175), explicitly comparing against the jit'd (not eager) forward because the weights are trained under it — the correct contract for a split inference/train numerics boundary.

**Findings:**

1. **[critical] mlp_jax.MlpJaxForward is documented as 'rejected' but is a LIVE fourth forward that silently drops the residual block**  
   *Pattern:* `SSOT-violation / leaky-boundary (duplicated forward graph that diverges; doc-as-SSOT lie)`  
   *Why cancerous:* mlp_jax_train.py:5 calls mlp_jax.py 'the rejected jax-jit inference experiment', but gumbel_search.py:117-122 imports MlpJaxForward and uses it as the leaf-eval forward whenever use_jax_mlp=True, and bench_hotpath.py:155 drives it on the real net. That forward (_forward_both, mlp_jax.py:41-56) reads `a2` straight into the heads with NO residual block, and refresh() (mlp_jax.py:71-79) only copies W1/b1/W2/b2/Wv/bv/Wp/bp — never Wr1/Wr2. So a net trained with --residual ON, evaluated through this path, silently computes head_in=a2 instead of a2+residual: a different, wrong net than training optimized, with no guard. The equivalence test pins _forward, _predict_both_f32, and forward_jax_jit (the TRAINING forward) — it never exercises MlpJaxForward, so this divergence is invisible to CI. Four hand-synced forwards is already three too many; the fourth is both lying about its status and quietly incorrect.  
   *Evidence:* mlp_jax.py:46 `v = (a2 @ Wv + bv).ravel() * ys + ym` (heads read a2, no residual); mlp_jax.py:77 `self.Wp = jnp.asarray(net.Wp,...)` (refresh copies no Wr*); gumbel_search.py:119 `fwd = MlpJaxForward(net)`; mlp_jax_train.py:5 `mlp_jax.py (the rejected jax-jit inference experiment)`  
   *Right abstraction:* ONE parametrized forward graph (params-dict in, (v_std, logits) out) defined once and shared by both the jax trainer and any jax inference wrapper — MlpJaxForward should call the same _forward_jax that the loss differentiates, not re-implement a residual-blind copy. If MlpJaxForward is truly dead, delete it and the use_jax_mlp seam; if it is kept, it must route through the canonical forward and be added to the equivalence test. The numpy f64/f32 forwards should likewise be generated from one source (e.g. a single op list interpreted at the chosen precision) rather than three transcriptions.

2. **[major] Three numpy/jax forward transcriptions kept bit-compatible by hand; the residual block math is copy-pasted in four places**  
   *Pattern:* `DRY/SSOT-violation (parallel hand-maintained implementations of one function)`  
   *Why cancerous:* The same trunk+residual+heads graph is written out independently in mlp.py:117-141 (_forward, f64), mlp.py:214-223 (_predict_both_f32, f32), mlp_jax_train.py:60-84 (_forward_jax, training) and mlp_jax.py:44-56 (_forward_both, the residual-less jax inference). The no-outer-ReLU residual skip `head_in = a2 + zr2` and its comment are transcribed verbatim three times (mlp.py:135, mlp.py:219, mlp_jax_train.py:79). An architecture change — the stated reason for moving to autodiff ('a one-line forward edit') — is in reality a four-site edit that the equivalence test only catches for three of the sites, and only at float32 tolerance. The masked softmax is similarly triplicated (mlp.py:225-230, mlp.py:251-256, mlp_jax_train.py:90-97). This is the classic 'authoritative value lives in N places' tax: every forward edit is a synchronization ritual.  
   *Evidence:* mlp.py:135 `head_in = a2 + zr2  # pre-activation skip, NO outer ReLU`; mlp.py:219 `head_in = a2 + zr2  # NO outer ReLU (matches _forward)`; mlp_jax_train.py:79 `head_in = a2 + zr2  # pre-activation skip, NO outer ReLU (matches numpy)`  
   *Right abstraction:* Express the forward ONCE as a precision-agnostic sequence of ops over a params dict, and obtain the f64-numpy, f32-numpy, and jax variants by substituting the array library / dtype — so the residual toggle, the head wiring, and the masked softmax exist in exactly one place. The equivalence test then guards a numerics boundary (f64 vs XLA-f32 roundoff), not a transcription boundary (did someone forget to copy the ReLU).

3. **[major] lr/l2 frozen at JaxTrainer construction; train_epochs still accepts them per-call and ignores them**  
   *Pattern:* `accepted-but-ignored-param / frozen-at-construction config`  
   *Why cancerous:* JaxTrainer freezes lr into optax.adam at __init__ (mlp_jax_train.py:215 `self.opt = optax.adam(learning_rate=self.lr, ...)`) and bakes l2 into the jit'd update closures (mlp_jax_train.py:219-220 `_make_az_update(self.opt, self.l2)`). The ExIt loop still threads lr/l2 down through train_epochs(trainer, X,...,lr,l2,...) (exit_loop.py:139, called at 301-302) where they are dead parameters — the docstring admits it: exit_loop.py:147-148 'lr/l2 are fixed at trainer construction (the loop varies neither); they are accepted here to keep the prior signature shape but the trainer's configured lr/l2 are authoritative.' This is a textbook accepted-but-ignored param plus a frozen-at-build knob: there is no live LR schedule, no warmup/decay, no way to anneal l2 across iterations without tearing down the trainer and resetting Adam's moments. For an experimentation harness whose whole point is sweeping training dynamics, the single most-swept knob (LR) is welded shut at construction and a phantom seam pretends otherwise.  
   *Evidence:* mlp_jax_train.py:206 `def __init__(self, net, lr, l2=0.0, ...)`; mlp_jax_train.py:215 `self.opt = optax.adam(learning_rate=self.lr, ...)`; exit_loop.py:147-148 `lr/l2 are fixed at trainer construction (the loop varies neither) ... the trainer's configured lr/l2 are authoritative`  
   *Right abstraction:* Make lr a live input to the update step: optax supports inject_hyperparams / a schedule fn so lr can vary per call without recompiling or resetting state, and l2 is already passed into _az_loss as a traced argument (mlp_jax_train.py:123) — closing it over in _make_az_update was a deliberate choice that forfeits live tuning. Either honor the per-step lr/l2 the callers pass (drive the schedule from them) or delete the dead parameters from train_epochs and own the frozen-ness honestly. A research trainer should treat LR as hot, not cold.

4. **[major] Two-headed net abstraction: weight layout knowledge (the _params registry, the residual toggle, y-scale) is split-brained across ValueMLP and JaxTrainer**  
   *Pattern:* `leaky-boundary / split source-of-truth`  
   *Why cancerous:* The net's structural truth is encoded once as ValueMLP._params() (mlp.py:103-111) and the 'W-prefix = weight matrix' L2 rule as ValueMLP._is_weight (mlp.py:113-114) — but JaxTrainer re-derives both: it reads params via net._params() (mlp_jax_train.py:225) yet re-implements the L2 scope with its own `name.startswith('W')` (mlp_jax_train.py:118) instead of reusing _is_weight, so the two definitions of 'which params get L2' can silently drift. Worse, JaxTrainer writes weights back by setattr on registry keys (mlp_jax_train.py:238 `setattr(net, k, arr)`), meaning the trainer must know that the dict key 'W1' corresponds to the attribute self.W1 — a layout coupling the registry was supposed to hide. The residual toggle lives as `self.residual` in numpy (mlp.py:68) but as `'Wr1' in params` in jax (mlp_jax_train.py:75) — two encodings of one fact. The net is now two objects that each hold half the layout knowledge and must agree by convention.  
   *Evidence:* mlp.py:113-114 `def _is_weight(self, name): return name.startswith("W")`; mlp_jax_train.py:118 `if name.startswith("W"):` (re-derived, not _is_weight); mlp_jax_train.py:238 `setattr(net, k, arr)`; mlp_jax_train.py:75 `if "Wr1" in params:` vs mlp.py:68 `self.residual = bool(residual)`  
   *Right abstraction:* A single weight-container/spec object that owns the param dict, the L2 mask, the residual flag, and the npz layout, consumed identically by numpy inference and the jax trainer. The trainer should never setattr by key into another object's attributes; it should hand a params dict back to a container method that knows how to absorb it. One definition of 'is this a weight', one definition of 'is residual on'.

5. **[major] Value standardization (y_mean/y_std) is hot live state mutated through a side-channel that all three forwards must re-read**  
   *Pattern:* `hidden-global-state / action-at-a-distance`  
   *Why cancerous:* y_mean/y_std are stored on the net (mlp.py:80-81), set at construction, AND re-pinned live mid-run via set_value_scale (mlp.py:295-299), called from exit_loop.train_epochs (exit_loop.py:150 `net.set_value_scale(float(Y.mean()), float(Y.std()))`) every iteration as the return distribution drifts. The trainer then has to re-read these off the net on every single step (mlp_jax_train.py:254-255, 268-269) to standardize the target, the f32 inference cache has to include ym/ys in its identity check (mlp.py:176 `c["ym"] == self.y_mean`), and MlpJaxForward has to copy them in refresh (mlp_jax.py:78-79). So one mutable scalar pair is action-at-a-distance across four readers, and forgetting to re-read it anywhere yields silently mis-scaled values. The correct-by-construction story ('net's params are the single source of truth', mlp_jax_train.py:188) quietly excludes the value scale, which is mutated out-of-band by the loop.  
   *Evidence:* mlp.py:295-299 `def set_value_scale(self, y_mean, y_std): ...`; exit_loop.py:150 `net.set_value_scale(float(Y.mean()), float(Y.std()))`; mlp_jax_train.py:254-255 `ys = np.float32(self.net.y_std) ...; ym = np.float32(self.net.y_mean) ...`  
   *Right abstraction:* Standardization belongs to the target/dataset pipeline, not to a mutable field smeared across four forward implementations. Either fold (mu,sigma) into the value-head weights at write-time (so the net is always in raw-return units and there is nothing to re-read), or make the standardizer an explicit object passed alongside the params — not a hidden attribute every reader must remember to consult.

6. **[minor] _JDTYPE is recomputed and frozen-at-import in two separate jax modules from the same DTYPE**  
   *Pattern:* `DRY-violation / frozen-at-construction`  
   *Why cancerous:* dtypes.py is the SSOT for precision, but the jax-float derivation `jnp.float32 if np.dtype(DTYPE)==np.dtype(np.float32) else jnp.float64` is copy-pasted at module-import time in both mlp_jax.py:37 and mlp_jax_train.py:54, and the same conditional is open-coded again four times inside JaxTrainer's hot methods (mlp_jax_train.py:254-255, 268-269). Because it is bound at import, the bench's documented 'flip CHOCO_AZ_DTYPE and reload' workflow (dtypes.py:21-23) only works if every one of these modules is reloaded in the right order — a fragile reload dance the SSOT was meant to prevent. The numpy/jnp dtype mapping should live in dtypes.py once (e.g. a JNP_DTYPE / a to_jax_dtype() helper), not be re-derived per module.  
   *Evidence:* mlp_jax.py:37 `_JDTYPE = jnp.float32 if np.dtype(DTYPE) == np.dtype(np.float32) else jnp.float64`; mlp_jax_train.py:54 (identical line); mlp_jax_train.py:254 `ys = np.float32(self.net.y_std) if _JDTYPE == jnp.float32 else np.float64(self.net.y_std)`  
   *Right abstraction:* Expose the jax dtype from dtypes.py itself (it already owns the precision knob and is jax-free, so add a tiny lazily-evaluated accessor or a JAX_DTYPE_NAME string) so there is exactly one place that maps DTYPE→jnp dtype, consumed by both jax modules and the trainer's scalar casts.

7. **[minor] train_value.train() and exit_loop.train_epochs() are near-duplicate epoch loops with copy-pasted r2_score and standardize/Adam scaffolding**  
   *Pattern:* `copy-paste / DRY-violation`  
   *Why cancerous:* r2_score is defined verbatim in two files (train_value.py:30-33 and exit_loop.py:54-57). Both modules then implement the same permute-into-batches-call-trainer epoch loop with the same `steps = max(1, n // batch)` / `b = idx[s*batch:(s+1)*batch]` / `if len(b)==0: continue` idiom (train_value.py:58-65 vs exit_loop.py:154-161) and the same build-net-then-JaxTrainer construction (train_value.py:47-53 vs exit_loop.py:215-226). Each new training entry point copies the harness rather than calling a shared train_epochs(trainer, batches, ...). This is the 'copy-paste main()' smell: bug fixes (e.g. the empty-batch guard, the standardization order) must be applied in N places.  
   *Evidence:* train_value.py:30-33 `def r2_score(y_true, y_pred): ss_res = ...` and exit_loop.py:54-57 (identical); train_value.py:62-63 `b = idx[s * batch:(s + 1) * batch]; if len(b) == 0: continue` vs exit_loop.py:157-158 (identical)  
   *Right abstraction:* One shared training utility module: a single r2_score, a single minibatch-epoch driver that takes a trainer + (X, targets...) and a step closure, reused by both the value-only Stage-1 path and the full AZ loop. The two CLIs differ only in which trainer.step they call; everything around it should be one function.

**Frozen-at-construction (config that should be live):**

- **Learning rate (lr) is baked into optax.adam at trainer construction; no schedule/warmup/decay is possible without rebuilding the trainer and resetting Adam moments.** — `chocofarm/az/mlp_jax_train.py:215` — LR is the single most-swept training knob in a research harness, and AZ-style runs routinely anneal it across iterations. optax supports per-step lr (inject_hyperparams / schedule); freezing it forfeits the central experimentation lever and makes the per-call lr argument exit_loop threads down (exit_loop.py:302) a dead parameter.
- **L2 coefficient is closed over inside the jit'd update kernels at construction (passed into _make_az_update/_make_value_update), so it cannot be annealed live even though _az_loss already accepts it as a traced argument.** — `chocofarm/az/mlp_jax_train.py:219-220` — _az_loss(params,...,l2) (mlp_jax_train.py:123) already takes l2 as a runtime arg — closing it over in the update step was a choice that converts a hot, sweepable regularization knob into a compile-time constant for no jit benefit (a traced scalar would not force recompiles).
- **_JDTYPE (the jax precision) is bound once at module import in each jax module from DTYPE, so changing CHOCO_AZ_DTYPE at runtime requires a multi-module reload in the right order.** — `chocofarm/az/mlp_jax.py:37 and chocofarm/az/mlp_jax_train.py:54` — dtypes.py advertises a per-run precision flip for the bench (dtypes.py:21-23 'The bench harness flips it via a small reload helper'); a value frozen at import in two derived modules makes that flip fragile. The mapping DTYPE→jnp dtype should be a single live accessor in the SSOT module.
- **Adam betas/eps are constructor defaults on JaxTrainer with no CLI/loop seam at all (unlike lr/l2 which at least appear as flags).** — `chocofarm/az/mlp_jax_train.py:206` — betas=(0.9,0.999), eps=1e-8 are hard-frozen at construction and not surfaced through exit_loop's argparse, so an optimizer-dynamics sweep (a routine experiment) requires editing source. For an experimentation codebase these should be configurable inputs, not buried constructor defaults.

### A.4 — az-search (Gumbel-AlphaZero search + featurization + action-slot mapping)

**Health:** `messy`

**Summary.** The math is faithful and the hot-path engineering is genuinely careful, but the subsystem leaks its feature layout across three modules (features.py authors it positionally; actions.py and feature_response.py re-derive the offsets by hand, the latter untested), and freezes every search hyperparameter except m/n_sims at construction with no config seam reaching the parallel executor. Pervasive smaller rot: an id(env)-keyed module-global slot cache (leak + address-reuse hazard), a build() parameter that is accepted-but-ignored while a caller still pays to compute it, and a dual source of truth for env (stored field plus threaded parameter, never asserted equal). None of it is currently wrong because every env is layout-identical and the same object, but every one of these is a silent-failure landmine the moment that invariant flexes during experimentation.

**Praise (earned, with citations):**

- gumbel_search.py:30-43 — the module docstring states the exact Danihelka faithfulness contract (Sequential Halving phase schedule, full-budget spend, prior-weighted v_mix vs visit-weighted) AND names the honest simplifications (single interior determinization vs leaf c_outcome widening) with a pointer to the test that verifies them; this is how non-trivial algorithmic code should document its invariants.
- gumbel_search.py:388-407 — _v_mix is deliberately extracted as a pure function of the node 'so the prior-weighting invariant is directly unit-tested', isolating the subtle paper-correctness point (prior- not visit-weighting matters precisely because SH makes visits unequal) into a testable seam.
- features.py:62-69 + actions.py:31-33 — feature_dim and n_action_slots derive EVERY dimension from env (N, detectors, teleports) and the docstrings explicitly forbid hardcoding 16/37/90/241; the design doc's stale constants are called out as stale rather than silently inherited. This is the right SSOT instinct (undercut only by the layout *order* not getting the same treatment).
- value_target.py — a textbook SSOT extraction: the λ-penalized return-to-go was duplicated across exit_loop and dataset; it is now one module, the pure-MC path is bit-identical (n_step=None / lam_blend>=1 dispatches verbatim to suffix_returns_to_go, value_target.py:105-106), the TD(λ) forward view is implemented as the O(D) backward recurrence with the boundary derived from first principles (boot[D] = -λ·exit_c), and the HONEST RISK of bootstrap optimism is documented with the observable signature (E[T]).
- features.py:143-181 — the _belief_cache verifies _belief_key hits with np.array_equal collision checking before returning, so a key collision can never hand back a different belief's features; the safety is correctness-preserving by construction and documented as such, and the cap (features.py:116) bounds memory for any caller regardless of reset discipline.

**Findings:**

1. **[critical] Feature layout is a three-writer SSOT violation: block offsets re-derived independently in actions.py and feature_response.py**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The feature vector's block order is *authored* positionally in features.py::build by sequential `o += N` accumulation (the only true source of truth). But the slice offsets are hand-transcribed in two other modules: actions.py::legal_mask_from_features hardcodes `2*N` (available) and `5*N` (informative), and feature_response.py::feature_names hardcodes the block *order* as a literal list. Reordering a sub-block in build() — e.g. swapping `unc` and `dist` — produces NO error: build still asserts `o == self.dim`, feature_names still asserts `len(names) == feature_dim` (a count check that is order-blind), and the mask path would grab the wrong column silently. The one guard that exists (tests/test_az_loop.py::test_legal_mask_paths_agree) only compares the two mask *outputs* at a snapshot; it would catch a break in the `available` slice but does NOT cover feature_names at all (no test references it), so the diagnostic would mislabel every importance row after a reorder. Three encodings of one layout, two of them untested against the authoritative one.  
   *Evidence:* features.py:208-212 `out[o:o + N] = marg; o += N` ... `out[o:o + N] = avail; o += N` ... `out[o:o + N] = unc; o += N` (positional author) vs actions.py:112,116 `avail = feat[2 * N:3 * N]` ... `informative = feat[5 * N:5 * N + nD]` vs feature_response.py:44-46 `for sub, tag in [("marg",...),("collected",...),("available",...),("dist",...),("unc",...)]`  
   *Right abstraction:* Have features.py own a single `FEATURE_LAYOUT(env)` descriptor (an ordered list of (name, width, start) blocks, computed once from env). build() writes through it, legal_mask_from_features slices `layout['available']`/`layout['informative']` by name, and feature_names iterates it. One reorder edits one structure; all consumers follow. A test should assert the descriptor's offsets equal a fresh positional recomputation.

2. **[critical] Search hyperparameters frozen at construction with NO config seam past m/n_sims**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* c_puct, c_visit, c_scale, c_outcome, max_depth are real algorithmic knobs (the PUCT exploration constant, the Danihelka σ-transform constants, the leaf outcome-averaging width, the depth cap) — exactly the things an ExIt researcher needs to sweep. They are captured as constructor defaults and never plumbed to any experiment driver. The exit_loop CLI exposes ONLY --m and --n-sims (exit_loop.py:372-373); it constructs the search with `m=args.m, n_sims=args.n_sims` and nothing else (exit_loop.py:281). The parallel executor is worse: its signature (parallel.py:318 `def __init__(self, n_workers, cores, base_seed, m, n_sims)`) physically cannot carry the other knobs to the workers, and the worker rebuild (parallel.py:209 `GumbelAZSearch(net, _W["env"], m=_W["m"], n_sims=_W["n_sims"])`) hardcodes that two-knob seam. So to try c_puct=2.0 you must edit source. Worse, the worker DOES rebuild the search live on every weight version (parallel.py:199-209) — proving a live-reconfig path exists — yet only the net flows through it; the search config is frozen for the whole run.  
   *Evidence:* gumbel_search.py:91-92 `def __init__(self, net, env, m=12, n_sims=48, c_puct=1.25, c_visit=50.0, c_scale=1.0, c_outcome=2, max_depth=24, use_jax_mlp=False):`; exit_loop.py:281 `search = GumbelAZSearch(net, env, m=args.m, n_sims=args.n_sims)`; parallel.py:318 `def __init__(self, n_workers, cores, base_seed, m, n_sims):`  
   *Right abstraction:* Pass a single `SearchConfig` dataclass (all hyperparams) from CLI/argparse through the executor initargs into GumbelAZSearch, instead of an ever-growing positional arg list that already truncated at two. The worker's per-version rebuild then naturally picks up config changes; ExIt can anneal c_puct/width across iterations.

3. **[major] id(env)-keyed module-global slot tables: unbounded leak + address-reuse aliasing hazard**  
   *Pattern:* `hidden-global-state`  
   *Why cancerous:* _SLOT_TABLES is a process-global dict keyed by id(env). Every Environment() instance (and the codebase creates many fresh ones — exit_loop, feature_response, dataset, parallel worker, eval, dozens in tests) installs an entry that is NEVER evicted: a slow memory leak in any long process that churns envs. More dangerous: id() is the memory address; if an env is garbage-collected and a *new* env is allocated at the same address, slot_action_tables silently returns the dead env's tables. Today the layout is the same for all envs so the bug is masked, but the moment two envs differ in N or len(detectors) (a different instance_path, a variant detector set), the cache hands back the wrong bijection with no error. The key is id() precisely because Environment defines no __eq__/__hash__ (confirmed: none in env.py) — so the 'cache' is keyed on identity-as-address, the least safe possible key.  
   *Evidence:* actions.py:48 `_SLOT_TABLES = {}`; actions.py:55-64 `key = id(env)` ... `_SLOT_TABLES[key] = tabs`; env.py has NO `__eq__`/`__hash__`  
   *Right abstraction:* Cache the tables ON the env (lazily-built `env._slot_tables` attribute) or on the FeatureBuilder/search that owns it — lifetime then tracks the object, no leak, no address reuse. Or build the bijection once in the search __init__ (it already hoists `self._s2a, self._a2s`) and drop the module global entirely; the free functions can take the env attribute.

4. **[major] build()'s `marg` parameter is accepted-but-ignored, and one caller pays to compute it for nothing**  
   *Pattern:* `accepted-but-ignored-param`  
   *Why cancerous:* FeatureBuilder.build(loc, bw, collected, marg=None) advertises a `marg` seam for reusing a pre-computed marginals call, but the body discards it: the fused kernel recomputes marginals internally and `marg` is never read except the dead default. The docstring openly admits this ('marg is accepted for backward compatibility but is now ADVISORY ... a supplied marg is not consumed'). This is a lying signature: three live callers still pass `marg=marg` (netvalue_ismcts.py:55, dataset.py:60, bench_hotpath.py:70/90) believing it does something. netvalue_ismcts is the sharp edge — _leaf_value computes `marg = env.marginals(bw)` on line 54 *specifically* to feed build(), and build throws it away and recomputes via the kernel: a wasted full marginals pass on every ISMCTS leaf. An ignored param is worse than no param: it invites exactly this wasted-work pattern and silently rots when the 'authoritative value lives elsewhere'.  
   *Evidence:* features.py:183-189 `def build(self, loc, bw, collected, marg=None)` ... `so a supplied marg is not consumed`; features.py:191 `marg_in, ... = self._belief_feats(bw, marg)` then _belief_feats:161-162 recomputes `marg, cnt = belief_marg_cover(...)` ignoring the arg; netvalue_ismcts.py:54-55 `marg = env.marginals(bw)` / `feat = self.fb.build(loc, bw, collected, marg=marg)`  
   *Right abstraction:* Remove the `marg` parameter from build() and _belief_feats() entirely (the kernel is now authoritative) and delete the dead `marg=marg` at every call site, including netvalue_ismcts.py's now-pointless `env.marginals(bw)` line. If marginal-reuse is genuinely wanted, make the cache the single owner and have callers fetch from it, not push into a sink.

5. **[major] GumbelAZSearch has two sources of truth for `env`: stored at construction AND threaded through every method**  
   *Pattern:* `leaky-boundary`  
   *Why cancerous:* The search binds `self.env` at construction (gumbel_search.py:94) and derives env-shaped state from it: FeatureBuilder(env), the slot tables `self._s2a/_a2s`, `self.n_slots`, `self.term_slot`. Yet every public and recursive method ALSO takes `env` as a parameter and uses it for the dynamics (env.apply, env.sample_world, env.exit_cost). _evaluate reads `self.env` (line 135) while _simulate_root_action/_descend read the passed `env`. Nothing asserts they are the same object. If a caller ever passes an env different from the construction-time one, the feature/mask/slot machinery (frozen from self.env) silently disagrees with the dynamics (passed env): masks for one action space, transitions for another, no error. In practice they are always identical, which makes the threaded `env` param pure dead weight inherited from the Policy.decide signature — parameter-passing hell where the value is already a field. Same dual-source smell in netvalue_ismcts (self.fb built from construction env; _leaf_value uses passed env).  
   *Evidence:* gumbel_search.py:94 `self.env = env` and gumbel_search.py:135 `env = self.env` (in _evaluate) vs gumbel_search.py:166,180,189,303,325 all `def ...(self, env, ...)` threading a separate env; netvalue_ismcts.py:46 `self.fb = FeatureBuilder(env)` vs :48 `_leaf_value(self, env, ...)`  
   *Right abstraction:* Pick one. Either the search is env-bound (drop the env parameter from all internal methods, use self.env throughout; the Policy adapter passes its own env only to assert-equal) or it is env-free (don't store self.env / FeatureBuilder at construction; build per-call from the passed env). The current 'both' guarantees a silent-mismatch landmine.

6. **[minor] decide_with_value / decide_with_target duplicate the empty-belief special case and the (action, pi) packaging**  
   *Pattern:* `DRY-violation`  
   *Why cancerous:* The two public entry points each open-code the identical `if len(bw) == 0:` short-circuit (build a one-hot TERMINATE pi, return), differing only in whether they also return the exit-toll value. decide_with_target is documented as 'the thin (action, pi) wrapper over' decide_with_value, but it is NOT implemented as one — it re-derives the empty-belief branch and re-calls _decide_root independently instead of dropping the third tuple element from decide_with_value. Two copies of the empty-belief contract will drift if the terminal semantics change (e.g. exit-toll sign).  
   *Evidence:* gumbel_search.py:172-176 `if len(bw) == 0:` ... `return TERMINATE, pi, -lam * env.exit_cost(loc)` vs gumbel_search.py:182-185 `if len(bw) == 0:` ... `return TERMINATE, pi`  
   *Right abstraction:* Implement decide_with_target as the literal wrapper the docstring claims: `a, pi, _ = self.decide_with_value(...); return a, pi`. One empty-belief branch, one _decide_root call.

7. **[minor] feature_response.py docstring hardcodes the stale dimension 220 (live env is 241)**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The module docstring states importance is '220 passes over the held-out set' and groups 'rather than 220 raw rows', while the actual live feature dim is 241 (features.py itself notes the 220→241 growth). The code is correct (it uses `d = X.shape[1]`), so only the prose is stale — but it's the same magic-number-in-two-places failure that bites: a reader sizing a sweep budget off the docstring is wrong by 21 features, and it signals the number was copy-pasted rather than derived. features.py made the same mistake survivable by routing everything through feature_dim(env); this docstring didn't get the memo.  
   *Evidence:* feature_response.py:11 `so the table is readable rather than 220 raw rows` and :18 `importance is one forward pass per feature (220 passes over` vs features.py:67 `... = 241`  
   *Right abstraction:* State the dimension as 'feature_dim(env) passes' in prose, never a literal; the one authoritative number is feature_dim(env).

8. **[minor] GumbelPolicy is the better config seam (**kw) but exit_loop/parallel still pin it to m/n_sims only**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* GumbelPolicy.__init__ correctly forwards `**kw` to GumbelAZSearch, so it CAN pass c_puct etc. — a small island of good plumbing. But the only construction sites (exit_loop.py:310 eval_pol, parallel.py:270) pass just `m=..., n_sims=...`, so the **kw seam is never exercised and the eval policy silently runs at default c_puct/c_visit regardless of what self-play used. The good abstraction exists and is unused, which is its own smell: the eval and the generator can diverge in search config with no single knob tying them.  
   *Evidence:* gumbel_search.py:440-441 `def __init__(self, net, env, m=12, n_sims=48, **kw):` / `self.search = GumbelAZSearch(net, env, m=m, n_sims=n_sims, **kw)`; exit_loop.py:310 `eval_pol = GumbelPolicy(net, env, m=args.m, n_sims=args.n_sims)`  
   *Right abstraction:* Thread the same SearchConfig (see the frozen-hyperparams finding) into both the generator search and GumbelPolicy so eval and self-play are guaranteed identical search settings by construction.

**Frozen-at-construction (config that should be live):**

- **c_puct, c_visit, c_scale, c_outcome, max_depth (PUCT exploration constant, Danihelka σ-transform constants, leaf outcome-averaging width, depth cap)** — `gumbel_search.py:91-102` — These are the core search knobs an ExIt researcher sweeps and would want to anneal across iterations. They are captured as constructor defaults with no CLI flag and no path through ParallelExecutor (parallel.py:318 signature carries only m/n_sims), so changing them requires a source edit. The worker already rebuilds the search live per weight-version (parallel.py:199-209) — a live-config path exists and is used for the net but withheld from the search hyperparameters.
- **search config used by the EVAL policy vs the GENERATOR search** — `exit_loop.py:281 (generator) and exit_loop.py:310 / parallel.py:270 (eval)` — Eval and self-play construct their searches independently, each pinned to m/n_sims only, so they silently run at default c_puct/c_visit and can diverge from each other; there is no single config object guaranteeing the apprentice is evaluated under the same search it was trained to imitate. A live, shared SearchConfig would make the two provably identical.
- **slot<->action bijection tables keyed by id(env)** — `actions.py:48-64 (_SLOT_TABLES module global)` — The tables are a fixed function of env shape and should live ON the env (or the FeatureBuilder/search) so their lifetime tracks the object and a new env shape rebuilds them. Frozen in a process-global address-keyed dict, they outlive their env, never evict, and would alias to a stale layout if a differently-shaped env reuses a freed address.

### A.5 — az-orch (ExIt loop + parallel redis executor)

**Health:** `messy`

**Summary.** The ExIt loop is a clean linear generate→train→eval→checkpoint pipeline with a genuinely good redis raw-bytes transport boundary and a well-factored param-registry-driven net (de)serializer. But it is choked by an argparse god-object threaded everywhere, a search configuration (m/n_sims/base_seed) frozen into pool initargs that cannot change mid-run, accepted-but-ignored lr/l2 params, reference constants duplicated instead of computed from the env SSOT, a version+1_000_000 overload hack, and a stack of five+ multiprocessing/JAX/redis deadlock band-aids fighting a substrate the design picked itself.

**Praise (earned, with citations):**

- chocofarm/az/parallel.py:92-128 — pack_net/unpack_net enumerate the weight set from the net's OWN param registry (`for k in net._params().keys()`) and ride the residual flag in the manifest, so an optional block transports with ZERO second edit site. This is the correct DRY abstraction: the wire format is derived from the single source of truth (the net's layout), not a parallel hardcoded tuple. Exemplary.
- chocofarm/az/parallel.py:14-47 — the redis-vs-pickle transport boundary is explicitly reasoned (raw `tobytes()` + tiny JSON manifest, `np.frombuffer` reconstruct, no pickle of array payloads) with a documented cost model. The actor/learner contract (version-gated `_ensure_net`, parallel.py:196-211, rebuilds the net only when the weight version changes) is a sound, standard distributed-RL pattern.
- chocofarm/az/parallel.py:213-221 — `_task_rng` folds base_seed+version+kind+idx so the SAME logical (iteration, kind, episode) draws the SAME stream regardless of worker count, giving a verified parallel≡serial invariant (workers=1 and workers=4 bit-identical). Determinism designed in, not bolted on.
- chocofarm/az/exit_loop.py:321-332 — checkpoint EVERY iteration (net npz + history.json + latest_net.npz) makes a timeout/restart lossless and the rate-per-iteration inspectable; this is what makes the loud-abort deadlock policy tolerable.

**Findings:**

1. **[major] train_epochs accepts lr/l2 then silently ignores them — authoritative value lives in the trainer**  
   *Pattern:* `accepted-but-ignored-param`  
   *Why cancerous:* The signature and call site advertise lr/l2 as live knobs, but they are dead. JaxTrainer.__init__ closes lr/l2 over at construction (mlp_jax_train.py:215 optax.adam(learning_rate=self.lr...), :219-220 _make_az_update(self.opt, self.l2)), and train_step (mlp_jax_train.py:247) takes neither. So the lr/l2 threaded into train_epochs and forwarded from args do nothing. The docstring confesses it. This is a lie in the API: a future experimenter who sets a per-iteration LR schedule by passing different lr gets a silent no-op — the worst kind of stale-config bug, because it looks configured.  
   *Evidence:* exit_loop.py:139 `def train_epochs(trainer, X, PI, M, Y, epochs, batch, lr, l2, alpha, beta, rng):`; exit_loop.py:147-148 docstring `‘lr’/‘l2’ are fixed at trainer construction (the loop varies neither); ... the trainer's configured lr/l2 are authoritative.`; body never references lr/l2; mlp_jax_train.py:206 `def __init__(self, net, lr, l2=0.0, ...)`  
   *Right abstraction:* Drop lr/l2 from train_epochs (and the call at exit_loop.py:301-302). If per-iteration LR scheduling is wanted, give JaxTrainer a real set_lr(lr) that re-derives the optax transform/jit update — make the knob LIVE, not vestigial. A param the receiver cannot honor must not be in the signature.

2. **[major] Search config (m / n_sims / base_seed) frozen into pool initargs — cannot change mid-run**  
   *Pattern:* `frozen-at-construction`  
   *Why cancerous:* m (Gumbel root actions), n_sims (sims/decision) and base_seed are passed ONLY into ctx.Pool(..., initargs=(self.cores, base_seed, m, n_sims)) and consumed once in _worker_init into the module-global _W. The per-iteration entry points generate(net, version, worlds, lam, ...) and evaluate(net, version, worlds, lam) carry NO search-budget argument and there is no setter. So search width — the single most important compute/quality dial in an ExIt loop, the thing you most want to anneal (cheap early, deep late) or sweep — is welded shut for the pool's multi-hour life. Worse, the serial path CAN vary it (rebuilds GumbelAZSearch per iteration at exit_loop.py:281), so parallel and serial have divergent capability: an experiment expressible serially is impossible in parallel.  
   *Evidence:* parallel.py:318 `def __init__(self, n_workers, cores, base_seed, m, n_sims):`; parallel.py:328 `initargs=(self.cores, base_seed, m, n_sims),`; parallel.py:209 `GumbelAZSearch(net, _W["env"], m=_W["m"], n_sims=_W["n_sims"])`; contrast parallel.py:343 `def generate(self, net, version, worlds, lam, explore_plies, lam_blend, n_step):` (no m/n_sims); serial path exit_loop.py:281 `GumbelAZSearch(net, env, m=args.m, n_sims=args.n_sims)`  
   *Right abstraction:* Pass per-iteration search budget (m, n_sims) and seed-salt through the task tuple alongside lam/explore_plies (already threaded per-iteration), and have _ensure_net/_gen_task rebuild/parametrize the search when the budget changes — exactly as version-gating already rebuilds the net. Freeze in initargs ONLY what is truly per-process-invariant (cores). The rule: a value the experimenter will want to anneal/sweep must be a per-call argument, not a constructor-frozen closure.

3. **[major] _W module-global worker dict is hidden mutable global state (action-at-a-distance)**  
   *Pattern:* `hidden-global-state`  
   *Why cancerous:* Every worker task reaches into a module-level mutable dict _W for env/fb/net/search/redis/m/n_sims/version. _gen_task/_eval_task/_ensure_net read and mutate it with no object owning it — the worker's entire state is a singleton smuggled through module scope. This is the classic spawn-pool workaround, but it makes the worker logic untestable without monkeypatching the module, couples _ensure_net's version cache to global identity, and means any second logical pool in one process collides on the same _W. The single global version slot shared by gen and eval is also what forces the version+1_000_000 hack.  
   *Evidence:* parallel.py:132-135 `_W = {"env": None, "fb": None, "net": None, "search": None, "version": -1, ...}`; parallel.py:199 `if _W["version"] == version and _W["net"] is not None:`; parallel.py:229+232 `_ensure_net(run, version)` then `generate_episode(_W["env"], _W["search"], _W["fb"], ...)`  
   *Right abstraction:* Wrap worker state in a Worker class instantiated once per process in the initializer and stashed behind one _WORKER reference, OR keep all reads behind a single accessor that encapsulates the version-cache invariant. The smell isn't process-local state — it's an unstructured dict reached into from three functions with no owner.

4. **[major] version overloaded with `it + 1_000_000` to disambiguate gen vs eval weight publishes**  
   *Pattern:* `band-aid`  
   *Why cancerous:* Eval republishes the now-trained weights and must not collide with the same iteration's generation weights (version it, the pre-train net). The fix adds a magic 1_000_000 offset so eval's version is it+1_000_000. This conflates two orthogonal facts — iteration and phase — into one integer via a numeric-range partition. It silently breaks at iters ≥ 1_000_000 (gen version collides with a prior eval version), and a reader cannot tell from the call site why the constant exists. It only exists because the worker version cache is a single global slot.  
   *Evidence:* exit_loop.py:308 `totR, totT, ets = executor.evaluate(net, it + 1_000_000, eval_worlds, lam)` (sole 1_000_000 in package); parallel.py:218 `* np.uint64(version + 1)` folds this inflated version into the eval RNG seed too  
   *Right abstraction:* Namespace the weight key by (run, phase, iteration) — az:w:<run>:<phase>:<version> with phase∈{gen,eval} — and pass phase as a first-class arg to publish_weights/_ensure_net; key the version cache on (phase, version). No magic offset, no overflow cliff, self-documenting.

5. **[major] Reference constants STATIC_FLOOR / CLAIRVOYANT_CEIL / DECOMP_ANCHOR hardcoded, duplicating the env-derived SSOT**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* 0.0855 (static floor) and 0.1454 (clairvoyant ceiling) are COMPUTABLE from the env — harness.realizable_static(env) and harness.clairvoyant_rate(env) derive them, and test_smoke.py:94-95 asserts those functions reproduce exactly these numbers. Yet exit_loop freezes them as module literals and uses them to compute %VoI (the loop's headline metric). The same numbers are copy-pasted across bounds/, eval/, dataset.py, bench/, probes/ (20+ sites). If the env is perturbed for an experiment, every hardcoded floor/ceiling goes silently stale and %VoI becomes a lie. DECOMP_ANCHOR=0.0941 is purely decorative (a flat TB line) yet still a frozen magic number.  
   *Evidence:* exit_loop.py:49-51 `STATIC_FLOOR = 0.0855` / `CLAIRVOYANT_CEIL = 0.1454` / `DECOMP_ANCHOR = 0.0941`; exit_loop.py:318 `voi = (rate - STATIC_FLOOR) / (CLAIRVOYANT_CEIL - STATIC_FLOOR) * 100`; SSOT at harness.py:17/:28; eval_az.py:33 imports realizable_static, clairvoyant_rate yet eval_az.py:34 still `LAM0 = 0.0855`  
   *Right abstraction:* Compute floor/ceiling once at run() start from the env via the harness functions (cache them — deterministic), or read them from one config object that owns the env-derived references. run() already builds env; STATIC_FLOOR = realizable_static(env) costs nothing and can never drift. One source: the env.

6. **[major] argparse Namespace is a 26-flag god-object threaded through run/generate/train**  
   *Pattern:* `god-object`  
   *Why cancerous:* main() builds a 26-argument argparse Namespace and hands the whole args blob to run(args), which reaches into ~25 distinct args.* fields across 90 lines. The Namespace conflates unrelated concerns — net topology, search budget, training, value-target, parallelism, eval, IO — with no typed config, no grouping, no validation seam beyond two scattered ad-hoc checks. CLI flags are the ONLY config seam, so a sweep harness or notebook must fake an argparse Namespace. The cross-cutting lr/l2 then get re-threaded individually into train_epochs (where they're dead), showing the blob both over- and under-couples.  
   *Evidence:* exit_loop.py:175 `def run(args):` referencing args.resume/init_weights/seed/hidden/residual/lr/l2/ckpt_dir/tb_logdir/window/lam/n_step/td_lambda/workers/cores/m/n_sims/iters/episodes/explore_plies/eval_seed/eval_n/epochs/batch/alpha/beta; exit_loop.py:366-409 the 26 `ap.add_argument` calls (grep count: 26)  
   *Right abstraction:* Introduce small frozen dataclasses grouped by concern (NetConfig, SearchConfig, TrainConfig, ParallelConfig, EvalConfig, IOConfig) built from args in main(); run() takes those. The CLI becomes one front-end; a sweep script constructs configs directly. Validation localizes (the --n-step/--td-lambda mutual-exclusion at exit_loop.py:245 belongs in TrainConfig.__post_init__).

7. **[major] Five-plus stacked deadlock band-aids fighting the multiprocessing+JAX+redis substrate**  
   *Pattern:* `band-aid`  
   *Why cancerous:* The executor carries a sediment of defensive patches, each fixing a symptom of the previous layer's substrate fight: (1) per-result imap timeout to escape unbounded list(imap) hangs; (2) bounded redis socket/connect timeouts so a stalled recv aborts loud; (3) 1h TTL on result blobs because aborted iterations leaked keys (~980 leaked az:res:* found post-mortem); (4) 1h TTL on weight keys; (5) worker-side faulthandler+SIGUSR1 solely to diagnose the wedge; (6) setdefault of OMP/OPENBLAS/MKL/NUMEXPR/NUMBA/XLA thread counts to sever the JAX→spawn-child residue the RCA fingers as the wedge trigger; (7) bounded close() with per-worker join-timeout+terminate because Pool.join takes no timeout. Each is individually reasonable, but together they wall off a design that combined three hostile substrates inside one pool. The patches treat symptoms, not the root coupling; the worker can still wedge — the fixes only make it abort loudly.  
   *Evidence:* parallel.py:67-82 socket timeouts (H2); parallel.py:150-153 `for _var in ("OMP_NUM_THREADS"...): os.environ.setdefault(_var, "1")` (H1a/Fix C); parallel.py:166-172 `faulthandler.enable(); faulthandler.register(signal.SIGUSR1...)`; parallel.py:255-259 result-blob `ex=_res_ttl`; parallel.py:282-310 `_drain_imap` per-result timeout (H1/Fix A); parallel.py:398-416 bounded close() join-timeout+terminate; docstring says 'deadlock fix' repeatedly + refs docs/notes/jaxtrain-deadlock-rca.md  
   *Right abstraction:* Stop fighting the substrate: (a) a worker entrypoint that imports NO jax/XLA (workers do pure-numpy search + redis; only the parent touches JAX), eliminating the fork/spawn/XLA interaction that caused the wedge and making bands (5)(6) and most of (1)(2)(7) unnecessary; or (b) replace the bespoke redis+Pool transport with an established job/queue runtime that owns liveness/timeouts/cleanup. Defensive timeouts belong as a thin safety net, not the load-bearing reliability strategy.

8. **[minor] Redis-as-transport partially reinvents a job-queue, with cleanup bolted on after the fact**  
   *Pattern:* `reinvented-wheel`  
   *Why cancerous:* The module hand-rolls a weight-broadcast + result-gather protocol over raw redis SET/GET with manual key namespacing (run-token + res-token + idx), manual TTLs for leak control, manual read-then-delete, and a manual (idx,n_rows) handshake through the Pool pipe — i.e. a bespoke distributed task queue. The raw-bytes payload choice is well-justified (pickle cost is real), but the surrounding orchestration (namespacing, expiry, eviction-window reasoning, leak post-mortems) is exactly the bookkeeping a queue/object-store provides for free. The eviction caveat is telling: the store is a 1GB allkeys-lru cache that 'can in principle drop a key under memory pressure', so correctness rests on a documented argument that the live set stays small.  
   *Evidence:* parallel.py:336-341 manual `pipe.set(f"az:w:{self.run}:{version}:m"...)` + `pipe.expire(...,3600)`; parallel.py:247-260 manual result-blob set with TTL; parallel.py:370-381 manual read-then-delete; parallel.py:38-40 'allkeys-lru eviction can in principle drop a key under memory pressure; weights carry a 1h TTL and are read-validated'  
   *Right abstraction:* If the perf win is solely avoiding pickle of float32 arrays, use shared memory (multiprocessing.shared_memory / Arrow Plasma) — no network round-trip, no LRU-eviction hazard, no namespacing/TTL/cleanup to own. If cross-machine distribution is the real goal, adopt a transport library that owns liveness and result lifecycle rather than re-deriving it from SET/GET + post-mortems.

9. **[minor] generate_episode + record-stacking duplicated across loop, worker, and three bench/probe mains**  
   *Pattern:* `copy-paste-mains`  
   *Why cancerous:* generate_episode is the one canonical episode-recorder, but it's invoked from five sites with divergent argument styles (positional vs keyword, hardcoded 0.0855, hardcoded explore_plies), and the serial loop and parallel worker each re-implement the recs → X/PI/M/Y stacking with the same float32 layout encoded twice — if the record tuple shape changes, both must change in lockstep. bench/capture_states.py and probes/gen_frozen.py each re-hardcode 0.0855, re-creating the SSOT drift.  
   *Evidence:* exit_loop.py:284 serial call; parallel.py:232 worker call; bench/capture_states.py:80 `generate_episode(env, search, fb, w, 0.0855, rng, 0)`; probes/residual_firewall/gen_frozen.py:28 `generate_episode(env, search, fb, w, 0.0855, gen_rng, n_explore_plies=4,`; stacking duplicated at exit_loop.py:288-293 and parallel.py:240-245  
   *Right abstraction:* Factor the record-list → (X,PI,M,Y) stacking into one helper next to generate_episode and call it from both the serial loop and the worker (single layout definition). Bench/probe scripts should import the loop's lam default rather than re-typing 0.0855.

10. **[minor] Worker-index parsing scrapes the multiprocessing process NAME string to assign cores**  
   *Pattern:* `leaky-boundary`  
   *Why cancerous:* Core assignment depends on parsing mp.current_process().name ('PoolWorker-1' → 0) — a CPython-internal naming convention, not a contract. If the name format changes (it has varied across Python versions: 'PoolWorker-N' vs 'SpawnPoolWorker-N'), every worker falls into the except and pins to widx=0, silently collapsing 4-core parallelism onto one core with NO error (fail-soft, 'a perf knob, not correctness'). The affinity intent is reduced to string-scraping an implementation detail.  
   *Evidence:* parallel.py:175-179 `name = mp.current_process().name` / `widx = int(name.rsplit("-", 1)[-1]) - 1     # PoolWorker-1 -> 0` / `except (ValueError, IndexError): widx = 0`; parallel.py:182 `os.sched_setaffinity(0, {core})`  
   *Right abstraction:* Pass an explicit worker index into the initializer (e.g. a multiprocessing.Value counter incremented under a lock, or ProcessPoolExecutor's initializer-with-index patterns), so core assignment derives from an owned integer rather than a scraped, version-fragile name string.

**Frozen-at-construction (config that should be live):**

- **m (Gumbel root actions) and n_sims (sims/decision) — the search-budget dial** — `chocofarm/az/parallel.py:328 (initargs), consumed once at parallel.py:191-192; per-iteration entry generate() at parallel.py:343 carries no budget arg` — Search width is the primary compute/quality knob in ExIt and the thing you most want to anneal (cheap early, deep late) or sweep; freezing it for the pool's multi-hour life means an annealing/sweep experiment the serial path supports (it rebuilds the search per iteration at exit_loop.py:281) is impossible in parallel — divergent capability between the two paths.
- **lr (learning rate) and l2 (weight decay)** — `chocofarm/az/mlp_jax_train.py:215 optax.adam(learning_rate=self.lr...) + :219-220 closures, frozen at JaxTrainer.__init__ (mlp_jax_train.py:206); exit_loop.py:226 trainer = JaxTrainer(net, lr=args.lr, l2=args.l2)` — LR scheduling/annealing across iterations is standard in AZ-style training; here lr/l2 are baked into the jit'd update closure at construction, and the train_epochs lr/l2 params that pretend to offer per-iteration control are dead no-ops, so any LR schedule is silently ignored. Should be a live JaxTrainer.set_lr that re-derives the update.
- **STATIC_FLOOR / CLAIRVOYANT_CEIL reference rates used in the headline %VoI metric** — `chocofarm/az/exit_loop.py:49-51 (module literals); used at exit_loop.py:318 in the %VoI formula` — These are deterministic functions of the env (computed by harness.realizable_static/clairvoyant_rate, asserted in test_smoke.py:94-95). Frozen as literals, they go stale the instant the env is perturbed for an experiment, making %VoI report progress against the wrong baseline with no error. They should be derived from the live env at run() start.
- **lam (λ penalty) pinned to args.lam for the whole run** — `chocofarm/az/exit_loop.py:242 lam = args.lam, used unchanged every iteration` — This one is DEFENSIBLY frozen (design §4.1 deliberately pins λ to the static-floor rate; a Dinkelbach outer loop is out of scope). Noted only to contrast with m/n_sims/lr: λ's freezing is a documented modeling decision, whereas the others are accidents of where the value happened to be threaded.

### A.6 — eval

**Health:** `messy`

**Summary.** The eval subsystem has one genuinely good architectural decision — harness.py recomputes the static floor and clairvoyant ceiling LIVE from the Environment, and six of eight scripts correctly import those two functions rather than hardcoding numbers. But that single shared seam is the only abstraction: there is no shared "evaluate a policy, print the VoI table" helper, so every solver gets a bespoke copy-paste main() that re-instantiates the env, re-prints the same floor/ceiling header, re-derives the same VoI-claw formula, and re-rolls its own arg parsing and PLAN literal. The SSOT win is then undermined by exit_loop.py and eval_az.py, which freeze the very numbers harness computes (0.0855 / 0.1454 / 0.0941) as module-level literals that will silently go stale the moment the env, value vector, or detector model changes.

**Praise (earned, with citations):**

- harness.py:17-50 — the reference lines are FUNCTIONS of the env (`realizable_static(env)` reads `env.value[r]`/`env.d`; `clairvoyant_rate(env)` enumerates `env.worlds` and `env.value[i]`), so they stay correct under any value/geometry/detector change. This is the right SSOT shape: derive, don't hardcode. Six scripts (eval_uct, eval_ismcts, eval_nmcs, eval_decomp, eval_faces, tb_runner) import them — `from chocofarm.eval.harness import realizable_static, clairvoyant_rate` — so the floor/ceiling have a single live source of truth.
- harness.py:1-9 and eval_az.py:2-21 — the module docstrings precisely state what each reference IS (floor = best fixed value-aware route; ceiling = clairvoyant max-VoI) and what the headline %VoI-clawed metric means. The eval contract is documented at the point of use, not buried in external docs.
- tb_runner.py:27-35,73-78 — the streaming evaluator is honestly bounded-by-construction: it holds only running sums plus a writer and self-aborts on an RSS cap (`if rss_mb() > a.rss_cap_mb: ... break`). That memory discipline is a real design property, not a comment.

**Findings:**

1. **[critical] Reference constants frozen as literals in exit_loop.py and eval_az.py — duplicates of what harness computes live**  
   *Pattern:* `SSOT-violation / frozen-at-construction`  
   *Why cancerous:* harness.py is the authoritative source: `realizable_static(env)` and `clairvoyant_rate(env)` recompute the floor/ceiling from the live env (and they depend on `env.value`, geometry, and the detector/face model). exit_loop.py and eval_az.py instead bake the *outputs* of those functions in as constants. The moment anyone changes the value vector (Environment accepts `value=...`), the instance geometry, or the detector model — exactly the experiments this codebase exists to run — these literals become silently wrong, and every %VoI / GO-NO-GO read-out computed against them is quietly miscalibrated. eval_az even prints `decomp anchor : ~0.094` and logs `ref/decomp 0.094` while exit_loop calls the same number `DECOMP_ANCHOR = 0.0941`: the duplicated constant is already internally inconsistent (0.094 vs 0.0941).  
   *Evidence:* az/exit_loop.py:48-51 `# reference lines ...\nSTATIC_FLOOR = 0.0855\nCLAIRVOYANT_CEIL = 0.1454\nDECOMP_ANCHOR = 0.0941`; eval/eval_az.py:34 `LAM0 = 0.0855  # static-floor rate`; eval/eval_az.py:79 `w.add_scalar("ref/decomp", 0.094, done + c)`; eval/eval_az.py:122 `print(f"decomp anchor       : ~0.094 (the value teacher)\n", ...)`. Contrast eval/harness.py:55-56 `static = realizable_static(env)\n    ceil = clairvoyant_rate(env)`.  
   *Right abstraction:* exit_loop and eval_az should call `realizable_static(env)` / `clairvoyant_rate(env)` like the other six scripts, and obtain the decomp anchor by running `DecompPolicy` (as eval_decomp already does) rather than pasting a number. If a frozen reference is genuinely wanted for a pinned training run, it should be captured once, written into the run's checkpoint/JSON metadata, and asserted against the live recompute at startup — not hand-typed as a literal in two files with two different roundings.

2. **[major] Copy-paste main(): the floor/ceiling header and the %VoI-claw formula are re-implemented in every eval script**  
   *Pattern:* `copy-paste main / missing abstraction`  
   *Why cancerous:* There is no `run_eval(env, plan)` or `print_voi_table(...)` helper. Every script repeats (a) `env = Environment(); static = realizable_static(env); ceil = clairvoyant_rate(env)`, (b) the identical 3-line floor/ceiling/headroom print block, and (c) the headline metric `(r - static) / (ceil - static) * 100` recomputed inline. That last formula — the single most important number the project tracks — is hand-typed in at least seven places; one transcription slip (e.g. forgetting the `* 100`, or swapping ceil/static) produces a plausible-looking wrong column with no shared test to catch it. Adding a new column (CI width, sec/ep) or fixing the formula means editing N files.  
   *Evidence:* Header block duplicated: eval/eval_uct.py:67-68, eval/eval_ismcts.py:25-26, eval/eval_nmcs.py:51-52, eval/eval_faces.py:81-83, eval/harness.py:57-59, eval/eval_az.py:120-122. Claw formula duplicated: eval/harness.py:74 `claw = (r - static) / (ceil - static) * 100`; eval/eval_ismcts.py:41; eval/eval_uct.py:81; eval/eval_nmcs.py:62; eval/eval_faces.py:94; eval/eval_decomp.py:39,92; eval/eval_az.py:46-47,76,92,96.  
   *Right abstraction:* A single `eval/report.py` with `references(env) -> (static, ceil)`, `print_reference_header(static, ceil)`, and a `voi_pct(r, static, ceil)` / `format_row(name, res, static, ceil)`. Each eval_*.py shrinks to a PLAN of (label, policy, budget) plus one call to a shared `run_plan(env, plan)`. The metric then has exactly one definition and one place to test.

3. **[major] No shared eval driver: every solver has a bespoke script with its own arg-parsing dialect**  
   *Pattern:* `god-script-per-solver / inconsistent CLI seam`  
   *Why cancerous:* Eight near-identical scripts differ mainly in which PLAN they hardcode, yet they disagree on *how to be configured*: eval_az.py and eval_decomp.py use argparse; eval_uct.py, eval_nmcs.py, eval_faces.py hand-roll `sys.argv[1:]` with ad-hoc token grammars (`N=<int>`, group names, bare ints). There is no uniform way to set N, seed, or budget across the suite, so driving the whole eval matrix from one orchestrator is impossible without learning three CLI dialects. This is the CLI-flags-as-only-config-seam smell at suite scale, and it makes experimentation (sweep a budget across all solvers) painful.  
   *Evidence:* Hand-rolled argv: eval/eval_uct.py:54-61 `for tok in args: if tok.startswith("N="): n_override = int(tok[2:]) else: wanted.add(int(tok))`; eval/eval_nmcs.py:45 `levels = [int(a) for a in sys.argv[1:]] or [1, 2]`; eval/eval_faces.py:74 `groups = set(sys.argv[1:])`. Argparse: eval/eval_decomp.py:44-48; eval/eval_az.py:105-115. tb_runner.py:52-60 has yet another argparse surface.  
   *Right abstraction:* One driver module exposing the solver registry + a uniform argparse (`--solver`, `--budget`, `--n`, `--seed`, `--subset`). Per-solver knowledge collapses to a registry entry mapping a name to a policy factory and default budget; the suite is then driven uniformly and a sweep is a loop, not eight invocations with three syntaxes.

4. **[major] Eval budgets / iteration sweeps frozen as module-level PLAN literals — the thing you most want to vary is the least reachable**  
   *Pattern:* `hot-where-cold / frozen-at-construction`  
   *Why cancerous:* The whole point of these scripts is to sweep search budget vs rate, yet the budgets are baked into module-global tuples (`PLAN`) or hardcoded inside `build_plan`/`make_plan`. The Dinkelbach schedule (`iters`, `warm_runs`, `final_runs`), the iteration ladder (200/400/800/1600), and the NMCS branching knobs are all literals. Only `final_runs` is overridable (and only in eval_uct, via the bespoke `N=` token). To try it=300 or a longer warm-up you edit source. This is exactly 'construct the experiment config at build time, close over it forever' — the config that should be the most live is the least reconfigurable.  
   *Evidence:* eval/eval_uct.py:45-50 `PLAN = [("uct(it=200)", 200, dict(iters=2, warm_runs=8, final_runs=40)), ...]`; eval/eval_ismcts.py:31-34 `plan = [("ismcts(it=150)", 150, dict(...)), ("ismcts(it=400)", 400, dict(...))]`; eval/eval_nmcs.py:33-40 `NMCSPolicy(level=2, playout_samples=2, step_samples=1, cand_det=1, cand_tre=3, max_steps=18)`; eval/eval_faces.py:46-70 build_plan with all budgets inline.  
   *Right abstraction:* Budgets and ladders belong in a config object/dataclass (or a small YAML/JSON the driver loads), with CLI overrides for every field, so a sweep is `--it 200,400,800 --warm 12 --final 40` rather than a source edit. The PLAN literals are fine as *defaults* but must not be the only seam.

5. **[minor] Implicit seed default: half the scripts pass seed=7, half pass nothing and rely on env's default**  
   *Pattern:* `hidden-default / action-at-a-distance`  
   *Why cancerous:* Reproducibility is the entire value proposition of an eval harness, yet the seed contract is split-brained. eval_uct, eval_ismcts, eval_decomp, eval_az explicitly pass `seed=7`; harness.main, eval_nmcs, eval_faces pass nothing and silently inherit `dinkelbach_rate(..., seed=7)`'s default — and separately the Dinkelbach warm phase is hard-pinned to `seed=1` inside env regardless of caller. The fact that they happen to agree on 7 today is luck, not design; change the env default and three scripts silently shift while four don't, breaking cross-script comparability with no error.  
   *Evidence:* Explicit: eval/eval_uct.py:78 `env.dinkelbach_rate(pol, seed=7, **budget)`; eval/eval_ismcts.py:40 `env.dinkelbach_rate(pol, seed=7, **budget)`. Implicit (no seed): eval/harness.py:73 `r = env.dinkelbach_rate(pol, **budget)["rate"]`; eval/eval_nmcs.py:60 `res = env.dinkelbach_rate(pol, **budget)`; eval/eval_faces.py:93 `r = env.dinkelbach_rate(pol, **budget)["rate"]`. Warm phase pinned in env: model/env.py:163 `lam = self.rate(policy, lam, warm_runs, seed=1)`.  
   *Right abstraction:* Seed should be a single explicit harness-level parameter threaded through `run_plan(..., seed=...)` and surfaced as a CLI flag, never silently defaulted, so every row in every script is provably reproduced from the printed seed.

6. **[minor] tb_runner logroot defaults to a hardcoded personal absolute path; --method help comment is stale**  
   *Pattern:* `hardcoded-environment / stale-doc`  
   *Why cancerous:* The default TensorBoard logroot is one developer's machine path, so on any other machine/CI the tool writes (or fails to write) to a nonexistent absolute directory unless the flag is remembered every time — a portability landmine baked as a default. Separately, the `--method` inline help advertises only 'nmcs | ismcts' while `make_policy` actually supports `uct` (and the main argparse help at line 54 also says 'nmcs | ismcts'); the documented config surface lies about the real one, the classic accepted-but-undocumented-third-option smell.  
   *Evidence:* eval/tb_runner.py:57 `ap.add_argument("--logroot", default="/home/bork/w/vdc/chocobo/tb")`; eval/tb_runner.py:54 `ap.add_argument("--method", required=True)         # nmcs | ismcts` vs eval/tb_runner.py:43-44 `if method == "uct": return UCTPolicy(iterations=int(cfg))`.  
   *Right abstraction:* Default logroot to a repo-relative or env-var-derived path (e.g. `os.environ.get("CHOCOFARM_TB", "tb/")`); generate the `--method` choices from the supported set (`choices=["nmcs","ismcts","uct"]`) so the help can never drift from `make_policy`.

7. **[minor] eval_az re-implements its own fixed-λ₀ accumulation and reference-line logging instead of reusing tb_runner / harness**  
   *Pattern:* `band-aid / parallel-implementation`  
   *Why cancerous:* eval_az.stream_compare hand-rolls a sumR/sumT/episode accumulator, cumulative-rate computation, and ref/floor|ceiling|decomp TB logging that tb_runner.py already does — but with a *different* λ policy (eval_az pins LAM0; tb_runner runs a 'gentle Dinkelbach nudge' `s["lam"] = 0.7*lam + 0.3*run_rate`). Two streaming evaluators with subtly different λ semantics and duplicated TB-scalar wiring means a fix or a metric rename has to be made in two places and the curves aren't actually comparable across tools. The comment 'the same cumulative-rate shape the solver runners stream' admits the duplication out loud.  
   *Evidence:* eval/eval_az.py:60-80 `acc = {"net": [0.0,0.0,0], "playout": [0.0,0.0,0]} ... w.add_scalar(f"rate/{tag}", cumrate, a[2]) ... w.add_scalar("ref/floor", static, ...); w.add_scalar("ref/ceiling", ceil, ...); w.add_scalar("ref/decomp", 0.094, ...)` vs tb_runner.py:84-99 which streams the same family of scalars with its own accumulator and λ-nudge.  
   *Right abstraction:* Factor a single streaming evaluator (running sums + ref-line logging + cumulative-rate scalar) parameterized by a λ strategy (pinned vs nudged) and a policy set; eval_az and tb_runner both call it. One place defines the TB scalar names and the ref lines.

**Frozen-at-construction (config that should be live):**

- **Dinkelbach schedule + iteration ladder + NMCS branching knobs frozen as module-level PLAN tuples (iters/warm_runs/final_runs, 200/400/800/1600, playout_samples/step_samples/cand_det/cand_tre/max_steps)** — `eval/eval_uct.py:45-50, eval/eval_ismcts.py:31-34, eval/eval_nmcs.py:33-40, eval/eval_faces.py:46-70` — Sweeping search budget vs rate is the explicit purpose of these scripts; the budgets are the experiment variable and should be CLI/config-overridable, not source-edits. Today only final_runs is reachable, and only in one script via an ad-hoc `N=` token.
- **LAM0 = 0.0855 — the fixed operating/training λ used for the apples-to-apples fixed-λ row and all the GO/NO-GO read-outs** — `eval/eval_az.py:34` — λ₀ IS the static-floor rate, which harness recomputes live; freezing it as a literal means a value/geometry/detector change silently evaluates 'the operating point the value was trained at' at the wrong point. It should be derived from realizable_static(env) (or read from the trained net's metadata), with a CLI override.
- **STATIC_FLOOR / CLAIRVOYANT_CEIL / DECOMP_ANCHOR reference lines baked as constants** — `az/exit_loop.py:49-51` — These are exactly what harness.realizable_static/clairvoyant_rate compute from the env (cheaply — present set is K=5, so the clairvoyant inner solve is bounded); hardcoding them duplicates the SSOT and goes stale on any env/value change, miscalibrating every %VoI and GO/NO-GO printed against them.
- **tb_runner logroot default = personal absolute path /home/bork/w/vdc/chocobo/tb** — `eval/tb_runner.py:57` — A machine-specific absolute path baked as a default is non-portable; it should derive from an env var or a repo-relative location so the same command works on any host/CI without remembering the flag.

### A.7 — bounds

**Health:** `sound`

**Summary.** The provable-bounds subsystem is mathematically careful and unusually well-documented, with a clean injected-callable V̂ seam that is genuinely the right abstraction for the dual penalty. Its architectural debt is concentrated in two places: MiniEnv is a hand-rewritten parallel copy of the real Environment's belief/dynamics mechanics (a second source of truth for filter/marginals/legal_actions/apply that will silently drift), and the bound code reaches into solver internals — DecompVhat copy-pastes decomp.py's per-λ _build setup and calls a private _live_occupancy_posterior. Reference rates and per-V̂ configuration are frozen as scattered literals across eval_bound.py rather than living in one place.

**Praise (earned, with citations):**

- The injected-callable V̂ seam is the correct abstraction and is genuinely decoupled: the penalty/inner solver treat V̂ as `vhat(env, loc, bw, collected, lam) -> float` (info_relaxation.py:61), so vhat_zero/vhat_analytic/DecompVhat/ExactBeliefVhat are interchangeable without touching PenalizedClairvoyant. This is the load-bearing design decision and it is clean.
- The validity-critical state cap ABORTS LOUDLY instead of truncating: info_relaxation.py:337-343 `raise RuntimeError(... "REFUSING to truncate (a truncated inner search can return a LOWER bound and silently break the upper-bound property")`. Refusing to silently degrade a *provable* bound into a non-bound is exactly the right call for this subsystem, and the docstring ties it to the proof (dual-bound.md §4).
- empirical_penalty_mean deliberately recomputes the penalty increment inline rather than calling _penalized_step, and says so: info_relaxation.py:467-468 `We recompute it directly here (not via _penalized_step) to keep the check independent of the inner-solve code path.` Keeping a validation check independent of the code it validates is principled — the one place where NOT reusing code is correct.
- vhat_zero's docstring (info_relaxation.py:65-70) precisely distinguishes V̂≡0 (a nonzero reward-deviation penalty) from the true z≡0 baseline (vhat=None), pre-empting the exact confusion a reader would have. This is documentation earning its keep.

**Findings:**

1. **[major] MiniEnv is a parallel reimplementation of Environment's belief/dynamics core (second source of truth)**  
   *Pattern:* `SSOT-violation / parallel-sim-model`  
   *Why cancerous:* MiniEnv hand-copies marginals, filter_treasure, filter_detector, sample_world, legal_actions, and apply from Environment with subtly different code rather than restricting/delegating. These are the belief mechanics whose correctness the *provable bound* depends on. If Environment's apply/legal_actions/marginals semantics ever change (e.g. the comment in env.py:38-46 already records one such re-keying of detectors from regions to faces), MiniEnv silently keeps the old semantics and the 'honest numbers' guarantee in its own docstring (minienv.py:11 'every number is honest') quietly becomes false — the validation would pass against a stale model. The whole point of MiniEnv is to be the *real* env minus worlds; copying the dynamics defeats that.  
   *Evidence:* minienv.py:81-115 reimplements `def marginals`, `def filter_treasure`, `def filter_detector`, `def sample_world`, `def legal_actions`, `def apply` — e.g. minienv.py:106-115 `def apply(self, loc, bw, collected, action, world): ... r = self.value[i] if (pres and i not in collected) else 0.0` is byte-for-byte env.py:125-135. Only d/exit_cost/nearest_exit/route_time delegate to `self._env` (minienv.py:69-79).  
   *Right abstraction:* MiniEnv should be a thin restriction view that delegates ALL belief/dynamics to the parent and overrides only the world-population and detector set. Either (a) give Environment a `restrict(keep, k_local)` factory that returns an Environment with a narrowed `worlds`/`detectors` (same code path), or (b) make MiniEnv subclass Environment / hold `self._env` and forward marginals/filter_*/legal_actions/apply to it (they already work bit-for-bit since bit positions are unchanged). Then there is exactly one implementation of the dynamics.

2. **[major] DecompVhat copy-pastes decomp.py's per-λ build pipeline and calls a private posterior helper (leaky boundary)**  
   *Pattern:* `leaky-boundary / copy-paste-internals`  
   *Why cancerous:* DecompVhat._build duplicates almost verbatim the cluster-discovery → anchors → micro-table → MacroPlanner construction that already lives inside DecompPolicy._build, and DecompVhat.__call__ reaches into the module-private `_live_occupancy_posterior`. The bound now depends on decomp's *internal* construction recipe, not its public interface. Any change to how decomp builds its macro (different anchor rule, extra micro modes, MacroPlanner signature) breaks the bound, and the two _build copies will drift — already they differ (the bound's omits the `joint`/`bit` tables). The penalty's V̂ is supposed to reuse 'the SAME object the decomp policy acts on' (info_relaxation.py:91-94), but it actually rebuilds a *similar* object, so they are not guaranteed identical.  
   *Evidence:* info_relaxation.py:108-127 `clusters = D.discover_clusters(env); sense = [...]; anchors = {...}; for c in sense: ... micro[(c.name,k)] = D.build_cluster_micro(...); macro = D.MacroPlanner(...)` mirrors decomp.py:549-560. info_relaxation.py:133 `post = D._live_occupancy_posterior(env, bw, macro)` calls a name-mangled-by-convention private (decomp.py:643).  
   *Right abstraction:* decomp.py should expose a public `build_macro(env, lam, horizon)` (or a `MacroValue` value-function object) that both DecompPolicy and DecompVhat consume, and `occupancy_posterior(env, bw, macro)` should be public API. DecompVhat then becomes a 5-line adapter that holds a macro value-function and forwards, with no knowledge of clusters/anchors/micro construction.

3. **[major] clairvoyant solve is implemented three times (harness, eval_bound, and the no-penalty inner DP all claim to be 'the' z≡0 ground truth)**  
   *Pattern:* `SSOT-violation / copy-paste`  
   *Why cancerous:* The clairvoyant subset×permutation Dinkelbach solve exists as harness.clairvoyant_rate (the canonical one every eval script imports) and is re-typed as clairvoyant_on_worlds in eval_bound.py with a different signature, different Dinkelbach iteration count (6 vs 5), and different run accounting (exact over all worlds vs Monte-Carlo sampling). Meanwhile the no-penalty PenalizedClairvoyant inner DP is asserted to reproduce the same value (info_relaxation.py:37-40, dual-bound.md §4.2). Three encodings of one mathematical object means the regression check (i) is comparing two independent reimplementations of the baseline against each other — a drift in any one silently weakens or fakes the PASS, and the 'reproduces 0.1454' claim is only as trustworthy as the least-maintained copy.  
   *Evidence:* eval_bound.py:52-77 `def clairvoyant_on_worlds(...)` with `for sub in itertools.combinations(present, s): ... bt = min(env.route_time(...) for p in itertools.permutations(sub))` is the same algorithm as harness.py:28-50 `def clairvoyant_rate` (`for sub in itertools.combinations(present, s): ... bt = min(env.route_time(...))`), but harness uses `for _ in range(5)` (harness.py:48) while eval_bound uses `dink_iters=6` (eval_bound.py:52).  
   *Right abstraction:* Factor a single `clairvoyant_value(env, worlds, keep, ...)` into harness (or model) that takes an explicit world-set/treasure-subset, and have both the headline harness and eval_bound call it. The no-penalty inner DP is allowed to be a separate codepath precisely because it is the thing under test — but the *baseline* it is regressed against must be the one canonical function.

4. **[minor] Bound reference λ values and per-V̂ config are scattered magic literals in the driver**  
   *Pattern:* `frozen-at-call-site / config-as-literals`  
   *Why cancerous:* The reference λ for analytic V̂ (0.10) and the decomp-achievable λ (0.094) appear as bare literals at multiple unrelated call sites in validate(), and the achievable-rate prior helper independently hardcodes vl=0.10. These are experiment knobs (which λ* to pin V̂ at, which achievable rate to clear) frozen into the driver body; changing the reference point for a sweep means hunting down each literal, and the 0.10 used for the dual-feasibility policy, the analytic bound, and the achievable helper are coincidentally-equal-but-independent (no shared name), so they can silently diverge.  
   *Evidence:* eval_bound.py:135-136 `vhat_lam=0.10` / `lam=0.10`; eval_bound.py:152 `("analytic", vhat_analytic, 0.10)`; eval_bound.py:173 `vhat_lam=0.094`; eval_bound.py:80 `def achievable_on_mini(mini, vl=0.10)`. The values 0.094/0.1454 also live only as prose in docstrings (info_relaxation.py:93, info_relaxation.py:37).  
   *Right abstraction:* Hoist the reference rates into named module constants (e.g. `DECOMP_ACHIEVABLE_LAM = 0.094`, `ANALYTIC_REF_LAM = 0.10`) or, better, derive them — the decomp-achievable λ should come from running the decomp Dinkelbach, not be pinned by hand to a number that prose elsewhere claims is its value. That keeps the 'achievable floor the bound must clear' honest as the policy evolves.

5. **[minor] MiniEnv silently omits the Policy/Environment simulation contract (partial duck-type)**  
   *Pattern:* `leaky-boundary / partial-interface`  
   *Why cancerous:* MiniEnv's docstring claims it 'exposes the env's Policy/bound interface' (minienv.py:28-30), but it implements only the belief/geometry subset and omits simulate, rate, and dinkelbach_rate. So a MiniEnv cannot be dropped into the standard eval harness to measure an achievable policy rate the way the full env can — which is exactly why eval_bound.py has to hand-roll achievable_on_mini and exact_optimal_rate instead of reusing env.dinkelbach_rate / realizable_static. The 'it's just the env minus worlds' mental model is false at the boundary, and the gap is invisible until something calls the missing method.  
   *Evidence:* minienv.py methods stop at `apply` (minienv.py:106-115); there is no `simulate`/`rate`/`dinkelbach_rate` (cf. env.py:138-165). Consequently eval_bound.py reimplements achievability: eval_bound.py:80-97 `def achievable_on_mini` duplicates harness.realizable_static (harness.py:17-25) with `p = mini.K/len(mini.keep)` standing in for `env.K/env.N`.  
   *Right abstraction:* If MiniEnv were a true restriction of Environment (see finding 1), simulate/rate/dinkelbach_rate would come for free and achievable_on_mini/exact_optimal_rate could reuse the harness references. At minimum the docstring should state the interface is the belief subset only, not 'the Policy interface'.

6. **[minor] restrict_faces is an accepted-but-ignored constructor parameter**  
   *Pattern:* `accepted-but-ignored-param`  
   *Why cancerous:* PenalizedClairvoyant takes `restrict_faces=True` and stores it, but the only use is a branch whose body is `pass` — it does literally nothing. It is documented as 'a hook for an over-approximating prune', i.e. a placeholder for unwritten code. A constructor flag that has no effect is a trap: a future experimenter sets restrict_faces=False expecting different (and bound-safe) behaviour and gets identical results, or worse assumes the prune exists. Dead config surface masquerading as a real knob.  
   *Evidence:* info_relaxation.py:249 stores `self.restrict_faces = restrict_faces`; info_relaxation.py:354-358 `if self.restrict_faces: # faces matter ONLY via their penalty rebate ... default True keeps all → exact.) pass`.  
   *Right abstraction:* Remove the parameter until the over-approximating prune actually exists; reintroduce it (with the prune) when there is behaviour to gate. A no-op flag should not ship as public API.

**Frozen-at-construction (config that should be live):**

- **vhat / vhat_lam / max_inner_states are frozen in PenalizedClairvoyant.__init__ and closed over for the object's life; the reference λ* the penalty pins V̂ at cannot be varied without constructing a whole new PenalizedClairvoyant** — `bounds/info_relaxation.py:243-249` — Route A vs Route B (fix V̂ at λ* vs rebuild per scanned λ) is the central experimental degree of freedom of this bound (info_relaxation.py:230-233), yet it is baked at construction. The Dinkelbach driver dual_bound_rate already scans λ; pinning the *penalty* reference should be a per-call argument so a sweep over reference λ* (the natural tightness experiment) does not require rebuilding the object and its lru_cache each time.
- **DecompVhat.horizon is fixed at construction and threaded into MacroPlanner; the macro lookahead depth (a search-width-like accuracy/cost knob for the V̂) is frozen for the whole bound run** — `bounds/info_relaxation.py:104-105 (`def __init__(self, horizon=1): self.horizon = horizon`) → info_relaxation.py:123/142` — Horizon trades V̂ accuracy (hence bound tightness) against cost — exactly the kind of thing one wants to sweep when investigating how tight the decomp-based bound can get. Freezing it at construction and caching built macros per-λ keyed only by λ means changing horizon needs a new object; it should be a parameter of the value query (or at least of a per-experiment config) so tightness-vs-horizon is a loop, not a re-instantiation.
- **Bisection bracket [lo, hi], tol, and max_iter for the dual root are positional defaults frozen at the dual_bound_rate call sites (lo=0.0, hi=0.4 in eval_bound; lo=0.0, hi=0.30 in the function default)** — `bounds/info_relaxation.py:402-403 and bounds/eval_bound.py:123/156` — The two call sites already disagree on hi (0.4 vs the 0.30 default), and the bracket is the thing most likely to need tuning when moving from a sub-instance to the full instance (where ρ* lives near 0.1454). A frozen, inconsistent bracket is a latent 'bracket did not straddle 0' degenerate-return (info_relaxation.py:435-438) waiting on a new instance — the search bounds should come from the instance/experiment config, not be retyped per call.

### A.8 — analysis

**Health:** `sound`

**Summary.** The analysis tools (analyzer.py, synthetic.py) are unusually disciplined: a clean abstract `Instance` boundary, one-function-per-quantity decomposition, bounded enumeration, and genuine reuse of the model's `arrangement()` so synthetic geometry flows through the SAME planar-arrangement code the real map uses. The real rot is at the data-pipeline boundary: the exactly-K-of-N prior is hardcoded independently in at least three places (env.py, analyzer.py, kernels.py) with K nowhere in instance.json, and the world-array combinatorics is copy-pasted verbatim between env and analyzer — a guaranteed stale-config divergence the moment K or N moves. Scripts carry hardcoded /home/bork paths, cwd-dependent artifact names, and one residual-firewall probe forks the entire training loop (with mismatched hyperparameters and a different net implementation) instead of reusing exit_loop.train_epochs.

**Praise (earned, with citations):**

- analyzer.py:59-95 — the abstract `Instance` dataclass is a real, earned abstraction boundary: `analyze(instance)` consumes only (treasures, faces, teleports, N, K) and never touches Environment or a solver, so the identical analysis runs on the frozen real map and on synthetic instances unchanged. This is the right shape for a research tool.
- synthetic.py:121 `faces = A.arrangement(regions)` — the generator pushes its random regions through the SAME `arrangement.py` that produces the real faces, instead of hand-assigning covers. Co-coverage, clusters, and indistinguishability are emergent from geometry, so the method is genuinely stress-tested rather than fit to one map (docstring lines 7-12 state this contract explicitly and the code honors it).
- analyzer.py — the per-function [DET-DEP]/[DET-IND] tagging (e.g. lines 102-107, 118-133, 311-320) is excellent discipline: each structural quantity declares whether it is a property of the prior (ports verbatim) or re-derived under the corrected face model, making the supersession of the contaminated hand-analysis auditable function-by-function.
- analyzer.py:378-456 `reachable_local_beliefs` has two hard, named boundedness guards (`max_cluster_bits` refusing oversize latent spaces, `max_states` aborting BFS with a `truncated` marker) and the docstring states plainly that 'exactly-solvable' is a property of small clusters only — fail-loud rather than hang/OOM, with the honest limitation surfaced in the output.
- probes/residual_firewall/gen_frozen.py:28 correctly REUSES `exit_loop.generate_episode` rather than reimplementing episode generation — the dataset-freezing probe is a thin, honest wrapper over the production generation path (only the training half, ab_train.py, forks).

**Findings:**

1. **[critical] The exactly-K-of-N prior is encoded in 3+ places with K/N absent from the data file (SSOT violation)**  
   *Pattern:* `SSOT-violation / same-knowledge-in-multiple-places`  
   *Why cancerous:* K=5 (and N=20) is a single domain fact about the instance, but it is hardcoded independently in env.py:31 (`self.N, self.K = len(self.treasures), 5`), analyzer.py:88 (`K: int = 5`), and az/kernels.py:72 (`warmup(N=20, nD=44)`), while instance.json stores NEITHER K nor N (verified: `'K' in d, 'N' in d` -> False, False). The analyzer claims authority in its own docstring (analyzer.py:27 'here 5 of 20 -> C(20,5)=15504') with no link to env. The instant someone studies a different K (the whole point of an experimentation harness), env and analyzer silently disagree and every world-count, prune-factor, and occupancy-partition in a StructuralReport is computed against a different prior than the simulator runs — with no error, just wrong numbers. The synthetic generator (synthetic.py:73 `K=4`) proves K is meant to be a free knob, which makes the real map's hardcoding the more glaring.  
   *Evidence:* env.py:31 `self.N, self.K = len(self.treasures), 5`; analyzer.py:88 `N: int = 20, K: int = 5`; instance.json has keys [treasures, teleports, regions_wkt, delta_treasures, overlaps] — no K/N  
   *Right abstraction:* Store K (and let N derive from len(treasures)) in instance.json as the single source, have Environment read it, and have analyzer.real_instance read K from the SAME instance.json (it already opens the file at analyzer.py:91) rather than defaulting K=5. The analyzer should never carry an independent literal for a fact the environment owns.

2. **[major] world_array re-implements env.worlds combinatorics verbatim instead of asking the model**  
   *Pattern:* `DRY-violation / copy-paste of load-bearing logic across a boundary`  
   *Why cancerous:* The C(N,K) equiprobable-world bitmask array is the prior. env.py builds it at line 57 and analyzer.py rebuilds it at line 106 with character-identical code. Two copies of the prior's construction means any change to world semantics (weighting, a different combinatorial structure, a bug fix) must be made in two files that don't import each other, and a silent divergence produces an analysis that describes a different probability model than the simulator. The analyzer deliberately avoids importing Environment (it imports only `arrangement`), so it had to reinvent this — the cost of the otherwise-clean Instance boundary is a duplicated prior.  
   *Evidence:* env.py:57 `[sum(1 << t for t in c) for c in itertools.combinations(range(self.N), self.K)]` vs analyzer.py:106 `[sum(1 << t for t in c) for c in itertools.combinations(range(inst.N), inst.K)]` — identical  
   *Right abstraction:* Hoist the world-array constructor to a single function in the model layer (e.g. `arrangement.world_array(N, K)` or a small `prior.py`) that both Environment.__init__ and analyzer import. The Instance can stay solver-free while still sharing the prior's one definition.

3. **[major] ab_train.py forks the entire training loop and monkeypatches private net internals**  
   *Pattern:* `one-off fork of the training loop / band-aid fighting the abstraction`  
   *Why cancerous:* The 'decisive A/B' probe reimplements the epoch loop, batch sampling, value-scale pinning, and full-dataset CE/MSE eval (ab_train.py:85-116) that already exist in exit_loop.train_epochs — but against the numpy ValueMLP, not the JaxTrainer the production loop actually uses (exit_loop.py:226). So the probe's verdict ('residual-ON must reach <= baseline loss or it's an optimization flaw') is measured on a DIFFERENT optimizer/codepath than ships. Worse, it reaches into private state (`net._forward`, `net._masked_softmax`, `net._init_adam`, `net._residual_backward`, `net.Wr2`, `net.__dict__[k]`) and monkeypatches `_forward`/`_residual_backward` onto a live instance (lines 42-71) — a maintenance trap that silently breaks the moment mlp.py's internal forward changes, with no test linking them. This is a stacked patch fighting the net abstraction rather than a supported ablation seam.  
   *Evidence:* ab_train.py:94-116 hand-rolled epoch/batch/eval loop; ab_train.py:69 `net._forward = types.MethodType(_forward, net)`; ab_train.py:104 `_, v_std, logits = net._forward(X)` reaching into a private method  
   *Right abstraction:* Expose the residual/no-outer-relu variants as first-class ablation flags on ValueMLP (the net already has `residual=` and an `on/off` axis is in exit_loop.py:385), and run the A/B through exit_loop.train_epochs / JaxTrainer so the probe measures the production optimizer. A probe that contradicts the shipped training stack proves nothing about the shipped net.

4. **[major] Probe hyperparameters are magic-number copies that silently diverge from the loop defaults**  
   *Pattern:* `frozen-at-construction config duplicated as literals / hot-where-cold`  
   *Why cancerous:* gen_frozen.py constructs the generation pipeline with literals m=12, n_sims=128, lam=0.0855, n_explore_plies=4, lam_blend=0.6 (lines 22, 28-29) and ab_train.py pins H=256, seed=0, l2=1e-4, batch=256, data_seed=7 (line 125). Several of these intentionally or accidentally differ from exit_loop's defaults (loop n_sims=48 vs probe 128; loop lam_blend/td_lambda=1.0 pure-MC vs probe 0.6). Because these are inline literals, not derived from a shared config, a reader cannot tell which divergences are deliberate (search strength) from which are stale copies (lam_blend). The 'frozen dataset' the whole probe rests on is generated under settings that no longer match the loop, and nothing flags it.  
   *Evidence:* gen_frozen.py:22 `search = GumbelAZSearch(net, env, m=12, n_sims=128)`; gen_frozen.py:28-29 `generate_episode(..., 0.0855, gen_rng, n_explore_plies=4, lam_blend=0.6, n_step=None)`; ab_train.py:125 `H, seed, l2, batch, data_seed = 256, 0, 1e-4, 256, 7`  
   *Right abstraction:* Take generation hyperparameters from the same defaults object exit_loop exposes (or accept them as explicit CLI args echoed into the npz metadata), so the frozen dataset records the regime it was produced under and divergences from the loop are visible, not buried in literals.

5. **[major] instance.json stores derived facts (overlaps, delta_treasures) that the live model recomputes and never reads**  
   *Pattern:* `SSOT-violation / authoritative-value-lives-elsewhere with stale redundant copies`  
   *Why cancerous:* chocobo_geometry.py writes `overlaps` and `delta_treasures` into instance.json (lines 156-157), but the live model never consumes them: arrangement.py reads only `regions_wkt` (line 48) and recomputes the arrangement; analyzer derives delta from faces (analyzer.py:82-85 `inst.delta`) and co-coverage from faces (analyzer.py:118-133), explicitly noting the two edge-sets 'happen to COINCIDE' on the real map. So `overlaps` is a frozen snapshot of the OLD pairwise-overlap model that only verify_faces.py:30 reads — to reconstruct the buggy `cover_mask` for a cross-check. Two encodings of co-coverage (the stored `overlaps` and the recomputed face co-occurrence) can drift apart silently if regions_wkt is ever re-edited without re-running chocobo_geometry.py, and a reader cannot tell which is authoritative.  
   *Evidence:* chocobo_geometry.py:156-157 `"delta_treasures": delta_treasures, "overlaps": [[i, j] for i, j, *_ in overlaps]`; verify_faces.py:30 `for a, b in data["overlaps"]: nbr[int(a)].add(int(b))` — the only live consumer, used to rebuild the OLD model  
   *Right abstraction:* regions_wkt is the SSOT for sensing geometry; delta and co-coverage are derivable from it (and from faces). Either drop overlaps/delta_treasures from instance.json, or mark them explicitly as a frozen legacy-model snapshot whose only purpose is the verify_faces regression check, so no one mistakes them for live inputs.

6. **[major] Scripts hardcode absolute /home/bork paths and a cwd-dependent artifact name**  
   *Pattern:* `leaky-boundary / environment baked into source (CLI-flag-or-literal as the only config seam)`  
   *Why cancerous:* The geometry source is a hardcoded absolute path in two scripts (chocobo_geometry.py:27 and build_faces_ggb.py:28, both `"/home/bork/chocobo.ggb"`), so the data-regeneration pipeline — the thing that PRODUCES instance.json and the GGB round-trip — only runs on one person's machine. verify_faces.py compounds it: it loads `GGB = "chocobo_faces.ggb"` as a bare relative name (line 19), but build_faces_ggb.py writes that file into the scripts/ directory via `os.path.dirname(__file__)` (line 29), so the verifier silently reads a stale or missing file depending on cwd. The model layer does this correctly (arrangement.py:28-29 anchors FACES/INSTANCE to `__file__`), making the scripts' regression a clear inconsistency, not a house style.  
   *Evidence:* chocobo_geometry.py:27 `GGB = "/home/bork/chocobo.ggb"`; build_faces_ggb.py:28 `SRC = "/home/bork/chocobo.ggb"`; verify_faces.py:19 `GGB = "chocobo_faces.ggb"` (cwd-relative) vs build_faces_ggb.py:29 `DST = os.path.join(os.path.dirname(os.path.abspath(__file__)), "chocobo_faces.ggb")`  
   *Right abstraction:* Take the source .ggb as a CLI argument (or an env var with a sensible default) and anchor every produced/consumed artifact to a single project-data directory the way arrangement.py already does. The data-generation boundary should be reproducible by anyone who has the source map.

7. **[minor] verify_faces.py re-implements the old cover_mask model to cross-check against the new one**  
   *Pattern:* `duplicated knowledge / leaky boundary between data-gen verification and model`  
   *Why cancerous:* The verifier rebuilds the deprecated `cover_mask[i] = {i} ∪ overlap-neighbours` model inline (verify_faces.py:30-31 from the stored `overlaps`) to assert the union of face-covers over each region reproduces it (line 49-55). This is legitimate as a regression check, but it means the OLD model's logic now lives in a script, divorced from the model layer, with the magic count '~44' and 'all 16' baked into print strings (lines 33, 55) that are really assertions-as-prose. If the instance grows, these literals silently misdescribe without failing.  
   *Evidence:* verify_faces.py:30 `for a, b in data["overlaps"]: nbr[int(a)].add(int(b)); nbr[int(b)].add(int(a))`; verify_faces.py:55 `print(f"\n[4] per-region ⋃(face covers) == old cover_mask, all 16: {ok}")`  
   *Right abstraction:* Make the checks real assertions (assert ok, assert len(faces) == expected) driven by counts derived from the instance, not printed prose with hardcoded '16'/'44'. The legacy cover_mask reconstruction belongs behind a named helper documented as 'legacy-model regression oracle', not inlined.

8. **[minor] analyzer mixes presentation (_print_report) into the analysis module; real_instance can silently mis-set N**  
   *Pattern:* `grab-bag module / accepted-but-overridable param`  
   *Why cancerous:* analyzer.py is otherwise clean, but it bundles a 22-line hand-rolled console formatter (_print_report, lines 581-602) and a `__main__` into the same module as the pure analysis, so the analysis layer carries a presentation concern (synthetic.py:134 imports `_print_report`, a private function, cementing the coupling). Separately, real_instance defaults N=20 (line 88) but reads treasures from instance.json (line 92); if the file ever holds != 20 treasures, world_array(inst) uses the stale N=20 while the treasure dict has a different size — a quiet inconsistency the function never checks (env.py:31 avoids this by deriving N=len(treasures)).  
   *Evidence:* analyzer.py:88 `def real_instance(..., N: int = 20, K: int = 5)`; analyzer.py:92 `treasures = {int(i): tuple(xy) for i, xy in data["treasures"].items()}` (N not derived from this); synthetic.py:134 `from chocofarm.analysis.analyzer import analyze, _print_report`  
   *Right abstraction:* Derive N from len(treasures) in real_instance (matching Environment) instead of accepting a default that can contradict the data. ### A.9 — config-xcut (configuration flow across the whole chocofarm codebase)

**Health:** `messy`

**Summary.** There is no central configuration object anywhere in chocofarm: every entrypoint owns its config via either an ad-hoc argparse namespace threaded as `args` or an inline `PLAN`/`build_plan` tuple-literal, and the project's three load-bearing reference constants (static floor 0.0855, clairvoyant ceiling 0.1454, decomp anchor 0.0941) are hand-copied across ~10 modules plus tests and docs, encoded inconsistently (0.0941 vs 0.094). The dynamic decision variable that genuinely needs to be live — the Dinkelbach penalty lambda — is, by contrast, threaded correctly as a function argument everywhere and owned by one fixed point loop, which is the one clean seam. The result is a codebase where the runtime MDP is reconfigurable but the experimental scaffolding (search widths, learning rate, reference lines, instance shape) is frozen at construction or duplicated, so changing the instance or the operating point means editing many files in lockstep.

**Praise (earned, with citations):**

- lambda is owned in exactly one place and flows live everywhere else. `Policy.decide(self, env, loc, bw, collected, lam, rng)` (chocofarm/solvers/base.py:18) takes lambda as a per-call argument, and `env.dinkelbach_rate` is its sole owner: `lam = lam0; for _ in range(iters): lam = self.rate(policy, lam, ...)[0]` (chocofarm/model/env.py:159-165). No solver caches lambda at construction; the decomp solver even keys its per-lambda micro-table cache on `round(lam, 6)` (chocofarm/solvers/decomp.py:547) so a changed lambda transparently rebuilds. This is the correct hot-where-it-should-be-hot design and it is consistent across all 100 `decide(...lam...)` call sites — a real bright spot.
- The action<->slot bijection is genuinely single-sourced. actions.py is explicit that the mapping 'is kept HERE, in one place, so mlp.py, gumbel_search.py, and exit_loop.py all agree on it by construction' (chocofarm/az/actions.py:8), `n_action_slots`/`slot_action_tables` derive everything from `env.N + len(env.detectors) + 1` with nothing hardcoded (actions.py:31-33,51-65), and a bijection test is asserted. This is exactly the right way to centralize a layout that three modules must agree on.
- feature_dim is derived, never hardcoded: `return env.N * 5 + len(env.detectors) * 3 + (6 + n_tel)` (chocofarm/az/features.py:62-69), and the module docstring explicitly warns against hardcoding 16/37/90/220. Downstream nets re-init their input layer from `feature_dim(env)` rather than a constant (exit_loop.py:178), which is the right SSOT discipline for a layout that has already changed twice.
- DTYPE is a single switchable precision read once from the environment with a fail-loud on bad values (chocofarm/az/dtypes.py:32-43), replacing the prior implicit float64/float32 mixing — a clean, well-scoped global config knob.

**Findings:**

1. **[critical] The three reference constants (0.0855 / 0.1454 / 0.0941) are hand-copied across ~10 modules and tests, with an inconsistent encoding**  
   *Pattern:* `SSOT-violation`  
   *Why cancerous:* The static floor, clairvoyant ceiling, and decomp anchor are the project's headline metric basis (every %VoI number divides by `(0.1454 - 0.0855)`). They are derived quantities (floor = realizable_static(env), ceiling = clairvoyant_rate(env), anchor = decomp rate) yet are pasted as literals in code paths that print/log/assert them. They have ALREADY drifted: exit_loop pins `DECOMP_ANCHOR = 0.0941` while eval_az logs `0.094` and bounds passes `vhat_lam=0.094`. If the instance, value vector, or teleport overhead ever changes, these literals silently become wrong everywhere — and the one place that recomputes them (harness.realizable_static / clairvoyant_rate) is NOT consulted by the consumers that hardcode them. A regression test (test_smoke.py:94-95) even pins the literals, so a legitimate instance change fails a test that should have read the value from the env.  
   *Evidence:* chocofarm/az/exit_loop.py:49-51 `STATIC_FLOOR = 0.0855` / `CLAIRVOYANT_CEIL = 0.1454` / `DECOMP_ANCHOR = 0.0941`; chocofarm/eval/eval_az.py:34 `LAM0 = 0.0855` and eval_az.py:79 `w.add_scalar("ref/decomp", 0.094, ...)`; chocofarm/bounds/eval_bound.py:173 `vhat_lam=0.094`; chocofarm/az/dataset.py:95 `default=0.0855`; tests/test_smoke.py:94 `assert abs(realizable_static(env) - 0.0855) < 1e-3`

   *Pattern:* `ad-hoc-parameter-passing-hell`  
   *Why cancerous:* There is no Config dataclass anywhere. ### A.10 — tests

**Health:** `messy`

**Summary.** The suite is a small (4-file, no-conftest) set of bounded gates that, on the smoke-test layer, pins genuine behavior (reference lines, env shape, solver wiring) cleanly. But the AZ layer is dominated by tests that exist ONLY to police architectural debt: cross-implementation float-equivalence among three hand-maintained forwards, the action-slot bijection that exists because the feature layout is encoded in three independent places, and white-box re-implementations of the search algorithm that reach into private state (_evaluate, _sequential_halving, _v_mix, root.prior/N/W). The deadlock test is honest and valuable, but its very existence documents a parallel substrate held together by a stack of band-aids (H1a/H1/H2/A/B/C) fighting the multiprocessing+redis+JAX+numba interaction.

**Praise (earned, with citations):**

- tests/test_parallel_deadlock.py:32-49 — `_FakeImap` is a clean, dependency-free seam: it stands in for the Pool's `IMapIterator` and lets `_drain_imap`'s bounded-wait contract be asserted with NO live Pool/redis/loop. - tests/test_smoke.py:91-95 `test_reference_lines_unmoved` — pins the detector-independent floor/ceiling (`realizable_static`≈0.0855, `clairvoyant_rate`≈0.1454) with a tolerance, a real invariant of the problem instance that SHOULD be guarded against accidental change. ## Appendix C — Systemic Anti-Patterns (verbatim, 8)

Synthesized by one subagent from the 11 component reports plus the 22 verdicts, with instruction to discount refuted/overstated claims and require cross-subsystem recurrence.

### C.1 — Config frozen at construction with no live-reconfiguration seam (the missing Config-ownership layer)  ·  `critical`

**Why cancerous.** This is the codebase's central disease and it recurs in EVERY compute-heavy subsystem. The pattern: a tuning knob is captured once in __init__ and closed over for the object's life, with no setter, schedule hook, or per-call override. The architecture proves it knows the right answer — `lam` (the Dinkelbach penalty) IS threaded as a live per-call argument through `decide`/`simulate`/`apply` (chocofarm/solvers/base.py:18, chocofarm/model/env.py:141) and DecompPolicy even rebuilds its tables when it changes — yet that treatment was extended to NOTHING else. For a research harness whose entire purpose is sweeping and annealing these exact knobs, the most-swept parameters are welded shut: to change search width or learning rate you reconstruct the object (or in the parallel case, tear down a multi-hour process pool). The damage compounds with the frozen-env monolith: because the Environment bakes its whole action space at construction, downstream caches key on env IDENTITY (next anti-pattern), so a reconfigured env silently aliases a stale cache. The deepest expression: config ownership lives NOWHERE — there is no Config object, so every value is owned by whichever constructor or argparse Namespace happened to capture it, and the only config seam is a CLI flag.

**Cross-subsystem examples:**

### C.3 — Hidden global state keyed by object identity (id(env) module-global caches; the _W worker dict)  ·  `major`

**Why cancerous.** Because the env freezes its action space at construction and owns no first-class action-space object, downstream code bolts on module-global caches keyed by id(env) — the least value-stable key possible, since Python reuses memory addresses after GC. Today the bug is masked (all envs are layout-identical) but the moment two envs differ in N or detector count — exactly the multi-instance/variant-detector experimentation this codebase exists to enable — the cache silently hands back the WRONG bijection with no error, and in any long-lived process it leaks one never-evicted entry per env constructed (the test suite alone builds ~20+). This is action-at-a-distance: correctness depends on identity and import-order invariants the code cannot enforce.

**Cross-subsystem examples:**

- chocofarm/az/actions.py:48 `_SLOT_TABLES = {}` and :55-64 `key = id(env); tabs = _SLOT_TABLES.get(key); if tabs is None: ... _SLOT_TABLES[key] = tabs` — a process-global dict keyed by id(env), never evicted; Environment defines no __eq__/__hash__ so identity-as-address is the only key

**Principled replacement.** The env owns its action-space mapping as an attribute computed in __init__ (env.slot_tables), so consumers read it directly — lifetime tracks the object, no module global, no id() key, no leak. Give workers an entrypoint that imports NO jax/XLA (pure-numpy search + redis; only the parent touches JAX), which eliminates the fork/spawn/XLA interaction that caused the wedge and makes the thread-pin, faulthandler, and most timeout bands unnecessary. If cross-machine distribution is genuinely needed, adopt an established job-queue/executor that owns liveness, timeouts, namespacing, and result cleanup rather than re-deriving them from SET/GET + post-mortems; if only pickle-avoidance is wanted, use shared memory (no network round-trip, no LRU-eviction hazard, no TTL bookkeeping). Defensive timeouts should be a thin safety net, not the load-bearing reliability strategy. For the cache, make weights an immutable params object swapped atomically so coherence is an invariant of one object identity, not a per-writer obligation — removing the contract and its three guard tests.

### D.3 — Config ownership


The codebase already gets the single most important config dimension right, and the target preserves it: lambda (the Dinkelbach penalty) is a per-call argument on decide()/simulate()/rate() (base.py:18, env.py:141/159-165), owned by exactly one fixed-point loop, and threaded live to ~100 call sites. DecompPolicy even keys its expensive per-lambda micro tables on round(lam,6) and rebuilds on change (decomp.py:546/590). This is the gold-standard "live cell" and it is the template for everything else. The synthesis classifies every config item into exactly one of four tiers, and the architectural malpractice in the findings is almost entirely items placed in the wrong tier.

TIER 1 — TRUE CONSTRUCTOR INVARIANT (immutable, expensive, set once): the geometry. The distance table (env.py:66-70, ~4.5k entries), the world-set bitmask array (env.py:56-58), and the parsed faces/arrangement. These genuinely cost real time to build and never change within a run. They belong in Environment.__init__ and nowhere else. The faces.json/instance.json files are their on-disk SSOT, loaded once by model.instance.load_instance().

TIER 2 — SCENARIO (cheap, copy-on-write on the env): value, entry, teleport_overhead, and K. The verdicts confirm none of these enter the distance table (value read only at env.py:131, entry at :139, tp at :85; K only shapes worlds, which is cheap relative to _dist). Today they are dead kwargs (env.py:24) that every one of ~40 call sites ignores, with K not even a kwarg but a bare literal (env.py:31). In the target they live in a frozen Scenario dataclass, and env.with_scenario(s) returns a new Environment sharing _dist (and, for value/entry/tp, sharing worlds too). A value/entry/K sweep becomes a list comprehension over Scenarios — the attic het-values experiment (attic/het_values_eval.py:17), which today monkeypatches a module global, becomes first-class. Because with_scenario yields a fresh object, env.slot_tables and FeatureBuilder snapshots cannot silently belong to the old config — the staleness the id(env) cache invites (actions.py:55) is structurally gone.

TIER 3 — LIVE CELL (hot, reconfigurable without rebuild): lambda (already correct), search budget (m, n_sims, c_puct, c_visit, ...), and learning rate / l2. These are the experimentation levers an ExIt/Dinkelbach researcher sweeps and anneals. The target makes them live in two complementary ways: (a) frozen-but-cheap-to-replace — a SearchConfig dataclass passed per-construction, so a sweep mutates one object instead of threading scalars through every eval main (tb_runner.py:66, eval_uct.py:76); and (b) genuinely per-call — SearchConfig is also acceptable as the optional cfg override on decide() and on the per-iteration parallel task, so budget can scale with belief width / remaining horizon and c_puct can anneal across iterations. lr/l2 become live via optax.inject_hyperparams, retiring the confessed dead parameters (exit_loop.py:147-148, mlp_jax_train.py:215). The litmus test: if the value changes during a run or across a sweep, it is a live cell, not a constructor invariant — m/n_sims/lr fail this today, lambda passes.

TIER 4 — DERIVED (computed, never stored twice): N=len(treasures), feature_dim(env), n_action_slots(env), env.slot_tables, the episode horizon, and the reference lines (static_floor/clairvoyant_ceiling/decomp_anchor). The codebase already nails feature_dim/n_action_slots (features.py:62, actions.py:31 — earned praise). The target extends that exact discipline to the three places that violate it: the reference constants (computed by harness.realizable_static/clairvoyant_rate but hand-copied as literals at exit_loop.py:49-51 and used in the headline %VoI divisor at :318), the horizon (one env.max_steps, not four magic 40s), and the feature layout (one FEATURE_LAYOUT descriptor, not three hand-transcribed offset encodings). A derived value is computed from its single source on demand (or cached on the object that owns the source), never frozen into a literal that goes stale when the source moves.


### D.4 — Initialization strategy


The initialization spine is a strict three-stage pipeline, each stage owning exactly one concern, so config enters at one seam and flows downward without back-references.

STAGE 0 — DATA: load_instance(path) parses instance.json/faces.json ONCE into a frozen Instance (treasures, teleports, K, faces). This is the only code that touches disk or resolves the data directory (replacing the three __file__-relative joins at env.py:26, arrangement.py:28, analyzer.py:90 with one importlib.resources resolution). K now lives in the file, so env, analyzer, and decomp read one source instead of three literals.

STAGE 1 — GEOMETRY: Environment(instance) builds the Tier-1 invariants (distance table, worlds, faces->SenseActions) exactly once. This is the expensive step and it happens once per process. env.with_scenario(Scenario) is copy-on-write: it allocates a new Environment that shares the immutable _dist (and worlds, for value/entry/tp scenarios), so producing N scenario variants costs N cheap shallow copies, not N table rebuilds. The env computes and caches its own derived data (slot_tables, BeliefRefs) lazily on itself — lifetime tracks the object, no module global, no id() key.

STAGE 2 — CONFIG: main()/from_args builds ONE RunConfig (nested Net/Search/Train/Parallel/Eval/IO frozen dataclasses). The CLI is a thin RunConfig.from_args adapter; a notebook or sweep harness constructs RunConfig directly (no fake argparse Namespace). Validation that is today scattered ad-hoc inside run() (the n-step/td-lambda exclusion at exit_loop.py:245) moves into the relevant dataclass __post_init__.

STAGE 3 — WIRING: ExItLoop.run(env, config) constructs the AZ components from the config object, NOT from 25 args.* reads. The net is a WeightContainer + ForwardSpec; the trainer takes the container and an opt config; the search takes (net, env, config.search). The ParallelExecutor takes config.parallel for genuinely per-process invariants (cores) ONLY, and receives SearchConfig+lam+seed per-iteration through the task tuple — exactly where the worker already rebuilds the search per weight-version, so per-iteration search config flows through the existing live path instead of being frozen into pool initargs. Workers initialize numpy-only (no jax import), so the spawn/XLA interaction that the deadlock band-aids fight never arises.

The ordering guarantees there are no construction-time back-edges: data has no knowledge of geometry, geometry has no knowledge of config, config has no knowledge of the loop. Each downward arrow passes a value object; nothing reaches back up to mutate a parent's frozen state.


### D.5 — How 'hot where it should be hot' falls out


"Hot where it should be hot" stops being heroic because heat is determined by WHICH TIER a value sits in, and the tier is a structural property of where the value lives — not a discipline each call site must remember. Three mechanisms make it automatic:

1. The per-call argument seam already proves it works for lambda. base.py:18 threads lam through decide() to 100 sites with zero ceremony, and the only reason m/n_sims/c/lr are cold is that they were assigned to self.X in __init__ instead of arriving as arguments. The fix is mechanical and local: the same seam that already carries lam carries an optional cfg. There is no global rewrite — the contract that makes lambda hot is the contract that makes search budget hot. Heat falls out of "is it an argument or a field," and the target puts the experimentation levers on the argument side.

2. Copy-on-write scenarios make the cold things cheap-to-replace instead of impossible-to-vary. The reason value/entry/K feel "frozen" today is that varying them means rebuilding the whole env (and silently orphaning id(env) caches). Once with_scenario shares the distance table, a sweep is a comprehension and the cost of "varying" collapses to a shallow copy. The knob doesn't need to be a live mutable cell — it needs to be cheap enough to re-bind that nobody is tempted to freeze it. The geometry stays a true invariant (correctly cold), the scenario becomes trivially re-bindable (appropriately warm).

3. Derived-from-source kills the "stale literal" class entirely. The reference lines, horizon, feature layout, and slot tables are hot in the sense that they MUST move when their source moves. The codebase already demonstrates the pattern is free: feature_dim(env) and n_action_slots(env) are recomputed from env and the design docs explicitly forbid hardcoding them (features.py:8, actions.py:13) — and those are the places with NO drift. The drift is concentrated exactly where someone froze a derived value into a literal (0.0855 at exit_loop.py:49, the 90/220/37 stale dims in docstrings, the four magic 40s). Routing every consumer through BeliefRefs(env)/env.max_steps/FEATURE_LAYOUT(env) means the value is correct-by-construction the instant the env changes, with no synchronization ritual. The smoke test that today PINS the literal (test_smoke.py:94, which breaks on any legitimate retune) becomes a sanity check that the computed floor is sane, not a tripwire forcing a 10-file hunt.

The net effect: a researcher who changes the value vector, the K, the search width, or the learning rate touches ONE place (a Scenario, a SearchConfig field, a config flag), and every dependent quantity — worlds, %VoI baseline, slot tables, rollout depth, the optimizer step — follows automatically because each is either a per-call argument, a copy-on-write derivative, or a computed-from-source quantity. Nothing is "kept in sync by nobody," because nothing is kept in sync at all — it is derived.
