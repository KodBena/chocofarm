## A. Factual errors about how the code actually works

### A1. **`Decision::leaf_requests` does not exist on the type the fiber drivers produce — the Phase C structural cross-check is built on a phantom field.** (CRITICAL)
The roadmap's Phase C / Q7 risk-mitigation hinges on: *"the leaf-request count per decision (`Decision::leaf_requests`, `search_runtime.hpp:69`) is a structural cross-check that must be identical across runtimes."*

- `TreeState::decision` is a **`GumbelAZPolicy::Decision`** (`fiber_tree.hpp:48`), defined at `gumbel.hpp:231-234`. It has **exactly two fields: `Action action` and `std::vector<double> improved`. There is no `leaf_requests`.**

The proposed "structural observable" is not available to the runtime the roadmap actually builds. The cross-check must be re-derived (e.g. count leaves driver-side per slot, which the wire driver *can* do since it owns the submit loop) — but the roadmap cites a field that the `run_episodes_wire_batched` path never sees. **MUST change:** Phase C must define a driver-side leaf counter, not read a non-existent `Decision` field.

### A2. **`wire_pool_bench.cpp` does NOT drive the production search path — it drives the *scripted* `CyclicGumbelSource` over a *constant* root state.** (CRITICAL — undercuts "leaf-transport swap, not a search rewrite")

RNG-free. There is **no per-episode `std::mt19937_64`, no world draw, no env stepping.**
- `wire_pool_bench.cpp:204` starts every tree at the **same constant** `loc`/`bw`/`coll` (`:155-157`) and never calls `env.apply`. Every "task" is the same root decision with a rotated gumbel script. It decides once and is discarded (`:228-230`).
- `run_episodes_batched` uses the **RNG ctor** `TreeState(gc, env, s.rng)` (`runner_batched.cpp:87`), with the per-slot persistent rng, the world-pick (`:170-172`), `env.apply` (`:116`), and the multi-ply episode loop.

So the wire engine to be "lifted verbatim" carries **none** of the determinism machinery the production path needs; it shares only the *transport* (DEALER + corr-id + greedy drain). The roadmap's §0 claim that the integration is "cross-breed (2) and (3) ... far less new code than the framing implies" is **backwards**: the only liftable part of `wire_pool_bench` is the ~40-line transport loop (`:172-236`); the entire episode/RNG/stepping machine comes from `runner_batched.cpp`, and the seam between a *multi-threaded* transport and that machine is genuinely new (see B1). The roadmap's own Phase B "Risk: medium" undersells this by leaning on "both lifted verbatim."

## B. Hand-waved hard parts

### B1. **The multi-thread lift of `run_episodes_batched` is asserted, not designed — and `run_episodes_batched` is single-threaded by construction.** (CRITICAL)
The roadmap (Q3, Phase B) says the T-thread structure "carries over" from `wire_pool_bench` and the episode driver is "lifted verbatim." But:

- `run_episodes_batched` is **one thread, K slots, one `predict_batch`** (`runner_batched.cpp:204-242`). Its `written`, `next_idx`, and `fill`/`on_decided`/`spawn_ply` closures all close over **shared single-thread state** (`next_idx` at `:79`, `written` at `:78`, captured by reference in lambdas `:86,101,165`).
- To run T threads each owning a disjoint episode subset, **`next_idx` and `written` become cross-thread** — `next_idx` must be partitioned (the `tid, tid+T` scheme) and `written` must be atomic. That is exactly the "merge" the roadmap waves at, but it means `fill`/`on_decided` cannot be lifted "verbatim": their capture set changes, and the single-`predict_batch` flush (`:222`) is replaced by a per-thread submit/poll loop. This is a genuine rewrite of the driver's control structure, not a substitution of one line.

The roadmap's Phase B "the one substitution: `runner_batched.cpp:204-242` ... becomes a per-thread greedy-async loop" hides that the gather-flush-resume in `:204-242` is a *barrier* (gather ALL parked, one batch, resume ALL) while the wire drain is *greedy-async* (resume one, immediately resubmit). These are different episode-advance disciplines; on_decided's re-gather assumption (`:233` "re-gather next flush") does not hold under greedy-async (there is no "next flush"). **MUST change:** Phase B must specify how `on_decided`'s `return true` (re-park) re-enters the greedy loop, not the barrier loop. This is the real integration and it is unscoped.

### B2. **Net-version straddle / lock-step argument has a hole the roadmap names but does not close.** (Q2)
The roadmap (Q2, §5 gap #4) argues straddle is impossible because the actor's generate is lock-step (`actor_transport.py:206-217`) so generate N fully drains before publish N+1. Verified, `_request` is lock-step. **But the lock-step is on the *control channel*, not on the *leaf channel*.** The actor's `generate` reply is sent only after `run_episodes_wire_batched` returns (`serve.cpp:187-195`), which is after all leaves resolve — so within the actor the drain completes. That part holds. The real hole the roadmap *understates*: `RedisParamsSource.poll()` reloads on the **version-supplier** (`inference_server.py:210-219`), and the executor sets `self._published_version = version` *before* `publish_weights` (Phase D). If the server's between-batch `poll()` fires *after* `_published_version` is bumped but *before* `publish_weights` has written the new blob to redis, `read_weights(want)` **raises loudly** (missing payload, `inference_server.py:204/215`) — aborting the reload mid-generate. The roadmap's Phase D sequences "set `_published_version = version` **before** `publish_weights`" — which is the **wrong order**: it opens exactly this window. **MUST change:** publish weights FIRST, then bump the supplier (`publish_weights(...); self._published_version = version`), so the blob exists before the server can want it. This is a real ADR-0002 fail-loud landmine the roadmap's own sequencing creates.

### B3. **Server `in_dim`/`n_actions` vs the actor's instance/faces geometry is never reconciled.**
`wire_server.py:build_server` builds the server from a Python `Environment()` with a **default instance and no faces concept** (`env.py:84`, `Environment(instance_path=None)`; Python has no faces arg). The C++ actor loads `instance`/`faces` from CLI and derives `feat_dim`/`n_slots` from *that* geometry (`serve.cpp:97`, `main.cpp:153`). For the wire leaves to decode, the server's `in_dim` must equal the C++ `fb.dim()`. The roadmap (Q1) says cpp_executor "builds the server over the live net it already holds" using `self.env`, `in_dim`, `n_slots` (`cpp_executor.py:82-83`) — so in production this *can* match, IF the Python `self.env` is constructed from the same instance the C++ actor loads. The roadmap never states this invariant or where it is enforced. A mismatch is a ragged-batch loud reject (`inference_server.py:113`) or silent dimension corruption. **MUST add:** an explicit "the server's `in_dim`/`n_actions` are derived from the SAME instance/faces the actor loads" invariant + where it is asserted.

### B4. **Teardown ordering is asserted but the failure mode is the reverse of what's claimed.**
Q1/Phase D: `close()` does `server.stop() → join → server.close()` **before** reaping the actor "so the actor never blocks on a dead server mid-generate." But generate is lock-step on the control channel; by the time `close()` runs, no generate is in flight (the executor is single-threaded over the actor). The actual hazard is the opposite: if the **actor** is reaped first while a leaf is outstanding, the server's `send_multipart` to a dead DEALER identity is a silent drop (ZMQ ROUTER drops unroutable). The ordering the roadmap picks is defensible, but the stated rationale ("never blocks on a dead server mid-generate") describes a state that lock-step already precludes. Low risk, but the reasoning is post-hoc.
