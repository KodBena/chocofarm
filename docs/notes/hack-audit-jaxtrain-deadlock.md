FINDINGS BEYOND VERDICT (required):
  - The self-instrumentation downgrade is the weakest point and rhymes with Case A. The RCA's central honest admission is "I cannot prove H1a without a worker-side dump," and the one change that would lift that admission — worker-side `faulthandler` in `_worker_init` — was set aside with "to keep the fix minimal." That is naming the exact tool that converts a guess into evidence and then declining it on a discipline-word. Fix C (the only cause-targeting change) is therefore shipping as un-falsifiable: if the deadlock recurs, the tree still has no way to confirm whether C addressed the cause, because the instrument that would tell was deferred. For a fix whose own RCA rests on "unconfirmed," shipping the bounding (A/B) and deferring the confirming (faulthandler) inverts the priority. The faulthandler registration is cheaper and lower-risk than C and should arguably have shipped *with* C, since C without it cannot be validated.
  - Fix C is honestly labeled "defensive," but its causal story is thin and the RCA half-concedes it. C re-sets thread-count pins in the child via `setdefault`. Under `spawn`, the child already inherits the parent's environment, in which `mlp_jax*.py` already `setdefault`'d `OMP_NUM_THREADS=1` and `XLA_FLAGS` at import time (RCA §2 H1a.1). So for the variables JAX touches, C's `setdefault` is frequently a no-op that re-affirms an already-inherited value; its only new effect is pinning `OPENBLAS/MKL/NUMEXPR/NUMBA_NUM_THREADS` that JAX did not set. Whether *those* are the wedge lever is exactly what is unconfirmed. C is "cannot regress" (true — core-pinned workers want one thread each) but its claim to "sever the residue the RCA fingers" is stronger than the evidence; it is a reasonable defensive guess, correctly tagged, but it is closer to dressed-up-plausible than to a demonstrated cause-fix.
  - `self.pool.join()` in `close():372` is an unbounded parent wait the fix did not bound and structurally cannot bound the same way (`Pool.join()` has no timeout). If a worker is wedged at end-of-run, `close()` hangs there instead. Low severity (end-of-run, once per process, not the observed per-iteration mid-loop hang) but it means the invariant "the parent never waits unbounded" is true for the hot loop and false for teardown. A complete statement of the invariant would terminate the pool with a bounded `Process.join(timeout)` + `terminate()` fallback, or document close() as a knowingly-unbounded best-effort.
  - The 600s per-result timeout is per-result, not per-fan-out. A pathological-but-not-wedged worker that returns each of E results just under 600s would let the whole fan-out run E×600s without tripping. Not the observed failure (the observed worker produced *no* event), and 600s is ~1000× a real episode, so this is a corner, not a defect — but the diagnostic message says "a worker failed to report within 600s," which is precisely true per-result and could mislead an operator into thinking the *fan-out* had a 600s bound.
  - Runtime claim mismatch (carried up from the RUNTIME line because it bears on trust, not just on verification status): the RCA says the loop was not run and only the targeted harness/unit test could validate; the relayed claim adds "unit-tested `_drain_imap` (28 passed)." The tree contains no such test. This is not necessarily dishonest — the test may have been run ad hoc and discarded — but a `_drain_imap` unit test (StopIteration drains clean; injected `TimeoutError` raises the RuntimeError with the right counts/phase/run) is cheap, deterministic, needs no redis or pool, and would be the one piece of *committed* evidence that Fix A behaves as written. Its absence from the tree means the load-bearing change ships with zero checked-in coverage. That test should exist in the change, not just in the prose.
