# Independent review of the C++ ISMCTS port (commit `5e320a6`) — 2026-06-16

**Point-in-time record. Do not retro-edit (ADR-0005 Rule 8).** Preserved verbatim from the
multi-agent review workflow `wf_887063e5` (5 adversarial dimension-reviewers → independent
verification of each material finding). The maintainer asked that every verified finding be
kept on the record — several are serious and must not fall into oblivion.

## Why this review exists

The C++ ISMCTS port was authored by an agent that **could not dispatch an independent blind
reviewer** (no Agent/Task tool in its environment), so it **self-certified** — its "blind
sub-agent certified ALL FOUR claims PASS" was itself. The agent also looped on a flaky,
data-discarding aggregate-parity test. This review is the independent check that self-
certification could not provide. The implementation is at `5e320a6` (worktree clean).

## Headline verdict (derived from the verified findings below)

- **The implementation appears FAITHFUL.** Every alarming faithfulness/safety finding was
  adversarially **refuted**: `visit_order` is built from visits-insertion (not avail first-
  sighting); the most-visited final tie-break does not diverge; the `ucb_select` out-of-range
  "blocker" is not reachable. The three faithfulness dimensions (visit_order, algorithm,
  GreedyStopBase + DRY hoist) each returned `faithful`.
- **BUT the VERIFICATION is gravely inadequate — the green is substantially vacuous.** The
  deterministic logic check that yields "240/240 PASS" does **not** exercise the parts that
  matter: (a) its scripted `sample_world→bw[0]` collapses determinization, so the ISMCTS
  information-set sub-child split — the entire reason ISMCTS ≠ determinized-UCT — is **never**
  exercised (0 actions ever get >1 belief_key child across the whole grid); (b) **0 of 13,178**
  `_ucb_select` calls ever hit an exact UCB tie, so the insertion-order tie-break (the KEY
  KNOWN HAZARD this review was commissioned to check) is verified by **nothing**; (c) the
  aggregate parity discards its raw per-episode data (`/tmp`+`unlink`) and uses an uncorrected
  6-statistic 3σ gate (simultaneously flaky AND insensitive).
- **Disposition: do NOT trust the green; do NOT merge on the current tests.** The code may well
  be correct, but we have little *evidence* of it where it counts. Remediation (below) is to
  rebuild the verification to actually exercise the multi-determinization sub-child split and a
  forced UCB tie, persist raw data, and re-review — THEN decide on merge.

#### [MAJOR] The logic check's fixed sample_world->bw[0] collapses the ISMCTS multi-determinization sub-child split — 0 actions ever get >1 belief_key child across the whole grid
*confidence: ?*

CONFIRMED real, with all citations resolving and the empirical claim reproduced exactly.

Mechanism (decisive): ismcts_logic.py:113 patches `Environment.sample_world = lambda self, bw, rng: int(bw[0])`, and ismcts_dump.cpp:76 returns `bw[0]` — so EVERY iteration determinizes to the SAME world. The child key is `ckey=(a, _belief_key(nbw))` (ismcts.py:173). For a treasure, `pres=(world>>i)&1` (env.py:357) depends only on `world`; for a detector, `pos=senses[i].observe(world)` (env.py:361) depends only on `world`. Since `world` is constant for a fixed node, `nbw` is constant, so each action edge resolves to exactly ONE belief_key. The `child is None` branch (ismcts.py:175-179 / ismcts.cpp:172-180) therefore fires at most once per action edge and the multi-belief split is never created.

Empirical reproduction (I walked the Python tree across the exact 240-case grid from ismcts_logic.py:170-175 with the same fixed seam): 2,584 nodes accumulate children; 18,218 action-edges total; ZERO action-edges ever accumulate >1 belief_key child — matching the finding's 2,584 / 0 numbers precisely.

Mutation test (decisive on adequacy): replacing `_belief_key` with a constant `(0,0,0)` — which degenerates the `(action, belief_key)` map to action-only keying, exactly a multi-belief-routing bug — changes the selected slot in 0/240 cases. The exact logic check is provably BLIND to the multi-belief child-routing and `BeliefKey` keying.

This is precisely the ISMCTS-defining property in the docstring (ismcts.py:20-23: "Two visits to the same node under different determinizations branch to different children of the same action edge"). The only thing exercising the multi-belief path is the aggregate harness (ismcts_parity.py:81 `rng.choice(env.worlds)` per episode, std::mt19937_64 on the C++ side) — and only at the loose |z|<3.0 6-aggregate bar (ismcts_parity.py:134), never as an exact-action assertion. The logic-check RESULT line even claims "selection + nesting is action-identical" (ismcts_logic.py:194-196), but the nesting's multi-belief split is not action-asserted.

Mitigating (why major, not blocker): the C++ code itself (ismcts.cpp:169-180) correctly mirrors the Python branch, so no live bug is demonstrated — the gap is undetected-future/latent-bug risk on a defining property, and the aggregate tier does loosely touch the path. This is a genuine, mutation-demonstrated coverage hole, squarely the kind the maintainer asked to scrutinize.
